<objective>
Close the SP-03 / B-01-followup race in `BuilderMap.tsx`: when a user creates a fresh empty map and clicks "Add to map" on a catalog dataset for the first time, the layer must render WITHOUT a hard refresh.

The prior B-01 fix (commit `85738f1c`) replaced a `tileTokensPending` boolean with a `tokenMap.has(dataset_id)` presence gate and added `tokenMap` to the main effect's deps. That partially helped but did NOT fully close it — workaround is still "refresh after add". The residual race is suspected to be:
- The main sync effect's `syncLayersToMap(...)` call uses a `useMemo`'d `syncInputs` that is keyed on `structuralKey` only. When `tokenMap` lands AFTER `structuralKey` flipped, the effect re-fires (good) but it re-uses the closure-captured `syncInputs` reference — which was correct in this case, but the broader pattern (closure-over-stale-`layers`) is fragile and the dep array contains `structuralKey` (a derived string) rather than `layers` itself.
- `ViewerMap.tsx:582-595` has the same surface (multi-layer sync with token gate) and works reliably. Its pattern: a `runSync(map)` useCallback that pulls EVERY input from a ref (`syncInputsRef.current` updated on every render), called from an effect whose deps include `layers` directly + `tokenMap` + the other primitive triggers.

Purpose: Converge BuilderMap's main sync effect onto ViewerMap's proven ref+callback architecture so the closure-stale-input class of bug is eliminated by construction.

Output: One-file refactor in `BuilderMap.tsx`; manual smoke confirms fresh-add renders without refresh; existing token-rotation / basemap-switch / cluster / terrain / reorder / visibility paths are preserved.
</objective>
