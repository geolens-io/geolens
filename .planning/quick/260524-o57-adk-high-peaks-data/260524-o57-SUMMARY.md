## 8. Follow-ups (recommended GeoLens phases)

Listed in priority order. Each has corresponding entry in `260524-o57-API-ISSUES.md` with verbatim repro + diagnosis.

1. **Issue 6 — Builder layer-stack visual sync regression** (BUILDER-DND-???): When the user drags a vector layer above a raster layer in `UnifiedStackPanel`, the API persists the new order (`PATCH /layers` with `order: […]`) but the MapLibre visual stack doesn't update. Likely a stale-layer-id reference in `reorderDataGeometry` or a missing call when raster layers change position. Repro: open a map with mixed raster+vector layers, drag a vector up; observe that the API gets the new order but the map keeps the old visual stack.
2. **Issue 3 — Hardcoded `maxzoom=18` in raster tile token** (TILE-API-???): `_build_tile_token_for_dataset` in `backend/app/processing/tiles/router.py:464-472` returns `minzoom=0, maxzoom=18` for every raster regardless of the COG's actual overview pyramid. MapLibre fetches tiles at z=17/18 which fall outside bounds → Titiler 404 → proxy 204 → "dem dimension mismatch" error. Fix: compute `maxzoom` from the dataset's native resolution (~`log2(40075016.686 * cos(lat) / 256 / res_x)`); compute `minzoom` from the lowest-resolution overview.
3. **Issue 5 — Basemap toast surfaces internal MapLibre errors as connection issues** (BUILDER-UX-???): The `errorHandlerRef` in `BuilderMap.tsx:408-437` catches MapLibre runtime errors without an HTTP `status` (e.g., terrain "dimension mismatch") and treats them as 5xx → triggers the "Basemap connection issue" toast. The toast text is misleading — there's no basemap connection problem; the basemap loaded fine. Fix: filter by error.message — terrain-related errors should not raise the basemap toast.
4. **Issue 2 — Map POST with partial `terrain_config` is overwritten by frontend** (MAP-API-???): The frontend's terrain auto-detect (when a DEM layer is added) silently overwrites the user's `enabled: false` choice. Either (a) honor an explicit `terrain_config.enabled: false` and never auto-enable, or (b) require the caller to explicitly opt-in to auto-enable behavior.
5. **Issue 4 — Basemap-error toast position collides with NavigationControl** (BUILDER-UX-???): Both anchored to `top-3 left-3`. Move toast to `bottom-3 left-3` or add collision-detection.
6. **Issue 1 — Upstream Openfreemap Positron style has broken sprite refs** — file an upstream issue or override the sprite manifest with empty placeholders for `road_*` and `us-state_*`.

```bash
  > /tmp/jwt.txt
JWT=$(cat /tmp/jwt.txt)
curl -s "http://localhost:8001/api/maps/c39be324-6815-40e5-8143-00a2723827b2" \
  -H "Authorization: Bearer $JWT" | python3 -m json.tool
```

**Aerial-quality note:** The NY Orthos source was fetched from the ArcGIS REST `exportImage` endpoint with `format=tiff&size=W,H` parameters. ArcGIS REST silently downsamples large export requests to fit within the server's `maxImageHeight`/`maxImageWidth` limits (typically ~4096 px). Even though we requested the full AOI, we got back a 3.5 MB tiff — orders of magnitude smaller than what native NAIP imagery would produce. For higher-quality aerials, swap to (a) direct NAIP COG download from USDA NASS, (b) a Sentinel-2 mosaic, or (c) chunk-and-stitch ArcGIS REST exportImage calls at sub-AOI tiles. For the marketing screenshot, the current aerial reads acceptably at zoom 12-13 but pixelates at zoom 14+.
