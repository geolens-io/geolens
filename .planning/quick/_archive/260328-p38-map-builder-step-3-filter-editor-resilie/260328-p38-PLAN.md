Output: Refactored LayerFilterEditor.tsx with tests, updated i18n keys.
</objective>

    **Step 1 — Write tests first** in `frontend/src/components/builder/__tests__/LayerFilterEditor.test.ts`. Import `parseFilterExpression` and `buildFilterExpression` from the component file. Cover all behaviors above. Tests will fail initially (RED).

    **Step 2 — Refactor parseFilterExpression** to return a discriminated union instead of FilterCondition[]:

    ```typescript
    type ParseResult =
      | { kind: 'editable'; combinator: 'all' | 'any'; conditions: FilterCondition[] }
      | { kind: 'opaque'; raw: FilterSpecification };
    ```

    Logic changes:
    - Detect top-level combinator: `expr[0] === 'any'` sets combinator to `'any'`, `expr[0] === 'all'` sets `'all'`, bare single expression defaults to `'all'`.
    - After parsing sub-expressions, if ANY parseSingle call returns null, the entire expression is opaque: return `{ kind: 'opaque', raw: expr }`.
    - For null/empty input: return `{ kind: 'editable', combinator: 'all', conditions: [] }`.

    **Step 3 — Update buildFilterExpression** to accept a `combinator: 'all' | 'any'` parameter (default `'all'`). When wrapping multiple conditions, use the combinator value instead of hardcoded `'all'`.

    **Step 4 — Refactor LayerFilterEditor component state:**
    - Add state: `combinator` ('all' | 'any'), `rawMode` (boolean), `rawText` (string), `rawError` (string | null), `opaque` (boolean).
    - On mount (the existing useEffect), call new parseFilterExpression. If `kind === 'opaque'`, set `opaque: true`, `rawMode: true`, `rawText: JSON.stringify(filter, null, 2)`. If `kind === 'editable'`, set conditions and combinator from result. Never call onFilterChange on mount (existing behavior preserved).
    - `emitChange(updated, combo)` calls `buildFilterExpression(updated, columnInfo, combo)` then `onFilterChange(result)`.

    **Step 5 — Combinator UI:** Between the title and the condition rows, render a small Select with two options: "Match all (AND)" / "Match any (OR)". Bind to `combinator` state. On change, re-emit the current conditions with the new combinator. Use i18n keys `filters.matchAll` and `filters.matchAny`.

    **Step 6 — Opaque banner:** When `opaque` is true and `rawMode` is false, show an info banner: "This filter uses advanced expressions that cannot be edited visually. Switch to JSON mode to edit." with a button to enable raw mode. Do NOT render the condition rows or Add Filter button when opaque and not in raw mode.

    **Step 7 — Raw JSON toggle:** Add a small button (e.g., Code icon from lucide-react or text "{}" ) next to the title that toggles `rawMode`.
    - Entering raw mode: set `rawText` to `JSON.stringify(buildFilterExpression(conditions, columnInfo, combinator), null, 2)` (or current filter if opaque).
    - In raw mode: render a `<textarea>` (plain HTML, no Textarea component exists) with monospace font, the rawText value, and an "Apply" button.
    - On Apply: try `JSON.parse(rawText)`. If valid, call `onFilterChange(parsed)`, re-run parseFilterExpression on the result to update conditions/opaque state. If invalid, show `rawError` below the textarea in red text.
    - Exiting raw mode (toggle back): re-parse current filter. If still opaque, show opaque banner.

    **Step 8 — i18n keys:** Add to `filters` section in all 4 locale files:
    - `filters.matchAll`: "Match all (AND)" / translations
    - `filters.matchAny`: "Match any (OR)" / translations
    - `filters.rawJson`: "JSON" / translations
    - `filters.rawJsonApply`: "Apply" / translations
    - `filters.rawJsonError`: "Invalid JSON" / translations
    - `filters.opaqueWarning`: "This filter uses advanced expressions. Use JSON mode to edit." / translations

    **Step 9 — Run tests (GREEN).** Fix any failures.

    - `buildFilterExpression` gets an optional third parameter (combinator) — backward compatible.
