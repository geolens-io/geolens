## Warnings

### WR-01: `known_source_columns` parameter is accepted but never read

**File:** `backend/app/ingest/metadata.py:448-538`
**Issue:** `rename_reserved_columns` declares `known_source_columns: list[dict] | None = None` as a keyword argument but the function body never references it. The parameter exists in the signature, is passed by two callers in `tasks.py` (lines 358, 900), and is documented in the function docstring implicitly — but it has zero effect. Callers are passing data expecting it to be used (possibly to short-circuit the DB query when column info is already known), but the function always queries `information_schema.columns` regardless. This is a dead parameter that misleads callers about the function's behaviour.

**Fix:** Either use the parameter to skip or supplement the `information_schema` query when provided, or remove it from the signature and the two call sites in `tasks.py`. If the intent was future optimization, add a `# noqa: ARG001` comment with a TODO, but removing is cleaner:

```python
# metadata.py — remove the unused parameter
async def rename_reserved_columns(
    session: AsyncSession,
    table_name: str,
) -> list[dict]:
    ...
```

```python
# tasks.py line 357-359 — remove the kwarg
reserved_renames = await rename_reserved_columns(session, table_name)

# tasks.py line 899-901 — same
reserved_renames = await rename_reserved_columns(session, staging_tn)
```

### WR-02: `compute_quality_score` still uses the old regex filter for column names

**File:** `backend/app/ingest/metadata.py:308-318`
**Issue:** `get_sample_values` was fixed (correctly) to SQL-quote column identifiers instead of filtering by `_TABLE_NAME_RE`. However, `compute_quality_score` (which was not part of this diff) still applies the same regex guard at line 312:

```python
non_geom_cols = [
    c
    for c in column_info
    if "geometry" not in c.get("type", "").lower()
    and _TABLE_NAME_RE.match(c.get("name", ""))   # <— old pattern
]
```

Any column whose name doesn't match `^[a-z0-9_]+$` — including the newly preserved `src_*` columns (which do match, so those are safe) but also any unicode or mixed-case column that survived GDAL LAUNDER — is silently excluded from the attribute completeness score. This is the same class of silent-skip bug that RESEARCH §2.5 documented for `get_sample_values`. The quality score will undercount attribute completeness for datasets with non-ASCII column names.

**Fix:** Apply the same SQL-quoting approach used in `get_sample_values`. The `col_exprs` string at line 315 already double-quotes the column name, so the only change needed is to drop the `_TABLE_NAME_RE` guard:

```python
non_geom_cols = [
    c
    for c in column_info
    if "geometry" not in c.get("type", "").lower()
    # No regex filter — col_exprs already double-quotes identifiers
]
```

### WR-03: `_drop_table` in test helper uses unquoted table identifier

**File:** `backend/tests/test_ingest_column_preservation.py:79-83`
**Issue:** `_drop_table` builds the `DROP TABLE` statement with an unquoted f-string:

```python
text(f"DROP TABLE IF EXISTS data.{table} CASCADE")
```

Table names are generated by `_table_id()` which produces `{prefix}_{8-hex-chars}`, so in practice the names are always lowercase alphanumeric with underscores and injection here is not a real risk. However, it's inconsistent with the production code (which uses `_qtable()`) and would silently fail if the table name contained special characters. This is in test code, so the impact is low.

**Fix:** Use the `_qtable` helper from `metadata.py`, or quote inline:

```python
async def _drop_table(test_db_session, table: str) -> None:
    await test_db_session.execute(
        text(f'DROP TABLE IF EXISTS "data"."{table}" CASCADE')
    )
    await test_db_session.commit()
```

### IN-01: Docstring example in `detect_dbf_truncation_collisions` shows wrong truncation value

**File:** `backend/app/ingest/metadata.py:548`
**Issue:** The docstring example reads:

```
[{"truncated": "populatio", "originals": ["population_2020", "population_2021"]}]
```

But `"population_2020"[:10]` is `"population"` (10 characters), not `"populatio"` (9 characters). The example truncated key is one character short. The implementation itself is correct — the test at `test_ingest_ogr_pure.py:202` asserts `result[0]["truncated"] == "population"` and passes — but the docstring will confuse readers.

**Fix:**
```python
# Change in docstring:
[{"truncated": "population", "originals": ["population_2020", "population_2021"]}]
```

### IN-02: Inconsistent structlog import style in `tasks.py`

**File:** `backend/app/ingest/tasks.py:378,398,919,937`
**Issue:** In the two DBF collision detection blocks, `structlog` is imported with a local alias (`import structlog as _sl` / `import structlog as _sl_ru`) and then called as `_sl.stdlib.get_logger(__name__).warning(...)`. This differs from every other logging call in `tasks.py` and from `metadata.py`, which both use `structlog.stdlib.get_logger(__name__)` at module level. The `_sl_ru` alias in the `reupload_file` block was apparently chosen to avoid a re-import collision, but both aliases shadow the same module. The result is two non-idiomatic local imports that are harder to read.

**Fix:** Use the module-level `logger` (which is already defined elsewhere in `tasks.py` via `structlog`) rather than local re-imports:

```python
# Replace both _sl / _sl_ru blocks with:
logger.warning(
    "Shapefile DBF 10-char truncation collision detected",
    table=table_name,
    collisions=dbf_collisions,
)
```
