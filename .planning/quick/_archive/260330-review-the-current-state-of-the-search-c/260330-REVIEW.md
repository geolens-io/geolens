# Search Card Layout + Style Review

## Overall Assessment

The current search cards are good enough to ship, but they are not yet the simplest or most intuitive version of this design. The overall direction is correct: calmer cards, clear title-first scanning, responsive preview behavior, and less ornamental chrome than earlier versions. The remaining issues are mostly about hierarchy, variant handling, and one real state bug that undermines trust in the card surface.

## Findings

### 1. Search can show an empty state and a valid collection card at the same time

Severity: High

A query like `Natural Earth Cultural` currently renders `No results found` while also showing a valid collection card. The API response behind that state reports `numberMatched: 0` but also returns one collection feature. The page then renders the empty state from `numberMatched` and the card list from `features.length`.

This is the most important issue because it makes the card surface feel logically broken before any styling discussion begins.

Likely cause:
- `frontend/src/pages/SearchPage.tsx:163-189`

Recommendation:
- Do not show empty-state UI while `features.length > 0`.
- Decide whether collection-only hits should participate in the main count model or be rendered in a separate result group with their own label.

### 2. Desktop cards waste horizontal space, so the preview feels detached from the content

Severity: Medium-High

At `1440px`, the first cards render at `1104px` wide with a `120px` preview. That is enough room for a strong balanced layout, but the source and description are capped with `max-w-3xl`, which creates a visible dead gutter between the text stack and the preview tile. The preview ends up reading like a separate ornament rather than part of the same card.

Likely cause:
- `frontend/src/components/search/SearchResultCard.tsx:193-245`
- `frontend/src/components/search/SearchResultCard.tsx:218-235`

Recommendation:
- Remove the artificial text-width cap in the card header.
- Keep the header grid, but let the text column actually occupy the column it has.
- Use a smaller preview at tablet widths (`96px`) and a slightly larger one on desktop (`112px` to `120px`).

### 3. The information hierarchy is flatter than it should be

Severity: Medium

The card currently asks the eye to parse too many “almost-equal” layers:

- source line
- description line
- facts row
- keyword pills

In practice, the keyword pills are visually stronger than the facts row, even though the facts are more important. In the long-source case, the source and description also collapse into one muted paragraph block.

Likely cause:
- `frontend/src/components/search/SearchResultCard.tsx:216-304`

Recommendation:
- Make the source line smaller and more clearly separate from the description.
- Let the description carry slightly more contrast than the source.
- Increase the visual weight of facts relative to tags.
- Quiet the tags so they read as optional metadata, not primary data.

### 4. Preview failure states still feel accidental instead of deliberate

Severity: Medium

Live Playwright inspection produced quicklook 404s for some cards. The cards stay structurally intact, but the preview area still reads as an empty tile with a faint failure state rather than a deliberate fallback. That weakens confidence in the cards.

Likely cause:
- `frontend/src/hooks/use-quicklook.ts:26-40`
- `frontend/src/components/search/SearchResultCard.tsx:245-266`

Recommendation:
- Use a stronger fallback state for preview failures.
- Prefer either an explicit “Preview unavailable” treatment or a reliable alternate graphic like `BBoxPreview`, rather than a nearly empty box.

### 5. Collection cards need an intentional variant, not just missing dataset bands

Severity: Low-Medium

The collection card basically works, but it feels like the dataset card after several sections were removed. On desktop it becomes a wide, sparse content block without enough composition to feel intentionally designed.

Likely cause:
- `frontend/src/components/search/SearchResultCard.tsx:127-170`
- `frontend/src/components/search/SearchResultCard.tsx:199-230`
- `frontend/src/components/search/SearchResultCard.tsx:272-346`

Recommendation:
- Give collections a dedicated compact variant:
  - collection badge + dataset count
  - title
  - one-line description
  - timestamp footer
- Avoid leaving empty structural expectations for preview/spec/tag bands that do not exist.

## Recommended Target

The next pass should not redesign the cards from scratch. It should tighten the existing design into a more disciplined version of itself.

### Recommended target layout

1. Header
- Left: type badge, optional status, title, source
- Right: preview sized by breakpoint (`96px` tablet, `112px-120px` desktop)

2. Summary
- One line on tablet, up to two lines on desktop
- More readable than the source, less dominant than the title

3. Facts
- Geometry / count / CRS / band data
- Slightly stronger than tags
- Keep inline, but avoid making them the faintest row in the card

4. Tags
- Max 2 on tablet, max 3 on desktop
- Lower-contrast than facts
- Strictly optional

5. Footer
- Updated time left
- Status right when relevant

### Styling rules for the next pass

- Remove `max-w-3xl` from source and description inside the header grid
- Reduce dead space before the preview
- Separate source and description with clearer contrast/size choices
- Make facts more legible than tags
- Create a collection-specific compact card variant

## Bottom Line

The current cards are close, but not finished. The main path forward is:

1. Fix the contradictory empty-state/collection-card bug.
2. Rebalance the header so the preview feels integrated, not detached.
3. Strengthen the hierarchy between source, summary, facts, and tags.
4. Give collections their own intentionally simple variant.
