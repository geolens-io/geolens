<objective>
Permanently close GitHub issue #101 (HIGH): large uploads (>511 MiB) returning 400 "There was an error parsing the body" because Starlette's `MultiPartParser` constructs a `SpooledTemporaryFile` that rolls over to `tempfile.tempdir` (default `/tmp`), which on the api service is `tmpfs:size=512m`.

Approach: Option 2 from the issue's fix table (the user-selected "Recommended" option). Override `tempfile.tempdir` once at api module-import top so all subsequent SpooledTemporaryFile rollovers land in the existing `/app/staging` named volume, which is already writable by the api user and large enough.

Purpose:
- Closes the open HIGH-severity demo-seeder blocker recorded in MEMORY.md and STATE.md (referenced from quick task 260508-nl9 Finding #5).
- Removes the need for the `docker-compose.upload-override.yml` workaround (which raised /tmp tmpfs to 2g — wastes RAM).
- Saves api container RAM by routing rollover spill to the persistent `upload_staging` volume instead of tmpfs.

Output:
- One ~5-line addition at the top of `backend/app/api/main.py` (imports + override + mkdir guard) sourced from existing `settings.upload_staging_dir`.
- One lightweight regression test under `backend/tests/` that asserts the override holds after module import, with no docker dependency.
</objective>

    **Step 2 — Create `backend/tests/test_tempdir_override.py`:**

    ```python
    """Regression test for GH-101: tempfile.tempdir override.

    Source: .planning/quick/260508-rr5-fix-tmp-tmpfs-cap-blocking-large-uploads/

    Asserts that importing app.api.main has the side effect of redirecting
    Python's stdlib tempfile to settings.upload_staging_dir, so Starlette's
    MultiPartParser SpooledTemporaryFile rollover lands on the staging
    volume instead of /tmp tmpfs (which is sized 512m on the api service).
    """
    from __future__ import annotations

    import importlib
    import sys
    import tempfile

    import pytest


    def test_tempdir_override_uses_staging_dir() -> None:
        """After importing app.api.main, tempfile.gettempdir() returns settings.upload_staging_dir."""
        # Ensure a clean import — drop any cached module so the side effect re-runs.
        sys.modules.pop("app.api.main", None)

        from app.core.config import settings  # noqa: WPS433 — intentional late import

        importlib.import_module("app.api.main")

        assert tempfile.gettempdir() == settings.upload_staging_dir, (
            "GH-101 regression: tempfile.tempdir was not redirected to "
            f"settings.upload_staging_dir (got {tempfile.gettempdir()!r}, "
            f"expected {settings.upload_staging_dir!r}). "
            "Multipart uploads >511 MiB will fail on /tmp tmpfs again."
        )


    def test_tempdir_override_does_not_crash_when_dir_missing(
        tmp_path, monkeypatch
    ) -> None:
        """Module import survives when upload_staging_dir does not yet exist on disk.

        Guards against breaking unit-test runs / alembic-only containers where
        the /app/staging volume is not mounted.
        """
        missing = tmp_path / "does-not-exist-yet" / "staging"
        assert not missing.exists()

        from app.core.config import settings  # noqa: WPS433

        # Monkeypatch the settings field BEFORE re-importing the module so the
        # side-effect block reads the patched value.
        monkeypatch.setattr(settings, "upload_staging_dir", str(missing))
        sys.modules.pop("app.api.main", None)

        importlib.import_module("app.api.main")

        assert missing.exists(), "Defensive mkdir guard did not create the staging dir"
        assert tempfile.gettempdir() == str(missing)


    def test_tempdir_override_no_hardcoded_path() -> None:
        """The override sources its value from settings, not a hardcoded literal."""
        # Read the source so we fail loudly if a future refactor regresses to '/app/staging'.
        from pathlib import Path as _P

        main_py = _P(__file__).resolve().parents[1] / "app" / "api" / "main.py"
        source = main_py.read_text(encoding="utf-8")

        # The exact override line must reference settings.upload_staging_dir.
        assert "settings.upload_staging_dir" in source, (
            "main.py tempdir override must source settings.upload_staging_dir, "
            "not a hardcoded path."
        )
        # No bare hardcoded '/app/staging' string assignment to tempfile.tempdir.
        # (This is a substring check, deliberately conservative — comments may
        # still contain '/app/staging'.)
        forbidden = 'tempfile.tempdir = "/app/staging"'
        assert forbidden not in source, (
            "main.py contains a hardcoded tempfile.tempdir = \"/app/staging\" "
            "assignment; route through settings.upload_staging_dir instead."
        )
    ```

    Note on test 3: it intentionally inspects file contents, not behavior. This is the lightweight regression guard the constraints called for — it catches a future refactor that drops the settings sourcing. Prefer this over an architecture-guard test in the architecture-guard module since this lives next to the behavior it protects.

    **Why this exact ordering, recapping for the executor:**
    1. The override is a module-level statement, not inside an `@app.on_event("startup")` or the `lifespan` async generator. Reason: lifespan runs AFTER imports complete and AFTER FastAPI registers all routes. By that point, any code that already constructed a SpooledTemporaryFile (none currently exists at import time, but the invariant is fragile) would have captured the old `tempfile.tempdir`. Module-top is the only ordering that holds for ALL future Starlette versions.
    2. It runs BEFORE `from fastapi import FastAPI, ...`. Reason: defense in depth — even though FastAPI/Starlette do not currently construct SpooledTemporaryFile at import time, future versions might. Module-top with stdlib-only prerequisites (`tempfile`, `pathlib.Path`, our own `app.core.config`) is the most robust position.
    3. The mkdir guard uses `parents=True, exist_ok=True` — never raises on existing dirs, creates parent chain if needed. This makes the override safe in unit-test mode where `/app/staging` may not exist on the host filesystem.
