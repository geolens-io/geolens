2. **Overview** (≈1 page)
   - One-paragraph problem statement: "GeoLens users want 3D terrain + extruded buildings + the ability to work with PostGIS 3D geometry. What is the cheapest path that uses the stack we already have?"
   - TL;DR of the three findings from RESEARCH.md Summary section:
     - Terrain is essentially free (Titiler 2.0.0 already ships `terrainrgb`).
     - Building extrusion is a single fill-extrusion style layer on the existing MVT pipeline.
     - PostGIS 3D geometry is structurally blocked at the ST_AsMVT tile boundary (the MVT 2.1 spec is 2D-only).
   - Primary recommendation box: ship Phase A (terrain + extrusions) first, defer Phase B (PostGIS-3D detection) and Phase C (3D geometry delivery endpoint) — per D-01 from CONTEXT.md: this is a design doc, not a commitment to implement.
   - Scope statement: three pillars in scope (terrain, extrusions, PostGIS 3D); deck.gl / 3D Tiles / Cesium / CityGML / glTF / true 3D meshes / point clouds are explicitly **out of scope** and deferred to future milestones.

3. **Current State of GeoLens** (≈1 page)
   Verified facts (cite "verified during planning 2026-04-08" for each):
   - **Frontend stack:** `maplibre-gl ^5.18.0`, `@vis.gl/react-maplibre ^8.1.0` (frontend/package.json). All MapLibre v5 3D features are available without an upgrade.
   - **Raster pipeline:** `ghcr.io/developmentseed/titiler:2.0.0` (docker-compose.yml:166-197). Ships built-in `terrainrgb` and `terrarium` algorithms — no rio-rgbify needed.
   - **MVT pipeline:** `backend/app/tiles/service.py` builds MVT tiles via `ST_AsMVT` / `ST_AsMVTGeom`; this is the 2D boundary for vector data.
   - **Ingestion:** `backend/app/ingest/ogr.py:319-350, 411-413` calls ogr2ogr with `PROMOTE_TO_MULTI` + `GEOMETRY_NAME=geom` but **no `-dim` flag** — Z coordinates survive ingestion unchanged.
   - **4326 column:** `backend/app/ingest/metadata.py:452-490` declares `geom_4326 geometry(Geometry, 4326)` — **unconstrained on dimension**, so Z survives the WGS84 reprojection too. The data is in PostGIS; only the MVT delivery strips it.
   - **Two map components:** `ViewerMap.tsx` (read-only, embed tokens) and `BuilderMap.tsx` (editing); they share `syncLayersToMap` + layer adapter registry.
   - **Known wrapper caveats (from CLAUDE.md):** `transformRequest` prop is silently ignored on the v8 `<Map>`; `<Source type="vector">` + declarative `<Layer>` children may not work — GeoLens uses imperative `map.addSource()` / `map.addLayer()`. Good news: declarative `terrain` and `sky` props **do** work on the v8 `<Map>` (verified via react-maplibre 1.0-release terrain example).

4. **Pillar 1: Terrain (DEM) via MapLibre + Titiler** (≈1.5 pages)
   - **What MapLibre native supports:** `raster-dem` source type, `setTerrain({source, exaggeration})`, `encoding` options (`mapbox`, `terrarium`, `custom`). Standard 3D viewer config is `pitch: 70, maxPitch: 85`. `NavigationControl` auto-gains pitch/rotate handles in 3D; `TerrainControl` is a dedicated toggle button.
   - **How Titiler closes the loop:** `GET /cog/tiles/WebMercatorQuad/{z}/{x}/{y}.png?url={dem_path}&algorithm=terrainrgb` produces Mapbox Terrain-RGB encoded PNGs on the fly from any single-band float DEM COG. GeoLens's existing raster proxy at `backend/app/tiles/router.py:261-291` already builds URLs in this form — adding terrain support is a query-string parameter plus a DEM flag.
   - **What's needed to ship:**
     - Backend: DEM detection heuristic at ingest (single-band float + optional user tag); a `is_dem` flag on the raster dataset record; add `&algorithm=terrainrgb` to the proxy URL for flagged DEMs.
     - Frontend: add `<Source type="raster-dem" encoding="mapbox">` when a DEM dataset is active; call `map.setTerrain(...)` (or use declarative `terrain` prop); add a 3D toggle to the viewer toolbar.
   - **Small illustrative snippet** (from RESEARCH.md — mark it as "illustrative only, not shipped in this spike"):
     ```tsx
     const terrain = { source: 'terrain-dem', exaggeration: 1.5 };
     <Map pitch={70} maxPitch={85} terrain={terrain}>
       <Source id="terrain-dem" type="raster-dem"
               url="/tiles/dem/{id}/tiles.json" tileSize={256} />
       <TerrainControl {...terrain} position="top-left" />
     </Map>
     ```
   - **Known MapLibre v5 3D bugs to flag:** `setPaintProperty` every-other-call issue when terrain is active; hillshade detail regression with terrain; `fill-extrusion` `setFilter` stale-until-repaint. (See RESEARCH.md "Known MapLibre v5 3D bugs" table — reproduce it in the doc.)
   - **Style-reload gotcha:** When the basemap style changes, `setStyle()` wipes custom sources/layers. The existing `style.load` re-sync hook in `ViewerMap.tsx` must also re-apply the terrain source and `setTerrain` call. Flag as a known integration point.
   - **Embed token compatibility:** The existing `X-Embed-Token` header injection in `ViewerMap.handleLoad` already covers terrain tiles because they flow through the same Titiler proxy. (Open Q4 in RESEARCH.md — verified by code reading.)

5. **Pillar 2: Building Extrusions via fill-extrusion** (≈1 page)
   - **What MapLibre supports:** `fill-extrusion` layer with data-driven `fill-extrusion-height`, `fill-extrusion-base`, `fill-extrusion-color`. Per-layer (not per-feature) `fill-extrusion-opacity` — flag this caveat. `fill-extrusion-vertical-gradient` gives free side-face shading. MapLibre v4.4.0+ computes per-polygon elevation on terrain so buildings sit correctly on uneven ground.
   - **How it plugs into GeoLens:** The existing MVT pipeline in `backend/app/tiles/service.py` already returns all non-geometry attribute columns alongside geometries. A polygon dataset with a numeric `height` column already delivers `height` as an MVT feature property — **zero backend changes needed**. The frontend work is in `fill-adapter.ts`: add a companion `fill-extrusion` layer (or swap `fill` → `fill-extrusion` when 3D mode is on).
   - **Small illustrative snippet** (reproduce the `coalesce + to-number` pattern from RESEARCH.md — mark as illustrative):
     ```ts
     map.addLayer({
       id: `${layerId}-extrusion`,
       type: 'fill-extrusion',
       source: sourceId,
       'source-layer': sourceLayer,
       minzoom: 14,
       paint: {
         'fill-extrusion-height': ['coalesce', ['to-number', ['get', heightColumn], 0], 0],
         'fill-extrusion-base':   ['coalesce', ['to-number', ['get', baseColumn], 0], 0],
         'fill-extrusion-color': fillColor,
         'fill-extrusion-opacity': 0.85,
         'fill-extrusion-vertical-gradient': true,
       },
     });
     ```
   - **Height attribute conventions** (reproduce the RESEARCH.md table): OSM `height` / `building:height` / `building:levels × ~3m`, CityGML `measuredHeight`, US datasets `HEIGHT` / `Height_m` / `BLDG_HT`. Recommendation: **don't auto-detect** — make it a per-layer dropdown of numeric columns in the layer style editor, with a free-form expression fallback, default off.
   - **Zero-height fallback:** `coalesce ... 0` renders as flat polygon at ground level — visually identical to the existing 2D fill. Safe default.
   - **Terra Draw limitation:** `terra-draw@1.25.0` draws at z=0; drawn features will land flat even in 3D mode. Acceptable for v1, flag as future limitation.

6. **Pillar 3: PostGIS 3D Geometry Support** (≈2 pages — the most complex pillar)
   - **What PostGIS supports:** `POINT Z`, `LINESTRING Z`, `POLYGON Z` (common — LiDAR, 3D footprints, surveying), `POLYHEDRALSURFACE`, `TIN`, `TRIANGLE` (rare — CityGML LOD2, IFC). Observation: ~99% of "3D data" GeoLens users will upload is XY polygons + height attribute, or POINT Z LiDAR-derived data.
   - **Detection functions GeoLens would use:** `ST_NDims`, `ST_CoordDim`, `ST_Is3D`, `GeometryType`, `ST_ZMin`, `ST_ZMax`, `ST_Z`. Casting: `ST_Force2D`, `ST_Force3D`, `ST_ForceSFS`.
   - **GeoLens's current state (verified 2026-04-08):** Z survives ingestion because ogr2ogr has no `-dim` flag, and it survives 4326 reprojection because `geom_4326` is declared `geometry(Geometry, 4326)` with no dimension constraint. **Z values are sitting in PostGIS right now for any 3D dataset that's been uploaded — they're just invisible to users and to the MVT pipeline.**
   - **THE CRITICAL FINDING: ST_AsMVT is 2D-only.** Reproduce the reasoning: MVT 2.1 spec defines only X/Y axes with integer coordinates — no Z axis, no `MoveToZ` command, no provision for elevation in the protobuf encoding. `ST_AsMVTGeom` therefore produces 2D output regardless of input dimensionality. This is a **structural limitation of the tile format, not a PostGIS bug**. Cite https://github.com/mapbox/vector-tile-spec/blob/master/2.1/README.md.
   - **Implications for the design:**
     - A PostGIS table with `POLYGON Z` features tiles correctly through the existing pipeline, but **all rendered geometries are flat at z=0** — Z is silently dropped.
     - Building extrusions (Pillar 2) work around this by carrying height as a **numeric attribute**, not as geometry Z.
     - True 3D geometry delivery requires a parallel non-MVT path.
   - **Three viable strategies for exposing 3D data to the client** (reproduce the RESEARCH.md list):
     1. **Attribute promotion (cheapest).** At ingest, if `ST_Is3D` and geometry is point-like, write `ST_Z(geom)` into a new `elev` numeric column. The existing MVT pipeline carries it as a normal attribute. Loses Z for non-point geometries but gives free 3D symbolization for LiDAR.
     2. **GeoJSON-Z streaming endpoint.** New `/api/datasets/{id}/features.geojson?include_z=true` endpoint streaming RFC 7946 three-coordinate positions. MapLibre GeoJSON sources accept Z pass-through. Best for small datasets (e.g. cap at 5000 features). No tiling, no LOD, no cache.
     3. **Custom binary format (3D Tiles / glTF / Cesium).** **Out of scope per CONTEXT.md.** Named only so readers understand what's being deferred.
   - **POLYHEDRALSURFACE / TIN / TRIANGLE:** defer entirely. PostGIS stores them today, but rendering them correctly requires 3D Tiles or glTF — out of scope. Flag in Open Questions for a future decision.
   - **Web Mercator clipping:** `clip_to_mercator_bounds` (`backend/app/ingest/metadata.py:432`) clips to ±85.06° latitude. DEMs of polar regions would be cropped. Out of scope for the spike, noted as a limitation.
