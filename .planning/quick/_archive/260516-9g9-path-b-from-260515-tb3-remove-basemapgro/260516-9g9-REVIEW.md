## Critical Issues

### CR-01: Path R `applyMasterOpacity` compounds on re-application; opacity becomes unrevertable

**File:** `frontend/src/lib/basemap-utils.ts:322-343` (helper) and `frontend/src/components/builder/map-sync.ts:222-258` (caller)

**Issue:** `applyBasemapConfigToMap` (map-sync.ts:222) takes its input from `map.getStyle()` — the **live** MapLibre style, which contains paint mutations from prior `applyMasterOpacity` calls. `applyMasterOpacity` then reads `existing = existingPaint[key]` and multiplies by the new `masterOpacity`. This makes every successive call compound, and the early-exit at `masterOpacity >= 1` prevents the value from ever returning to 1.

Concrete scenario (raster basemap — Positron, Dark, OSM, Bright — the most common case):

1. User drags master opacity 1.0 → 0.5.
   - First call: `existing = undefined` on raster layer (raster basemaps don't ship with explicit `raster-opacity`), so `nextPaint['raster-opacity'] = 0.5`. Stamped onto live map. **Looks correct visually.**
2. User drags 0.5 → 0.8 (expects basemap to brighten slightly).
   - `map.getStyle()` returns live style with `raster-opacity: 0.5`. `existing = 0.5`. `nextPaint = 0.5 * 0.8 = 0.4`. Map dims **further**. **Opposite of intent.**
3. User drags 0.4 → 1.0 (expects full restoration).
   - `applyMasterOpacity` early-exits at line 323 (`masterOpacity >= 1`). `next.paint['raster-opacity']` stays at 0.4 (the live value). Diff is empty (`current.paint === next.paint`). `setPaintProperty` is never called. **Map stuck at 0.4. Unrevertable without a page reload.**

The vector-line subtle case is masked because `applyProminence` always re-stamps `line-opacity` to literal 0.35/0.4 in `subtle` mode, giving Path R a fresh baseline. But (a) the `full` mode does NOT re-stamp (it only sets visibility), so once a `full`-mode road has been dimmed by master opacity, it compounds the same way; (b) `subtle` mode itself compounds when master opacity changes more than once without a road-visibility change in between.

The 4 new test cases all start with a freshly constructed `StyleSpecification` literal, so the test fixtures match the "first call" path only. The compounding manifests only against a live `map.getStyle()` output that has already been mutated — which is exactly the runtime path. A round-trip test (`apply 0.5 → apply 0.8` against the SAME style, then assert raster-opacity ≈ 0.8) would have caught this.

The PARTIAL Gate 8 (manual smoke) coincidentally passes because the persisted value (`0.55`) matches the live (compounded) state on reload — the save/load loop hides the bug from the smoke checklist. The user-perceived bug is "slider only goes down" once they try to drag it back up.

**Fix:** the helper must operate against the **original** style, not the live-mutated one. Two clean options:

```ts
// Option 1 (recommended): make applyBasemapConfigToMap idempotent by
// re-fetching the unmodified basemap style. Requires plumbing the
// "original" style or stashing it in a ref. Larger change.

// Option 2 (smallest diff): treat applyMasterOpacity as an absolute setter
// rather than a multiplier — track the "original" per-key opacity in a
// WeakMap keyed by (layer.id, key), or store it on the layer's metadata
// the first time we touch it.

// Option 3 (in-place, minimum surface): scrap the early-exit at >=1 and
// always set the per-type *-opacity keys to (originalOpacity * master).
// originalOpacity must be cached separately (NOT read from live paint).

// Sketch for Option 3 — change applyMasterOpacity to write *absolute*
// opacity sourced from the original (pre-mutation) layer paint:
function applyMasterOpacity(layer: StyleLayer, masterOpacity: number, baselinePaint?: Record<string, unknown>): StyleLayer {
  if (!Number.isFinite(masterOpacity) || masterOpacity < 0) return layer;
  const type = typeof layer.type === 'string' ? layer.type : '';
  const keys = OPACITY_PAINT_KEYS_BY_TYPE[type];
  if (!keys || keys.length === 0) return layer;
  const existingPaint = (layer.paint ?? {}) as Record<string, unknown>;
  const baseline = baselinePaint ?? existingPaint;
  const nextPaint: Record<string, unknown> = { ...existingPaint };
  for (const key of keys) {
    const original = baseline[key];
    if (original == null) {
      nextPaint[key] = masterOpacity; // MapLibre default 1.0 * master
    } else if (typeof original === 'number' && Number.isFinite(original)) {
      nextPaint[key] = original * masterOpacity; // absolute, not iterative
    }
    // Expressions: untouched
  }
  return { ...layer, paint: nextPaint } as StyleLayer;
}
```

The plumbing for `baselinePaint` requires `applyBasemapConfigToMap` to cache the original basemap style on first load (a `Map<layerId, originalPaint>` keyed on style URL or `style.load` event). That's where the real work lives.

Also add a vitest case that explicitly catches the regression:

```ts
it('master opacity is reversible when reapplied (does not compound)', () => {
  const style: StyleSpecification = {
    version: 8,
    sources: { osm: { type: 'raster', tiles: ['x'], tileSize: 256 } },
    layers: [{ id: 'osm', type: 'raster', source: 'osm' }],
  };
  // First pass: 0.5
  const afterFirst = applyBasemapConfigToStyle(style, { opacity: 0.5 });
  // Second pass against the mutated style: 1.0
  const afterSecond = applyBasemapConfigToStyle(afterFirst, { opacity: 1 });
  const layer = afterSecond.layers[0] as unknown as { paint: { 'raster-opacity': number } };
  expect(layer.paint['raster-opacity']).toBeCloseTo(1, 5); // currently fails — stuck at 0.5
});
```

This is a BLOCKER because Path R was the explicit deliverable of Task 4 and SUMMARY frames it as closing CAVEAT-1; in reality it only works for the very first slider drag of a session and silently corrupts on every drag thereafter.

## Warnings

### WR-01: `label_mode='hidden'` followed by master-opacity dim leaves `text-halo-width: 0.8` on hidden labels

**File:** `frontend/src/lib/basemap-utils.ts:373-388`

**Issue:** When `road_visibility` or `boundary_visibility` is `hidden`, the label early-exit branch at line 373-375 (`sublayerHidden`) skips the prominence/subtle stamp on text layers. But the prior pass through `applyProminence(layer, 'hidden')` for the road/boundary sets `layout.visibility = 'none'` — fine. The label layer itself, however, is reached only via `isTextLabelLayer(next) && !sublayerHidden`. If a layer is BOTH a road label (matched by `isRoadLayer` for symbol type) AND a text label, the road-prominence branch (lines 357-365) sets `text-opacity: 0.45, icon-opacity: 0.35` for subtle, OR `visibility: 'none'` for hidden. That's the only stamp for the symbol case, and `applyMasterOpacity` then multiplies into 0.45 / 0.35 — correct.

The real edge: when a layer matches `isBoundaryLayer` (symbol case) AND `isTextLabelLayer`, the boundary branch only stamps `text-opacity: 0.45` (no `icon-opacity`), then the `!sublayerHidden` guard at line 376 prevents the label branch from running — so any `icon-opacity` on the layer is left at MapLibre default 1.0. Then `applyMasterOpacity` for `symbol` type writes both `text-opacity` (existing 0.45 × master) AND `icon-opacity` (undefined → master). Result: the icon ends up at full master opacity while the text is master × 0.45. This is unintentional asymmetry. Low frequency (only OFM boundary symbol layers that ship icons), but it'll look like "the boundary glyph stays solid while the label dims".

**Fix:** Either include `icon-opacity: 0.45` in the boundary subtle paint, or restructure so `applyMasterOpacity` doesn't seed missing keys for symbol layers that were partially stamped by prominence. The latter is cleaner — track which keys came from prominence vs default and only multiply (not seed) for layers that already passed through a prominence branch.

### WR-02: `markDirty()` after `setBasemapConfig` is redundant; could mask future setBasemapConfig regressions

**File:** `frontend/src/pages/MapBuilderPage.tsx:748-753`

**Issue:** `onMasterOpacityChange` calls `layers.setBasemapConfig({...current, opacity})` followed by `layers.markDirty()`. The `setBasemapConfig` setter in `use-builder-layers.ts` is the React `useState` setter — it does NOT itself mark dirty. So `markDirty()` is required here. **However**, the same pattern at line 461 (`handleResetBasemapAppearance`) calls `setBasemapConfig(null)` then `markDirty()` — and the comment at line 463 says "basemapConfig reset IS persisted (null → saved)". That comment misleadingly implies setBasemapConfig handles dirty-tracking. It doesn't.

This is a **convention drift risk**: every call site of `setBasemapConfig` must remember to call `markDirty()`. Several existing call sites (e.g., `BasemapAppearanceControls`'s `onChange` is passed via `setBasemapConfig` indirectly) rely on the parent doing it. A grep across the repo would find inconsistency.

**Fix:** Either (a) wrap `setBasemapConfig` in a setter that also marks dirty (cleanest — one source of truth, matches the spirit of Option B), or (b) document the convention with a JSDoc on the setter export and audit existing callers. Option (a) is preferred because Option B was supposed to be "single source of truth" — having a setter that doesn't track its own dirty state is a half-measure.

### WR-03: Schema `opacity` field ordering breaks the "scalars + nullable enum last before model_config" convention

**File:** `backend/app/modules/catalog/maps/schemas.py:182-214`

**Issue:** Per RESEARCH.md §1, the existing convention is: lower-bound enums first (label_mode, road_visibility, boundary_visibility), bool, enum, then optional/nullable enum (relief_contrast), then `model_config`. The new `opacity: float` is positioned **after** the optional `relief_contrast` and **before** `model_config`. That's the position the PLAN prescribed, but it breaks symmetry because `opacity` is a non-nullable scalar with a default — it should sit with `building_visibility` (the other non-nullable scalar), not between the nullable enum and `model_config`.

The functional impact is zero (Pydantic doesn't care about order), but JSON/OpenAPI doc readers see fields out of logical groupings. Also, the existing field-order convention has been the implicit shape that downstream SDK consumers' generated types follow — `attrs`-based Python SDK will emit `opacity` last, which differs from "alphabetical" or "logical-domain" ordering they might expect.

**Fix:** Move `opacity: float = Field(...)` to sit immediately after `building_visibility: bool` (line 198), before `land_water_tone`. Net effect on functional behavior: zero. Net effect on docs/SDK: more obvious grouping.

Optional alternative: insert a comment block at the position to flag intentional convention deviation if the author specifically wants opacity last for "master appearance" semantics.

### IN-02: Test "opacity=1 is a no-op (paint unchanged for raster)" asserts the wrong invariant

**File:** `frontend/src/lib/__tests__/basemap-utils.test.ts:349-358`

**Issue:** The test starts with `paint: { 'raster-opacity': 0.8 }` and asserts that applying opacity=1 leaves it at 0.8. This is **correct as a no-op test for a single call** but masks the CR-01 unrevertable-opacity bug. The test name implies the broader invariant "opacity=1 is identity," which would be violated by CR-01's intended fix (where opacity=1 means "restore to original 1.0, not preserve current 0.8").

When fixing CR-01, this test will need to change. The new semantics: "opacity=1 restores to baseline (the original style value)." This test will then assert 1.0, not 0.8. The current shape was correct for the as-shipped (buggy) semantics but ossifies the bug.

**Fix:** Bundle the test update with CR-01's fix. Document the change in the commit message.

### IN-03: `applyMasterOpacity` skips `circle-stroke-opacity` for circles without stroke

**File:** `frontend/src/lib/basemap-utils.ts:318`
