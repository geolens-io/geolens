<objective>
Fix SP-07 — make `has_quicklook` a truthful predicate on the OGC search response.

Today `service_records.py:312` derives `has_quicklook` from `dataset.quicklook_256_uri is not None`. That column is set when ingest writes the URI, but stale demo-seeded rows (and any storage drift) can leave the column non-null after the PNG has been removed from MinIO/S3/local storage. The frontend uses `has_quicklook` as license to fire `GET /datasets/<id>/quicklook?size=256` — divergence becomes a 404 in the browser console (m-02 evidence: 3 of 6 seeded MultiPoint datasets in the 2026-05-15 smoke run).

v1009.1 shipped a frontend-only mitigation (`quicklook-cache.ts` negative-caches 404s in sessionStorage). The honest fix is upstream — at the predicate. Going forward, ingest already writes the URI **only after a successful `storage.put()`** (verified in `tasks_common.py:586-587`, `tasks_raster.py:374-380`, `tasks_vrt.py:312-321`), so the URI-presence predicate is reliable for any row created from now on. The only dishonest rows are pre-existing stragglers.

**Approach: Option A — Backfill-only, no schema change.** Add a one-shot async script that walks every vector dataset with `quicklook_256_uri IS NOT NULL`, calls `storage.exists()` once per row, and clears the URI on miss. After the sweep, the existing predicate becomes truthful. This matches the "small, mechanical" framing — no Alembic migration, no enterprise overlay coordination, no `service_records.py` edit, no per-request storage IO. The storage abstraction at `app/platform/storage/provider.py:27` already exposes `async def exists(key) -> bool` on both `s3.py:108` and `local.py:52`, so the helper exists — no new storage method needed.

**Approaches considered and rejected:**
- **B. Add a `quicklook_generated_at` verified-flag column.** Rejected: requires an Alembic migration on the OSS migration chain, which must coordinate with the enterprise overlay (feedback_alembic_squash_enterprise_chain). Backfilling the new column from URI presence carries forward the same staleness, so a HEAD sweep is still needed at migration time — strictly more work than Option A.
- **C. Per-request HEAD in `service_records.py:312`.** Rejected explicitly in the planning context: N storage HEADs per search page render is unacceptable IO amplification. Each search page renders 12-50 records; this would be 12-50 storage requests per page.
- **D. Celery periodic reconciliation.** Rejected: heaviest blast radius, requires worker schedule wiring, and only marginally better than a one-shot script given ingest now writes URI post-upload (so the stale set is bounded — backfill once and the problem is closed).

Purpose: Cut the search-page 404 noise at the source so the frontend's negative cache is defense-in-depth, not the only line of defense.
Output: One-shot reconcile script under `backend/scripts/` (matching the existing `generate_vector_quicklooks.py` pattern) + a targeted pytest covering the predicate contract and the sweeper behavior.
</objective>
