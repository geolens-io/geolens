# Quick Task 20260425 Plan

## Local Research Findings

- No repository-root `Dockerfile` or root `.dockerignore` exists today.
- `backend/Dockerfile` builds the current Python runtime from `python:3.14.3-slim`, installs `uv` from `ghcr.io/astral-sh/uv:0.11.3`, installs GDAL/XMLSec runtime packages, syncs locked backend deps, creates `appuser`/`appgroup` uid/gid 1001, and sets the API entrypoint and API healthcheck.
- `docker-compose.yml` builds `api`, `worker`, and `migrate` from `./backend`; API and worker deliberately run as root in compose so their entrypoints can repair mounted volume ownership and then drop to uid/gid 1001 with `setpriv`.
- `backend/scripts/api-entrypoint.sh` and `backend/scripts/worker-entrypoint.sh` are the runtime split between API and worker. The worker must get its own final target with `worker-entrypoint.sh`, `uv run --no-dev python -m app.worker`, and worker healthcheck semantics.
- `frontend/Dockerfile` is already multi-stage: `node:25.9.0-alpine` builds `dist`, then `nginxinc/nginx-unprivileged:1.29.8-alpine` serves it. The root target should preserve that pattern and copy `frontend/nginx.conf` plus `frontend/docker-entrypoint.sh`.
- `frontend/nginx.conf` listens on container port `8080`, not `80`. When touching the demo overlay, keep production frontend port mapping aligned to container `8080`.
- `docker-compose.demo.yml` is the production-like overlay: it clears dev bind mounts and uses the production frontend image. Base `docker-compose.yml` keeps Vite dev server behavior through `frontend/Dockerfile.dev`.
- `docker-compose.enterprise.yml` mounts `../geolens-enterprise` at runtime for `api`, `worker`, and `migrate`. Do not copy enterprise code into the root Docker build context.
- `.github/workflows/publish.yml` currently publishes separate `geolens-api` and `geolens-frontend` images from separate contexts. The root Dockerfile change should keep separate publish artifacts and add a separate `geolens-worker` target/image rather than folding worker into API.
- Current compose syntax passes: `docker compose config --quiet`, `docker compose -f docker-compose.yml -f docker-compose.demo.yml config --quiet`, and `docker compose -f docker-compose.yml -f docker-compose.enterprise.yml config --quiet`.

## Task 1 - Add Root Dockerfile Targets And Root Ignore Rules

files:
- `Dockerfile`
- `.dockerignore`

action:
- Create a repository-root `Dockerfile` using BuildKit syntax and named runtime targets: `api`, `worker`, and `frontend`.
- Keep shared backend setup in a `backend-base` stage that copies backend files from root-qualified paths (`backend/pyproject.toml`, `backend/uv.lock`, `backend/app`, `backend/alembic`, `backend/alembic.ini`, `backend/scripts`) into `/app`, installs the same apt packages and locked uv dependencies as `backend/Dockerfile`, creates uid/gid 1001, and owns `/app`, `/app/staging`, and the app home/cache paths.
- Put API-only runtime metadata on the `api` target: `geolens-api` OCI title/description, API healthcheck against `http://localhost:8000/health`, `ENTRYPOINT ["/app/scripts/api-entrypoint.sh"]`, and the existing uvicorn command.
- Put worker-only runtime metadata on the `worker` target: `geolens-worker` OCI title/description, healthcheck against `http://localhost:8001/health/live`, `ENTRYPOINT ["/app/scripts/worker-entrypoint.sh"]`, and `uv run --no-dev python -m app.worker`.
- Add `frontend-build` and `frontend` stages that reproduce `frontend/Dockerfile`: install frontend dependencies from `frontend/package.json` and `frontend/package-lock.json`, run `npm run build:app`, serve with `nginxinc/nginx-unprivileged:1.29.8-alpine`, keep the nginx upgrade step, copy `dist`, `frontend/public/env-config.template.js`, `frontend/nginx.conf`, and `frontend/docker-entrypoint.sh`, expose `8080`, and keep the existing frontend healthcheck.
- Do not introduce a combined `app` target or copy built frontend assets into the backend runtime.
- Add a root `.dockerignore` before any root-context compose/CI wiring. Prefer an allowlist-style context that includes only backend and frontend build inputs needed by the new Dockerfile, while excluding `.git`, `.env*` except examples, `.planning`, `.claude`, `.agents`, docs/audit output, coverage/test output, local virtualenvs, dependency folders, Playwright reports, generated SDK output, and any enterprise/private directories.
- Leave `backend/Dockerfile`, `frontend/Dockerfile`, and `frontend/Dockerfile.dev` in place for compatibility unless later verification proves they are unused and the user asks to remove them.

verify:
- `docker build --target api -t geolens-api:root-dockerfile .`
- `docker build --target worker -t geolens-worker:root-dockerfile .`
- `docker build --target frontend -t geolens-frontend:root-dockerfile .`
- `docker run --rm --entrypoint sh geolens-api:root-dockerfile -c 'test "$(id -u)" = "1001" && test -x /app/scripts/api-entrypoint.sh'`
- `docker run --rm --entrypoint sh geolens-worker:root-dockerfile -c 'test "$(id -u)" = "1001" && test -x /app/scripts/worker-entrypoint.sh'`
- `docker run --rm --entrypoint sh geolens-frontend:root-dockerfile -c 'test "$(id -u)" != "0" && test -f /usr/share/nginx/html/env-config.template.js && nginx -t'`

done:
- Root Docker builds produce three independently runnable runtime images with the same dependency baselines, entrypoints, healthchecks, non-root runtime posture, and frontend nginx behavior as the current split Dockerfiles.

## Task 2 - Wire Compose To Root Targets Without Collapsing Services

files:
- `docker-compose.yml`
- `docker-compose.demo.yml`
- `docker-compose.enterprise.yml`

action:
- Update backend-backed services in `docker-compose.yml` to build from the root Dockerfile without changing service boundaries: `api` uses `context: .`, `dockerfile: Dockerfile`, `target: api`; `worker` uses `target: worker`; `migrate` uses the API-compatible backend runtime target unless a dedicated migration target is added in Task 1.
- Preserve API and worker compose runtime behavior: existing `user: "0:0"` root-then-drop comments, entrypoints, commands, bind mounts, volume ownership expectations, healthchecks, `init`, `stop_grace_period`, restart policy, environment anchors, and `depends_on` behavior remain intact.
- Preserve the base development frontend path in `docker-compose.yml` unless intentionally adding a `frontend-dev` root target. The Vite dev server should continue to use `frontend/Dockerfile.dev`, bind mount `./frontend:/app`, and expose container port `5173`.
- Update `docker-compose.demo.yml` so the production frontend image is built from the root Dockerfile `frontend` target. Keep dev bind mounts cleared, preserve API/worker production-like commands and volumes, and map the host frontend port to the nginx container port `8080` used by `frontend/nginx.conf`.
- Keep `docker-compose.enterprise.yml` as runtime-only enterprise wiring. It may inherit the new root build targets from base compose, but it must continue mounting `../geolens-enterprise:/enterprise` and setting `GEOLENS_ENTERPRISE_PATH`/`GEOLENS_EDITION` without adding enterprise build context or Dockerfile COPY steps.
- Keep TiTiler, database, seeder, backup, MinIO, Valkey, and other support-service image/build definitions separate from the root API/worker/frontend runtime image targets.
