# Search Page UI/UX Review

## Overall Assessment

The current search page is better than average and has a clear search-first interaction model, but it does not fully meet best-practice UI/UX yet. The result-card redesign largely succeeded, and the mobile filters sheet is directionally good. The remaining issues are not mostly matters of taste; several are concrete quality gaps in accessibility, state clarity, and responsive polish.

## Findings

### 1. Closed spatial search dialog remains exposed off-canvas

Severity: High

Playwright found a mounted `Search area` dialog even when the panel was visually closed. It remained `display: block`, `visibility: visible`, and contained 9 focusable descendants while positioned off-screen. That is a real accessibility issue because closed dialogs should not stay active in the accessibility tree or tab order.

Likely cause:
- `frontend/src/components/search/SpatialFilterPanel.tsx:279-288`

Recommendation:
- Use a proper hidden/unmounted dialog or sheet primitive for the closed state.
- If it must stay mounted, apply the full accessibility contract: remove it from the a11y tree and keyboard path when closed.

### 2. Scrolling the landing page removes filter access while users are still browsing results

Severity: High

The default landing experience already includes a browsable result list. Once the hero scrolls away, the sticky shell preserves only the search input and drops the filter controls entirely. That means users remain in a browse context but lose refinement access at the moment the sticky header is supposed to help.

Likely cause:
- `frontend/src/pages/SearchPage.tsx:53-67`
- `frontend/src/pages/SearchPage.tsx:73-89`

Recommendation:
- Treat “default results shown” as browse mode for sticky controls, not just “query/filter state exists”.
- Or keep a condensed filter affordance available in the sticky landing state.

### 3. Type counts and visible type controls do not match the records actually shown

Severity: High

On the live landing page, the toolbar showed `All (59)` and `Vector (59)` while the page itself showed `60 results` and visibly included a `Table` result card. That is confusing and weakens trust in the filter model.

Likely cause:
- `frontend/src/components/search/FilterPanel.tsx:372-386`
- `frontend/src/components/search/FilterPanel.tsx:686-698`

Recommendation:
- Derive the visible “All” count from the same source of truth as the result count.
- Decide whether tables are first-class filter types in this UI; if yes, expose them, and if not, stop labeling them inconsistently in the results/counts model.

### 4. Mobile default state promotes Save Search too early, and pagination breaks visually

Severity: Medium

On mobile landing, the hero action row shows `Filters` and `Save` even when the user has not entered any query or refinement. That adds clutter and promotes a weak action. Separately, the pagination center label wraps vertically on narrow screens, which reads as a broken layout.

Likely cause:
- `frontend/src/components/search/FilterPanel.tsx:253-280`
- `frontend/src/components/search/FilterPanel.tsx:520-529`
- `frontend/src/components/layout/Pagination.tsx:23-53`

Recommendation:
- Gate mobile Save Search behind meaningful search state, matching the desktop rule.
- Add a mobile-specific pagination layout that stacks or compresses intentionally.

### 5. Table cards still produce failing quicklook requests

Severity: Medium

The first table result on landing triggers failing quicklook requests and falls back to `Preview unavailable`. The fallback UI is acceptable, but the repeated 404s are unnecessary noise and make the search page feel less polished in dev/staging.

Likely cause:
- `frontend/src/components/search/SearchResultCard.tsx:86-89`
- `frontend/src/hooks/use-quicklook.ts:20-47`

Recommendation:
- Only fetch quicklooks for record types that actually support them, or expose quicklook capability explicitly in the result payload.

### 6. Browse sticky shell is still heavier than it needs to be

Severity: Low

The browse header works, but once filters are active it takes up a lot of vertical space before the first result card is visible. It feels closer to a pinned control dashboard than a streamlined sticky search bar.

Likely cause:
- `frontend/src/pages/SearchPage.tsx:58-89`
- `frontend/src/components/search/FilterPanel.tsx:361-536`

Recommendation:
- Collapse secondary filters more aggressively in the sticky state.
- Keep only the highest-value controls persistently visible.

## Strengths

### Search-first framing is clear

The page still immediately communicates the core GeoLens flow: search, refine, inspect, open.

### Result cards are mostly readable and responsive

The card hierarchy is materially better than older dense versions, especially on mobile where the current text/spec/tag structure remains scannable.

### Mobile filters sheet is a good direction

Moving the dense filter surface into a focused sheet is the right interaction model for small screens, even though a few details still need polish.

## Cleanup / Simplification Opportunities

1. Extract a shared search-controls shell instead of duplicating hero and sticky composition logic in `SearchPage`.
2. Replace the custom spatial side panel with the same dialog/sheet primitives already used elsewhere in search.
3. Unify the save-search display rule across breakpoints.
4. Consolidate type/count derivation so chips, counts, and actual results cannot drift apart.
5. Move quicklook eligibility to a capability check rather than optimistic fetch-and-fallback.
