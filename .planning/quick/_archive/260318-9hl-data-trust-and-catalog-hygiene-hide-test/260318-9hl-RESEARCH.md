# Data Trust & Catalog Hygiene - Research

**Researched:** 2026-03-18
**Domain:** Search filtering, publication state, synthetic content detection
**Confidence:** HIGH

## Summary

The codebase already has a `record_status` field on the `Record` model with values `draft | ready | internal | published` (default: `draft`). This field is already exposed in search API responses (`properties.record_status`) and used for ranking boosts (published records get 2x boost), but it is **not filterable** via the search API -- there is no `record_status` query parameter. The seed script (`scripts/seed-perf-data.py`) tags all synthetic datasets with `perf-seed` and `synthetic` keywords, providing a reliable detection mechanism.

**Primary recommendation:** Add a `record_status` filter param to the search API and store, then use keyword-based detection to auto-badge synthetic content on cards. Default the search to `exclude_synthetic=true` with a toggle.

## 1. Publication State Model

### Record Model (`backend/app/datasets/models.py`)

- **Field:** `Record.record_status` (line 75)
  - Type: `String(20)`, default `"draft"`
  - Check constraint: `IN ('draft', 'ready', 'internal', 'published')` (line 36-38)
- **Published timestamp:** `Record.published_at` (line 117-119) -- `DateTime(timezone=True)`, nullable
- **Migration:** `backend/alembic/versions/2026_03_16_186_01_publication_status.py` migrated legacy `archived`/`deprecated` to `draft`

### Search API Response

- `record_status` IS exposed in the OGC record properties (line 994 of `search/service.py`):
  ```python
  "record_status": record.record_status,
  ```
- The `OGCRecordProperties` schema in `search/schemas.py` does NOT list `record_status` as a field, but it's included in the raw dict output from `dataset_to_ogc_record()`.

### Search Ranking (Already Uses Status)

In `search/service.py` lines 736-739:
```python
published_boost = case(
    (Record.record_status == "published", literal(2.0)),
    else_=literal(1.0),
)
```
Published datasets already get a 2x ranking boost. Draft/ready/internal datasets are not penalized, just not boosted.

### Not Yet Filterable

The search API does NOT accept `record_status` as a filter parameter. `SearchParams` in `search/schemas.py` (lines 9-25) does not include it. The `search_datasets()` function does not filter by it.

### Default Value on Import

New datasets are created with `record_status='draft'` (the column default). The ingest commit endpoint does not set a different status.

## 2. Test/Synthetic Content Detection

### Seed Script: `scripts/seed-perf-data.py`

- All synthetic datasets get keywords: `["perf-seed", "synthetic"]` (line 51: `BASE_TAGS = ["perf-seed", "synthetic"]`)
- Additional category tags added per dataset (lines 54-65)
- Filenames follow pattern: `perf_seed_NNNN.geojson` (line 36: `SEED_FILENAME_PREFIX = "perf_seed_"`)
- Titles follow pattern: `"{source_name} - Variant {index}"` (line 362)
- Descriptions contain: `"Synthetic performance test dataset #..."` (line 363)

### Detection Strategies

| Strategy | Reliability | Notes |
|----------|------------|-------|
| Keyword `perf-seed` | HIGH | All seed data has this tag |
| Keyword `synthetic` | HIGH | All seed data has this tag |
| Filename prefix `perf_seed_` | HIGH | `Dataset.source_filename` field |
| Title contains "Variant" | LOW | Too broad, could match real data |

**Recommendation:** Use keyword `synthetic` as the canonical marker. It's already applied to all test data and is semantically clear.

### No `source` or `origin` Field

There is no field distinguishing "imported by user" vs "generated by script". The `source_format` field tracks file format, not provenance origin.

## 5. Ranking and Default Behavior

### Current Ranking

In `search/service.py` lines 736-753:
- **Relevance sort with text:** FTS rank * published_boost (2x) * freshness_boost (1.5x for last 30 days)
- **Relevance sort without text:** published_boost * freshness_boost, then updated_at descending
- **Other sorts:** date_added, name, last_updated (no boosting)

### Recommended Approach: Exclude by Default, Filter to Include

Rather than just biasing ranking, the cleanest approach is:
1. Add `exclude_synthetic: bool = true` param (or `show_test_data: bool = false`) to search
2. When true, add `.where(NOT EXISTS keyword='synthetic')` to the query
3. Add a toggle in the filter bar: "Show test data" (off by default)
4. This keeps test data accessible to admins while cleaning up the default catalog view

### Alternative: Record Status Filter

Add `record_status` as a proper filter param. Default could be `record_status=published` for anonymous users, but this would hide ALL draft content including newly imported real data. Better to combine both approaches:
- Filter by `record_status` for publication lifecycle
- Separate `exclude_synthetic` toggle for test data
