This is an operational task — no code changes, just running existing infrastructure.

Key facts:
- Services already running: api (healthy), db (healthy), worker (healthy), frontend, titiler
- Demo overlay: `docker-compose.demo.yml` defines `seeder` service
- `.env.demo` exists — contains admin credentials and config
- Seeder Dockerfile is a multi-stage build that downloads ~7GB of data at build time (GEBCO, NE shaded relief, WB/OWID indicators, OSM buildings, UCDP conflict data, UNHCR refugees)
- First build will be slow (10-15 min for GEBCO download alone); subsequent builds use BuildKit cache
- The seeder is idempotent — safe to re-run
- The demo overlay overrides api/worker/frontend to production mode (no bind mounts, no --reload)

IMPORTANT: The demo overlay replaces api, worker, and frontend service configs (removes dev bind mounts, sets production commands). Running `docker compose -f docker-compose.yml -f docker-compose.demo.yml up seeder` will ALSO affect those services if they need recreating. Use `--no-deps` to only start the seeder, or be aware the api/worker/frontend may restart in demo mode.

<task type="auto">
  <name>Task 1: Build and run the seeder service</name>
  <files></files>
  <action>
1. Build the seeder image using the demo overlay. The build downloads ~7GB of source data and processes it — expect 15-30 min on first run:

```bash
docker compose -f docker-compose.yml -f docker-compose.demo.yml build seeder
```

2. Once built, run the seeder. Use `--no-deps` to avoid recreating api/worker/frontend in demo mode (they are already running and healthy in dev mode):

```bash
docker compose -f docker-compose.yml -f docker-compose.demo.yml run --rm seeder
```

Note: If the above fails because the seeder needs the api/worker to be started via the demo overlay, then bring up just the seeder with deps:

```bash
docker compose -f docker-compose.yml -f docker-compose.demo.yml up seeder
```

3. Monitor output for:
   - "Authenticating..." and successful login
   - API key creation
   - Per-theme dataset creation/upload messages
   - Map fixture application
   - Any errors or retries

4. The seeder should exit 0 on success. If it exits non-zero, check logs:

```bash
docker compose -f docker-compose.yml -f docker-compose.demo.yml logs seeder
```
