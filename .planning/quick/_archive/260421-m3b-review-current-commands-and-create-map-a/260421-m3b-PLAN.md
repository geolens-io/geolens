Create `.claude/commands/map-audit.md` following the exact structural conventions of `builder-audit.md` and `design-audit.md`. The command audits a single saved map by its UUID.

**Header section:**
- Title: `/map-audit — Saved Map Quality, Style & Access Audit`
- Description: Audit a specific saved map by ID — covering style quality, data integrity, performance concerns, design quality, MapLibre Style Spec compliance, and sharing/access configuration.
- Usage: `/map-audit <id>` (full audit) or `/map-audit <id> <scope>` where scope is `style`, `data`, `perf`, `design`, `spec`, or `access`
- `Arguments: $ARGUMENTS`
- Scope dispatch table mapping each keyword to its subagent (same pattern as builder-audit)

**INTAKE section (Serial):**

Step 1 — Fetch the map via live API:
```bash
MAP_ID=$(echo "$ARGUMENTS" | awk '{print $1}')
curl -s http://localhost:8000/api/maps/${MAP_ID}/ | python3 -m json.tool
```
- If curl fails or returns non-200, print error and abort with message: "Server not running or map not found. Start the dev server (`docker compose up -d`) and verify the map ID."
- Parse the JSON response and extract: map metadata (name, description, visibility, basemap_style, viewport, widgets), layers array, share token status.

Step 2 — For each layer in the response, extract and catalog:
- `layer_type` (vector_geolens, raster_geolens, geojson)
- `dataset_geometry_type` (Point, LineString, Polygon, MultiPoint, etc.)
- `paint` properties (the raw MapLibre paint object)
- `layout` properties
- `filter` expression (if any)
- `label_config` (if any)
- `style_config` (if any — data-driven style settings)
- `opacity`, `visible`, `show_in_legend`

Step 3 — Check share/access state:
```bash
curl -s http://localhost:8000/api/maps/${MAP_ID}/share/
curl -s http://localhost:8000/api/maps/${MAP_ID}/visibility-check/
```

Step 4 — Read relevant source files for cross-reference:
- `frontend/src/components/builder/layer-adapters/registry.ts` (adapter type mapping)
- `frontend/src/components/builder/layer-adapters/shared.ts` (resolveAdapterType)
- `frontend/src/lib/layer-capabilities.ts` (capability gates)
- `frontend/src/lib/basemap-utils.ts` (basemap validation)

**SUBAGENT DISPATCH section (Parallel — 6 subagents):**

Each subagent must: (1) analyze the fetched map JSON, (2) cross-reference against source code where needed, (3) produce numbered findings with severity tags [CRITICAL]/[HIGH]/[MEDIUM]/[LOW], (4) include evidence and concrete fix for every finding, (5) cap at 20 findings max.

**Subagent 1: Style Quality**
Goal: Verify every layer's paint/layout properties are well-formed, appropriate for the geometry type, and produce a reasonable visual result.
Process:
- For each layer, check paint properties are valid for the layer's adapter type (circle-* only on points, fill-* only on polygons, line-* on lines, raster-* on raster, heatmap-* on heatmap)
- Check opacity values are in 0.0-1.0 range
- Check color values are valid CSS color strings (hex, rgb, rgba, hsl, hsla, named) — NOT OKLCH
- Check circle-radius > 0, line-width > 0
- Check line-dasharray values are positive numbers
- If style_config exists (data-driven), verify match/interpolate/step expressions have correct arity
- Check that default/fallback colors are defined in data-driven expressions
- Flag layers with empty paint objects (no styling applied — will be invisible)
- Flag layers where paint properties don't match the geometry type (e.g., fill-color on a Point layer)

**Subagent 2: Data Integrity**
Goal: Verify layers reference valid datasets, have reasonable configurations, and data relationships are intact.
Process:
- Check each layer's dataset_id exists and has a name (dataset_name not null/empty)
- Check dataset_geometry_type is set for vector layers (null geometry = potential problem)
- Check sort_order values are unique and sequential (no gaps or duplicates that could cause z-order ambiguity)
- Check for duplicate dataset references — same dataset_id appearing multiple times (acceptable but flag for awareness)
- Check filter expressions reference properties that exist in dataset_column_info
- Check label_config references a text-field property that exists in dataset_column_info
- If dataset_extent_bbox is null, flag (map may not be able to zoom-to-fit)
- Check dataset_feature_count — if 0 or null, flag (layer will render nothing)
- Check layer_type matches dataset_record_type expectations (raster dataset should have raster_geolens layer_type)

**Subagent 3: Performance**
Goal: Identify configuration choices that could cause slow rendering or excessive resource usage.
Process:
- Count total layers — flag if > 10 (MapLibre performance degrades with many layers)
- Check for layers with very large feature_count (> 100k) without appropriate style simplification (e.g., should use heatmap or clustering at low zoom)
- Check for layers at sort_order extremes that may be hidden beneath others
- Check if all layers are visible — many hidden layers still consume tile bandwidth
- Check viewport — zoom level 0-2 with large feature counts = expensive
- Check for data-driven styles with many categories (> 20 match arms = complex expression)
- Check heatmap layers — verify radius is reasonable (< 50px recommended)
- Flag layers with both fill and line styling that could be simplified

**Subagent 4: Design Quality**
Goal: Assess visual/cartographic quality of the map configuration.
Process:
- Check color palette diversity — are all layers the same default color? (lazy configuration)
- Check opacity usage — multiple layers at opacity 1.0 stacked on polygons = occlusion
- Check label readability — if label_config exists, verify text-halo-color/text-halo-width are set (labels without halos are hard to read on varied backgrounds)
- Check basemap choice — is basemap_style a known valid style ID? Cross-reference with basemap-utils.ts
- Check viewport defaults — center_lng/center_lat/zoom set or null? (null = map opens at default world view, not data extent)
- Check layer naming — display_name set or null? (null = shows dataset name, which may be a table name like "my_upload_2024")
- Check show_in_legend consistency — all layers hidden from legend defeats the purpose
- Check widget configuration — if widgets array exists, are the IDs valid?

**Subagent 5: MapLibre Spec Compliance**
Goal: Validate every paint, layout, and filter property against the MapLibre Style Spec.
Process:
- For each layer, enumerate every key in `paint` and `layout` and verify it's a valid MapLibre property name for the corresponding layer type
- Check expression syntax: `["match", input, label1, output1, ..., fallback]` has even pairs + fallback
- Check `["interpolate", interpolation, input, stop1, output1, ...]` has `["linear"]` or `["exponential", base]` as interpolation
- Check `["step", input, default, stop1, output1, ...]` syntax
- Check filter expressions use expression form `["==", ["get", "prop"], value]` not deprecated `["==", "prop", value]`
- Check enum properties use valid values: line-cap (butt/round/square), line-join (bevel/round/miter), text-anchor (center/left/right/top/bottom/etc), symbol-placement (point/line/line-center)
- Verify numeric ranges: opacity 0-1, circle-radius >= 0, line-width >= 0, text-size > 0
- Check for unknown/misspelled property names (e.g., "fill-colour" instead of "fill-color")

**Subagent 6: Sharing & Access**
Goal: Verify the map's sharing configuration is correct and secure.
Process:
- Check visibility setting (private/internal/public) — is it intentional?
- If public: run visibility-check endpoint to see if any non-public datasets are referenced (data leak risk)
- If share token exists: check expiry (expired tokens = broken share links)
- If share token exists: verify the shared view URL works (curl the /shared/{token} endpoint)
- Check if thumbnail_url is set — maps without thumbnails look incomplete in the list view
- Check created_by — orphaned maps (null created_by) may indicate a deleted user
- If forked_from is set, verify the source map still exists
- Check that non-public layers in a public map are flagged by visibility-check

**Playwright MCP Visual Verification section (after subagents):**

Prerequisite: dev server must be running at http://localhost:8080. Check with browser_navigate. If not available, skip visual verification and note which checks were skipped.

Steps:
1. Log in with admin credentials (admin/admin) at http://localhost:8080
2. Navigate to the map viewer: `http://localhost:8080/maps/{MAP_ID}/view`
3. browser_take_screenshot — verify map renders, layers visible, basemap loads
4. browser_snapshot — check for console errors, missing tile requests, broken layers
5. Toggle dark mode — browser_take_screenshot — verify no hardcoded colors bleed through, controls readable
6. browser_resize to 390px width — browser_take_screenshot — verify responsive behavior
7. browser_resize back to 1440px
8. If map has a share token, navigate to the shared URL and verify it renders without auth
9. Check the legend — verify layer names and visibility match the map configuration

**SYNTHESIS section:**

Scoring table with 6 dimensions:
| Dimension | What it measures | Subagent |
|-----------|-----------------|----------|
| Style Quality | Paint/layout correctness, color validity, data-driven expressions | 1 |
| Data Integrity | Dataset references, column existence, type consistency | 2 |
| Performance | Layer count, feature density, expression complexity | 3 |
| Design Quality | Color diversity, label readability, viewport, naming | 4 |
| Spec Compliance | MapLibre Style Spec conformance for all properties | 5 |
| Sharing & Access | Visibility, tokens, data exposure, thumbnails | 6 |

Grade A-F with same rubric as builder-audit. Overall map health = weighted average (Style Quality and Spec Compliance weighted 2x).

Action items table with fields: ID (M-001, M-002...), Priority (P0/P1/P2), Severity, Finding, Layer (which layer), Fix, Dimension, Effort.

**WHAT NOT TO FLAG section:**

Include these exemptions (adapted from builder-audit where applicable):
- Hardcoded hex colors in paint properties — MapLibre cannot use CSS variables
- Default paint objects from the builder (standard circle-color, fill-color, line-color defaults)
- Empty layout objects — layout properties are optional in MapLibre
- filter: null — no filter is a valid state
- label_config: null — not all layers need labels
- style_config: null — simple (non-data-driven) styling is fine
- opacity: 1.0 for single layers — only flag when multiple polygon layers stack
- show_in_legend: true for all layers — this is the default
- Raster layers with empty paint (opacity-only styling is normal)
- Missing dataset_sample_values — this is optional enrichment data
- sort_order starting from 0 — this is the default
