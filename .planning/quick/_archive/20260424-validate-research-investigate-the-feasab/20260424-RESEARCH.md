# Feasibility: One Dockerfile For GeoLens Frontend And Backend

Date: 2026-05-04

## Decision

Creating one Dockerfile is feasible, but the useful target is **one repository-root multi-stage Dockerfile with multiple build targets**, not necessarily one final container that owns the whole application runtime.

The Open WebUI example is relevant as a build pattern: it builds frontend assets in a Node stage, starts a Python runtime stage, then copies built frontend files into the final backend image. Docker explicitly supports this with multiple `FROM` stages and `COPY --from`, and supports `--target` builds for selecting a named stage.

For GeoLens, there are three different interpretations:

| Option | Feasible? | Recommendation | Why |
|---|---:|---|---|
| One root Dockerfile with named targets for `api`, `worker`, `frontend`, and optionally `app` | Yes | Recommended first implementation if reducing Dockerfile sprawl is the goal | Preserves current service boundaries while centralizing build definitions. Compose and publish can build `--target` images. |
| One final "app" image that serves the React build from FastAPI | Yes, with backend code changes | Viable later, not a quick Dockerfile-only change | Requires mounting static files in FastAPI, SPA fallback behavior, runtime `env-config.js` generation outside nginx, and equivalent cache/security headers. |
| One final container running nginx and FastAPI together | Yes, but poor fit | Avoid unless packaging for a single-process-hosting constraint | Docker recommends one service per container where possible. This path needs a supervisor/wrapper, signal handling, health semantics, logs, and root/non-root permission work. |

## Current GeoLens Container Shape

GeoLens currently has cleanly separated production image responsibilities:

- `backend/Dockerfile` builds the API/worker Python runtime on `python:3.14.3-slim`, installs `uv`, GDAL/runtime XMLSec dependencies, syncs locked Python deps with BuildKit cache mounts, exposes a healthcheck on `:8000`, and drops to `appuser`.
- `frontend/Dockerfile` already uses a multi-stage build: Node builds `/app/dist`, and `nginxinc/nginx-unprivileged` serves the output on `:8080`.
- `frontend/docker-entrypoint.sh` generates runtime `env-config.js` from `API_BASE_URL` and `TILE_BASE_URL` before starting nginx.
- `frontend/nginx.conf` is more than a static file server. It handles SPA fallback, `/api/` proxying, `/raster-tiles/` auth requests, TiTiler proxying, auth/tile caches, gzip, security headers, upload size, and long API proxy timeouts.
- `docker-compose.yml` keeps `api`, `worker`, `frontend`, `db`, and `titiler` as separate services. The worker uses the backend image but a different entrypoint/command. TiTiler remains an external raster tile service.
- `docker-compose.demo.yml` intentionally switches frontend from Vite dev to production nginx, clears dev bind mounts, and keeps API/worker as separate services.
- `.github/workflows/publish.yml` publishes two separate GHCR images today: `geolens-api` from `./backend` and `geolens-frontend` from `./frontend`.

Compose validation passed for:

```bash
docker compose config --quiet
docker compose -f docker-compose.yml -f docker-compose.demo.yml config --quiet
docker compose -f docker-compose.yml -f docker-compose.enterprise.yml config --quiet
```

## External References

- Docker multi-stage builds: multiple `FROM` instructions can use different bases; `COPY --from` moves only selected artifacts into later stages; `docker build --target <stage>` can build a named stage. Source: https://docs.docker.com/build/building/multi-stage/
- Docker multi-process containers: Docker documents that the main process is `ENTRYPOINT`/`CMD`, recommends separating areas of concern by using one service per container, and says multiple services in one container require wrapper scripts, job control, or a process manager. Source: https://docs.docker.com/engine/containers/multi-service_container/
- Open WebUI Dockerfile: Node frontend build stage followed by Python backend/runtime stage, with built frontend assets copied into the final image. Source: https://github.com/open-webui/open-webui/blob/main/Dockerfile

## What A Good One-Dockerfile Design Would Look Like

The most practical design is a repository-root `Dockerfile` with named targets:

```dockerfile
# syntax=docker/dockerfile:1

FROM node:25.9.0-alpine AS frontend-build
WORKDIR /app/frontend
COPY frontend/package.json frontend/package-lock.json ./
RUN npm ci
COPY frontend/ ./
RUN npm run build:app

FROM python:3.14.3-slim AS backend-base
COPY --from=ghcr.io/astral-sh/uv:0.11.3 /uv /uvx /bin/
WORKDIR /app
# install backend runtime deps, copy backend/pyproject.toml + backend/uv.lock,
# uv sync, copy backend source, create appuser, healthcheck, entrypoint

FROM backend-base AS api
ENTRYPOINT ["/app/scripts/api-entrypoint.sh"]
CMD ["sh", "-c", "uv run --no-dev uvicorn app.api.main:app --host 0.0.0.0 --port 8000"]

FROM backend-base AS worker
ENTRYPOINT ["/app/scripts/worker-entrypoint.sh"]
CMD ["sh", "-c", "uv run --no-dev python -m app.worker"]

FROM nginxinc/nginx-unprivileged:1.29.8-alpine AS frontend
COPY --from=frontend-build --chown=nginx:nginx /app/frontend/dist /usr/share/nginx/html
COPY frontend/public/env-config.template.js /usr/share/nginx/html/env-config.template.js
COPY frontend/nginx.conf /etc/nginx/conf.d/default.conf
COPY frontend/docker-entrypoint.sh /docker-entrypoint.sh
```

Compose could then build from the same file with different targets:

```yaml
services:
  api:
    build:
      context: .
      dockerfile: Dockerfile
      target: api

  worker:
    build:
      context: .
      dockerfile: Dockerfile
      target: worker

  frontend:
    build:
      context: .
      dockerfile: Dockerfile
      target: frontend
```

The publish workflow could keep publishing separate images while changing only the build context and target:

```yaml
matrix:
  include:
    - image: geolens-api
      target: api
    - image: geolens-frontend
      target: frontend
```

This gives "one Dockerfile" without forcing one container.

## Required Work Before Implementing

1. Add a root `.dockerignore`.
   The current `backend/.dockerignore` and `frontend/.dockerignore` are scoped to their existing build contexts. A root build context without a root `.dockerignore` would send `.env`, `.git`, reports, local venvs, dependency folders, and planning artifacts unless explicitly excluded.

2. Move build paths from context-relative to repo-relative.
   `backend/Dockerfile` currently assumes `uv.lock` and `pyproject.toml` are at the build-context root. A root Dockerfile needs `backend/uv.lock`, `backend/pyproject.toml`, and `COPY backend/ /app`.

3. Preserve frontend nginx behavior if the frontend remains an nginx target.
   The one-Dockerfile/multiple-target design can preserve the current `frontend/nginx.conf` unchanged except for path-qualified `COPY` sources.

4. Decide whether publish artifacts remain separate.
   Separate `geolens-api` and `geolens-frontend` images are still the cleanest publish contract. A combined `geolens-app` image would be an additional artifact, not a direct replacement, because API/worker/frontend have different runtime responsibilities.

5. Keep worker separate.
   Even if API and frontend assets are combined, the worker is a distinct long-running process with separate healthcheck, shutdown timeout, and scaling behavior. Folding it into an app container would reduce operational clarity.

6. Do not fold TiTiler into the app image.
   `frontend/nginx.conf` currently proxies raster tile rendering to `titiler:8000`. TiTiler is an external image with its own GDAL tuning and should remain a separate service.

## If We Want One Final App Image Later

A Python-only app image can serve built frontend assets, but it requires application changes:

- Add FastAPI `StaticFiles` serving for the built `dist` directory.
- Add SPA fallback to `index.html` for non-API routes.
- Generate `env-config.js` at startup or serve it dynamically from an endpoint.
- Reproduce nginx headers and asset cache policy.
- Decide how `/raster-tiles/` proxies to TiTiler; today nginx handles auth subrequests, cache keys, anonymous rate limiting, and empty-tile handling.
- Revisit `root_path="/api"` in `backend/app/api/main.py`; serving both SPA and API from one ASGI app may need router/path adjustments.

This is feasible, but it is not equivalent to the Open WebUI pattern unless GeoLens moves proxy/static responsibilities into the backend.

## Risk Assessment

| Risk | Applies To | Severity | Mitigation |
|---|---|---:|---|
| Root build context leaks local files into Docker build context | One root Dockerfile | High | Add root `.dockerignore` before changing compose/publish. |
| Losing nginx raster auth/cache/proxy behavior | One final Python app image | High | Keep nginx frontend target or port behavior into backend with tests. |
| Worse operational scaling | One container runtime | Medium | Keep API, worker, frontend, TiTiler as separate services even if build definitions are centralized. |
| More complex publish workflow | One root Dockerfile | Low | Use `target` in the existing matrix. |
| Larger image if Node artifacts or build tools leak into runtime | Bad multi-stage implementation | Medium | Only `COPY --from=frontend-build /dist`; keep Node out of final targets. |

## Recommendation

Implement, if desired, in two phases:

1. **Low-risk phase:** introduce a root multi-stage Dockerfile with `api`, `worker`, and `frontend` targets, plus a root `.dockerignore`. Update compose and publish workflow to build those targets while preserving separate runtime services and separate published images.

2. **Optional later phase:** add a new `app` target that copies frontend `dist` into the backend runtime. Treat this as a new deployment mode with dedicated tests for static serving, SPA routes, `/api`, `/raster-tiles`, runtime config, headers, and healthchecks.

Do not start by replacing the existing two production images with a single all-in-one runtime container.
