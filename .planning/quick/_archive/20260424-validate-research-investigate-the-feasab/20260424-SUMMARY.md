# Quick Task 20260424 Summary

Task: `--validate --research investigate the feasability of creating one dockerfile for frontend and backend see https://github.com/open-webui/open-webui/blob/main/Dockerfile for example`

## Outcome

GeoLens can move to one Dockerfile, but the low-risk version is a repository-root multi-stage Dockerfile with named targets for separate `api`, `worker`, and `frontend` images. That preserves the current runtime topology while centralizing build definitions.

I do not recommend starting with one all-in-one runtime container. The current frontend nginx image is not just static hosting: it owns `/api/` proxying, raster auth subrequests, TiTiler proxying, tile/auth caches, SPA fallback, runtime frontend config generation, gzip, security headers, and long API upload/proxy timeouts. Replacing that with a single Python runtime would require real backend/proxy implementation work.
