# Quick Task 260515-k49: Fix img Bearer-JWT mismatch — Context

**Gathered:** 2026-05-15
**Status:** Ready for planning

<domain>
## Task Boundary

Fix the structural bug surfaced during the live verify of SP-07 (quick task 260515-i45):

Browser `<img>` tags do NOT carry the `Authorization: Bearer <jwt>` header — only cookies. The OGC search response is fetched by `apiFetch()` with the Bearer JWT in TanStack Query, so authenticated users see all records they're authorized for (including private "sample" datasets in the dev environment). `SearchResultCard.tsx:155-156` then constructs `${API_BASE}/datasets/${id}/quicklook?size=256` and hands it to `<img src>`. The image GET goes to the backend **anonymously**, hits `check_dataset_access_or_anonymous()` at `backend/app/modules/catalog/authorization.py:75-95`, and gets a 404 (existence-masking, intentional security behavior).

**Proof point (captured during 260515-i45 live verify via Playwright MCP):**
- Same URL, same browser session:
  - `fetch(url, { headers: { Authorization: \`Bearer ${token}\` } })` → **200**
  - `fetch(url)` (no header) → **404**
- 3 console 404s observed on the Search page for the 3 `visibility=private` datasets (UUIDs 777ddb26, 9773fbcc, bbb61bdb).

**Out of scope for this task:**
- Backend access control logic itself (intentional and correct as-is)
- The SP-07 sweeper (260515-i45) — independent defensive tool, keep as-is
- The raster/vrt dispatch fix (260515-ilt) — already landed, unaffected
- SP-12 (representative-fraction "1:N" pane) — separate v1009.1 followup

</domain>

<decisions>
## Implementation Decisions

### Auth transport mechanism

**Locked:** `fetch() + URL.createObjectURL(blob)` pattern in the frontend.

- The `<img src>` becomes a blob URL backed by an authenticated `fetch()`.
- Reuse the existing `apiFetch()` helper from `frontend/src/api/client.ts` — it already attaches the Bearer JWT from the zustand `useAuthStore`.
- Add explicit blob-URL lifecycle management: `URL.revokeObjectURL(blobUrl)` on component unmount and on URL change to avoid memory leaks.
- Recommended shape: a `useQuicklook(datasetId, size)` hook returning `{ url: string | null, status: "loading" | "ready" | "missing" | "error" }`. Components consume the hook and render conditionally.
- Rationale: zero auth-surface change, no backend touches, no cookie/CORS/SameSite negotiation, and the blob URL approach is well-trodden React territory.

### Scope of UI surfaces

**Locked:** Audit and fix ALL `has_quicklook` consumers.

- Required call sites (known from the bug evidence + the v1009.1 work):
  - `frontend/src/components/search/SearchResultCard.tsx:155-164` — Search page cards
  - `frontend/src/components/builder/DatasetSearchPanel.tsx:120-130` — Builder dataset search panel
- Plus any other `<img src=".../quicklook">` references discovered during planning. Use `grep -rn "quicklook" frontend/src/` to enumerate before finalizing scope.
- Build the `useQuicklook` hook once; convert each call site. Slightly bigger patch than Search-only but eliminates the inconsistency risk.
- Rationale: the bug is structural — if it exists on Search, it exists everywhere `<img>` references the endpoint. Better to fix the pattern, not the symptom.

### `quicklook-cache.ts` strategy

**Locked:** Adapt for the new transport.

- The v1009.1 cache exists because `<img>` 404s were unrecoverable; once cached, no retry. With `fetch()`, we now have access to the response status, so:
  - Cache **genuine** misses (status 404 from a dataset that legitimately has no quicklook, e.g. ingest with `quicklook_failed` log path) — these are stable signals.
  - Do NOT cache transient auth/network errors (4xx that isn't 404 from access logic — though with the new auth path, the access 404 should disappear entirely for valid sessions).
  - Implementation note: with blob URLs, "cache" becomes "remember `missing` status for this dataset id and skip the fetch on next render"; the in-memory cache is per-tab as today.
- The cache module stays; its semantics shift from "negative-cache image-tag 404s" to "negative-cache fetch() 404s." Likely a few lines of edits + updated comment block.

### Backend changes

**Locked:** Frontend-only. Zero backend edits.

- `/api/datasets/{id}/quicklook` endpoint stays exactly as-is. It already accepts `Authorization: Bearer <jwt>` (verified via Playwright `fetch()` test).
- The `has_quicklook` OGC predicate stays as-is (260515-i45 + 260515-ilt locked the truthful version).
- `check_dataset_access_or_anonymous()` stays as-is — the 404-mask is intentional and shouldn't change.
- SP-07 reconcile sweeper (260515-i45) stays as-is — defensive tool.

### Claude's Discretion

- The exact API shape of the `useQuicklook` hook — return type, status enum, AbortController behavior on unmount, integration with TanStack Query vs plain useEffect — is up to the planner/executor as long as:
  - Bearer JWT is sent on every quicklook fetch
  - blob URLs are revoked on unmount
  - Card placeholder renders gracefully on `missing` and `error`
  - No regression to existing 200/404 behavior for already-working public datasets
- Whether to colocate the hook with `quicklook-cache.ts` or split into a new `use-quicklook.ts` file — planner decision.
- Whether to add a Playwright MCP UAT step to the plan — recommended but not required (we can self-verify with `browser_console_messages` after the fix lands).

</decisions>

<specifics>
## Specific Ideas

- **Existing apiFetch pattern:** `frontend/src/api/client.ts` exposes `apiFetch()` which already pulls the JWT from `useAuthStore` (zustand with `persist`, key `geolens-auth`). For non-React contexts (the hook will be inside React, but the helper might be reused) it's `useAuthStore.getState().token`. Per project memory: "Auth store: useAuthStore (zustand with persist) — `getState().token` for non-React contexts".
- **Existing quicklook-cache module:** `frontend/src/lib/quicklook-cache.ts` — read its current shape (likely a Map<datasetId, "missing">). Bring it along for the new transport.
- **Existing tests:** `frontend/src/components/builder/__tests__/DatasetSearchPanel.test.tsx`, `frontend/src/pages/__tests__/SearchPage.test.tsx`, `frontend/src/components/builder/__tests__/DatasetSearchPanel.dragdrop.test.tsx` — they construct fake records with `has_quicklook: false` (which sidesteps the image fetch). After the refactor, add at least one test asserting the hook fetches with the Bearer header attached and renders a blob URL on success.
- **Playwright MCP verification target:** After the fix, the same Search page reload with localStorage `geolens-auth` populated should yield **zero** `/quicklook` 404s in the console (down from the current 3).
