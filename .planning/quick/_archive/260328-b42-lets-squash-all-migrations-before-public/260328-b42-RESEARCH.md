## Summary

The project has 12 Alembic migrations (0001-0012) that need squashing into one initial migration before public release. Migration 0001 already executes a raw `initial_schema.sql` (2500-line pg_dump) containing the full schema from an earlier 56-migration squash. Migrations 0002-0012 are incremental DDL changes (new columns, indexes, CHECK constraints, type narrowing, column rename) plus two data migrations (0008, 0012) that are irrelevant for fresh databases.

The approach should NOT use `alembic revision --autogenerate` because the schema contains substantial infrastructure that autogenerate cannot produce: 24 custom SQL functions (5 application-specific immutable functions, 18 procrastinate job queue functions), 3 custom ENUM types, 8 triggers, a GENERATED ALWAYS stored column with custom function references, GIN/GiST indexes, and pgvector columns. Instead, the cleanest approach is to produce a new `initial_schema.sql` via `pg_dump` of the fully-migrated database (same technique 0001 already uses), then write a thin Python migration wrapper identical to the current 0001.

**Primary recommendation:** Dump the current fully-migrated schema with `pg_dump --schema-only --schema=catalog`, write a new 0001_initial.py wrapper, delete 0002-0012 and the old SQL file, verify round-trip parity.

## Key Findings

### Why NOT autogenerate (HIGH confidence)

Alembic's `--autogenerate` reflects SQLAlchemy model metadata into DDL operations. It will miss everything that lives outside the ORM models:

| Category | Count | Example | Autogenerate? |
|----------|-------|---------|---------------|
| Procrastinate ENUM types | 3 | `procrastinate_job_status`, `procrastinate_job_event_type`, composite type `procrastinate_job_to_defer_v1` | No |
| Procrastinate functions | 18 | `procrastinate_defer_jobs_v1()`, `procrastinate_fetch_job_v2()` | No |
| Procrastinate tables | 4 | `procrastinate_jobs`, `procrastinate_events`, `procrastinate_periodic_defers`, `procrastinate_workers` | No |
| Procrastinate triggers | 7 | `procrastinate_jobs_notify_queue_job_inserted_v1` | No |
| Application functions | 6 | `immutable_array_camel_to_spaced()`, `immutable_camel_to_spaced()`, `set_updated_at()` | No |
| Application trigger | 1 | `trg_records_updated_at` on `catalog.records` | No |
| GIN/GiST indexes | 4 | `idx_records_search_vector` (GIN), `idx_records_spatial_extent` (GiST), FTS indexes | Partial -- some GIN/GiST may not render correctly |
| Partial indexes | 1 | `idx_records_source_organization` with `WHERE source_organization IS NOT NULL` | Maybe -- depends on model definition |

The CONTEXT.md suggests autogenerate as Claude's discretion area. Research shows `pg_dump --schema-only` is the correct approach -- the existing 0001 already uses this exact technique, and it captured everything correctly. Autogenerate would require manually supplementing 30+ objects as raw SQL, defeating the purpose.

### Recommended Approach: pg_dump (HIGH confidence)

The current 0001 migration already demonstrates the pattern:

1. `pg_dump --schema-only --schema=catalog` produces the SQL
2. A thin Python migration reads and executes the SQL via raw asyncpg connection
3. The asyncpg bypass is needed because the SQL contains multi-statement blocks that SQLAlchemy's prepared statement wrapper can't handle

This is the proven, battle-tested pattern already in the codebase.

### Procrastinate Tables: Not Managed by SQLAlchemy (HIGH confidence)

Procrastinate manages its own schema (4 tables, 3 types, 18 functions, 7 triggers) within the `catalog` schema (configured via `settings.procrastinate_schema = "catalog"`). These are NOT represented in SQLAlchemy models at all -- `env.py` does not import any procrastinate models. The current `initial_schema.sql` captured them because it was a full `pg_dump`, and procrastinate's `open_async()` call in `app/main.py` is idempotent (applies schema if missing, no-ops if present).

**Important:** Procrastinate can apply its own schema on startup. However, including it in the migration SQL is the correct approach because:
- The `migrate` docker-compose service runs `alembic upgrade head` BEFORE the API starts
- Having everything in one place avoids ordering dependencies

## Architecture Patterns

### Migration File Pattern (existing, proven)

```
backend/alembic/versions/
  0001_initial_schema.py    # Thin wrapper: reads + executes initial_schema.sql
  initial_schema.sql        # Full pg_dump --schema-only --schema=catalog output
```

The Python wrapper handles the asyncpg multi-statement execution:
```python
def upgrade() -> None:
    sql = Path(__file__).with_name("initial_schema.sql").read_text()
    bind = op.get_bind()
    asyncpg_conn = bind.connection.dbapi_connection._connection
    await_only(asyncpg_conn.execute(sql))
```

## Don't Hand-Roll

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Schema dump | Manual SQL assembly from 12 migrations | `pg_dump --schema-only --schema=catalog` | Guaranteed accurate, handles ordering, includes all objects |
| Schema comparison | Manual diff | `pg_dump` both databases, `diff` the output | Catches every difference including indexes, constraints, triggers |
| Multi-statement SQL execution | SQLAlchemy `text()` | Raw asyncpg `execute()` via `await_only` bridge | SQLAlchemy wraps with prepared statements (single-statement only) |

## Common Pitfalls

### Pitfall 1: pg_dump Includes Owner/Privileges
**What goes wrong:** `pg_dump` includes `OWNER TO` and `GRANT` statements that may reference roles that don't exist on target.
**How to avoid:** Use `pg_dump --no-owner --no-privileges` flags. The existing `initial_schema.sql` already uses `-- Owner: -` style (no-owner flag was used).

### Pitfall 2: alembic_version Table Left Stale
**What goes wrong:** After deleting old migration files, the `alembic_version` table in dev databases still points to `0012_map_thumbnail_to_storage`. Running `alembic upgrade head` fails because it can't find that revision.
**How to avoid:** For dev databases: `alembic stamp head` after the squash. But per CONTEXT.md, no existing databases need upgrade paths -- clean slate only.

### Pitfall 3: SET Statements at Top of pg_dump
**What goes wrong:** The `pg_dump` output starts with `SET statement_timeout = 0;` etc. These are fine for multi-statement asyncpg execution but would fail if passed through SQLAlchemy's normal `execute()`.
**How to avoid:** Keep using the asyncpg bypass pattern from current 0001.

### Pitfall 4: Missing CREATE SCHEMA IF NOT EXISTS
**What goes wrong:** The SQL needs `CREATE SCHEMA IF NOT EXISTS catalog;` at the top. pg_dump includes this, but verify it's present and uses `IF NOT EXISTS`.
**How to avoid:** Check the dump output. Current file already has it correctly.

### Pitfall 5: Sequences/Identity Columns
**What goes wrong:** pg_dump may include `ALTER SEQUENCE ... OWNED BY` statements that reference specific sequence names. These are fine for fresh installs.
**How to avoid:** Non-issue for fresh installs, but verify the dump doesn't include `SELECT setval()` calls (which set sequence positions from data).

## Verification Strategy

### Schema Parity Check

1. Start with current fully-migrated DB (run all 12 migrations)
2. `pg_dump --schema-only --schema=catalog --no-owner --no-privileges > current_schema.sql`
3. Create fresh DB, apply new squashed migration
4. `pg_dump --schema-only --schema=catalog --no-owner --no-privileges > squashed_schema.sql`
5. `diff current_schema.sql squashed_schema.sql` -- should be empty (or only cosmetic differences like statement ordering)

### Test Suite Verification

The existing test suite (`conftest.py`) creates a fresh test DB and runs `alembic upgrade head` before every session. After the squash, running the full test suite validates that:
- The single migration creates a valid schema
- All application code works against it
- All ORM operations succeed
