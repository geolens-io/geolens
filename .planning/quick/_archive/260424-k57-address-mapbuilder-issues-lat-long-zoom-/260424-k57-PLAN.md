<task type="auto">
  <name>Task 3: Fix measurement widget click conflict with BuilderMap</name>
  <files>
    frontend/src/components/builder/BuilderMap.tsx
  </files>
  <action>
The measurement widget is broken because BuilderMap's click handler (line 240) and mousemove handler (line 272) both fire on every map interaction, interfering with the MeasurementWidget's click handler. When the measurement widget is active, BuilderMap's click handler queries features and opens/closes popups, and its mousemove handler overrides the cursor away from crosshair.

**Fix in BuilderMap.tsx:**

1. Import `useWidgetStore` at the top:
   ```typescript
   import { useWidgetStore } from '@/components/map-widgets/map-widget-store';
   ```

2. Inside the component, create a ref that tracks measurement-active state. This avoids re-registering the map event handlers on every widget toggle:
   ```typescript
   const measureActiveRef = useRef(false);
   ```

3. Subscribe to the widget store to keep the ref current. Add after the existing refs/state:
   ```typescript
   useEffect(() => {
     return useWidgetStore.subscribe((state) => {
       measureActiveRef.current = state.activeWidgets.has('measurement');
     });
   }, []);
   ```

4. In the click handler `handleClick` (inside the useEffect at line 236), add an early return at the very top:
   ```typescript
   if (measureActiveRef.current) return;
   ```

5. In the mousemove handler `handleMouseMove` (line 272), add an early return inside the rAF callback, right after the `if (!map.isStyleLoaded()) return;` check:
   ```typescript
   if (measureActiveRef.current) return;
   ```

This ensures that when the measurement widget is active:
- Map clicks go only to the MeasurementWidget's handler (not consumed by feature popup logic)
- The cursor stays as `crosshair` (set by MeasurementWidget) instead of being overridden to `''` or `'pointer'`
- When measurement is deactivated, BuilderMap's handlers resume normal behavior immediately (no re-registration needed)
