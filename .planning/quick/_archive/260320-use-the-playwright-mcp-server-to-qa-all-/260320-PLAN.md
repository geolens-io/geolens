## Objective

Use the Playwright MCP browser tools to QA the 4 record-detail experiences currently exposed in the product (vector, raster, VRT dataset detail plus collection detail), and produce a comprehensive, actionable UI/UX findings list with evidence and easy-win suggestions.

## Scope

- Pages: `GET /datasets/:id` for `record_type` = `vector_dataset`, `raster_dataset`, `vrt_dataset`, plus `GET /collections/:id` for collection detail
- Viewports: Desktop (1280x800) and Mobile (375x812)
- Coverage: Overview/header, map/top summary, each visible tab, empty states, and obvious a11y regressions (labels, focus, keyboard)
- Non-goals: Implementing fixes in this quick task (output is findings + evidence)

<action>
Preparation:
1. Load `qa-targets.json`. For any `null` target, add a "Blocker: missing target record" entry in FINDINGS.md and skip QA for that type (unless the user explicitly wants us to generate seed data).

For each available record type target:
1. Desktop pass:
   - `mcp__playwright__browser_resize` to 1280x800.
   - Navigate directly to `http://localhost:8080<path>` (append `#overview` only for dataset detail pages that use tabs).
   - Validate the top-of-page hierarchy and core affordances:
     - Exactly one H1 title for the active detail surface.
     - Header action budget/overflow behavior where the page exposes primary actions.
     - Map canvas or bbox preview visible (where applicable) and no obvious layout overlaps with sticky tabs/top summary.
     - Dataset top summary block visible on dataset pages (`data-testid="dataset-top-summary"` and its key sub-blocks if present); collection detail gets equivalent scrutiny on header, metadata card, and members section.
   - Tab sweep (only tabs that appear for that record_type/page): click each tab and confirm “first meaningful content” renders (not just skeletons).
   - Trigger 1-2 “high leverage” actions per type to surface UX gaps:
     - Vector: Access export trigger, Metadata edit affordance (read-only vs editable), Attributes/structure content.
     - Raster: Quicklook / raster facts / download action, Access export.
     - VRT: Sources/members management, generation status / regenerate affordance if shown.
     - Collection: Members list, add/remove affordances, bbox preview, metadata density, and any collection-specific guidance/empty states.
   - Capture evidence:
     - Screenshot full viewport (`mcp__playwright__browser_take_screenshot`) to `evidence/<type>-desktop.png`.
     - If console errors appear, dump console logs via `mcp__playwright__browser_console_messages` to `logs/console-<type>.log`.
     - If a network error is suspected (500/401 loops), dump network requests via `mcp__playwright__browser_network_requests` to `logs/network-<type>.log`.

2. Mobile pass:
   - `mcp__playwright__browser_resize` to 375x812.
   - Reload the same URL and validate:
     - Sticky tabs do not obscure content on dataset pages; collection detail remains readable without horizontal overflow.
     - Overflow menus are usable (kebab trigger tappable, items not off-screen).
     - Primary actions remain discoverable.
   - Capture `evidence/<type>-mobile.png`.

3. Findings write-up (in `.planning/.../FINDINGS.md`):
   - Use a consistent schema per finding:
     - `ID`, `Severity (P0/P1/P2)`, `Type (Bug/UX/A11y/Content/Perf)`, `Record type`, `Tab/area`, `Steps`, `Expected`, `Actual`, `Evidence (screenshot/log)`, `Suggested fix`, `Effort (Easy win / Milestone)`.
   - Organize sections:
     - P0 Blocking Bugs
     - P1 UX Gaps
     - P2 Polish / Nice-to-haves
     - A11y & Keyboard
     - Consistency and Content (labels, taxonomy, empty-state language)
     - Easy wins (explicit list of items that look <= 30 min each)
     - Milestone candidate breakdown (group larger items into coherent workstreams)

Optional automated sanity check (only if the app is already running and seeded):
- Run `npm run e2e -- e2e/dataset-detail.spec.ts` to ensure the vector baseline spec still passes; if it fails, capture the failure summary in FINDINGS.md as a P0 regression.
</action>
