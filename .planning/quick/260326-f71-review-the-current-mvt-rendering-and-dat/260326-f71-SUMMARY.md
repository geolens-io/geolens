## Changes Made

### Fix 1: Bounds CTE precomputation (service.py)
Added `ST_Transform(ST_TileEnvelope(...), 4326) AS geom_4326` to the bounds CTE. Replaced both `ST_Transform(bounds.geom, 4326)` references in the WHERE clause with `bounds.geom_4326`. Guarantees single evaluation regardless of query planner behavior.

### Fix 2: Per-tile logging to DEBUG (router.py)
Changed `logger.info("tile_access", ...)` to `logger.debug(...)`. A single map view generates 20-50 tiles; this eliminates 20-50 INFO entries per view.

### Tests
Updated `test_tile_query_uses_correct_params` with bounds.geom_4326 assertions. Added `test_tile_query_single_transform_in_where` verifying no redundant ST_Transform.

### Findings Document
133-line document covering: architecture overview, 8 strengths, 2 implemented fixes, 2 deferred fixes (no tile cache layer), 2 medium-effort recommendations, 6 confirmed non-issues.

## Deviations from Plan

### Adaptation: No tile cache layer in codebase

**Found during:** Task 1, reviewing Fix 2 and Fix 3

**Issue:** The plan references a `TileCacheProvider` with `(table, z, x, y)` interface and cache-hit/miss code paths in router.py. The current codebase has no tile-specific cache in the request path. The generic `CacheProvider` (Redis/in-memory) uses JSON serialization, incompatible with binary MVT bytes.

**Resolution:** Implemented the 2 applicable fixes (bounds CTE, logging). Documented the 2 cache-related fixes as deferred items in the findings document with clear implementation guidance for when a tile cache layer is added.

**Files affected:** None skipped -- the findings document captures all recommendations.
