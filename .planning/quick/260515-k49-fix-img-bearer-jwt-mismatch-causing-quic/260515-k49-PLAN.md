<objective>
Fix the structural bug where `<img src=".../quicklook">` requests cannot carry `Authorization: Bearer <jwt>`, causing 404s on Search and Builder for non-public datasets the user is otherwise authorized to see.

Replace every quicklook `<img>` with a `useQuicklook` React hook that drives an authenticated `apiFetchBlob()` and renders the result as a `URL.createObjectURL(blob)` blob URL. Adapt `quicklook-cache.ts` to negative-cache genuine `fetch()` 404s (file missing on disk) without caching transient errors.

Purpose: zero backend changes, zero auth-surface change. Eliminates 3 console 404s observed on the Search page (UUIDs 777ddb26, 9773fbcc, bbb61bdb during 260515-i45 live verify) and makes private-dataset quicklooks render for the owner/admin.

Output: new `use-quicklook.ts` hook + tests, adapted cache module, and two converted call sites with blob-URL lifecycle managed via TanStack Query (existing pattern from `use-map-thumbnail.ts`).
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser -> API | Bearer JWT travels in `Authorization` header for every `apiFetchBlob` call. `<img src>` does NOT cross this boundary anymore for the quicklook endpoint. |
| component -> blob URL | `URL.createObjectURL` allocates browser memory backed by the fetched Blob; URLs leak unless revoked. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-k49-01 | Information Disclosure | useQuicklook | mitigate | Route every request through `apiFetchBlob` (which attaches Bearer from `useAuthStore`). Never inline `Authorization` construction. Existence-mask behavior on 404 is preserved — backend still 404s anonymous probes. |
| T-k49-02 | Denial of Service (client memory) | useQuicklook | mitigate | `useEffect` cleanup revokes blob URL on unmount AND on `data` change (queryKey-driven). Unit tested. |
| T-k49-03 | Repudiation | quicklook-cache | accept | Negative cache is per-tab sessionStorage; no audit trail needed (cache state is non-authoritative UI hint). |
| T-k49-04 | Tampering | quicklook-cache | accept | sessionStorage is origin-scoped; tampering only causes a false-negative placeholder (no security impact — the backend remains the source of truth for access control). |
| T-k49-05 | Elevation of Privilege | backend authorization | accept | Zero backend edits. `check_dataset_access_or_anonymous` continues to enforce access. The frontend fix only ensures the Bearer token reaches the endpoint for users who already have access. |
</threat_model>
