## Phase Goal

Add backend detection of 3D geometry on ingest. Record `ST_NDims`, `ST_Is3D`, `ST_ZMin`, `ST_ZMax` on the dataset record. Surface Z range in the dataset detail UI ("This dataset has 3D geometry — Z range: 12m to 847m"). Optional attribute-promotion strategy for point geometries (`elev` column).

## Locked Decisions

- **D-01**: Alembic migration adds `is_3d` (bool), `n_dims` (int), `z_min` (float), `z_max` (float) to the dataset model
- **D-02**: Detection runs during ingest via PostGIS functions: `ST_NDims`, `ST_Is3D`, `ST_ZMin`, `ST_ZMax`
- **D-03**: Attribute promotion for point geometries — if `ST_Is3D(geom)` is true AND geometry is point-like, extract `ST_Z(geom)` into a new `elev` numeric column in the data table
- **D-04**: POLYHEDRALSURFACE / TIN handling: defer — no special UI warning needed this phase
- **D-05**: Z range surfaced on dataset detail page UI as informational badge/text
- **D-06**: Detection is post-ingest (after ogr2ogr import completes and table exists in PostGIS)

## Claude's Discretion

- Where exactly to place the 3D detection in the ingest pipeline (after table creation, during metadata extraction, or as a separate step)
- Exact UI placement and styling of the Z range indicator on the dataset detail page
- Whether `elev` column promotion happens during ingest or as a post-processing step
- Error handling for datasets with mixed 2D/3D geometries (some features have Z, some don't)
- Whether to show `n_dims` in the UI or keep it backend-only
