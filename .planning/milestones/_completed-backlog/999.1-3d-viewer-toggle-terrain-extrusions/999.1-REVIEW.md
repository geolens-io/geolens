# Phase 999.1: Code Review Report

**Reviewed:** 2026-04-12T00:00:00Z
**Depth:** standard
**Files Reviewed:** 17
**Status:** issues_found

## Summary

This phase adds DEM/terrain support: an `is_dem` boolean on `raster_assets`, auto-detection during ingest from single-band float heuristic, a `TerrainControl` toggle in `ViewerMap`, and a fill-extrusion companion layer in the fill adapter. The backend migration chain is correct. The feature logic is sound overall, but there are three warnings and three info-level observations worth addressing.

## Warnings

### WR-01: VRT DEM datasets silently get a vector tile URL in the shared-map path

**File:** `backend/app/maps/service.py:833-838`
**Issue:** The tile URL branch for shared maps only checks for `record_type == "raster_dataset"` to assign the raster-proxy URL. A `vrt_dataset` that has `is_dem = True` will fall through to the vector branch and receive a `.pbf` URL. In `ViewerMap`, `demLayer` is selected by `l.is_dem && l.dataset_record_type?.startsWith('raster')`, which would match `vrt_dataset` — but `seedTerrainSource` would receive a `.pbf` URL, causing the terrain source to silently fail to load.
**Fix:** Extend the raster-proxy URL branch to cover `vrt_dataset` as well, consistent with how the rest of the codebase treats these two types identically:
```python
if ds_record_type in ("raster_dataset", "vrt_dataset"):
    tile_url = f"/tiles/raster-proxy/{layer.dataset_id}/{{z}}/{{x}}/{{y}}.png"
```

### WR-02: `fill-outline-color` syncPaint always sets `transparent` regardless of `_stroke-disabled`

**File:** `frontend/src/components/builder/layer-adapters/fill-adapter.ts:100-102`
**Issue:** In `syncPaint`, the branch that syncs `fill-outline-color` based on `_stroke-disabled` sets the value to `'transparent'` in both the enabled and disabled branches — the conditional is present but both arms write the same value:
```ts
map.setPaintProperty(layerId, 'fill-outline-color',
  strokeDisabled ? 'transparent' : 'transparent');  // both arms identical
```
This means the native 1px fill outline is always suppressed, even when stroke is enabled. The companion `line` layer provides the visible stroke, but MapLibre's default fill outline (which can bleed through at low opacity) is never restored when stroke is re-enabled.
**Fix:**
```ts
map.setPaintProperty(
  layerId,
  'fill-outline-color',
  strokeDisabled ? 'transparent' : undefined,  // undefined removes the override
);
```
Note: `setPaintProperty` with `undefined` removes the property override and restores the default. If MapLibre rejects `undefined`, use `map.removeProperty` or set to the actual outline color.

### WR-03: Terrain source seeded with wrong URL encoding — `window.location.origin` prefix applied to already-absolute raster-proxy URL

**File:** `frontend/src/components/viewer/ViewerMap.tsx:82-84` and `288`
**Issue:** `seedTerrainSource` prepends `window.location.origin` unconditionally to the tile URL, but `demTileUrl` comes from `SharedLayerResponse.tile_url`, which is a path like `/tiles/raster-proxy/{id}/{z}/{x}/{y}.png`. In the dev Vite proxy this will be `http://localhost:8080/tiles/raster-proxy/...`, which is correct. However, the `map.setTransformRequest` installed in `handleLoad` (line 317) already absolutifies any relative URL via `url.startsWith('http') ? url : \`${window.location.origin}${url}\``. Since `addSource` fires before `setTransformRequest` runs on tile requests, this is actually fine — but only because both paths produce the same absolute URL. The real risk is that `seedTerrainSource` is called inside a `useEffect` that can fire **before** `handleLoad` registers `setTransformRequest` (because `mapReady` is set inside `handleLoad`). Verify the ordering is guaranteed: `handleLoad` sets `mapReady = true`, so the `useEffect([mapReady, demTileUrl])` that calls `seedTerrainSource` only runs after `handleLoad` completes. The order is correct, but the double-absolutification in `seedTerrainSource` creates a maintenance hazard.

More critically: the `style.load` handler at line 554-563 calls `seedTerrainSource(m, currentDemTileUrl)` where `currentDemTileUrl` is already the raw path. This is fine because `seedTerrainSource` prepends `window.location.origin`, but this logic should be centralized. Consider extracting the full URL construction out of `seedTerrainSource` to avoid subtle bugs when the tile URL changes format.

**Fix:** Make `seedTerrainSource` accept the full absolute URL rather than performing URL construction internally, and build the absolute URL at the call site where the format is known:
```ts
function seedTerrainSource(map: MaplibreMap, absoluteTileUrl: string) {
  if (!map.getSource('terrain-dem')) {
    map.addSource('terrain-dem', {
      type: 'raster-dem',
      tiles: [absoluteTileUrl],
      tileSize: 256,
      encoding: 'mapbox',
    });
  }
}
// call site:
const fullUrl = `${window.location.origin}${demTileUrl}`;
seedTerrainSource(map, fullUrl);
```

## Info

### IN-01: `is_dem` not exposed on `RasterMetadata` response schema or `DatasetResponse`

**File:** `backend/app/datasets/schemas.py:114-151` and `frontend/src/types/api.ts:35-54`
**Issue:** `is_dem` is settable via `DatasetMeta` (line 356), persisted on `RasterAsset`, and surfaced in shared-map layer responses — but it is not included in `RasterMetadata` or `DatasetResponse`. This means the dataset detail page and builder layer panel have no way to know whether a dataset is marked as a DEM, making it impossible to show a "DEM / Terrain" indicator in the UI or to set `is_dem` without a separate API call.
**Fix:** Add `is_dem: bool = False` to `RasterMetadata` in `backend/app/datasets/schemas.py` and populate it from `RasterAsset.is_dem` in the service layer. Mirror it in `frontend/src/types/api.ts`.

### IN-02: `_is_float_dtype` partial-string match can produce false positives

**File:** `backend/app/raster/cog.py:13-15`
**Issue:** The implementation uses `any(f in dtype.lower() for f in _FLOAT_DTYPES)`, and `_FLOAT_DTYPES` includes the bare string `"float"`. Any dtype string containing the substring `"float"` will match, including unusual driver-reported strings like `"nonfloat"` or future dtype names. While no current GDAL dtypes would trigger this, the pattern is fragile.
**Fix:** Match against exact dtype tokens using equality rather than substring containment:
```python
def _is_float_dtype(dtype: str) -> bool:
    return dtype.lower() in _FLOAT_DTYPES
```
The set already contains `"float32"`, `"float64"`, `"float16"`, `"float"`, and `"complex"`, so exact matching is sufficient.

### IN-03: Test description comment for `getLayerIds` is stale

**File:** `frontend/src/components/builder/__tests__/layer-adapters.test.ts:271`
**Issue:** The test at line 271 is described as `'getLayerIds returns [layerId, outlineId] (two layers without _height_column)'` but the assertion immediately below checks `ids.toHaveLength(3)` and verifies a third `extrusion` ID. The description says "two layers" but the test correctly asserts three. This is a documentation-only inconsistency, but it will mislead future readers.

```ts
it('getLayerIds returns [layerId, outlineId, extrusionId] (three layers — extrusion always reserved)', () => {
```
