<objective>
Replace the legacy comma-separated `Input` in `ShareLinkSettings` (SharePanel.tsx:252-270) with a chip-based allowed-origins UI. Each chip is a shadcn Badge with a remove X; the canonical-form text comes from Plan 01's `normalizeOrigin`. Add/remove is optimistic + rollback on PATCH failure. Wildcard input shows inline error and does not fire PATCH; backend 422 (Plan 02 contract) surfaces the SAME inline error so dev and prod failure modes look identical.

Purpose: chips are the source of truth post-Save (UI-SPEC Surface 1 lines 96-200). The comma-separated input is fragile, drops case-normalization, allows non-canonical entries, and gives no per-origin remove affordance. Chips fix Pitfall #8 (canonical form) at the moment-of-render boundary AND give the user a clean remove path.

`rawShareToken` and `embedTokenRaw` survival (Pitfall #6) is preserved because chip add/remove uses the `updateEmbedToken` mutation (not `createEmbedToken`) â€” the existing race-free path. Plan 06 adds the `inflightEmbedCreate` ref pattern for the createEmbedToken path; THIS plan does not touch that surface.

Output: chip-based UI in ShareLinkSettings; 5 new regression pins in SharePanel.test.tsx; new i18n keys per UI-SPEC table.
</objective>
