<objective>
Eliminate the `/maps/new` 422 console-noise pattern. The current `<Route path="maps/:id">` matches `/maps/new` with `id="new"`, then `MapBuilderPage` calls `useMap('new', ...)` which fires `GET /api/maps/new` (returns 422 — `new` is not a valid UUID), and subsequent hooks (`useEnabledWidgets`, etc.) may fire additional 422s or 4xx responses against the same invalid id surface.

Two spurious 422s on every browser visit to `/maps/new` is cosmetic console noise today (returns blocks the editor from loading), but blocks future surfaces (per-prefix rate-limit middleware, error-rate dashboards, smoke-test logs) and signals a brittle routing seam.

Fix at source: either route `/maps/new` to a dedicated handler that opens the Create dialog without mounting the live editor, OR guard `MapBuilderPage` to short-circuit BEFORE any data hook fires when `id === 'new'`. The plan picks Option A by default but documents the Option B fallback in case Option A introduces unanticipated routing edge cases (e.g., interactions with the existing `<Route element={<MapViewerGate />}>` and the editor/viewer split).

Purpose: TD-11 close. Restores a clean network log on `/maps/new` and unblocks the future per-prefix middleware mentioned in REQUIREMENTS.md.

Output: 1-2 files modified (`App.tsx` and optionally `MapBuilderPage.tsx`), zero 422 entries on `/maps/new` in Playwright MCP smoke session, no regression on valid `/maps/:uuid` URLs.
</objective>

`useMap` hook (`src/hooks/use-maps.ts` line 48):

```
export function useMap(id: string | undefined, opts?: { refetchOnWindowFocus?: boolean }) {
  return useQuery({
    queryKey: queryKeys.maps.detail(id),
    queryFn: () => getMap(id!),
    enabled: !!id,
    staleTime: 60_000,
    refetchOnWindowFocus: opts?.refetchOnWindowFocus,
  });
}
```

`getMap` (`src/api/maps.ts` line 53):

```
export async function getMap(id: string): Promise<MapResponse> {
  const resp = await apiFetch<MapResponse>(`/maps/${id}`);
  // ... normalize ...
  return resp;
}
```

So when `id='new'`, the request goes to `GET /api/maps/new` and the backend returns 422 (FastAPI UUID validation on the path parameter — `new` is not a valid UUID4 string).

The Create dialog itself (`src/components/maps/MapCreateDialog.tsx`) is a standalone Radix Dialog opened by `setMapOpen(true)` in `Navbar.tsx` and `setCreateOpen(true)` in `MapsPage.tsx`. After a successful create, `MapCreateDialog` calls `navigate(\`/maps/\${newMap.id}\`)` (line 79 — concrete UUID, not `/maps/new`).

There is NO source-level `navigate('/maps/new')` call in the current codebase (planner verified via `grep -rn`). The `/maps/new` URL only arrives from external sources: bookmarks, manual address-bar entry, a stale link in user-facing copy, OR an SSR placeholder. The fix must handle the URL gracefully without forcing the user to know that `/maps/new` is "wrong".

Decision tree for the executor:

Option A (route-level — preferred): Add `<Route path="maps/new" element={<MapsPage />} />` BEFORE the `<Route path="maps/:id" ...>` line. react-router resolves static segments before dynamic segments, so this matches first. The Create dialog will not open automatically (Option A renders the maps list, with the existing `Create Map` button as the affordance) — this is the cleaner UX because the user lands on a useful page rather than an isolated dialog with no surrounding navigation context.

Option B (in-component guard — fallback if Option A breaks `MapViewerGate` lazy-load contract): Modify `MapBuilderPage.tsx` so the very first statement after `const { id } = useParams(...)` is `if (id === 'new') return <Navigate to="/maps" replace />;`. This MUST be placed BEFORE the `useMap(id, ...)` call to prevent the network request. The downstream `MapViewerGate` lazy-load contract is unaffected because the early return runs after the lazy chunk loads but before any hook fires.

Note on the `useMaps` (list) hook that `MapsPage` calls: this fires `GET /api/maps/?limit=20` (with a real trailing slash, no `new` segment), which returns 200 OK. No 422 noise from Option A's redirect target.
