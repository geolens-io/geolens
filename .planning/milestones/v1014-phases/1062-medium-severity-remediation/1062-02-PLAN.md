<objective>
Close SEC-S10 (MEDIUM, CVSS 5.3) and SEC-S11 (MEDIUM, CVSS 5.3). Per the audit (`docs-internal/audits/sec-audit-20260519.md:374-380`), there is currently no per-route `@limiter.limit(...)` decoration on the three highest-cost / highest-exposure surfaces:

- `/search/datasets/` (`search/router.py:603-615`) — every unique query string triggers an OpenAI embeddings API call. Attacker pace × $0.02/1M tokens = unbounded cost.
- `/datasets/{id}/related/` (`datasets/api/router_data.py:56-65`) — same embedding-cost surface, different shape.
- `/settings/basemaps/` (`settings/router.py:707-727`) — unauthenticated by design; substitutes the configured `api_key` value into the URL template at response time. A commercial-tier basemap key (Mapbox / Stadia / etc.) misused here is replayable at attacker pace.

The slowapi `Limiter` is already wired (`backend/app/api/main.py:469`, `auth/router.py:48`) and `_global_rate_limit` (60/sec) already gates every route via `default_limits=[...]`. This plan adds **route-specific decorators** that tighten the budget on these three endpoints, plus a documentation paragraph for SEC-S10 stating that the `api_key` field is publicly resolved by design (matches the audit's recommendation: "Acceptable for client-side tile-provider keys ... but worth documenting and adding rate limiting to prevent commercial-tier key replay if an operator uses a backend-only key by mistake").

Purpose:
- Cap OpenAI embedding cost-DoS (audit's specific framing of S11).
- Cap commercial-tier basemap key replay (audit's specific framing of S10).
- Make limits configurable so the e2e S11 test can dial down for fast assertion without changing prod defaults.

Output:
- `@limiter.limit(_semantic_search_rate_limit)` on `search_datasets_endpoint` and `search_facets_endpoint` (both call into the same embedding cache).
- `@limiter.limit(_semantic_search_rate_limit)` on `list_related_datasets` (`router_data.py:57`).
- `@limiter.limit(_basemap_proxy_rate_limit)` on `get_basemaps` (`settings/router.py:707`).
- SEC-S10 docstring extension on `get_basemaps` explaining the `api_key` exposure model.
- Two new `PersistentConfig` knobs: `semantic_search_rate_limit` (default 30, per-minute) and `basemap_proxy_rate_limit` (default 120, per-minute), plumbed through `get_cached_*_rate_limit` accessors mirroring the existing `get_cached_login_rate_limit` pattern.
- `.env.example` documents the two new env keys.
- Backend pytest in `test_rate_limits.py` asserting 429 after threshold (uses a monkeypatched low limit to keep the test fast).
- e2e `/sec-audit.spec.ts` S11 test goes from `test.skip` -> assertion-pass without code change (the test detects the 429s).

S11 audit text (sec-audit-20260519.md:378-380):
> No per-route @limiter.limit(...) on /search/datasets/ (semantic) or /datasets/{id}/related/. Global default = 60/sec/IP. The 300-second TTL LRU embedding cache (service_semantic.py:33-37) collapses identical strings, but unique novel queries each call OpenAI's embedding API at attacker's pace. Cost-DoS risk: $0.02 per 1M tokens × bursts. Recommend @limiter.limit("30/minute") matching cost-sensitive endpoints.

S10 audit text (sec-audit-20260519.md:374-376):
> /api/settings/basemaps/ is unauthenticated (by design — frontend needs basemap config). When an admin configures a basemap with {api_key} in the URL template plus an api_key value, the resolved URL with the literal key substituted is returned to anyone. Acceptable for client-side tile-provider keys (Mapbox/Stadia/etc. are designed for browser exposure), but worth documenting and adding rate limiting to prevent commercial-tier key replay if an operator uses a backend-only key by mistake. Add a docstring + admin-UI warning at configuration time.

The CONTEXT.md (decision 2) specifies: per-IP + per-token caps configurable via .env. **For this plan**: ship per-IP only. Per-token caps require a custom `key_func` (read JWT subject) — that's a SEC-FU enhancement; not in audit scope. Document the decision in the plan SUMMARY.
