<objective>
Close SEC-S14 (MEDIUM, CVSS 5.4). Per audit (`docs-internal/audits/sec-audit-20260519.md:398-400`): Zustand persists access + refresh tokens to `localStorage` under `geolens-auth`. Any future XSS exfiltrates both. No current XSS sinks (zero `dangerouslySetInnerHTML`, popups render via React text nodes), so the risk is **conditional on future regressions**. The audit recommends:

1. ESLint rule `react/no-danger` to harden the boundary (this is **SEC-FU-03**, deferred to Phase 1063).
2. Migrate to `httpOnly` cookie + CSRF token medium-term (large architectural change, NOT in v1014 scope).
3. **This plan's job**: Add an ESLint guard that prevents NEW code from regressing the surface (writing tokens to localStorage outside the existing auth-store path).

**Scope discipline**: The ban is on the **pattern**, not on `geolens-auth` specifically. The existing zustand persist call at `frontend/src/stores/auth-store.ts:30` is the SINGLE legitimate write site — and that file does NOT call `localStorage.setItem` directly (zustand persist middleware wraps it internally inside node_modules). So we can ban the bare pattern across the entire `frontend/src/` tree without an exemption list.

If a future contributor reaches for `localStorage.setItem('myjwt', token)` to short-circuit the auth flow, ESLint fails the build. The rule does NOT prevent legitimate uses (theme, sidebar collapsed, i18n language, view-storage) because those keys don't match `/token|jwt|auth/i`.

Output:
- `frontend/eslint.config.js` extended with a `no-restricted-syntax` rule (or equivalent AST selector) targeting `localStorage.setItem(<token-shape-string-literal>, ...)`. **OR** a `no-restricted-properties` rule on `localStorage.setItem` with a custom message — pick the form that lets us regex-match the first arg.
- Regression-test files in `frontend/src/__tests__/`:
  - `sec-s14-eslint-regression.ts` — contains intentional violations; ESLint MUST flag them. Tracked via a make/npm target.
  - `sec-s14-eslint-regression.skip.ts` — contains patterns that LOOK risky but should pass.
- `docs-internal/audits/security-lessons.md` extended with a section documenting:
  - The ESLint rule's pattern + scope.
  - The medium-term httpOnly-cookie + CSRF migration plan, with the explicit tradeoffs:
    - Pro: XSS-resistant token storage.
    - Con: requires CSRF token primitive on every state-mutating request.
    - Con: requires backend `/auth/login/` to emit a Set-Cookie response (currently returns JSON body).
    - Con: cross-subdomain deployments (api.geolens.com vs app.geolens.com) need careful cookie scoping.
    - Estimate: 1-2 phase milestones, deferred to a future hardening pass.
- e2e/sec-audit.spec.ts has no S14 test (frontend lint is the gate — see audit's recommendation).

Purpose:
- Pin the current "no token in localStorage outside auth-store" invariant so it survives future contributors.
- Surface the httpOnly migration plan in a discoverable doc so the next operator who picks up v1015+ has the prior-art context.
- Establish the ESLint-regression-test pattern that Phase 1063 (SEC-FU-03 `react/no-danger`) can reuse.

The `no-restricted-syntax` rule supports AST selectors. For matching `localStorage.setItem('<token-shape>', ...)`, the selector is:
```
CallExpression[callee.object.name='localStorage'][callee.property.name='setItem'][arguments.0.type='Literal'][arguments.0.value=/token|jwt|auth/i]
```

The selector matches when:
- The call is `localStorage.setItem(...)`,
- The FIRST argument is a string literal (not a variable or template literal — those slip through; that's a known limitation; document it),
- The literal's value matches the regex `/token|jwt|auth/i`.

Add to `rules`:
```js
'no-restricted-syntax': [
  'error',
  {
    selector: "CallExpression[callee.object.name='localStorage'][callee.property.name='setItem'][arguments.0.type='Literal'][arguments.0.value=/token|jwt|auth/i]",
    message: 'SEC-S14 (sec-audit-20260519.md): writing token/jwt/auth values to localStorage is banned outside the central auth-store. Use useAuthStore.setAuth() or, for non-session data, pick a non-token-shaped key.',
  },
],
```

**Template-literal limitation**: ESLint AST selectors can match `Literal.value` but NOT `TemplateLiteral` directly with the same expression. A motivated attacker can write `` localStorage.setItem(`token-${userId}`, ...) `` and slip through. Document this in security-lessons.md as a known gap; closing it requires a custom ESLint plugin (out of scope). The pattern this rule catches is **accidental copy-paste / cargo-cult regression**, not deliberate evasion — which matches the audit's "future regression" framing.

    ### httpOnly-cookie + CSRF migration plan (deferred to future hardening pass)

    **Context.** The audit's secondary recommendation for SEC-S14 is to migrate JWT storage from `localStorage` to `httpOnly` cookies. This eliminates XSS exfiltration risk entirely (JS cannot read httpOnly cookies). The current zustand-persist-to-localStorage architecture (`frontend/src/stores/auth-store.ts:29-54`) was chosen for simplicity at v1.0 and remains correct for the open-core single-tenant deployment model. The audit notes no current XSS sinks (zero `dangerouslySetInnerHTML`, popups render via React text nodes) — so the present risk is **conditional on future regressions**.

    **Decision (deferred).** Migration to httpOnly cookies + CSRF token is the long-term hardening path. It is **NOT scheduled** in v1014; this plan ships the ESLint guard (cheap, regression-pinning) and defers the architectural change.

    **Consequences if migration happens later:**

    Positive:
    - XSS-resistant token storage (httpOnly cookie cannot be read by `document.cookie` or `localStorage` access).
    - No more "logout doesn't clear OS-level password manager" UX edge cases (cookie scope is clean).
    - Cleaner mobile/SDK token handoff (some SDKs default to cookie auth).

    Negative:
    - Requires a CSRF token primitive on every state-mutating request (POST/PUT/PATCH/DELETE). Current GeoLens has no CSRF token surface — adding one touches every API client call site (`frontend/src/api/client.ts` + every mutator hook).
    - `/auth/login/` must shift from "returns JSON body with access_token" to "sets Set-Cookie response header". OAuth2-compatible callers (OpenAPI-spec consumers, SDKs, CLI tools) expect the JSON body. Need a dual-mode: cookie for browser, JSON for non-browser. Detect via `Accept` / `Sec-Fetch-Mode` headers.
    - Cross-subdomain deployments (api.geolens.com vs app.geolens.com) need careful `Domain=` cookie scoping. SameSite=Lax breaks cross-origin POST without explicit allowlist.
