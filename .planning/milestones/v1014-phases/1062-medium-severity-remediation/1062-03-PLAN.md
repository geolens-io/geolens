Close SEC-S12 (MEDIUM, CVSS 5.0) and SEC-S13 (MEDIUM, CVSS 4.3). Per audit (`docs-internal/audits/sec-audit-20260519.md:382-396`):

- **S12 (TRGM-INDEX-SIMPLE)**: `backend/app/modules/catalog/search/service_filters.py:78-87` builds `to_tsvector('simple', concat_ws(...))` inline per request and matches under `OR english_vector_match`. The English side has a GIN index (`idx_records_search_vector` from migration 0010), but the `simple` side has **no** matching functional GIN. Every non-English query (CJK, accented Latin without unaccent, names with diacritics not normalized) triggers a sequential scan on the entire `records` table.
- **S13 (TRGM-INPUT-CAP)**: `backend/app/modules/catalog/search/router.py:530` declares `q: str | None = Query(None, ...)` with no `max_length`. The peer `/search/datasets/` caps at 1000 (`SearchQueryParams.q`). A multi-megabyte `q` amplifies the un-indexed `simple` vector scan from S12 (and the existing ILIKE matches in `_build_text_filter`).

These two findings are paired because the S13 input cap is the cheapest defense against the same DoS surface S12 mitigates structurally. Closing them in the same plan lets a single regression test scaffold cover both.

Purpose:
- S12 turns a per-request seq-scan on `records` into a GIN index probe for the non-English query path. Performance win + DoS surface reduction.
- S13 caps the worst input that amplifies the S12 fix. Closes the trivially-fuzzable input vector before the structural fix lands.

```python
record_simple_vector = func.to_tsvector(
    "simple",
    func.concat_ws(
        " ",
        func.coalesce(Record.title, ""),
        func.coalesce(Record.summary, ""),

```

**CRITICAL**: The runtime expression is `concat_ws(' ', ..., array_to_string(theme_category, ' '))`. The audit's recommended migration (`docs-internal/audits/sec-audit-20260519.md:385-392`) uses simple `||` concat WITHOUT the theme_category column:
```python
op.create_index(
    "ix_records_simple_search_vector",
    "records",
    [sa.text("to_tsvector('simple', coalesce(title,'') || ' ' || coalesce(summary,'') || ' ' || coalesce(lineage,''))")],
    postgresql_using="gin",
)
```
The audit's `lineage` column does not exist — the actual column is `lineage_summary`. The audit also omits `theme_category`.

**For the index to actually serve the runtime query, the indexed expression MUST match the runtime expression exactly**. Postgres function indexes require expression-tree equality (the planner compares parsed expression trees).

The runtime currently uses `concat_ws` with theme_category. The runtime is the source of truth — **the migration must mirror it, OR the runtime must be refactored to match the migration**. Pick ONE:

- **Path A (recommended)**: write the migration to match the current runtime exactly — `to_tsvector('simple', concat_ws(' ', coalesce(title,''), coalesce(summary,''), coalesce(lineage_summary,''), coalesce(array_to_string(theme_category, ' '), '')))`. Index covers theme_category too, so it serves the runtime query 1:1.
- **Path B**: change the runtime expression in `service_filters.py:78-87` to drop theme_category (and use `||` instead of `concat_ws`), match the audit's simpler form, then write the migration to match the new runtime.

**Pick Path A**. It's a strictly-additive migration with no service_filters.py change — lower regression risk on the close path of v1014, and theme_category appearing in the simple-vector match is semantically correct (it's keyword-shaped data).

Verifying expression-tree equality is hard without applying the migration and running EXPLAIN. The Task 2 test does exactly that: `EXPLAIN (FORMAT JSON)` on a non-English query against a seeded record and asserts the plan contains `ix_records_simple_search_vector`.
