<objective>
Close SEC-FU-02, SEC-FU-09, and SEC-FU-10 — three configuration-level defense-in-depth additions that all live in operator-facing files and have zero impact on runtime code paths.

**SEC-FU-02 (validate_demo_credentials_guard literal verification):** Phase 1061 Plan 05 already extended the guard to refuse 3 literal committed credentials: `DEMO_JWT_SECRET = "demo-only-do-not-use-in-production-change-me"`, `DEMO_ADMIN_PASSWORD = "demodemo"`, `DEMO_POSTGRES_PASSWORD = "geolens-demo-2026"`. SEC-FU-02 explicitly calls out `JWT_SECRET_KEY=demo-only-do-not-use-in-production-change-me` — already covered. This plan **verifies** that coverage is correct by reading `config.py` lines 19-23 and adds a single named pytest test (`test_sec_fu_02_jwt_demo_literal_refused`) that explicitly references SEC-FU-02 in its docstring so the audit traceability is unambiguous. If the literal is already pinned by an existing test, name the new test as a redundant-but-explicit pin (no functional duplication penalty — different test ID = different reproduction guarantee).

**SEC-FU-09 (nginx server_tokens off):** Move/add `server_tokens off;` into `frontend/nginx.conf`'s server block. This suppresses the nginx version in the `Server:` header on response and inside the default error pages, so a remote scanner can't fingerprint the nginx minor version. The directive can live at either `http {}` or `server {}` scope; the audit's wording ("Move into the prod server block") implies the server-block placement, which is what we ship.

**SEC-FU-10 (DATABASE_URL_OVERRIDE role-scoping):** `.env.example` line 360 currently shows `# DATABASE_URL_OVERRIDE=` with one inline example. Extend the surrounding commentary block with explicit least-privilege guidance: the configured Postgres role should not be the cluster superuser, and an example SQL recipe showing how to create a `geolens_app` role with the privileges the app actually needs (`USAGE` on the `catalog` schema, `SELECT`/`INSERT`/`UPDATE`/`DELETE` on tables, `USAGE`/`SELECT` on sequences, `EXECUTE` on functions). This is a documentation change only — no enforcement at runtime.

Purpose: All three closures are configuration / documentation hygiene. They make a misconfigured deployment harder to ship and codify the explicit literal refusal for SEC-FU-02 traceability.
