## Decisions Made

- **Unconditional guard over demo-mode bypass:** The audit's headline scenario (GEOLENS_DEMO_MODE=true deployed to internet with committed creds) was exploitable via the early-return. Removing the early-return is the correct fix; `GEOLENS_DEMO_MODE` still controls seeding/reset but not credential-literal acceptance.
- **DEMO_POSTGRES_PASSWORD added:** The postgres password `geolens-demo-2026` was committed in `.env.demo` but the original guard only checked JWT + admin. Closing all three literals in one change.
- **Double-dollar in entrypoint:** `$$MINIO_ROOT_USER` in docker-compose YAML causes compose to emit the literal `$MINIO_ROOT_USER` (single dollar) into the container's shell script, where the container's environment (populated via `environment:`) then expands it. This avoids the compose-level expansion that would cause a parse error.
- **docker-compose.demo.yml unchanged:** No MinIO services in the demo overlay — it relies on `.env.demo` for MinIO creds. No changes needed.

## Happy-Path Demo Deployment Recipe

```bash
# 1. Generate per-deploy credentials
./scripts/init-demo-env.sh
# Capture the admin password from stdout (only chance to see it)

# 2. Start demo overlay
docker compose -f docker-compose.yml -f docker-compose.demo.yml --env-file .env.demo up -d

# 3. Confirm no validation errors in boot log
docker compose logs api --tail 50 | grep -i "validate_demo_credentials"
# Expected: silent (no error lines)

# 4. Confirm demodemo is refused
curl -sL -o /dev/null -w "%{http_code}" -X POST http://localhost:8000/api/auth/login/ \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d 'username=admin&password=demodemo'
# Expected: 401

# 5. Login with generated password
curl -sL -X POST http://localhost:8000/api/auth/login/ \
  -H "Content-Type: application/x-www-form-urlencoded" \
  -d "username=admin&password=<generated-password>"
# Expected: 200 with access_token
```
