<objective>
Close SEC-S05 (HIGH, CVSS 7.5) — pgvector `/datasets/{id}/related/` visibility bypass. The seed dataset's embedding is read via `_load_self_record_and_embedding` WITHOUT a visibility check; only the *neighbors* are filtered by `apply_visibility_filter`. An anonymous attacker can probe any private dataset's UUID — if the seed has any public neighbors, the response returns those neighbors with cosine-similarity scores that act as an oracle on the private title/summary/keywords/lineage content. Three "empty" states (nonexistent UUID, private-no-embedding, private-no-public-neighbors) are timing-distinguishable.

Purpose: Closes the third anonymous-read HIGH finding in the audit. Sibling routes in the same file (`/rows/`, `/validate/`) already call `check_dataset_access_or_anonymous`. The fix is a single-line addition mirroring those siblings.

Output:
- `list_related_datasets` (router_data.py:56-65) calls `await check_dataset_access_or_anonymous(db, dataset, dataset_id, user)` after `get_dataset` and before `get_related_datasets`.
- Backend pytest coverage in `test_related_datasets_idor.py`: 5 tests covering anonymous + owner + non-owner against private + public dataset seeds.
- `e2e/sec-audit.spec.ts` S05 test passes with `SEC_AUDIT_PRIVATE_DATASET_ID` env var.
- Timing-uniform: nonexistent UUID and private-deny both return 404 from the same `check_dataset_access_or_anonymous` path (which raises 404 from `get_permission_extension().can_access_dataset` failure). No timing oracle remains.
</objective>
