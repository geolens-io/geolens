## Decisions Made

- **Removed `chk_users_auth_provider` from e002's docstring.** The original draft mentioned it as a "does NOT touch" marker, but the plan's literal acceptance criterion required zero occurrences anywhere in the file. Replaced with neutral phrasing ("the users-table auth_provider CHECK") that documents the same intent without the literal symbol.
- **Generator runs locally, not in Docker.** The plan suggested running `generate_fixtures.py` inside the migrate container, but the geolens API Docker image's pyOpenSSL/cryptography binary stack is incompatible with Python 3.14's `X509_V_FLAG_NOTIFY_POLICY` removal -- a pre-existing platform issue out of Phase 217's scope. I instead generated fixtures via a host-side temporary uv venv on Python 3.13 with `pysaml2>=7.5.4` and Homebrew `libxmlsec1`. The generated `.xml.b64` outputs are committed; tests do not need pysaml2 installed locally to consume them.
- **XSW fixture uses standalone `xmlns:saml="..."` declaration on the evil assertion.** pysaml2 emits namespace-prefixed elements (`ns0:Response`, `ns1:Assertion`) but the evil assertion is self-contained with its own `saml:` prefix, so the wrapped XML still parses cleanly. The XSW property (signature wraps the legitimate inner assertion via `<Advice>`, evil assertion is at the top level unsigned) is preserved.
- **Reused existing `IdPConfig` shape from pysaml2 docs (Context7-fetched).** The `_make_server()` config dict mirrors the canonical IdP server example: SP metadata loaded from a temp file, `xmlsec_binary` auto-detected across Linux + macOS Homebrew paths, `subject_data` pointed at `/tmp` for ephemeral use.

## Issues Encountered

- **Docker container pyOpenSSL/cryptography incompatibility on Python 3.14.** The existing geolens-api docker image errors during `OpenSSL.crypto` import with `AttributeError: module 'lib' has no attribute 'X509_V_FLAG_NOTIFY_POLICY'`. This blocks pysaml2 import inside any of the standard service containers. Pre-existing issue (image was built before the Python 3.14 + pyOpenSSL upgrade caused this regression); out of scope for Phase 217. Worked around by generating fixtures on the host via a separate Python 3.13 uv venv. Recommended follow-up: pin pyOpenSSL to a version compatible with Python 3.14 in the api Dockerfile (or update cryptography), and rebuild the image. Not blocking subsequent plans because (a) generated fixtures are already committed, (b) plan 02's tests will run via `cd backend && uv run pytest` against the local Python 3.13 venv (worktree backend has `.venv` already provisioned), not via docker.

- **alembic check against enterprise overlay deferred.** The plan's verify block calls for `docker compose -f docker-compose.yml -f docker-compose.enterprise.yml run --rm migrate sh -c "uv add --editable /enterprise && uv run alembic upgrade heads && uv run alembic check"`. The migrate container's docker image is stale -- it lacks `app/core/db/models.py` (added in source post-image-build) and `sqlalchemy` is not on the system Python -- so the auto-run `migrate` command fails with `ModuleNotFoundError: No module named 'app.core.db.models'`. The entry-point discovery for `geolens.migrations` works correctly (verified by direct python invocation: `paths: ['/enterprise/geolens_enterprise/migrations/versions']`), but the version_locations is set in `env.py` which fails to import for the same reason as the `app.core.db.models` issue. Recommended follow-up: rebuild the api/migrate docker image so `app.core.db.models` and the full venv are present, then re-run the alembic check. The migration files themselves are structurally correct -- AST-validated revision/down_revision/branch_labels and python-imported cleanly. The deferral is documented in the SUMMARY rather than blocking the task because (a) the structural correctness is verified, (b) Plan 02 will exercise the migration end-to-end as part of its own verification, and (c) the docker-image-staleness issue is independent of Phase 217.
