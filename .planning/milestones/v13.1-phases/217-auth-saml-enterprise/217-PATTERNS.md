```python
@router.get("/{provider_slug}/callback", response_class=Response)
async def oauth_callback(
    provider_slug: str, request: Request, db: AsyncSession = Depends(get_db),
) -> Response:
    """Handle IdP callback: exchange code, find/create user, issue JWT, redirect to frontend."""
    from app.modules.auth.oauth.service import find_or_create_oauth_user

    # Compute frontend URL before try block (needed in except for error redirect)
    frontend_url = await get_public_app_url(db, request=request)

    try:
        client, provider = await build_oauth_client(provider_slug, db)

        # Exchange authorization code for tokens
        token = await client.authorize_access_token(request)

        # Extract userinfo
        userinfo = token.get("userinfo")
        if userinfo is None:
            userinfo = await client.userinfo(token=token)

        # Find or create the GeoLens user
        user = await find_or_create_oauth_user(
            db, provider, dict(userinfo), dict(token)
        )

        # Record login timestamp
        user.last_login_at = func.now()

        # Issue GeoLens JWT
        expire_minutes = await ACCESS_TOKEN_EXPIRE_MINUTES.get(db)
        expire_days = await REFRESH_TOKEN_EXPIRE_DAYS.get(db)

        identity = AuthenticatedIdentity(
            user_id=user.id, username=user.username, email=user.email
        )
        service = AuthService(db)
        access_token = service.create_access_token(identity, expire_minutes=expire_minutes)
        refresh_token = service.create_refresh_token(user.id, expire_days=expire_days)
        await db.commit()

        redirect_url = (
            f"{frontend_url}/oauth/callback"
            f"#token={access_token}"
            f"&refresh_token={refresh_token}"
            f"&expires_in={expire_minutes * 60}"
        )
        return RedirectResponse(url=redirect_url, status_code=302)

    except HTTPException:
        raise  # Let 404s from build_oauth_client pass through
    except Exception:
        correlation_id = uuid.uuid4().hex[:12]
        logger.exception(
            "OAuth callback failed",
            provider=provider_slug, correlation_id=correlation_id,
        )
        error_url = f"{frontend_url}/oauth/callback#error=oauth_failed&correlation_id={correlation_id}"
        return RedirectResponse(url=error_url, status_code=302)
```

## Patterns NOT to Mirror (anti-patterns)

### A1. **Old import paths in the existing scaffold MUST NOT be retained**

`~/Code/geolens-enterprise/geolens_enterprise/auth/saml/router.py:12-23` and `config.py:10-11` reference pre-v13.0 paths like `from app.auth.oauth.service import ...`, `from app.dependencies import get_db`, `from app.persistent_config import ...`, `from app.public_urls import ...`, `from app.extensions.guards import ...`. **All ten paths are dead** (RESEARCH §9 verified each). The Phase 217 modernization is a hard cutover — no compatibility shims, no `try/except ImportError`. Use the verified replacements in the table at RESEARCH §9.

### A2. **CONTEXT.md D-10's `from app.api.deps import get_db` is wrong**

CONTEXT.md D-10 lists `from app.api.deps import get_db (verify exact path during planning)`. RESEARCH §9 verified: `app/api/deps.py` does not exist. The correct path is `from app.core.dependencies import get_db` (also confirmed by `backend/app/modules/auth/oauth/router.py:17` itself). Plan-time verifier MUST use the RESEARCH-verified path.

### A3. **Do NOT touch `chk_users_auth_provider` in `e002_add_saml_columns`**

The downgrade analog at `2026_04_08_0001:88-101` re-adds `'saml'` to **both** CHECK constraints. CONTEXT.md D-02.3 + D-04 are explicit: SAML users get `auth_provider='oauth'` from `find_or_create_oauth_user()`. Only `chk_oauth_providers_type` should be relaxed in `e002`. Mirroring the full downgrade verbatim would over-relax.

### A4. **Do NOT define a discriminated union for `OAuthProviderCreate`**

CONTEXT.md D-12 mentions discriminated unions as the preferred form. RESEARCH §6 explains why this is a trap: existing schema is flat with no nested per-type sub-models; converting to `Union[GoogleProvider, MicrosoftProvider, OIDCProvider, SAMLProvider]` is a much larger refactor that risks breaking existing OAuth admin endpoints. **Use `model_validator(mode='after')` per RESEARCH §6** (the explicit fallback CONTEXT.md D-12 endorses).

### A5. **Do NOT log `idp_certificate` in audit-log diff**

Pitfall 9 (RESEARCH §14): the encrypted PEM would land in audit logs without redaction. The `SECRET_FIELDS = {"idp_certificate", "client_secret_encrypted", "client_secret"}` redaction step in Pattern Detail 10 is mandatory, not optional.

### A6. **Do NOT inherit from `IdentityExtension` Protocol**

Phase 214 discipline: `EnterpriseSamlExtension` satisfies the Protocol structurally (PEP 544 implicit conformance). Adding `class EnterpriseSamlExtension(IdentityExtension):` would force a `from app.core.identity import IdentityExtension` edge from enterprise repo into core's `app.core.*` — violates the Phase 214 layering rule. Match `DefaultIdentityExtension`'s style: no inheritance, just the method.

### A7. **Do NOT generate `metadata` XML by string concatenation**

The scaffold's `_build_idp_metadata_xml()` is fine for the IdP-side metadata input (`config.py:14-34`) because the only tainted input is `certificate`, decrypted from a trusted DB column. But for the **SP-side** metadata endpoint at `GET /auth/saml/{slug}/metadata` (D-08), use pysaml2's `create_metadata_string(configfile=None, config=client.config, sign=False)` — RESEARCH §9 item 4. Hand-rolling SP metadata XML risks XML injection if any provider field contains XML-special chars.

### A8. **Do NOT mix SAML CRUD into `SettingsAuthTab.tsx`**

RESEARCH §7 explicitly considered and rejected option 2 ("Tab inside SettingsAuthTab"): pollutes a community-edition file with enterprise-only logic. The standalone `AdminSamlPage.tsx` keeps gating clean. Reuse the **shape** of `OAuthProvidersSection` (TanStack Query + Dialog + Table + AlertDialog) but in a new `SamlProvidersSection` component.

### A9. **Do NOT add SAML routes to `backend/openapi.json`**

The OpenAPI snapshot is generated from core only (no enterprise overlay loaded at snapshot time). SAML routes registering at runtime via the extension loader is the correct behavior — they intentionally don't leak into the SDK (RESEARCH integration-points / Phase 215 scope). Plan-time `make openapi-check` should pass without changes; if it fails, the wrong test environment is loading the overlay.

### A10. **Do NOT add `provider_type='saml'` rows in non-enterprise test environments**

Pitfall 10 (RESEARCH §14): without `e002` applied, the DB CHECK rejects `'saml'`. Tests that exercise SAML provider creation MUST run after the enterprise migration, OR mock the `OAuthProvider` row creation, OR use a programmatic registry insertion (per the `saml_overlay_registered` conftest fixture).
