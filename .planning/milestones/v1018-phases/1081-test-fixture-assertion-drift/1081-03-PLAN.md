**Diagnostic evidence (read before editing):**

The PYTEST-BASELINE-2026-05-21.md baseline (row #7 of the NEW-DISCOVERY table) captures the exact error: `"Task got Future attached to a different loop"`. This is asyncpg's signature error when a connection pool was bound to one event loop and someone tries to await on it from a different one. It's NOT an InvalidCatalogNameError (that was TI-01, closed in Phase 1075). It's NOT a CannotConnectNowError (that's parallel-mode environmental cap, deferred). It's a per-test event-loop binding issue.

**Why this test is special (vs the other 4 in the file):**

The 4 sibling tests in `test_tasks_common_phase_brackets.py` all request `test_db_session`:
- `test_phase_session_loads_existing_job(test_db_session)` (line 78)
- `test_phase_session_yields_none_when_job_missing()` (line 98) — actually no fixture either, but it does NOT use the helper's internal session for any DB read; it just opens the bracket and yields
- `test_job_phase_session_none_branch_rolls_back_on_exception()` (line 121) — **TD-06 target**
- `test_phase_session_rolls_back_on_exception(test_db_session)` (line 152)
- `test_phase_session_commit_persists_on_normal_exit(test_db_session)` (line 182)

The `test_db_session` fixture (`conftest.py:531-541`) transitively requests `client`, which monkey-patches `db_module.async_session = test_session_factory` (conftest.py:368-369). So the helper's lazy `from app.core.db import async_session` resolves to the TEST-DB-BOUND factory inside any test that pulls `test_db_session`.

**Important nuance:** `test_phase_session_yields_none_when_job_missing` (line 98) ALSO bare-calls the helper without a session fixture, but it does NOT raise an exception inside the `async with` block — it just yields and exits cleanly. So even if the session is on a defunct loop, the failure mode might be a silent "Task pending" warning rather than a hard exception. The TD-06 test (line 121) DOES raise inside the block, which forces the helper's broad-except path (line 232: `await session.rollback()`) — and `session.rollback()` is what surfaces the cross-loop binding as a hard error. So the asymmetric failure is real.

**Why a `db_module.async_session` rebind via the `client` fixture is the right fix (not a fixture-scope tweak to `anyio_backend`):**

- `anyio_backend` is session-scoped (`conftest.py:61-63`). Promoting it to function-scope would create a new loop per test, but it would ALSO churn `_test_db_lifecycle` (session-scoped autouse, lines 134-334) which creates/migrates the test DB. That's a 30+ second per-test overhead → unacceptable.
- A new fixture (`function`-scoped `async_session` rebinder) would duplicate the work the `client` fixture already does. The simpler fix is to request `client` directly.
- The Plan 1080-01 broad-except justification at lines 232/238 of `tasks_common.py` is load-bearing: the `session.rollback()` MUST run before re-raise to avoid asyncpg pool leak. We cannot weaken that path. Fix has to be at the test side.

**Why not change `_job_phase_session`:** Production-code change is OUT of scope per REQUIREMENTS.md TD-06 + the Phase 1080 broad-except justification just landed. The helper's contract is correct. The lazy import is correct (avoids circular import). The session-rollback path is correct (REMED-03 / P2-05 contract). Changing it would also touch Plan 1080's just-landed work.

**Why not promote `anyio_backend` to function scope:** see above — DB churn cost.

**Why not run the test in a subprocess:** xdist-style isolation is heavyweight and inconsistent with the project's "fix-at-root-cause" Plan 1075 protocol.

**The chosen fix is the smallest possible diff:** add `client` as a fixture arg, add a 6-line comment explaining the cross-loop binding root cause and the fix mechanism, run verification including a FULL-SUITE reproducer.
