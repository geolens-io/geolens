<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| anonymous client → /datasets/{id}/export | Untrusted, unauthenticated caller requests a file export; the gate must distinguish public+published (allow) from everything else (deny) |
| authenticated client → /datasets/{id}/export | Authenticated caller; RBAC visibility + `export` capability must both hold |
| anonymous/authenticated client → /collections/{id}/items[/] | Read path; alias must not widen access beyond the existing handler's own auth (`get_optional_user` inside the handler is unchanged) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1157-01 | Information Disclosure | `export_dataset_endpoint` anonymous branch | mitigate | Anonymous path calls `check_dataset_access_or_anonymous` (enforces `visibility=='public' AND record_status=='published'`, raises 404 to hide existence) PLUS a defense-in-depth 403 guard requiring `visibility=='public'`. Unpublished/private/restricted anon export stays denied. This is the inverse risk of EXP-01 and is pinned by the EXP-02 regression test (Plan 02). |
| T-1157-02 | Elevation of Privilege | `export_dataset_endpoint` authenticated branch | mitigate | The `export` capability check is preserved on the `else` (authenticated) branch via `get_effective_permissions` + per-role lookup; a viewer with `export` revoked still gets 403. Removing `require_permission` does NOT drop the capability gate for authenticated callers — it relocates it to the authenticated branch only. |
| T-1157-03 | Denial of Service (over-gating regression) | `export_dataset_endpoint` anonymous branch | mitigate | Risk that the new gate over-denies legitimate public+published anonymous export. Pinned by the EXP-02 positive guard asserting anon export of public+published returns a real body. |
| T-1157-04 | Spoofing / Information Disclosure | port-routed authorization | mitigate | The Phase 1156 GOTCHA — `port.check_dataset_access_or_anonymous` silently AttributeErrors at runtime, which could leave the anon branch unenforced. Mitigated by importing the helper directly from `app.modules.catalog.authorization` and asserting (Task 1 verify) that `port.check_dataset_access_or_anonymous` appears nowhere in the file. |
| T-1157-05 | Information Disclosure | API-01 trailing-slash alias | accept | The alias points at the unchanged `get_collection_items` handler, whose own `get_optional_user` + dataset-access checks are unchanged. The alias adds a URL shape, not an access path; no new authorization surface. Low risk. |
| T-1157-SC | Tampering | npm/pip/cargo installs | accept | No package-manager installs in this plan (existing-files-only constraint; no new deps). No `[ASSUMED]`/`[SUS]` packages introduced. |
</threat_model>
