<objective>
Three audit findings live in the same code path (the backend `run_ogr2ogr_service` subprocess + the seeder script that surfaces its errors). Closing them in one plan keeps the GDAL-related changes co-located and gives a single regression-test target.

SEED-02 raises the GDAL HTTP timeout from a hardcoded 120s to a configurable value (env-driven, default 300s — 2.5× the audit's observed bottleneck) and adds a per-layer single-retry-with-doubled-timeout in the seeder so the audit's `2/4 success` rate becomes `4/4` for layers that fit inside a 600s budget. SEED-03 adds a `Data quality skips: N` count in the Import Summary so AGO upstream noise is summarized rather than implicit. SEED-04 strips ogr2ogr's 150-line GDAL driver enumeration from `stderr` before raising IngestionError, so seeder error output shows only the actionable line.

Purpose: A new user running `seed-ago-data.py` immediately after `bash scripts/install.sh` sees a high-quality run output — no 120s timeouts on common AGO layers, no 150-line driver dumps when something does fail, and a one-line summary of how many AGO services were skipped due to upstream data quality.

Output: Updated backend ingest path + seeder script + one regression test file.

The 120s timeout the audit observed comes from the backend, not the seeder. From `backend/app/processing/ingest/ogr.py:497-519`, the ogr2ogr command is built with:

    cmd = [
        "ogr2ogr", "-f", "PostgreSQL", db_conn_str, gdal_source, "-overwrite",
        "-nln", f"data.{table_name}", "-lco", "FID=gid", "-lco", "PRECISION=NO",
        "--config", "PG_USE_COPY", "YES",
        "--config", "GDAL_HTTP_TIMEOUT", "120",  # ← the audit's 120s
    ]

`GDAL_HTTP_TIMEOUT` is GDAL's per-HTTP-request timeout (curl's `CURLOPT_TIMEOUT`). The subprocess-level kill-on-hang timeout is the separate `timeout: float = 1800.0` parameter of `run_ogr2ogr_service` — that's the 30-minute wall-clock killswitch and is NOT what's firing in the audit.

Error path at `ogr.py:561-564`:

    if proc.returncode != 0:
        raise IngestionError(
            f"ogr2ogr failed (exit {proc.returncode}): {stderr.decode().strip()}"
        )

`stderr.decode()` includes ogr2ogr's full driver enumeration before the operational error — 150+ lines like `-> 'FITS' -> 'PCIDSK' -> 'netCDF' ...` per the audit's SEED-04 finding.

Settings module at `backend/app/core/config.py` uses Pydantic Settings; the established add-a-knob pattern (around line 95-122) is:

    field_name: type = default_value

with optional `description=` for docs. New env vars are auto-resolved by Pydantic from `FIELD_NAME` uppercase. The "ingest" prefix is appropriate — there's no existing OGR/GDAL prefix in this file.

Seeder retry shape at `scripts/seed-ago-data.py:692-758`:

    async with sem:
        try:
            for attempt in range(1, MAX_RETRIES + 1):
                try:
                    # ... ingest_via_service / update_via_service ...
                    if result.get("status") == "failed":
                        raise RuntimeError(result.get("error_message", "Unknown ingest error"))
                    # ... success path ...
                    break  # success, exit retry loop
                except httpx.HTTPStatusError as exc:
                    if exc.response.status_code >= 500 and attempt < MAX_RETRIES:
                        # retry with backoff
                        continue
                    raise
        except Exception as exc:
            results.append({"name": layer_name, "status": "failed", "error": str(exc)})

The retry loop only catches `httpx.HTTPStatusError` for 5xx. The audit's two timeouts came back as `RuntimeError` from line 729-731 (the `result["status"] == "failed"` branch) and bypassed retry entirely.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| seeder → GeoLens API | seeder is run by admin user with API key; API trusts the key |
| GeoLens API → ogr2ogr subprocess | subprocess-level; ogr2ogr runs in worker container with PG conn string |
| ogr2ogr → upstream AGO service | network-layer; AGO is untrusted; GDAL handles the HTTP |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1054-01-DOS | Denial of Service | `run_ogr2ogr_service` | accept | A larger GDAL_HTTP_TIMEOUT (300 vs 120) extends the window during which a slow upstream service can hold the subprocess. The wall-clock kill via `_communicate_with_timeout(timeout=1800)` still bounds total runtime. Worst case: 6× longer-running subprocesses on slow networks; not a DoS amplifier because each ingestion is admin-triggered, not user-input-driven. |
| T-1054-01-INFO | Information Disclosure | `_strip_ogr_driver_list` | mitigate | Driver-list stripping is conservative — only removes lines that match the strict `-> 'NAME' (modes)` shape. Cannot accidentally strip an attacker-controlled error message, because the regex requires the leading `-> ` pattern that GDAL emits. If a future GDAL version changes the format, the worst case is that nothing gets stripped and we revert to today's behavior — never the inverse (stripping real error content). |

No new external dependencies. No npm/pip/cargo installs. No new endpoints or auth surfaces.
</threat_model>
