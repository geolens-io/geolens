<objective>
The audit's IMPORT-05 (LOW) finding: with 109+ seeder-created tables all auto-registered via the API at seed time, the Register Table tab shows `No unregistered tables found in the database.` — technically correct but framed as a problem rather than a success.

Per CONTEXT.md's locked decision: "Conditional copy: when backend returns 0 registrable tables (with 'all tables already registered' semantically) → success message; when actually empty/no-postgis → absence message."

The discriminator: if the backend's `useDiscoverTables` returns `tables.length === 0`, that's the empty state. To tell whether it's "no tables in DB" vs "all tables registered", we need a second signal. Options:

(a) Add a backend field to `DiscoverTablesResponse` (e.g. `total_registered_datasets: int`) that counts existing datasets from `data.*` schema sources. Frontend then renders the success variant when `tables.length === 0 && total_registered_datasets > 0`.

(b) Reuse an existing public datasets count via the `/api/datasets/` list endpoint with `limit=1` to read the total count from `numberMatched`. Frontend fires this conditionally only when `tables.length === 0`.

Path (b) is preferred — no backend change, smaller blast radius, uses existing read-only endpoint that's accessible to anyone with `upload` permission (the same permission required to see the Register Table tab anyway). The extra GET is cheap and only fires in the empty-state branch.

Purpose: Close IMPORT-05 — reframe the Register Table empty state from absence to success when the catalog is already fully populated.
Output: Updated `RegisterForm.tsx` (conditional render of two empty-state variants) + 4-locale i18n keys + regression test.
</objective>
