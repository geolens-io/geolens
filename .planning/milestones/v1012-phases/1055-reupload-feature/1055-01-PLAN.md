<objective>
Close the cross-record-type swap gap in the existing reupload backend.

Purpose: CONTEXT.md locked the decision that vector→raster and raster→vector swaps must return 400 ("too many downstream invariants break"). The shipped backend at `backend/app/modules/catalog/datasets/api/router_reupload.py` validates the file *extension* against `allowed_extensions` but does NOT validate that the file extension is *compatible with the dataset's record_type*. A user can upload a `.tif` to a vector dataset today — extension validation passes (`.tif` is in the global allowed list for raster uploads), the Celery worker runs ogr2ogr against a GeoTIFF, and either fails opaquely or worse, succeeds with garbage. The audit's outstanding decision (CONTEXT.md `<decisions>` block, sub-bullet 1) is unresolved in code.

Output: A `_assert_compatible_record_type` guard called from both `reupload_dataset` and `request_presigned_reupload` that returns HTTP 400 with a clear message when the file extension is incompatible with the dataset's record_type. New tests pin the behavior in both directions.
</objective>
