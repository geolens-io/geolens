## Interaction Contract

This section is the primary deliverable for Phase 1040 — it defines every visual state and interaction rule the executor must implement.

### 1. Catalog row drag affordance (POL-01)

**Trigger:** User hovers a dataset row in `DatasetSearchPanel`.

**Visual changes on hover:**
- Grab handle (`GripVertical` icon, lucide-react, `h-3.5 w-3.5`) appears at the left edge of the row, INSIDE the existing row layout before the type-icon.
  - At rest: opacity 0 (hidden; do not reserve column space — modal rows use a simpler layout than stack rows)
  - On row hover: opacity 35% (matches `StackRow.tsx:183` pattern: `opacity-35 group-hover/row:opacity-70`)
  - During drag (row being dragged): opacity 70%
- Cursor changes to `cursor-grab` on the entire row (not just the handle). Source: `StackRow.tsx:183`.
- During active drag of THIS row: cursor changes to `cursor-grabbing` on the dragged element.

**Decision (Claude's discretion):** Modal rows use a simpler row anatomy than stack rows (no eye, no opacity slider, no kebab, no caret column). The grab handle is the ONLY new affordance added to the left side. Column reserved space is NOT added — modal rows are not a grid, they are flex rows. The handle floats at the left edge inside the existing row padding.

### 2. Drag-in-flight state (the ghost / drag overlay)

**When a catalog row is picked up:**
- The source row in the modal becomes semi-transparent: `opacity: 0.4`, `background: var(--surface-2)` (matches sketch 007 `.row.dragging` spec).
- A `DragOverlay` ghost appears attached to the pointer. The ghost renders as a simplified pill/card showing only the type-icon swatch + dataset name, no interactive controls.
  - Ghost background: `var(--surface-2)`; border: `1px solid var(--border)`;  border-radius: `var(--radius-md)`; box-shadow: `0 4px 12px oklch(0 0 0 / 15%)` (light elevation).
  - Ghost height: min 36px; padding: `8px 12px`; max-width: 260px; text overflow ellipsis.
  - Ghost cursor: `cursor-grabbing`.
- `document.documentElement.classList.add('dragging-active')` — exactly the same pattern already wired in `UnifiedStackPanel.tsx:527`. All `.kebab` controls hide in the stack panel (`opacity: 0 !important`) per the existing `dragging-active` class (AUD-03 notes this rule is missing from index.css — Phase 1042 owns that fix, but this phase MUST add it if targeting the catalog-drag flow).

**Decision (Claude's discretion):** The ghost does NOT replicate the full 7-column row anatomy — that would be visually complex and out of context. A compact pill with type-icon + name is sufficient to communicate "this dataset is moving."

### 3. Stack drop targets during catalog drag

The unified stack (`UnifiedStackPanel`) must detect when a catalog drag (from outside the stack) is in progress and activate drop targets.

#### 3a. Drop between loose rows (reorder / insert position)

**Visual:** 2px insertion line between rows.

```
.insert-line {
  height: 2px; margin: -1px 0;
  background: var(--color-primary);
  border-radius: var(--radius-full);
  box-shadow: 0 0 0 2px oklch(0.55 0.18 250 / 25%);  /* 25% bloom */
  pointer-events: none;
}
```

Insertion position: above or below a row based on pointer Y vs row midpoint. Source: sketch 007 (locked). This insertion line component already exists from v1008 intra-stack reorder; the executor must confirm it activates for cross-DnD-context catalog drags, not only intra-stack drags.

#### 3b. Drop onto a folder-group row (POL-03)

**Visual when folder-group row is the active drop target:**
- Group row background → `var(--primary-50)` (same as `.row.selected`)
- Group row left rail → `box-shadow: inset 2px 0 0 var(--color-primary)` (same as `.row.selected`)
- If group is expanded, children container also gets `background: oklch(0.97 0.02 250 / 60%)` wash + `border-radius: var(--radius-md)`

No insertion line when a group row is the active drop target — the tint IS the affordance.

**Drop behavior:** On drop, `parent_group_id` is set to the group's id. The layer inserts at the end of the group's children. Source: POL-03.

#### 3c. Drop onto the basemap group row (POL-04)

**Visual:** Same as 3b (primary-50 tint + left rail on the basemap group row header).

**Constraint:** The basemap group row accepts drops ONLY from basemap-typed rows in the modal (tab: `basemap`). Dropping a vector or raster row onto the basemap group → show no drop affordance; cursor: `not-allowed` (via `e.dataTransfer.dropEffect = 'none'`).

**Drop behavior:** On drop, call `onBasemapChange(entry.id)` (the existing callback in `DatasetSearchPanel.tsx:287`). Do NOT create a new overlay layer. Source: POL-04, CONTEXT.md.

#### 3d. Drop rejected

Dropping a non-basemap row onto the basemap sublayer container or onto individual basemap sublayer rows: no affordance, cursor `not-allowed`. These are the cross-stack constraints flagged in sketch 007 README.

### 4. Keyboard fallback (POL-05 / POL-23)

Phase 1044 owns the full a11y audit, but Phase 1040 MUST wire the keyboard API so it is not retrofitted. The required keyboard path:

| Key | Action |
|-----|--------|
| Space (on a dataset row) | Pick up / confirm drop (toggles drag mode) |
| ArrowUp / ArrowDown | Move the "floating" layer up/down in the stack while in keyboard-drag mode |
| Enter | Confirm drop at current position |
| Escape | Cancel drag, return layer to origin |

Implementation: `@dnd-kit` keyboard sensor (`KeyboardSensor`) with `coordinateGetter: sortableKeyboardCoordinates`. Source: POL-23 (a11y), CONTEXT.md specifics note.

ARIA: When keyboard drag is active, announce position via `aria-live="polite"` region reading "Layer {name} — position {n} of {total}" on each arrow key press.

### 5. Modal stays open (POL-05)

On successful drag-drop:
- The `DatasetSearchPanel` modal remains mounted and open. No `onClose()` call.
- A toast fires: `toast.success(...)` using the existing `sonner` import pattern from `BuilderMap.tsx:4`. Deduplication key: `add-layer-${datasetId}`.
- The stack row for the newly-added layer appears at the drop position with a brief entry animation: `opacity: 0 → 1` over 150ms (`transition-opacity duration-150`).

### 6. Global drag-active state rules

While any catalog drag is in progress:
- `document.documentElement.classList.add('dragging-active')` — matches existing UnifiedStackPanel pattern.
- `cursor: grabbing` on `html` body to prevent cursor flicker.
- All stack row kebab controls: `opacity: 0 !important` (must be in index.css under `.dragging-active .kebab` — AUD-03 finding; this phase must add this rule since it is the first phase introducing catalog→stack drag).

### 7. Drop animation

On successful drop:
- Insertion line or group tint fades out: `opacity: 0` over 100ms.
- The `DragOverlay` ghost dismisses with `dropAnimation={null}` (matches existing UnifiedStackPanel pattern at `UnifiedStackPanel.tsx:794`).
- The new stack row slides in from opacity 0.

## Empty / Error States for This Phase

This phase does NOT introduce new empty or error states of its own. The catalog modal's existing error state (AUD-11) is out of scope (Phase 1043 owns it). However:

- **Drop with no valid target:** No toast, no error. The drag is simply cancelled (ghost returns). Standard DnD non-drop behavior. No copy needed.
- **Add-layer mutation failure on drop:** Surface `toast.error(t('builder:toasts.addLayerFailed', { defaultValue: 'Failed to add layer — try again' }))`. Matches error tone in `BuilderMap.tsx:400`.

## What to Avoid

Source: sketch 007 rejected variants + BUILDER-UX-AUDIT findings.

- **Auto-expand collapsed groups on hover dwell (Variant C).** Rejected in sketch 007. If a user drags onto a collapsed group and wants to insert at a specific child position, the right path is to first expand the group, then drag. No 500ms auto-expand.
- **Bordered container drop target (Variant B).** Rejected in sketch 007. The primary-50 tint + left rail (Variant A) re-uses existing selection vocabulary; a 2px border adds too much visual weight to the drag-over target and breaks row rhythm.
- **Explicit footer drop pad on groups.** Rejected in sketch 007 (Variant C). The drop target is the group row and its children container — no persistent "drop here" pad.
- **Closing the modal on drag-drop.** POL-05 explicitly requires the modal to stay open.
- **New design tokens.** v1009 hard constraint: all work uses existing `sketch-findings-geolens` token set.
- **New backend endpoints.** Hard constraint from CONTEXT.md: use existing `handleAddDataset` / `onBasemapChange` callbacks.
- **Drag-to-reorder within the modal results list.** Explicitly listed as a Future Requirement in REQUIREMENTS.md — not in scope.
- **Drag a group onto another group (nesting).** Explicitly flagged as non-goal in sketch 007 README. Implementation must reject the drop.
- **Basemap sublayer escape from basemap group.** Sublayer rows (`SublayerRow` in `UnifiedStackPanel.tsx:340`) are drag-disabled (`disabled: true`). The catalog modal does not surface sublayer rows.
