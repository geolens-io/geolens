<objective>
Lift the `@dnd-kit` `DndContext` out of `UnifiedStackPanel.tsx` and into `MapBuilderPage.tsx` so that draggable items inside the Add Dataset modal (Plan 02) and droppable rows inside the unified stack live in the SAME DnD context. Without this lift, cross-panel drag from the modal to the stack is architecturally impossible — `@dnd-kit` collision detection only fires within one DndContext.

This is a pure refactor: behavior of the intra-stack reorder, the dragging-active class toggling, and the DragOverlay ghost MUST stay identical. The plan also lands two AUD findings that are blocking for the catalog-drag path: AUD-03 (the missing `.dragging-active .kebab` CSS rule) and AUD-04 (replace the no-op `useSortable` on the basemap row with `useDroppable` so it can be a drop target).

Purpose: Foundational structural change required by every subsequent plan in this phase. The catalog row useDraggable, the cross-context drop handler, the group drop tint, and the basemap swap-on-drop all need one shared context.

Output:
- `MapBuilderPage.tsx` owns sensors + DndContext + handleDragStart/End/Cancel + activeId state.
- `UnifiedStackPanel.tsx` is reduced to: SortableContext over layer ids, row rendering, DragOverlay (still owned here because it needs row layout knowledge).
- `index.css` carries the new `.dragging-active .kebab` rule and `html.dragging-active { cursor: grabbing }` rule.
- Basemap group row is wrapped in `useDroppable` (not `useSortable`) and exposes `data-basemap-drop-target="true"` when `isOver`.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser DOM ↔ DnD state | Drag events come from untrusted DOM but only mutate local React state; no backend write occurs in this plan |
| @dnd-kit context boundary | Lifting the context expands the surface that listens to drag events; risk = unrelated DOM nodes inside the lifted scope inadvertently registering as draggable/droppable |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1040-01 | Tampering | UnifiedStackPanel intra-stack reorder | mitigate | Preserve activationConstraint `distance: 8` and `closestCenter` collision detection verbatim; assert via existing builder vitest suite that reorder still works |
| T-1040-02 | Denial of Service | DndContext sensor leak across re-renders | mitigate | Use `useSensors` + `useMemo` for sensor list at MapBuilderPage level so sensor identities are stable across renders |
| T-1040-03 | Information Disclosure | Lifted DnD state | accept | DnD events expose only layer ids the user already owns via the map; no cross-user risk |
</threat_model>
