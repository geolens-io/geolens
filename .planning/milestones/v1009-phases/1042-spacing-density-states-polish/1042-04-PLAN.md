## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| build-time JSON author -> runtime i18next | builder.json is bundled at build time; no runtime input crosses any boundary. |
| client browser <- React component tree | All component edits are presentation-layer + a single setTimeout state lifecycle. No new request boundary, no untrusted input sink. |
| pointer/keyboard input -> useDraggable / handleAddDataset | freshLayerId is set in response to the existing handleAddDataset callback; no new event surface. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1042-04-01 | T (Tampering) | builder.json dedup | accept | JSON file is bundled at build time; deletion of duplicate keys cannot introduce inconsistent runtime behavior because i18next would have non-deterministically picked one of the two duplicates anyway. The dedup makes behavior deterministic — net safety improvement. |
| T-1042-04-02 | I (Information Disclosure) | freshLayerId state | accept | freshLayerId is a UUID that is already visible in the layer list; no new disclosure. |
| T-1042-04-03 | D (Denial of Service) | setTimeout-based freshLayerId reset | mitigate | Per-add timeout could theoretically leak if user spams add. Mitigation: timeout ref clears any pre-existing timer before scheduling a new one (single-flight); useEffect cleanup cancels timer on unmount. Worst case is N adds = 1 active timer. |
| T-1042-04-04 | E (Elevation of Privilege) | StackRow isFresh prop | accept | isFresh is a boolean className flag; no privilege change. |
</threat_model>
