CONTEXT D-08 / D-09 lock the bulk-button alternative over a per-layer "+ add another" button. CONTEXT D-10 locks the naming convention `{filename}: {layer_name}` (collision-safe via existing `generate_table_name` logic at `tasks_vector.py:144`). CONTEXT D-11 locks Promise.allSettled + results modal with retry affordance.

Independent of Plans 1058-01 and 1058-02: this plan touches `BulkReviewList.tsx` and `UploadForm.tsx` exclusively — zero file overlap with `ReuploadDialog.tsx`. Hence Wave 1 (parallel with Plan 1058-01).

- New "Ingest all layers" button rendered inside the expanded row for a multi-layer entry.
- Fan-out handler in `UploadForm.tsx` that issues N parallel `commitImport` calls with a concurrency cap of 4 (security mitigation per CONTEXT threat-model note).
- Results modal showing per-layer status (succeeded / failed) after the fan-out settles.
- Single-layer entries unchanged (no new affordance, no new behavior).

Concurrency-pool helper (small, self-contained). Implement INLINE in UploadForm.tsx
(not a separate utility file unless the executor judges otherwise):
```typescript
async function runWithConcurrency<T, R>(
  items: T[],
  worker: (item: T) => Promise<R>,
  concurrency: number,
): Promise<PromiseSettledResult<R>[]> {
  const results: PromiseSettledResult<R>[] = new Array(items.length);
  let nextIndex = 0;
  const runners = Array.from({ length: Math.min(concurrency, items.length) }, async () => {
    while (true) {
      const i = nextIndex++;
      if (i >= items.length) return;
      try {
        results[i] = { status: 'fulfilled', value: await worker(items[i]) };
      } catch (err) {
        results[i] = { status: 'rejected', reason: err };
      }
    }
  });
  await Promise.all(runners);
  return results;
}
```

| Threat ID | Category | Component | Disposition | Mitigation Plan |

| T-1058C-01 | Denial of Service | Unbounded parallel commits hammering /import/upload/commit | mitigate | Concurrency cap of 4 in `runWithConcurrency` helper (CONTEXT D-15 threat model). A 10-layer GPKG sends 4 in-flight at peak, not 10. For a 100-layer pathological file (impossible to upload through the existing 100MB upload limit anyway), the cap still applies. |
| T-1058C-02 | Tampering | `layer_name` injection via fan-out | mitigate | `layer.name` originates from `entry.previewData.layers` — populated server-side by `_extract_common_layer_metadata` from ogrinfo JSON output. User does not directly write this field; the only attack vector would be a malicious GPKG with embedded layer names, which is identical to the existing single-layer reupload's `layer_name` attack surface (covered by Plan 1058-01's T-1058A-02 disposition). No new exposure. |
| T-1058C-03 | Race condition | Two layers in the same fan-out reuse the same `entry.jobId` | mitigate | Backend `/import/upload/commit` for vector files dispatches a single Procrastinate task per call. Two concurrent commits against the same job_id may either succeed-twice (creating two separate datasets) or fail-one (race on the IngestJob.status check). Mitigation: backend already updates `job.status` to `'queued'` BEFORE returning from commit, and re-checks `job.status == 'pending'` (existing logic at router_reupload.py:390). Behavior under fan-out: the FIRST commit wins; subsequent commits return 400 "Job already processed". **Resolution:** the fan-out shape requires the backend to permit multiple commits against the same job_id with different `layer_name` values. **VERIFICATION REQUIRED:** confirm during execution that `/import/upload/commit` supports this pattern. If it does NOT, the executor must either (a) re-upload the file N times to get N distinct job_ids, OR (b) add backend support in a separate task. Either way, this is documented at execution time, not pre-baked into the plan. **Default assumption per CONTEXT D-08:** fan-out works against the existing endpoint; backend changes are out of scope. If runtime reveals this is wrong, escalate via Plan SUMMARY's "Deviations from Plan" section and update the plan in revision mode. |
| T-1058C-04 | Information Disclosure | Results modal shows error messages from rejected commits | accept | Error messages are derived from backend responses, which already redact sensitive details (no stack traces, no secrets).
