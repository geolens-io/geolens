# Plan 1058-04 — Backend Fan-Out for GPKG-03

## Objective

Close GPKG-03 acceptance by adding a single backend endpoint that converts one IngestJob with multiple selected layers into N independent ingest tasks. The Plan 1058-03 UI fan-out cannot succeed today because `/ingest/commit/{job_id}` rejects commit N+1 with `400 Job already processed`. This plan adds the missing backend path; the Plan 03 UI is rewired to call it instead of looping the single-commit endpoint.

<action>
Add new endpoint to `backend/app/processing/ingest/router.py`:

`POST /ingest/commit-fan-out/{job_id}` — accepts a JSON body matching `FanOutCommitRequest`:

```yaml
job_id: UUID  # path param (existing upload job)
layers:       # body list
  - layer_name: str
    title: str | None  # optional override; defaults to "{file_basename}: {layer_name}"
  - layer_name: str
    title: str | None
  ...
```

Returns `FanOutCommitResponse`:

```yaml
fan_out_id: UUID  # the original job_id (kept for client correlation)
results:           # list per layer in request order
  - layer_name: str
    new_job_id: UUID  # the cloned IngestJob ID
    dataset_id: UUID  # the new dataset row
    status: "queued"  # all start as queued; client polls /jobs/{new_job_id}
  - ...
```

Endpoint behavior:
1. Resolve the original IngestJob by `job_id`. 404 if missing. 400 if `job.status != "pending"`.
2. Validate every `layer_name` in the request appears in `job.user_metadata.all_layers`. If any missing → 422 with the list of unknown names.
3. For each requested layer (sequentially is fine — Plan 03 already concurrency-bounded the frontend):
   - Call `service.create_fan_out_jobs(original_job, layer)` (new helper, see Task 2)
   - Append result to response list
4. Mark `job.status = "fanned_out"` and `job.completed_at = now()` (terminal state — distinguishes from "completed" which means a single committed dataset).
5. Commit DB session, return 202 Accepted with the response payload.

Permission: same as `/ingest/commit/{job_id}` — `require_permission("register")`.

Pydantic models go in `backend/app/processing/ingest/schemas.py`:
- `FanOutLayerRequest(BaseModel)` — `layer_name: str = Field(min_length=1, max_length=500)`, `title: str | None = None`
- `FanOutCommitRequest(BaseModel)` — `layers: list[FanOutLayerRequest] = Field(min_length=1, max_length=50)` (cap fan-out at 50 layers per request as a sanity guard)
- `FanOutLayerResult(BaseModel)` — `layer_name: str`, `new_job_id: UUID`, `dataset_id: UUID`, `status: Literal["queued", "failed"]`, `error: str | None = None`
- `FanOutCommitResponse(BaseModel)` — `fan_out_id: UUID`, `results: list[FanOutLayerResult]`
</action>

<acceptance_criteria>

<action>
Implement `service.create_fan_out_jobs(original_job: IngestJob, layer: FanOutLayerRequest, session: AsyncSession) -> FanOutLayerResult`:

1. Determine the new dataset title:
   - `title = layer.title if layer.title else f"{original_job.source_filename or 'dataset'}: {layer.layer_name}"`
2. Generate a unique `table_name` via existing `generate_table_name(title, session)` — handles slug collisions
3. Create a new `IngestJob` row:
   - `file_path = original_job.file_path` (same file on disk)
   - `source_filename = original_job.source_filename`
   - `status = "pending"`
   - `user_metadata = { ...original_job.user_metadata, layer_name: layer.layer_name, title, fan_out_parent_id: original_job.id }`
   - DO NOT copy `dataset_id` — fan-out creates new datasets
4. Create a new `Dataset` row:
   - `title = title`
   - `slug = derived from table_name`
   - `source_layer = layer.layer_name` (preserves the existing source_layer convention)
   - Other defaults per existing dataset creation pattern
5. Link new IngestJob to new Dataset (`job.dataset_id = new_dataset.id`)
6. Dispatch the existing Procrastinate task: `commit_ingest_job.defer_async(job_id=new_job.id, ...)`
7. Return `FanOutLayerResult(layer_name=layer.layer_name, new_job_id=new_job.id, dataset_id=new_dataset.id, status="queued")`
8. On error: catch, log, return `FanOutLayerResult(..., status="failed", error=str(exc))`

Caveat: this MUST NOT touch the file on disk; multiple ingest tasks read the same `file_path` and `ogr2ogr` is read-only on the source. If file cleanup happens after the first task completes, all fan-out tasks would lose their source. Verify that file cleanup is gated by `original_job.id` (not the per-fan-out job IDs); if not, defer file cleanup to a separate sweep keyed on `fan_out_parent_id`.
</action>

## Threat Model (ASVS L1)

| ID | STRIDE | Vector | Mitigation | Status |
|---|---|---|---|---|
| T-1058D-01 | Tampering | User submits layer_name not in job's `all_layers` | 422 with explicit unknown-layer list; no subprocess invocation | mitigated |
| T-1058D-02 | DoS | User submits 10000 layer names in one fan-out | `max_length=50` cap on layers list | mitigated |
| T-1058D-03 | Spoofing | User specifies title="<script>alert(1)</script>" | Title flows into `Dataset.title` which is rendered with React JSX escaping; no SSR template injection | mitigated |
| T-1058D-04 | Info disclosure | Per-layer error messages leak internal paths | `FanOutLayerResult.error` is `str(exc)` — verify exception messages are user-safe; wrap with `_user_safe_error()` if needed | verify-at-impl |
| T-1058D-05 | Privilege escalation | User without `register` permission can fan-out | Endpoint inherits `require_permission("register")` from single-commit path | mitigated |
| T-1058D-06 | Repudiation | Fan-out result is not auditable | `fan_out_parent_id` in each new job's user_metadata + `status="fanned_out"` on the original job provides full traceability | mitigated |
| T-1058D-SC | Supply chain | No new dependencies | N/A | N/A |

Block on: T-1058D-04 must be verified at implementation time. All others mitigated by design.

## Out of Scope

- Per-layer commit rollback / atomic fan-out (Plan 03 explicitly accepts partial-success outcome modal as the UX)
- Multi-file fan-out (one IngestJob = one file; multi-file requires N uploads)
- Backend-side concurrency limit on the per-layer task dispatch (Procrastinate handles task queue concurrency at the worker level)
