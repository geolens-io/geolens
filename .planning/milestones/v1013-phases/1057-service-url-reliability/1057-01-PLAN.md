<objective>
Fix WFS-04 (P0): User can import polygon-heavy WFS layers declaring abstract OGC geometry types (MultiSurface, MultiCurve, CompoundSurface, CompoundCurve, GeometryCollection) end-to-end without the post-ingest bounds-clip UPDATE failing on a PostGIS type mismatch.

Purpose: Per D-01, replace column-level subtype discipline with a generic `geometry(Geometry, 4326)` column on the service-ingest path. The `-nlt PROMOTE_TO_MULTI` flag at `backend/app/processing/ingest/ogr.py:575` causes ogr2ogr to declare the column with the abstract type emitted by GeoServer (e.g., `MultiSurface`), then the bounds-clip UPDATE at `backend/app/processing/ingest/metadata.py:934` fails because feature geometries are concrete (`MultiPolygon`). Dropping `-nlt` (or replacing with `CONVERT_TO_LINEAR`) yields a constraint-free `geometry` column where concrete subtypes coexist freely; `get_geometry_type` at `metadata.py:165` continues to derive the concrete type via first-feature inspection, so `Dataset.geometry_type` and downstream UX are unchanged.

Output: Updated `run_ogr2ogr_service` argv list + regression test fixture.
</objective>

GDAL flag semantics (verified):
- Dropping `-nlt` entirely makes ogr2ogr emit the source-declared subtype into the column declaration → this is the current bug.
- `-nlt PROMOTE_TO_MULTI` promotes singletons to multis → not the fix; still subtype-constrained.
- `-nlt CONVERT_TO_LINEAR` converts curve geometries (CompoundCurve, CurvePolygon) into linear approximations, but DOES NOT remove subtype constraint — only useful when source has parametric curves.
- `-nlt GEOMETRY` (or `-nlt PROMOTE_TO_MULTI` combined with `-nlt GEOMETRY` flag override): emits a generic `geometry` column with no subtype constraint. **This is the canonical GDAL flag for "no subtype constraint".**
- Executor should verify against `ogr2ogr --help -nlt` that `GEOMETRY` is the correct token; if not, `-nlt CONVERT_TO_LINEAR` is the documented fallback in CONTEXT.md D-01, OR drop `-nlt` entirely and rely on column-emission default. Pick the option that produces `"f_geometry_column = 'geom' AND type = 'GEOMETRY'"` (or equivalent constraint-free state) in PostGIS `geometry_columns`.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| WFS/ArcGIS/OGC service → backend HTTP fetch | Remote-controlled XML/JSON; SSRF already gated upstream by `validate_url_for_ssrf` |
| User-supplied `layer_name` → ogr2ogr subprocess argv | User-controllable layer identifier embedded in subprocess argv |
| ogr2ogr stdout/stderr → backend logs | Subprocess output may include service-supplied error text |

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1057A-01 | Tampering | `run_ogr2ogr_service` argv composition | mitigate | argv is a Python list passed to `asyncio.create_subprocess_exec` (NOT a shell string); no shell metacharacter expansion possible. Verified at ogr.py:595. No change in this plan. |
| T-1057A-02 | Information Disclosure | ogr2ogr stderr decoded into IngestionError message | mitigate | Existing `_strip_ogr_driver_list` (ogr.py:608) sanitizes driver-list noise. The asyncpg error path (the bug this plan fixes) surfaces a structured error from clip_to_mercator_bounds, not arbitrary remote content. No change. |
| T-1057A-03 | Denial of Service | Constraint-free geometry column accepting any subtype | accept | A generic `geometry` column accepts any valid PostGIS geometry, so an upstream service emitting unexpected subtypes (e.g., GeometryCollection) does not crash ingest — this is the desired property per D-01. `get_geometry_type` derives the concrete subtype post-ingest for UX purposes. No new attack surface; the existing PostGIS-level validity check via `ST_IsValid` (run elsewhere in the pipeline) remains the gate. |
