<objective>
Rename the ~64 `widget`-namespaced i18n keys to the `plugin` namespace across all 8 locale files (en/es/fr/de × builder.json/admin.json), translating the user-visible values appropriately per locale, while keeping full 4-locale key parity and preserving the plugin **ID literals** `measurement`/`legend`.

This is the first half of **I18N-01** (the only requirement in Phase 1163). It deliberately does NOT touch any `t()` call site — that is Plan 02 (wave 2), which depends on these renamed keys existing first. Renaming the keys here without the call-site update in Plan 02 would temporarily break i18n lookups at runtime; that is why Plan 02 runs in the same phase immediately after and the phase is only complete when BOTH plans are done.

Purpose: Give the i18n surface an honest name consistent with the already-renamed backend (`maps.plugins`), API (`plugins`/`enabled_plugins`), and frontend identifiers (`Plugin*`, `map-plugins/`).
Output: 8 modified locale JSON files; zero `widget` in locale files; parity preserved; valid JSON.
</objective>

## Value-translation rule (swap the noun, keep the loanword "Plugin"/"Plugins")

"Plugin"/"Plugins" is an accepted loanword in es/fr/de. For each renamed VALUE, replace only the widget-noun token inside the existing localized phrasing — do NOT re-translate the whole sentence:
- "widget" (lowercase, singular) → "plugin"; "Widget" (capitalized) → "Plugin"; "widgets"/"Widgets" → "plugins"/"Plugins"; "WIDGETS" → "PLUGINS".
- Values with NO widget word (`{{count}} enabled` and its localized forms `activados`/`activés`/`aktiviert`; `Enable/Disable {{name}}` and localized forms) keep their value BYTE-IDENTICAL — only the KEY changes.

Concrete resulting values:
- en mapPlugins "Map plugins"; es "Plugins del mapa"; fr "Plugins de carte"; de "Karten-Plugins"
- badges en "{{count}} plugin"; es/fr "{{count}} plugin"; de "{{count}} Plugin"
