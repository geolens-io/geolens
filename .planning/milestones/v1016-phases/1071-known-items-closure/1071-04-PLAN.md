## Current behavior
The mint endpoint at `POST /auth/download-token/{dataset_id}` issues a JWT
with `typ='download'`, `scope='dataset:{id}'`, and TTL ≤ 120s. For
authenticated callers it includes a `sub=<user-id>` claim; for anonymous
callers on a public dataset it OMITS sub entirely.

The consumer `_resolve_download_user` decodes the token, validates
`typ`/`scope`/`exp`, then attempts a User lookup using `payload.get("sub")`.
When sub is missing it falls through to the unconditional 401 at line 215.

The dependency that calls `_resolve_download_user` is `download_cog` at line
221, which is typed `user: Identity = Depends(...)` — meaning a User object
is required by the downstream code. The audit log at line 277-290 uses
`user.id`. The visibility check at line 256 (`check_dataset_access`) wants
a User to be present.

## What "anonymous download" must mean
1. Public dataset: visibility is "public" — no User context is needed for
   the access check.
2. Audit emit: `user_id=None` is structurally accepted by AuditEvent (the
   ORM column is nullable for SAML/JIT pre-provisioning rows; confirm by
   reading `backend/app/modules/audit/service.py` if uncertain).
3. The download_cog function should not reach the `user.id` line if user
   is None — branch the audit emit on `user is not None`.

## Two viable implementation shapes — executor chooses
**Shape A — accept Identity | None in download_cog:**
- Change `_resolve_download_user` return type to `Identity | None`.
- Change `download_cog`'s parameter to `user: Identity | None = Depends(_resolve_download_user)`.
- Branch the audit emit and the permission check on user truthiness.
- For the visibility check, change `check_dataset_access` call to
  `check_dataset_access_or_anonymous` (already used in router_export.py:116;
  the function is in `app.modules.catalog.authorization`).
- For the export permission check at lines 238-245, branch:
  - User present → enforce as today (matrix['user_role']['export']).
  - User absent → enforce that the dataset is public (the mint endpoint
    already enforces this via `check_dataset_access_or_anonymous` at
    router.py:259, so the consume side can trust the token was minted
    legitimately — the typ/scope/exp validation in _resolve_download_user
    is the gate).

**Shape B — return a sentinel AnonymousIdentity:**
- Create `app.core.identity.AnonymousIdentity` (or reuse a Null Object
  pattern) with id=None, username="anonymous", roles=().
- `_resolve_download_user` returns this sentinel when sub is absent.
- `download_cog` continues to type user as `Identity` but branches on
  `user.id is not None`.
- Audit emit handles None id.

CONTEXT.md does NOT lock the choice. Pick Shape A (it's the simpler,
fewer-files-touched route and matches the existing
`check_dataset_access_or_anonymous` pattern). If Shape B's invariant
("download_cog always has SOME user object") matters more than the type
discipline, switch — document the choice in the SUMMARY.

fix(download): consume anonymous download tokens end-to-end (KNOWN-01)

POST /auth/download-token/{id} for a public dataset has been issuing a
no-sub anonymous token since v1015 Phase 1065, but
_resolve_download_user has always rejected such tokens with 401 — the
mint side comment at auth/router.py:262-267 stated the consumer "handles
a missing sub gracefully" without it ever doing so.

This closes the loop. _resolve_download_user now returns Identity | None
for valid no-sub tokens; download_cog branches on user-None to enforce
public visibility (defense-in-depth on top of the mint-side
check_dataset_access_or_anonymous gate) and emit the audit row with
user_id=None.

Tests cover authenticated, anonymous-public, mint-rejected anonymous-private,
expired, and wrong-scope paths.

Closes KNOWN-01.
