The export endpoint at `backend/app/processing/export/router.py:32`:

```python
@router.get("/{dataset_id}/export", response_class=FileResponse)
async def export_dataset_endpoint(
    ...
    user: Identity = Depends(require_permission("export")),
    ...
):
```

`require_permission` (at `backend/app/modules/auth/dependencies.py:270`) is
a factory that returns a closure `_permission_checker`. The closure:

1. Reads `get_cached_user_roles(request, db, current_user)` to know the
   user's role names.
2. Reads `get_effective_permissions(db)` to load the full role→capability
   matrix.
3. Returns 403 if NO role in the user's set has the requested capability
   enabled in the matrix.

The DEFAULT permission matrix grants `export` to viewer, editor, and admin
(per `backend/app/modules/auth/permissions.py:DEFAULT_ROLE_PERMISSIONS`).
The admin can REVOKE export from viewer via the admin permissions endpoint
(`PUT /admin/permissions/role-permissions/`) — that flow IS tested elsewhere
(`backend/tests/test_permissions.py::test_get_put_permissions` at line 140).

What's missing: a test that does BOTH operations in one flow — revoke export
from viewer, then attempt the export as a viewer, expect 403.

1. First, revoke `export` from viewer via the admin matrix PUT.
2. Then, hit the export endpoint as viewer.
3. Expect 403.
4. Also verify the EDITOR (which we did NOT revoke) STILL gets 200 — proves
   the revoke is scoped to viewer, not blanket.
5. After the test, restore the matrix to defaults (or rely on the

passes and we exercise the capability gate's 403 branch (NOT the
