### Structure Rationale

- **`docs/` as independent subdirectory with own `package.json`:** The docs project has a completely different dependency tree (Starlight, starlight-openapi, no Satori/resvg-js) than the marketing site. An independent `package.json` keeps `npm install` fast and avoids version conflicts. No workspace setup needed — Cloudflare Pages sets root directory per project, so each project's `npm install` + `npm run build` runs in its own directory. This is the simplest approach that works natively with Cloudflare's monorepo support.
- **`docs/src/content/openapi/geolens.json`:** A committed file rather than a runtime fetch. This means the docs repo always has a buildable snapshot even without network access. The CI workflow refreshes it on each deploy.
- **No npm/pnpm workspaces:** Workspaces add value when packages share code between sub-projects. The marketing site and docs site share no JavaScript — only CSS design tokens. A workspace setup would add `pnpm-workspace.yaml`, a root `node_modules/`, and hoisting complexity for zero gain. Simpler: independent `package.json` files, each installed by their respective CI job in their own directory.

## Architectural Patterns

### Pattern 1: Design Token Bridge via `customCss`

**What:** A single `docs/src/styles/custom.css` file maps the GeoLens OKLCH tokens (defined in the marketing site's `global.css`) to Starlight's `--sl-*` CSS custom properties. The file is registered in `astro.config.mjs` via the `customCss` array.

**When to use:** Always. This is the only approved approach for sharing brand tokens.

**Trade-offs:** No shared file, no symlink, no build-time copy script. The `custom.css` file is a hand-written mapping layer. The OKLCH values are duplicated from the marketing site's `:root` block — approximately 15 lines of color values. When the GeoLens brand changes, both `src/styles/global.css` (marketing) and `docs/src/styles/custom.css` (docs) must be updated. Given brand token changes happen infrequently (last change was months ago), the duplication cost is low and the tooling complexity cost of a shared package or symlink is high.

**Why not other options:**
- **Symlink `../src/styles/global.css`:** Git does not guarantee symlink portability across CI runners; Cloudflare Pages does not guarantee the `..` parent directory is available during build when root dir is set to `docs/`.
- **npm workspace package `@geolens/tokens`:** Adds a third package, a publish step, and version sync overhead for what is ~15 OKLCH color values.
- **Build-time copy script:** A `prebuild` npm script copying the marketing CSS adds a dependency on relative paths that breaks when Cloudflare Pages sets `docs/` as the root directory.
- **`@import` from sibling:** Vite/Astro CSS `@import` paths are resolved relative to the CSS file, not the project root. Cross-project `@import '../src/styles/global.css'` breaks when root dir is `docs/`.

**Example:**
```css
/* docs/src/styles/custom.css */
/* Maps GeoLens OKLCH brand tokens → Starlight design tokens */
/* Keep primary-500/600/700 in sync with src/styles/global.css */

:root {
  /* Accent = GeoLens primary blue */
  --sl-color-accent-low:  oklch(0.93 0.05 250);   /* primary-100 */
  --sl-color-accent:      oklch(0.55 0.18 250);   /* primary-500 */
  --sl-color-accent-high: oklch(0.46 0.16 250);   /* primary-700 */

  /* Font */
  --sl-font: 'Inter Variable', ui-sans-serif, system-ui, sans-serif;

  /* Content width */
  --sl-content-width: 48rem;
}
```

### Pattern 2: Two Cloudflare Pages Projects, One GitHub Repo

**What:** Each Pages project is created independently in the Cloudflare dashboard, pointed at the same GitHub repo but with different root directory and build command settings. Each has its own `wrangler.toml` in its subdirectory.

**When to use:** Exactly this scenario — two independently deployable sites from one repo.

**Trade-offs:** Each project deploys independently; a push to `docs/**` only triggers the docs build if CI path filters are configured correctly. Max 5 Pages projects per repo (more than enough). Build System V2 required (already active on newer projects).

**Configuration — marketing (existing, no change needed):**
The existing `wrangler.toml` at repo root stays as-is:
```toml
name = "getgeolens-com"
compatibility_date = "2024-01-01"
pages_build_output_dir = "dist"
```
Cloudflare dashboard: root directory = `.` (default), build command = `npm run build`.

**Configuration — docs (new):**
```toml
# docs/wrangler.toml
name = "geolens-docs"
compatibility_date = "2025-01-01"
pages_build_output_dir = "dist"
```
Cloudflare dashboard: root directory = `docs`, build command = `npm run build`.

**CI approach (preferred over Cloudflare auto-deploy):** The existing repo uses `cloudflare/pages-action` in GitHub Actions rather than Cloudflare's native GitHub integration. Keep this pattern — it gives path filtering control. The docs deploy job runs `cd docs && npm ci && npm run build` and deploys from `docs/dist`.

### Pattern 3: OpenAPI Ingestion — Committed Snapshot + CI Refresh

**What:** `docs/src/content/openapi/geolens.json` is a committed file. The `fetch-openapi.ts` script fetches the latest from `https://api.getgeolens.com/api/openapi.json` (or the production GeoLens API URL) and writes it before the build. In CI, this script runs as a build step before `npm run build`.

**When to use:** Always — for this project, production API access from CI is reliable and the spec changes with every meaningful feature.

**Why not build-time fetch inside Astro config:** The `starlight-openapi` plugin accepts a remote URL directly as `schema`. This is elegant but makes the build dependent on the production API being reachable. If the API is down or rate-limited, the build fails. A committed snapshot decouples build reliability from API availability. The snapshot serves as fallback; CI refreshes it on each docs deploy.

**Why not multi-repo fetch (geolens monorepo → docs repo):** The FastAPI `openapi.json` endpoint is public (unauthenticated). A simple `curl` in CI is sufficient — no GitHub Actions cross-repo token needed.

**Implementation:**
```typescript
// docs/scripts/fetch-openapi.ts
import { writeFileSync } from 'fs';

const url = process.env.OPENAPI_URL ?? 'https://YOUR_API_HOST/api/openapi.json';
const res = await fetch(url);
if (!res.ok) {
  console.warn(`Failed to fetch openapi.json (${res.status}), using committed snapshot`);
  process.exit(0); // non-fatal — committed snapshot remains
}
const json = await res.json();
writeFileSync('src/content/openapi/geolens.json', JSON.stringify(json, null, 2));
console.log('Updated src/content/openapi/geolens.json');
```

**`astro.config.mjs` usage:**
```javascript
starlightOpenAPI([{
  base: 'api',
  schema: './src/content/openapi/geolens.json',  // local after fetch
  label: 'API Reference',
  collapsed: true,
}])
```

### Pattern 4: Canonical Docs Migration — Move to Docs Site, Leave Pointer

**What:** `backend/docs/install.md` and `backend/docs/admin.md` are migrated to `docs/src/content/docs/quickstart/install.md` and `docs/src/content/docs/admin-guide/index.md` respectively. The originals in the geolens monorepo are replaced with a one-line pointer: `Docs moved to https://docs.getgeolens.com`. The docs site becomes the canonical home.

**Why not keep geolens monorepo as source-of-truth and pull at build:** A pull step would require either a GitHub token (to access the private geolens repo from the public CI context) or publishing the docs as a separate artifact. The content in `backend/docs/` is human prose that belongs in the docs site — it has no build dependency on the backend code. Moving it avoids the two-source problem entirely. The geolens `backend/docs/` directory is not used programmatically by any test or CI step.

### Pattern 5: Cross-linking Without Hardcoded URLs

**What:** Define site base URLs as constants in each project's config, not scattered across content files.

**Marketing site:** The `getgeolens.com` URL is already in `astro.config.mjs` as `site: 'https://getgeolens.com'`. Add `DOCS_URL = 'https://docs.getgeolens.com'` as a JS constant in a `src/lib/links.ts` file used by any component that links to docs.

**Docs site:** `astro.config.mjs` sets `site: 'https://docs.getgeolens.com'`. Add `MARKETING_URL = 'https://getgeolens.com'` in `src/lib/links.ts` for footer/nav links back to the marketing site.

**Content files (`.md`/`.mdx`):** Use absolute URLs for cross-site links in prose. Starlight does not provide a template variable mechanism for markdown links. Keep cross-site links to a minimum in content — they live in nav/footer components where the constant is accessible.

## Data Flow

### Build-Time Data Flow (Docs Site)

```
CI: docs deploy job
    │
    ├── fetch-openapi.ts
    │       └── GET /api/openapi.json → writes docs/src/content/openapi/geolens.json
    │
    └── astro build (in docs/)
            ├── Starlight processes src/content/docs/**/*.md(x)
            │       → HTML pages for quickstart, user guide, admin guide
            ├── starlight-openapi reads geolens.json
            │       → generates static HTML API reference pages
            ├── Pagefind runs post-build
            │       → writes pagefind/ index to dist/
            └── @astrojs/sitemap generates sitemap-index.xml
```

### Deploy Data Flow

```
git push to main (changes in docs/**)
    │
    ├── GitHub Actions: deploy-docs.yml
    │       ├── cd docs && npm ci
    │       ├── npx tsx scripts/fetch-openapi.ts
    │       ├── npm run build  (outputs docs/dist/)
    │       └── cloudflare/pages-action → geolens-docs project → docs.getgeolens.com
    │
    └── GitHub Actions: ci.yml (path filter: NOT docs/**)
            └── (skips if only docs changed)
```

### Pagefind Static Search

Pagefind runs automatically as part of Starlight's build step — no configuration needed, no external service. The generated `pagefind/` directory is included in `docs/dist/` and served as static assets by Cloudflare Pages. This works natively. Confidence: HIGH (verified in Starlight docs).

## Build Order — Prerequisites

The following must exist before the first docs deploy succeeds:

1. **DNS:** `docs.getgeolens.com` CNAME pointing to Cloudflare Pages. Must exist before the Pages project can serve traffic on the subdomain — but the Pages project itself can be created and deployed without DNS (it gets a `*.pages.dev` URL first).
2. **Cloudflare Pages project `geolens-docs`:** Created in Cloudflare dashboard. Root directory = `docs`. Build command = `npm run build`. Build output = `dist`. Can be created before DNS.
3. **GitHub Actions secrets:** `CLOUDFLARE_API_TOKEN` and `CLOUDFLARE_ACCOUNT_ID` are already in the repo secrets. The new `deploy-docs.yml` reuses them.
4. **`docs/` subdirectory committed:** At minimum `package.json`, `astro.config.mjs`, `docs/src/content/docs/index.mdx` must exist for the first build to succeed.
5. **`docs/src/content/openapi/geolens.json` committed snapshot:** CI script is non-fatal on failure, but the initial commit needs a valid snapshot to bootstrap API reference pages.
6. **`@astrojs/sitemap` in `docs/package.json`:** Starlight includes sitemap generation via the `@astrojs/sitemap` integration. The `astro.config.mjs` must set `site: 'https://docs.getgeolens.com'` for the sitemap URLs to be correct.

**Correct order:** Create Cloudflare Pages project → commit `docs/` directory → push to main → CI deploys to `*.pages.dev` → configure DNS CNAME → add custom domain in Pages dashboard.

## Sitemap, robots.txt, and llms.txt

**Sitemap:** Each site generates its own independent sitemap. The marketing site already has `@astrojs/sitemap` configured. The docs site adds its own `@astrojs/sitemap` instance in `docs/astro.config.mjs`. The two sitemaps do not merge — they live at `https://getgeolens.com/sitemap-index.xml` and `https://docs.getgeolens.com/sitemap-index.xml`. Search engines treat them as separate properties. No coordination required.

**robots.txt:** `docs/public/robots.txt` is a static file:
```
User-agent: *
Allow: /
Sitemap: https://docs.getgeolens.com/sitemap-index.xml
```

**llms.txt:** Starlight does not generate `llms.txt` out of the box. A static `docs/public/llms.txt` file with a manual summary is sufficient for v15.0. Dynamic generation (using `getCollection('docs')` to enumerate all pages) can be added as a `docs/src/pages/llms.txt.ts` endpoint if desired.

## Anti-Patterns

### Anti-Pattern 1: npm/pnpm Workspaces

**What people do:** Set up a workspace root `package.json` that treats marketing and docs as workspace packages, hoisting shared deps.

**Why it's wrong:** The two sites share zero JavaScript. Workspaces add root-level `node_modules/`, a `pnpm-workspace.yaml` or `workspaces` field, and force both projects to be installed together. The existing CI and Cloudflare Pages configuration for the marketing site works today and should not be touched.

**Do this instead:** Independent `package.json` files in `./` and `docs/`. Each CI job `cd`s into its directory and runs `npm ci` + `npm run build` independently.

### Anti-Pattern 2: Fetching openapi.json at Astro Config Resolution Time

**What people do:** Pass a remote URL directly to `starlightOpenAPI([{ schema: 'https://api...' }])` and rely on the plugin to fetch it during the build.

**Why it's wrong:** If the production API is unreachable during the CI build (network restrictions, API downtime, rate limiting), the entire docs build fails. There is no fallback.

**Do this instead:** Committed snapshot + pre-build fetch script. The script is non-fatal on failure — if the fetch fails, the committed snapshot is used and the build succeeds with slightly stale API docs.

### Anti-Pattern 3: Symlinked Design Tokens

**What people do:** Create a symlink from `docs/src/styles/global.css` → `../src/styles/global.css` to avoid duplication.

**Why it's wrong:** Git tracks symlinks as files, but Cloudflare Pages sets the build root to `docs/` — the `..` path resolves outside the build root and is unavailable. The symlink works locally but breaks in CI.

**Do this instead:** A mapping CSS file (`docs/src/styles/custom.css`) that declares only the `--sl-*` override values. ~15 OKLCH values. Accept the minimal duplication.

### Anti-Pattern 4: Auto-deploy via Cloudflare Dashboard Git Integration

**What people do:** Connect both Pages projects to the GitHub repo via the Cloudflare dashboard's native Git integration instead of using GitHub Actions.

**Why it's wrong:** The existing marketing site uses `cloudflare/pages-action` in GitHub Actions (not the dashboard integration). Mixing the two approaches creates two deploy pathways. The dashboard integration also does not support fine-grained path filtering out of the box — every push triggers both projects unless you configure watch paths in the Cloudflare dashboard UI, which is less transparent than CI path filters in code.

**Do this instead:** Consistent `cloudflare/pages-action` approach for both sites. Path filters in `on.push.paths` control which CI job runs.
