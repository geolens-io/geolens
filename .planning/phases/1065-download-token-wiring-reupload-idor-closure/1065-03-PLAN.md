<objective>
Add the missing `_assert_compatible_record_type` call to `reupload_service_preview`, mirroring the guard
already present in `reupload_dataset` (line 128) and `request_presigned_reupload` (line 589).

Purpose: Without this guard, a vector-dataset owner can submit a service-URL reupload pointing to a raster
service. The validation failure happens deep in the worker (ogr2ogr processing), surfacing as a pipeline 500
or a stale failed job — not a useful 400. Adding the guard at handler entry gives immediate, actionable feedback.

This plan depends on Plan 02 (which adds `check_dataset_access` to `reupload_service_preview`). The insertion
point is: after `check_dataset_access`, before `validate_url_for_ssrf`. The `_assert_compatible_record_type`
helper is already defined in the same file at line 67 — no new imports needed.

Note on the filename parameter: `reupload_service_preview` receives a `ReuploadServicePreviewRequest` body,
not a filename. Read the helper's logic (lines 67-107) to determine what to pass as the `filename` argument.
The helper uses `Path(filename or "").suffix.lower()` to extract the extension. For a service-URL reupload,
there is no file extension — pass `None` or `""` as the filename. This means only the VRT-guard branch
(record_type == "vrt_dataset") will trigger for service-URL paths, which is the correct behavior: VRT
datasets cannot be reuploaded via service URL. The vector↔raster cross-type check requires a file extension
to detect; without one, those branches are silently skipped — which is acceptable because the ogr2ogr
layer will handle type mismatches at commit time for service URLs. Confirm this interpretation by reading
lines 80-107 of _assert_compatible_record_type before coding.

If the helper's behavior for `filename=None` already correctly gates VRT but allows other types to pass
through for service URLs, proceed. If the project's design intent requires blocking ALL record-type mismatches
for service URLs (not just VRT), add a `service_type` parameter to the helper or add an inline guard in
`reupload_service_preview` specifically for service-URL paths. Document the choice in a code comment.

Output:
- 1-line insertion in `reupload_service_preview` (after check_dataset_access, before validate_url_for_ssrf)
- New test file covering the guard behavior in the service-preview path
</objective>
