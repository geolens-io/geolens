# Phase 1023 Plan: Cluster Source Eligibility And GeoJSON Contract

**Phase:** 1023 — cluster-source-eligibility-and-geojson-contract
**Milestone:** v1005 Builder Point Cluster Foundation
**Status:** Complete
**Requirements:** SRC-01, SRC-02, SRC-03, SRC-04, SRC-05

## Goal

Prove when Point Cluster can be safely offered and wire the bounded GeoJSON source loading contract without changing persisted schemas or replacing the existing vector-tile path.

## Implementation

1. Add a cluster eligibility helper for map-layer metadata.
   - Eligible only for vector point datasets.
   - Requires `dataset_feature_count` / `feature_count` to be known and at or below the bounded GeoJSON cap.
   - Excludes raster, DEM, non-point, missing-count, and oversized layers.

2. Extend `renderAs` capability plumbing.
   - Add `cluster` to the renderer id/type surface.
   - Filter the `Cluster` option through eligibility.
   - Keep `Cluster` out of unsupported or ineligible layers.
   - Build Cluster patches with existing writable fields only.

3. Generalize bounded GeoJSON fetching.
   - Keep existing GeoJSON-Z behavior working.
   - Add a reusable bounded GeoJSON helper that preserves JWT, API-key, and embed-token paths.
   - Do not fetch for non-cluster point renderers.

4. Wire builder/viewer cluster source prefetch.
   - Builder uses JWT-authenticated fetch for local authoring.
   - Viewer uses the same API-key/embed-token path as GeoJSON-Z.
   - Failed, truncated, or unsupported cluster source loads stay nonfatal and degrade to normal Point rendering.

5. Add focused coverage.
   - renderAs eligibility and patch tests.
   - bounded GeoJSON request-path tests.
   - builder/viewer fetch selection helpers where practical.
