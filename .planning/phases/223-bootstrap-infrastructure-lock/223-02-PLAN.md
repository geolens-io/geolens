**Critical notes (do NOT modify these):**

1. **`paths:` includes the workflow file itself** (`'.github/workflows/docs-ci.yml'`) — this is the self-modification trigger per RESEARCH §7.1. Without it, edits to the workflow won't trigger CI and you'll get "I changed the workflow but it didn't run" surprises.

2. **`defaults.run.working-directory: docs`** at the job level — per RESEARCH §7.6, this prevents build contamination from running `npm install` at the repo root and hoisting marketing deps. It applies to all `run:` steps in the job (but NOT to `uses:` action invocations).

3. **`directory: docs/dist`** in `pages-action@v1` — this path is **relative to the repo root, NOT to `working-directory`**, per RESEARCH §3.5. Verified by inspecting marketing's `ci.yml` which uses `directory: dist`.

4. **`cache-dependency-path: docs/package-lock.json`** — explicit path because `actions/setup-node@v4` defaults to repo-root `package-lock.json` (which doesn't exist for the marketing repo, and even if it did, would point at the wrong tree).

5. **`Verify wrangler.toml name matches projectName` step** — cheap insurance per pitfall §7.5. If a future edit accidentally diverges `wrangler.toml` `name` from `projectName:` input, this step fails fast with a readable error.

6. **`bash scripts/verify-build.sh` step** — runs the load-bearing build-artifact gate from Plan 01. If the canonical URL, noindex meta, sitemap, `_redirects`, or flat-URL assertions fail, CI goes red.

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| GitHub Actions runner → Cloudflare Pages API | CI uploads built artifacts via `cloudflare/pages-action@v1` using `CLOUDFLARE_API_TOKEN`. Token leakage exposes the entire CF account. |
| GitHub repo → CF Pages preview/production | Every push to `main` (and every PR with `docs/**` changes) ships HTML to public CDN. Build isolation and noindex insurance are the controls. |
| Public DNS → custom-domain TLS | `docs.getgeolens.com` resolves via CF; TLS is provisioned by CF Universal SSL. Cert misconfig or stalled provisioning = broken initial deploy. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-223-CROSS-CONTAM | Tampering (build isolation) | docs-ci.yml + ci.yml interaction | mitigate | Two-layer defense: (1) `defaults.run.working-directory: docs` at job level in `docs-ci.yml` (Task 1) prevents `npm install` from contaminating root; (2) symmetric `paths-ignore: ['docs/**']` patch on marketing `ci.yml` (Task 2) prevents marketing CI from running on docs-only commits. Verified by Task 3 Step 6 with two probe PRs. |
| T-223-TLS | Denial of Service / availability | `docs.getgeolens.com` HTTPS endpoint | mitigate | Manual verification post-attach: `curl -I https://docs.getgeolens.com` MUST return HTTP/2 200 without `-k` flag (Task 3 Step 5). Apex `getgeolens.com` is already on CF DNS, so CNAME auto-creation + Universal SSL provisioning is the easy path (RESEARCH §3.13). Recovery procedure documented: delete + re-create custom domain attachment if TLS hangs >15 min. |
| T-223-SECRETS | Information Disclosure (credentials) | CLOUDFLARE_API_TOKEN, CLOUDFLARE_ACCOUNT_ID | mitigate | Reuse existing GitHub secrets per D-05 — no new tokens minted, narrowing the secret-leak surface. `cloudflare/pages-action@v1` is a maintained-by-Cloudflare action that does NOT echo secret inputs to logs (verified via marketing CI logs over the past 12 months).
