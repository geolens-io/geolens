<spike_hypotheses>
Pre-spike hypotheses to prove or disprove (in order of suspicion). The spike output must explicitly check off which hypotheses are TRUE/FALSE based on observed log/state. The audit doc must NOT just list these — it must commit to ONE root cause.

**H1 — Stale session-state after `session.commit()` at tasks_common.py:822, then re-entering `_generate_quicklook` which itself calls `await session.commit()` at line 662.** The MissingGreenlet may be triggered when the post-commit session has an internal asyncpg cursor that the next `await db.execute(...)` inside `generate_vector_quicklook` reuses incorrectly. Connection-killing query (the bounds SELECT on the 6018-feature multipolygon table) may leave the connection in a partial-rollback state. This matches the antecedent warning "Can't reconnect until invalid transaction is rolled back" — a classic SQLAlchemy 2.0 + asyncpg signal that the session needs explicit rollback before next execute().

**H2 — `asyncio.wait_for(...)` at quicklook.py:231 cancelling the inner SQLAlchemy `db.execute` mid-flight without cleaning up the asyncpg cursor.** When `wait_for` cancels a coroutine that's inside an `await db.execute(...)`, the cancellation propagates to the asyncpg protocol but may leave the connection-level state inconsistent. The next call into the session triggers MissingGreenlet because the greenlet bridge sees the cursor in a state it can't proceed from. This is the "Was IO attempted in an unexpected place?" shape from the SQLAlchemy error doc /e/20/xd2s.

**H3 — Lazy relationship access on `dataset.record` post-`commit()`.** Tasks_common.py:802 sets `dataset.record.record_status` BEFORE the commit at line 822. After commit, ORM identity-map should still hold the relationship (expire_on_commit=False), but if `_generate_quicklook` later triggers an implicit emit (e.g. session.flush during the inner commit at line 662) and the relationship needs to refresh, the refresh would happen via a sync greenlet bridge from a code path the planner expected to be pure-async. Less likely given expire_on_commit=False, but worth ruling out.

**H4 — Feature-shape pathological geometry on `urban_areas_landscan_10m`.** 6018 multipolygon features with non-trivial vertex counts; the `ST_Simplify(ST_MakeValid(geom_4326), 0.01)` at quicklook.py:153-154 may take long enough to trip the asyncio.wait_for timeout (_GENERATION_TIMEOUT_SECONDS — not visible in the read; spike must capture it from the file). H2 + H4 may combine: timeout fires during a long execute, cancellation leaves cursor wedged.

**H5 — Connection-pool-level issue: the failing path catches the inner Exception broadly at quicklook.py:652-659 (`tasks_common.py:_generate_quicklook` `except Exception as _ql_exc`), but the SECOND `try/except` for the commit at lines 661-672 catches a NEW exception emitted by the next call into the broken session.** The MissingGreenlet might be raised from the commit phase try block (lines 662-672), not the generate phase try block. Spike must determine which `try` block emitted the MissingGreenlet by reading the worker log carefully.

The audit doc names ONE root cause, cites the precise line, and proposes a fix that closes that root cause.
</spike_hypotheses>
