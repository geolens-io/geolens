<root_cause_analysis>
The 1363 `InvalidCatalogNameError` errors observed in v1016 Phase 1074 trace to one or more of:

1. **Connection-pool stale entries.** The session-scoped DB is created with a uuid-suffixed name (e.g. `geolens_test_a1b2c3d4`). The `client` fixture creates per-test `create_async_engine(settings.test_database_url, ...)`, which uses asyncpg's connection pooling. If a prior test's engine.dispose() didn't complete before teardown drops the DB, **the next test's connection pool tries to acquire a connection to a now-deleted DB** → `InvalidCatalogNameError`.

2. **Worker collision under `-n auto`.** pytest-xdist runs N workers in parallel. Each worker imports `conftest.py` and calls `_session_test_database_name(...)` independently, generating different uuid suffixes — so workers each get their OWN DB. BUT: workers also share `settings.postgres_db_test` *at module load time* and mutate it inside the fixture. Test modules that read `settings.test_database_url` AT IMPORT TIME (not inside a fixture) will see whichever value was set first. Result: many tests resolve a DB name that ANOTHER worker just dropped.

3. **`pytest-xdist` not installed.** `backend/pyproject.toml:58-71` does NOT include `pytest-xdist`. So `pytest -n auto` currently fails outright — TI-01 acceptance requires we add it.

**Fix strategy:**

A. **Add `pytest-xdist>=3.6` to `[dependency-groups].dev`.** Required to satisfy the TI-01 "`pytest -n auto` exits 0" success criterion.

B. **Compute the test DB name from `worker_id`** (xdist sets `PYTEST_XDIST_WORKER=gw0`, `gw1`, … via `os.environ`; in-process when no xdist is present, the value is `"master"`). Suffix the DB name as `geolens_test_{worker_id}_{uuid8}` so workers are physically separated AND uuid still randomizes across re-runs of the same worker.

C. **Tighten teardown ordering.** Before DROP DATABASE, `pg_terminate_backend` ALL connections to that DB (the existing query already does this — keep it). Add `await engine.dispose()` calls before the session fixture teardown reaches the DROP step (the `client` fixture already calls `await test_engine.dispose()` — verify it does NOT race the session teardown).

D. **Mutate `settings.postgres_db_test` ONCE at fixture startup, BEFORE the `yield`, and ensure no test module reads it at import time.** The existing code does this — verify no fixture (in `conftest.py` OR in any test file) accesses `settings.test_database_url` at module-collection time. If any do, lift them inside a function.

E. **Add a regression test** `backend/tests/test_conftest_lifecycle.py` that:
   - Asserts the DB name contains the worker_id (or `"master"` when running solo)
   - Asserts the DB exists after the fixture yields
   - Asserts an async engine can connect, run a trivial `SELECT 1`, and dispose cleanly
   - Asserts `settings.postgres_db_test` is restored to its original value after the fixture tears down (use a stub fixture or session-scoped marker to inspect this)

This is a code-level refactor in ONE file (conftest.py) plus one new test file plus one pyproject.toml line — total ~3 file modifications. Well under the 30% context budget for a Wave 1 plan.
</root_cause_analysis>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| developer machine → test DB | test code runs against a real Postgres+PostGIS instance (typically localhost or docker compose db); no untrusted input |
| pytest worker → pytest worker | parallel xdist workers share the same Postgres server; isolated via per-worker DB names |
| CI runner → test DB | Phase 1078 will land the alembic clean-DB CI workflow; this plan does not interact with that surface |

|-----------|----------|-----------|-------------|-----------------|
| T-1075-01 | Tampering | conftest.py worker_id env-var consumption | accept | `PYTEST_XDIST_WORKER` is set by pytest-xdist itself; a malicious value would only cause an invalid DB identifier (PG would reject), not data corruption. No production code path consumes this env var. |

Bandit + pip-audit (already in dev group) catch CVE drift. |

| T-1075-04 | Information Disclosure | test DB seed includes admin user with default password | accept | The test DB is created against `settings.database_url_sync` (typically localhost or docker compose); the seeded admin uses `settings.geolens_admin_password` which the developer controls. Legitimacy verified at https://pypi.org/project/pytest-xdist/. RESEARCH.md Package Legitimacy Audit is N/A for this plan (no `pytest-xdist` audit table required — this is a single well-known plugin, not a research-grade package choice). |

`T-1075-SC` is accepted under the rule that single, ecosystem-canonical pytest plugins do not require the full slopcheck protocol. If the next plan adds 3+ new packages, the audit table protocol applies.
