## User Constraints (from CONTEXT.md)

### Locked Decisions
- **D-01:** Use MapLibre's built-in `TerrainControl` component for the 3D toggle — zero custom UI.
- **D-02:** 3D mode always enables pitch/rotate navigation and extrusions even without DEM terrain. Terrain stays flat if no DEM is present.

## Summary

This phase adds a 3D toggle to `ViewerMap.tsx` that activates MapLibre terrain (via Titiler's `terrainrgb` algorithm) and building extrusions (via a `fill-extrusion` layer reading a user-configured height column). The design doc (260408-aa5) resolved all 7 open questions before this research ran — findings below confirm that design doc's conclusions against the live codebase.

The work splits into three independent tracks: (1) backend `is_dem` flag — one Alembic migration, one heuristic in `extract_raster_metadata`, one URL param in `raster_tile_proxy`, one field in `DatasetMeta` PATCH; (2) frontend terrain toggle — `TerrainControl` in `ViewerMap.tsx`, terrain source management in the existing `style.load` hook, `raster-dem` source created imperatively; (3) frontend extrusion — height column dropdown in `LayerStyleEditor.tsx` stored in `paint._height_column`, new `fill-extrusion` companion layer in `fill-adapter.ts` conditional on 3D mode + height column presence.

All MapLibre v5 3D features needed are available without version changes. `maplibre-gl` is at `^5.22.0` [VERIFIED: package.json]. `@vis.gl/react-maplibre` is at `^8.1.0` [VERIFIED: package.json]. The declarative `terrain` prop on `<Map>` works in v8 [CITED: design doc §2 / react-maplibre v8 terrain example]. Terrain tiles flow through the existing `X-Embed-Token` proxy path without new auth plumbing [CITED: design doc §3].

**Primary recommendation:** Implement `is_dem` backend first (Wave 0), then terrain toggle and extrusion adapter in parallel (Wave 1), then layer style editor height dropdown (Wave 2), then embed-token smoke test (Wave 3).

## Architecture Patterns

### Existing Patterns This Phase Must Follow

**Pattern 1: Imperative map.addSource() + map.addLayer() for data layers**
All vector tile sources and layers in this project use the imperative API. The `fill-extrusion` companion must follow this pattern, not declarative `<Source>/<Layer>`. [VERIFIED: ViewerMap.tsx, fill-adapter.ts, CLAUDE.md known issue]

**Pattern 2: Declarative `terrain` and `sky` props on `<Map>` DO work**
In contrast to `<Source type="vector">` which is unreliable, the `terrain` prop on `<Map>` works correctly in @vis.gl/react-maplibre v8. Use the declarative prop for terrain activation. [CITED: design doc §2, react-maplibre 1.0-release terrain example]

```tsx
// Source: design doc §3 illustrative snippet
const terrain = { source: 'terrain-dem', exaggeration: 1.5 };
<MapGL terrain={is3D && hasDem ? terrain : undefined} ...>
```

`setStyle()` (triggered by basemap changes) wipes all custom sources including the terrain source. ViewerMap already has a `style.load` listener at lines 474-492. Terrain source + `setTerrain` must be re-applied inside `onStyleLoad`. [VERIFIED: ViewerMap.tsx:474-492; CITED: design doc §3 "Style-reload integration point"]


**Pattern 4: fill-extrusion as companion layer in fill-adapter**
`fill-adapter.ts` today emits `fill` + `line` (outline) companion layers. The 3D extrusion layer is a third companion: `${layerId}-extrusion`, added only when `_height_column` is present in `paint` AND the map is in 3D mode. [VERIFIED: fill-adapter.ts]


**Pattern 5: triggerRepaint() after terrain paint mutations**
MapLibre v5 has a known bug where `setPaintProperty` applies on only every other call when terrain is active. Always call `map.triggerRepaint()` after paint/filter changes to extrusion or fill layers when terrain is on. [CITED: design doc §3 known bugs table, maplibre-gl-js issues #1650, #2035, #3001, #3039]

**Pattern 6: is_dem detection heuristic in extract_raster_metadata**
`extract_raster_metadata` in `backend/app/raster/cog.py` already records `band_count` and `band_info[].dtype`. The heuristic is: `band_count == 1 AND _is_float_dtype(dtype)`. [VERIFIED: raster/cog.py:61-82]


**Pattern 7: Alembic migration chaining**
Latest migration is `2026_04_12_1102-989ae68d7859_tighten_pydantic_varchar_constraints_p1_`. New `is_dem` migration must chain from this revision. [VERIFIED: backend/alembic/versions/]

### Data Flow: DEM Terrain Path

```
User uploads DEM COG
  → extract_raster_metadata detects single-band float → is_dem_candidate=True
  → RasterAsset.is_dem = True (auto)
  → Admin can PATCH /datasets/{id} with {"is_dem": true/false} to override

ViewerMap receives layers (SharedLayerResponse[])
  → Finds layer where token.kind === 'raster' AND layer.is_dem === true
  → Adds 'terrain-dem' raster-dem source imperatively (URL = tile_url + "&algorithm=terrainrgb")
  → Passes terrain prop to <MapGL>
  → On style.load: re-applies terrain source + setTerrain
```

### Data Flow: Fill-Extrusion Path

```
Builder layer style editor (LayerStyleEditor.tsx, polygon type)
  → User selects height column from numeric column dropdown
  → Stored as paint._height_column = "height_col_name"
  → Saved to DB via existing layer PATCH

ViewerMap syncLayersToMap
  → fill-adapter.addLayers sees paint._height_column
  → If 3D mode active: adds fill-extrusion companion layer
  → If 3D mode off: skips extrusion, renders flat fill as normal
```

## Common Pitfalls

### Pitfall 1: Terrain source wiped on basemap change
**What goes wrong:** `setStyle()` (called when user switches basemap) removes all custom sources, including the `raster-dem` terrain source. Terrain silently disappears.
**Why it happens:** MapLibre `setStyle` with `styleDiffing={false}` fully replaces the style — no custom sources survive.
**How to avoid:** Re-add the terrain source and call `setTerrain()` inside the existing `style.load` listener (ViewerMap.tsx:474-492). Use a ref to track current 3D state and DEM URL.
**Warning signs:** Terrain works initially but disappears after switching basemap.

### Pitfall 2: setPaintProperty only applies every other call under terrain
**What goes wrong:** When terrain is active, `setPaintProperty` on `fill-extrusion` layers sometimes silently no-ops.
**Why it happens:** Known MapLibre v5 bug (issues #1650, #2035, #3001, #3039).
**How to avoid:** Always call `map.triggerRepaint()` after paint mutations to extrusion layers when 3D mode is active.
**Warning signs:** Layer color/opacity changes in style editor don't take effect until user moves the map.

### Pitfall 3: fill-extrusion setFilter stale until next zoom
**What goes wrong:** After calling `map.setFilter()` on a `fill-extrusion` layer when terrain is active, the visual doesn't update until the user zooms.
**Why it happens:** Known MapLibre v5 bug.
**How to avoid:** Call `map.triggerRepaint()` after `setFilter` calls on extrusion layers.
**Warning signs:** Filter changes in builder don't render until zoom interaction.

### Pitfall 4: `_height_column` stored in paint bleeds to other adapters
**What goes wrong:** The `paint._height_column` custom key is passed through `stripCustomProps` in the adapter shared utilities — but only if it's in `CUSTOM_PAINT_PROPS`.
**Why it happens:** `fill-adapter.ts` uses `stripCustomProps(basePaint)` which strips known custom props before passing to `map.addLayer`. If `_height_column` is not in `CUSTOM_PAINT_PROPS`, it will be passed through to MapLibre and may cause a style spec error.
**How to avoid:** Add `'_height_column'` to `CUSTOM_PAINT_PROPS` in `frontend/src/components/builder/layer-adapters/shared.ts`, then read it before `stripCustomProps` strips it.
**Warning signs:** MapLibre console error "unknown paint property _height_column".

### Pitfall 5: raster-dem source URL must NOT include the `&algorithm=terrainrgb` via transformRequest override
**What goes wrong:** The terrain tile URL must include `algorithm=terrainrgb` in the URL itself (appended in the backend proxy), not injected via `transformRequest`. `transformRequest` is already used for auth headers; adding query params there creates coupling.
**How to avoid:** The backend `raster_tile_proxy` appends `&algorithm=terrainrgb` when `is_dem=True`. The frontend just uses the standard tile URL path — the backend handles encoding. Confirm via smoke test (D-11).

### Pitfall 6: TerrainControl shows even when no DEM layers are active
**What goes wrong:** `TerrainControl` is visible in the toolbar even on maps with no DEM layers, potentially confusing users who click it and see no effect.
**How to avoid:** Per D-07, this is acceptable — 3D mode = "3D camera controls + extrusions", flat terrain if no DEM. The TerrainControl is always shown when `is3D` is available. No conditional rendering needed.

### Pitfall 7: fill-extrusion layer renders at all zoom levels, causing perf issues on large polygon datasets
**What goes wrong:** Extruding thousands of polygons at low zoom levels tanks GPU performance.
**How to avoid:** Set `minzoom: 14` on `fill-extrusion` layers (Claude's Discretion item; design doc recommends 14 for building-scale feature density).
**Warning signs:** Sluggish panning on maps with polygon datasets below zoom 14 when 3D is active.

## Code Examples

### Adding raster-dem source for terrain

```typescript
// Source: design doc §3 illustrative, adapted for ViewerMap imperative pattern
function applyTerrain(map: MaplibreMap, demTileUrl: string) {
  if (!map.getSource('terrain-dem')) {
    map.addSource('terrain-dem', {
      type: 'raster-dem',
      tiles: [`${window.location.origin}${demTileUrl}`],
      tileSize: 256,
      encoding: 'mapbox',  // matches Titiler terrainrgb output
    });
  }
  map.setTerrain({ source: 'terrain-dem', exaggeration: 1.5 });
}

function removeTerrain(map: MaplibreMap) {
  map.setTerrain(null);
  if (map.getSource('terrain-dem')) {
    map.removeSource('terrain-dem');
  }
}
```

## Open Questions

1. **Where does `is_dem` live on `SharedLayerResponse`?**
   - What we know: `SharedLayerResponse` in `frontend/src/types/api.ts:819` currently does not include `is_dem`. The backend serializes this from a JOIN across `datasets + records + raster_assets`.
   - What's unclear: Which backend endpoint serves `SharedLayerResponse` — need to confirm it JOINs `raster_assets` already or needs a new join.
   - Recommendation: Search `backend/app/maps/router.py` for the shared-map endpoint. If `raster_assets` is already joined, add `is_dem` to the SELECT. If not, add a LEFT JOIN.

2. **Should `TerrainControl` appear always or only on maps with DEM layers?**
   - What we know: D-07 says 3D with no DEM = flat map with pitch/rotate. D-02 says 3D toggle always enables pitch/rotate. TerrainControl is MapLibre's standard toggle.
   - What's unclear: Is it better UX to always show the control, or hide it when no DEM layer is present?
   - Recommendation: Always show it (per D-02/D-07 — "3D mode = 3D camera + extrusions, not terrain required"). The design doc calls this the intended behavior.

3. **`raster_tile_proxy` vs nginx path for terrainrgb**
   - What we know: The dev path uses `raster_tile_proxy` (lines 283-369 in tiles/router.py). Production uses nginx `auth_request`. Both ultimately build the Titiler URL via the same `render_params` logic.
   - What's unclear: The nginx path uses `X-GeoLens-Render-Params` header and proxies through a separate nginx config; `algorithm=terrainrgb` must also be injected there, OR only on the FastAPI proxy side.
   - Recommendation: Inject `algorithm=terrainrgb` in `_titiler_render_params` when `is_dem=True`, so both the FastAPI proxy path and the nginx-proxied path pick it up via the same header. Alternatively, add a separate `X-GeoLens-Algorithm` header and configure nginx to append it. Investigate nginx config before implementing.
