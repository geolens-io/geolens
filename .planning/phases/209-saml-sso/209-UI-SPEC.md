## Interaction States

### Metadata XML Paste Flow

1. Admin selects `provider_type: 'saml'` from dropdown
2. Form shows Metadata XML textarea (empty, placeholder: "Paste IdP metadata XML here...")
3. Admin pastes XML content
4. On save, backend parses and returns extracted entity_id, sso_url (admin does not see these until save)
5. After save, edit view shows read-only SP Entity ID and ACS URL fields with copy buttons

### Provider Type Switching

1. When admin changes type dropdown from OIDC to SAML: OAuth-specific fields (client_id, client_secret, discovery_url, scopes) hide; SAML metadata textarea appears
2. When admin changes type from SAML to OIDC: Reverse transition
3. No animation needed -- conditional render is sufficient (matches existing Microsoft tenant_id pattern)

### SAML Login Flow (Frontend)

1. User sees SAML provider button(s) alongside OIDC buttons on login page
2. Click triggers full-page redirect to `${API_BASE}/auth/saml/{slug}/login`
3. Backend redirects to IdP
4. IdP authenticates, POSTs back to ACS endpoint
5. Backend issues JWT, redirects to `${frontend_url}/oauth/callback#token={jwt}`
6. Existing `OAuthCallbackPage` extracts token from hash fragment -- no changes needed

### SAML Error Flow

1. Backend redirects to `${frontend_url}/oauth/callback#error={encoded_message}`
2. `OAuthCallbackPage` detects error, redirects to `/login` with error in location.state
3. `LoginPage` shows toast with error message -- existing pattern, no changes needed

### Enterprise Gate Behavior

1. `isEnterprise === false`: `'saml'` option does not appear in provider_type Select dropdown
2. `isEnterprise === true`: `'saml'` appears as a selectable provider type
3. No tooltip or upsell message for community edition -- option is simply absent
