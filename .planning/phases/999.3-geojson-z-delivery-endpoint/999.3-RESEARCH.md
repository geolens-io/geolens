## Summary

Phase 999.3 adds a `/api/datasets/{id}/features.geojson` endpoint that returns an RFC 7946 FeatureCollection with Z coordinates preserved, capped at 5,000 features. The backend work is straightforward: one new router endpoint with auth/RBAC parity to `features/router.py`, one new service function using `ST_AsGeoJSON` with Z passthrough, and a wrapped response object carrying truncation metadata alongside the FeatureCollection. The critical insight is that `ST_AsGeoJSON(geom_4326)` already preserves Z coordinates — no special flag is needed; Z is output natively when the geometry has Z. The existing codebase strips Z only at the MVT boundary (`ST_AsMVTGeom`), which this endpoint bypasses entirely.

The frontend work is a coordinated change across three surfaces: (1) `SyncLayerInput` needs `is_3d` and `feature_count` so `syncLayersToMap` can branch on source type; (2) `map-sync.ts` gets a new GeoJSON source branch alongside the existing vector MVT branch; (3) a `geojson-z-adapter.ts` is added to the adapter registry handling `geojson` source type, since existing adapters assume a `source-layer` (MVT-specific) but GeoJSON sources have no `source-layer`. The indicator badge lives in the layer list UI, not in the map itself.

**Primary recommendation:** Model the new endpoint directly after `list_features` in `features/router.py` — same auth chain (`get_current_active_user` + `check_dataset_access`), same dataset lookup, same 404 pattern. The response structure wraps the FeatureCollection in a thin envelope with `truncated` and `total_count` at the top level so MapLibre can still load `response.features` as a FeatureCollection.

## Architecture Patterns

### Backend: Response Envelope Pattern

The locked decisions require both `truncated: true` and `total_count: N` in the response alongside the FeatureCollection. RFC 7946 GeoJSON has no standard field for truncation metadata. Two options:

**Option A — GeoJSON extension (top-level extra fields):** RFC 7946 §6.1 states "GeoJSON objects MAY have other members". Add `truncated` and `total_count` as extra top-level fields on the FeatureCollection object itself. MapLibre's `addSource` with `type: "geojson"` accepts a `data` object that is a FeatureCollection — it only looks at `type` and `features`, and ignores unknown fields. This is simpler and still parseable by strict GeoJSON consumers.

**Option B — Outer envelope:** `{ "type": "FeatureCollection", "features": [...], "truncated": true, "total_count": N }`. This is structurally the same as Option A since a FeatureCollection IS the outer object.

**Recommendation (Claude's discretion):** Use Option A — a FeatureCollection with extra top-level fields. The frontend passes the entire response to `map.addSource(..., { type: 'geojson', data: responseBody })` directly, since MapLibre ignores unknown fields. This avoids a response.features destructuring step. [ASSUMED — MapLibre ignores unknown FeatureCollection fields; should be confirmed against maplibre-gl v5 source code or docs]

### Backend: SQL Query for Z-Inclusive GeoJSON

**Critical finding:** `ST_AsGeoJSON(geom_4326)` already outputs Z coordinates when the geometry has them. No special flags or parameters are needed. Z passthrough is the default behavior of ST_AsGeoJSON. The existing service.py call `ST_AsGeoJSON(geom_4326, 6)::json` strips Z only if the geometry itself lacks Z — it does not actively strip Z. [VERIFIED: DESIGN.md §5 confirmed Z survives ingestion and reprojection; ST_AsGeoJSON behavior is standard PostGIS]

The new service function `get_features_with_z` is a simplified version of `get_features` without pagination, with LIMIT 5001 (fetch one extra to detect truncation), and using `geom_4326` directly. No `include_geometry=False` path needed since Z delivery is the entire purpose.

```python
# Source: features/service.py pattern + PostGIS ST_AsGeoJSON docs
select_cols = (
    "gid, ST_AsGeoJSON(geom_4326, 6)::json AS geometry, "
    "to_jsonb(t.*) - 'gid' - 'geom' - 'geom_4326' AS properties"
)
sql = (
    f"SELECT {select_cols} FROM data.{table_name} t "
    f"ORDER BY gid LIMIT :limit"
)
# Fetch 5001; if len(rows) > 5000: truncated=True, return rows[:5000]
```

### Frontend: Source Type Branching in syncLayersToMap

`syncLayersToMap` in `map-sync.ts` currently has two branches:
1. `token?.kind === 'raster'` → raster adapter
2. All others → vector (MVT) source with `type: 'vector'`

A third branch is needed for GeoJSON-Z:

```typescript
// New branch in syncLayersToMap (after raster branch)
if (layer.is_3d && layer.feature_count != null && layer.feature_count <= 5000) {
  // GeoJSON-Z path
  const geojsonUrl = `/api/datasets/${layer.dataset_id}/features.geojson?include_z=true`;
  if (!map.getSource(sourceId)) {
    // Fetch FeatureCollection then addSource
    fetchGeoJsonZ(geojsonUrl, token).then(data => {
      map.addSource(sourceId, { type: 'geojson', data });
      adapter.addLayers(map, adapterInput);  // geojson-z adapter
    });
  }
  desiredSources.add(sourceId);
  continue;
}
```

However, `syncLayersToMap` is synchronous. Async data fetch introduces complexity. The cleaner approach: fetch the GeoJSON-Z URL as part of the tile token fetch step (or as a separate effect in the component), store the fetched FeatureCollection in a Map keyed by `layer.id`, and pass it into `syncLayersToMap` as an additional parameter.

**Recommended approach:** Add an optional `geojsonDataMap?: Map<string, GeoJSON.FeatureCollection>` parameter to `syncLayersToMap`. The caller (ViewerMap, BuilderMap) fetches GeoJSON-Z data in a `useEffect` and passes it in. This keeps `syncLayersToMap` synchronous and makes the fetch lifecycle explicit in the component. [ASSUMED — alternative is adding async support to syncLayersToMap; both work but sync is simpler]

### Frontend: GeoJSON-Z Adapter

Existing adapters all use `'source-layer': sourceLayer` in their `addLayer` calls. This is required for vector (MVT) sources but is invalid for GeoJSON sources. A new adapter (or adapter branch) handles the GeoJSON case. The simplest approach: check `adapterInput.sourceType === 'geojson'` in each adapter's `addLayers` and omit `'source-layer'` when true. Or: add a `geojson-circle`, `geojson-fill`, `geojson-line` adapter family.

**Simpler approach (recommended):** Add `sourceType?: 'vector' | 'geojson'` to `AdapterLayerInput`. All existing adapters default to vector behavior. In each adapter's `addLayer` call, spread `...('source-layer' in spec ? { 'source-layer': sourceLayer } : {})` based on `sourceType`. Since only three adapters (circle, fill, line) are ever used for vector datasets and Z-capable datasets are vector datasets, this minimal change avoids duplicating all adapter code.

[VERIFIED: types.ts — AdapterLayerInput has no sourceType field today; adding it is non-breaking as existing consumers omit it]

## Common Pitfalls

### Pitfall 1: `source-layer` in GeoJSON MapLibre Layers

**What goes wrong:** Adding a layer with `'source-layer': 'data.tablename'` against a `type: 'geojson'` source causes a silent no-render (MapLibre ignores the layer or throws in strict mode).

**Why it happens:** `source-layer` is a concept exclusive to vector tile sources. A GeoJSON source has exactly one layer — specifying `source-layer` is undefined behavior.

**How to avoid:** When registering map layers for a GeoJSON source, omit `'source-layer'` entirely. The `sourceType` flag on `AdapterLayerInput` drives this.

**Warning signs:** Layer added successfully via `map.addLayer()` with no error, but no features render on the map.

### Pitfall 2: `ST_AsGeoJSON` Precision and Z Passthrough

**What goes wrong:** Using `ST_AsGeoJSON(geom_4326, 6)` — existing code uses precision=6. This rounds coordinates to 6 decimal places, which is ~11 cm for XY and applies the same rounding to Z. For elevation data this may be acceptable (cm precision) but could be a surprise.

**Why it happens:** The precision argument applies to all coordinate components, including Z.

**How to avoid:** Use precision=3 for elevation-heavy data (millimeter precision) or keep 6. For phase scope, keep the same precision=6 used everywhere else. [ASSUMED — no user requirement specified precision; 6 decimal places on Z is fine for most use cases]

### Pitfall 3: Fetch Timing — GeoJSON Source Added Before Map Ready

**What goes wrong:** The GeoJSON fetch completes before `map.isStyleLoaded()` is true, and `map.addSource()` fails silently.

**Why it happens:** `useEffect` fetch can complete quickly; if map initialization is slower, the source add hits the map before it's ready.

**How to avoid:** Gate `map.addSource()` on `mapReady && map.isStyleLoaded()` — same guard already used by the existing sync effects in `ViewerMap.tsx` (line 469). Store fetched data in state; the sync effect re-runs when `mapReady` becomes true.

### Pitfall 4: 999.2 Frontend Gaps Block This Phase

**What goes wrong:** `DatasetResponse` in `api.ts` has `is_3d` as an optional field (added at lines 127-130 per verification), but `MapLayerResponse` and `SharedLayerResponse` do NOT have `is_3d`. The frontend auto-switch logic (D-07) requires `is_3d` on the layer response types.

**Why it happens:** Phase 999.2 added `is_3d` to `DatasetResponse` only. The map layer API responses are separate serialization paths with their own Pydantic schemas.

**How to avoid:** Phase 999.3 Plan 01 must include: add `is_3d: bool | None` to backend `MapLayerResponse` and `SharedLayerResponse` Pydantic schemas, and their corresponding TypeScript interfaces.

[VERIFIED: api.ts lines 707-729 (`MapLayerResponse`), 821-839 (`SharedLayerResponse`) — neither has `is_3d`]

### Pitfall 5: Style Reload Wipes GeoJSON Sources

**What goes wrong:** When the basemap changes, `map.setStyle()` wipes all custom sources. The `style.load` handler in `ViewerMap.tsx` (line 544) calls `runSync()` which re-adds vector sources from tile URLs. GeoJSON-Z data was fetched earlier and stored — it needs to be re-added too.

**Why it happens:** MapLibre's `setStyle()` is a full style reset. The existing `managedSourcesRef` + `style.load` hook re-syncs everything, but only if the GeoJSON data is available in state (not re-fetched).

**How to avoid:** Store fetched GeoJSON-Z data in a React ref or state map. The sync function picks it up from there on every call, including after style reload. Pattern already established by `demTileUrlRef` in ViewerMap for terrain source.

### Pitfall 6: RBAC on the New Endpoint Must Match Exactly

**What goes wrong:** Using `get_optional_user` instead of `get_current_active_user` would allow anonymous access to private datasets.

**Why it happens:** `get_optional_user` returns `None` for unauthenticated requests; `check_dataset_access` handles `None` through `check_dataset_access_or_anonymous` but that function allows public+published datasets to anonymous users.

**How to avoid:** Use `get_current_active_user` (requires authentication) as used by the existing `list_features` endpoint. D-06 says full auth + RBAC parity, not optional-user parity. [VERIFIED: visibility.py lines 127-157 — `check_dataset_access` requires a non-None user]

## Code Examples

### Backend: New Service Function

```python
# Source: features/service.py pattern (lines 63-150)
async def get_features_with_z(
    db: AsyncSession,
    table_name: str,
    *,
    cap: int = 5000,
) -> tuple[list[dict], bool, int]:
    """Fetch up to `cap` features with Z coordinates preserved.

    Returns (rows, truncated, total_count).
    Uses ST_AsGeoJSON which natively outputs Z when geometry has Z.
    """
    _validate_table_name(table_name)

    select_cols = (
        "gid, ST_AsGeoJSON(geom_4326, 6)::json AS geometry, "
        "to_jsonb(t.*) - 'gid' - 'geom' - 'geom_4326' AS properties"
    )
    # Fetch cap+1 to detect truncation without a separate COUNT query
    data_sql = (
        f"SELECT {select_cols} FROM data.{table_name} t "
        f"ORDER BY gid LIMIT :limit"
    )
    result = await db.execute(text(data_sql).bindparams(limit=cap + 1))
    rows = [dict(row._mapping) for row in result.all()]

    truncated = len(rows) > cap
    if truncated:
        rows = rows[:cap]

    # Get total count (use cached feature_count from caller when available)
    count_sql = f"SELECT COUNT(*) FROM data.{table_name} t"
    count_result = await db.execute(text(count_sql))
    total_count = count_result.scalar_one()

    return rows, truncated, total_count
```

### Backend: New Router Endpoint

```python
# Source: features/router.py pattern (lines 49-185)
@features_router.get(
    "/{dataset_id}/features.geojson",
    response_class=JSONResponse,
    responses={200: {"content": {"application/geo+json": {}}}, **ERROR_RESPONSES_AUTH},
)
async def get_features_geojson_z(
    dataset_id: uuid.UUID,
    include_z: bool = Query(False, description="Include Z coordinates in geometry output"),
    user: User = Depends(get_current_active_user),
    db: AsyncSession = Depends(get_db),
) -> JSONResponse:
    """Return up to 5,000 features as RFC 7946 GeoJSON with Z coordinates."""
    dataset = await get_dataset(db, dataset_id)
    if dataset is None:
        raise HTTPException(status_code=404, detail="Dataset not found")

    await check_dataset_access(db, dataset, dataset_id, user)

    if dataset.geometry_type is None:
        raise HTTPException(status_code=400, detail="Dataset has no geometry")

    rows, truncated, total_count = await get_features_with_z(
        db, dataset.table_name, cap=5000
    )

    features = [
        GeoJSONFeature(id=row["gid"], geometry=row["geometry"], properties=row["properties"])
        for row in rows
    ]

    body = {
        "type": "FeatureCollection",
        "features": [f.model_dump(mode="json") for f in features],
        "truncated": truncated,
        "total_count": total_count,
    }

    return JSONResponse(content=body, media_type="application/geo+json")
```

## Open Questions

1. **Should `get_features_with_z` use the cached `feature_count` for `total_count` or always query COUNT?**
   - What we know: `get_features` uses `cached_feature_count` when no filters are active (service.py line 140). The cap+1 trick detects truncation without a COUNT query.
   - What's unclear: If `feature_count` is passed from the caller, we can skip the extra COUNT. But total_count in the response should reflect the actual total, not the cached value which might be stale.
   - Recommendation: Accept `cached_feature_count: int | None` as a parameter; use it for `total_count` when `truncated` is False (all features returned = cached count is accurate). When `truncated` is True, run `COUNT(*)` for accuracy.

2. **Which layer types need GeoJSON-Z support?**
   - What we know: 3D datasets are typically POINT Z (LiDAR), LINESTRING Z (road centerlines), POLYGON Z (building footprints). All three map to circle, line, fill adapters.
   - What's unclear: Heatmap adapter (point-based) also needs GeoJSON-Z support if heatmap render mode is used on a 3D point dataset.
   - Recommendation: Add `sourceType` branching to all four adapters (circle, line, fill, heatmap) for completeness.

3. **Does the viewer (embed token path) need special handling for the GeoJSON-Z fetch?**
   - What we know: `ViewerMap.tsx` uses `setTransformRequest` to inject `X-Embed-Token` header on all map tile requests. But `fetch()` calls from React are not subject to `transformRequest`.
   - What's unclear: If a viewer embed token path tries to load GeoJSON-Z, the fetch needs to send the embed token as a header. The existing token flow sends `X-Embed-Token` on map-tile requests (via transformRequest), not on API fetches.
   - Recommendation: The GeoJSON-Z fetch function must accept an optional `embedToken` parameter and add `X-Embed-Token` header when present. The `apiFetch()` client in `api/client.ts` handles auth headers for JWT/API-key paths; embed token needs explicit handling.
