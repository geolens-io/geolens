## Phase Goal

Add `/api/datasets/{id}/features.geojson?include_z=true` endpoint that streams RFC 7946 GeoJSON with three-coordinate positions. Full auth + RBAC parity. Feature count cap at 5,000. Frontend auto-switches between MVT and GeoJSON-Z for small 3D datasets.

## Locked Decisions

- **D-01**: Single FeatureCollection response (not NDJSON streaming) — MapLibre GeoJSON source needs the full collection; 5,000 cap keeps payloads manageable
- **D-02**: Subtle indicator on auto-switch — show info badge "3D preview · 5,000 feature limit" so user understands why panning doesn't load new tiles
- **D-03**: Hard cap with silent truncation — return first 5,000 features; include `"truncated": true` and `"total_count": N` in response metadata; no error
- **D-04**: Single response, no pagination — one request returns up to 5,000 features; existing features API covers pagination needs
- **D-05**: No Z coordinate filtering — no `z_min`/`z_max` params; client-side filtering on ≤5,000 features is trivial
- **D-06**: Full auth + RBAC parity with existing dataset endpoints (from roadmap)
- **D-07**: Frontend auto-decides: use GeoJSON-Z source when `is_3d && feature_count <= 5000`, fall back to MVT otherwise (from roadmap)
- **D-08**: Feature count cap at 5,000 (from roadmap)

## Claude's Discretion

- Exact response JSON structure (wrapping FeatureCollection with metadata vs. pure GeoJSON)
- SQL query construction for Z-inclusive geometry extraction (ST_AsGeoJSON with Z)
- How to integrate the GeoJSON source into the existing layer adapter system
- Placement and styling of the "3D preview" indicator badge
- Whether to use `ST_AsGeoJSON` directly or build the GeoJSON in Python

## Deferred Ideas

- NDJSON streaming for large datasets (reconsider if cap is raised beyond 5,000)
- Server-side Z range filtering (`z_min`/`z_max` params)
- User-triggered 3D toggle (currently auto-switches silently with indicator)
