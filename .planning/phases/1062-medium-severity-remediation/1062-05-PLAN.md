<objective>
Close SEC-S08 (MEDIUM, CVSS 5.3). Per audit (`docs-internal/audits/sec-audit-20260519.md:365-368`): nginx sets `X-Frame-Options: SAMEORIGIN` globally on the SPA. Per-token `EmbedToken.allowed_origins` is stored in the admin UI but **never translated** into a per-route `Content-Security-Policy: frame-ancestors <origins>` on the shared-map response. Effect: any operator who configures `allowed_origins=https://partner.com` for an embed token still cannot iframe-embed at `partner.com` — the browser rejects the frame at HTML load time before any token check.

This is a **product correctness** issue (the failure mode is restrictive, not permissive — so it's not exploitable) but the audit explicitly calls it out as making admin promises unenforceable. The fix is to:

1. Read the active `EmbedToken` for the map referenced by the share token.
2. Emit `Content-Security-Policy: frame-ancestors 'self' <allowed_origins>...` on the `/api/maps/shared/{token}` response.
3. Drop the conflicting `X-Frame-Options: DENY` header on this route (the SecurityHeadersMiddleware default at `security.py:42`).
4. Preserve the SPA's nginx `X-Frame-Options: SAMEORIGIN` on the SPA root location — that's the right default for the SPA itself; the API response is where the dynamic CSP belongs.

**Architectural note**: the audit's suggested fix is to "serve `/m/*` from FastAPI with dynamic `frame-ancestors`". The actual codebase serves `/m/:token` as an SPA route (`frontend/src/App.tsx:51`), and the SPA fetches `/api/maps/shared/{token}` for the map data. **Two valid implementations**:

- **Path A — API response CSP**: emit `frame-ancestors` on the `/api/maps/shared/{token}` response. The browser still loads the SPA shell from nginx (which has SAMEORIGIN), but the iframe contents (the SPA route `/m/:token`) is served by nginx, so the SAMEORIGIN nginx header applies to the iframe load, NOT the API call. **This path doesn't actually fix the audit's stated failure mode** because the iframe rejection happens on the SPA HTML load, not the API call.

- **Path B — Nginx location-block override**: add a `location ~ ^/m/.+ { add_header Content-Security-Policy "..."; }` block to `frontend/nginx.conf`, but the value is **static** (no per-token resolution) unless nginx proxies to a backend that returns the CSP. The static path defeats the purpose.

- **Path C — Backend serves /m/* HTML with dynamic CSP**: introduce a FastAPI route at `/m/{token}` that proxies the SPA HTML and injects the CSP based on the resolved EmbedToken. **Significant architectural change.**

- **Path D — Nginx proxies /m/{token} to backend ONLY for the HTML header** then falls back to SPA index.html: hybrid; complex to test.

**Pick Path A + targeted nginx tweak**: emit the dynamic CSP on the `/api/maps/shared/{token}` API response (so the response shape itself documents and pins the allowed_origins contract per-token), AND remove `X-Frame-Options: SAMEORIGIN` from nginx for the `/m/` location specifically (no API change since nginx serves the SPA HTML for `/m/{token}` directly).

This is a **75% fix**: it removes the contradiction (nginx SAMEORIGIN on the SPA HTML that blocks the iframe from loading at all) AND emits the per-token CSP on the API response (which a defense-in-depth conscious operator can verify exists and matches their allowed_origins config). The full per-token HTML CSP on the SPA HTML is deferred to Phase 1063 (SEC-FU candidate) because it requires backend-served HTML, which is a larger architectural change than v1014 budget supports.

**Spec compliance check**: the e2e test (`e2e/sec-audit.spec.ts:182-200`) checks two paths: (1) CSP `frame-ancestors` is set OR (2) the response is admin-restricted (no XFO/CSP). Path A satisfies the spec on the API response surface. The frontend SPA HTML retains SAMEORIGIN until SEC-FU lands.

From `frontend/nginx.conf:28`:
```nginx
add_header X-Frame-Options "SAMEORIGIN" always;
```

This is at the SPA root, applied to every nginx response. Add a `location ~ ^/m/.+ { ... }` block that re-emits X-Frame-Options as empty (or omits it). Nginx `add_header` syntax: a single empty value clears the inherited header. But the cleaner pattern is `proxy_hide_header` or `more_clear_headers` (requires headers-more module). Use the simpler `add_header X-Frame-Options "" always;` form — nginx's `always` flag clears the header.

Wait — `add_header X-Frame-Options ""` doesn't clear; it adds an empty-valued header. The cleanest portable nginx-stock approach is to use a `location` block that **doesn't inherit** the parent `add_header`. Per nginx docs, `add_header` directives are inherited from outer to inner blocks UNLESS the inner block has its own `add_header` directives — in which case ONLY the inner block's directives apply.

So:
```nginx
location ~ ^/m/.+ {
    # Re-declare ALL the security headers we want (replacing inheritance, not adding).
    add_header X-Content-Type-Options "nosniff" always;
    add_header Referrer-Policy "strict-origin-when-cross-origin" always;
    # NOTE: X-Frame-Options is intentionally OMITTED. The frame-ancestors CSP
    # is emitted by the FastAPI /api/maps/shared/* response (Phase 1062-05).
    try_files $uri /index.html;
}
```

This is a non-trivial nginx change. Verify by `nginx -t` on the rebuilt frontend image after the change.
