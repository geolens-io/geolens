<objective>
The audit's IMPORT-03 (HIGH) finding: every Upload File commit fires:

    [ERROR] Cannot update a component (%s) while rendering a different component (%s).
    To locate the bad setState() call inside %s, follow the stack trace
    ImportPage UploadForm UploadForm

Root cause (verified by reading `UploadForm.tsx`): three callsites of `setPhase(...)` live inside `setEntries(prev => { ... })` updater functions. React 19's strict-mode rules forbid calling setState on one component while React is computing a setState on another. The three offending sites in `UploadForm.tsx`:

1. **Line 163** in `handleCommitSingle`:
        setEntries((prev) => {
          const updated = prev.map(...);
          const allDone = updated.every(...);
          if (allDone) setPhase('tracking');   // ‚Üê setState during setEntries updater
          return updated;
        });

2. **Line 208-212** in `handleCommitAll`:
        setEntries((prev) => {
          const hasTracking = prev.some((e) => e.status === 'tracking');
          if (hasTracking) setPhase('tracking');   // ‚Üê setState during setEntries updater
          return prev;
        });

3. **Line 236-240** in `removeEntry`:
        setEntries((prev) => {
          const updated = prev.filter((e) => e.id !== entryId);
          if (updated.length === 0) setPhase('idle');   // ‚Üê setState during setEntries updater
          return updated;
        });

The fix: derive the phase transition from `entries` via `useEffect`. The effect dep'd on `entries` runs AFTER React commits the setEntries change, satisfying React's strict-mode rule. The three callsite-specific decisions ("all done ‚Üí tracking", "has tracking ‚Üí tracking", "empty ‚Üí idle") collapse into a single effect that inspects the current `entries` shape.

Purpose: Close IMPORT-03 ‚Äî eliminate the React setState-in-render warning on every upload commit.
Output: Refactored `UploadForm.tsx` (the three setEntries blocks no longer call setPhase inline) + regression test asserting `console.error` is not called with "Cannot update a component" during a simulated commit flow.
</objective>

<interfaces>
<!-- BatchPhase + FileEntry status states from frontend/src/types/api.ts -->

`BatchPhase` is the discriminated string union for the form's overall phase:
  - 'idle'
  - 'uploading'
  - 'reviewing'
  - 'tracking'

`FileEntry.status` values referenced in the existing transition logic:
  - 'tracking' ‚Äî terminal-success per entry (a commit landed; backend is processing)
  - 'upload-failed' ‚Äî terminal-failure (upload step never succeeded)
  - 'commit-failed' ‚Äî terminal-failure (commit step rejected)
  - 'committing' ‚Äî transient (commit in flight)
  - 'preview', 'previewing', 'uploading' ‚Äî transient (earlier in pipeline)

The transition rules (factored from the three current inline branches):
  - If `entries` is empty AND phase is `reviewing`: go to `idle`.
  - If `entries` is non-empty AND every entry is in a terminal state (`tracking | upload-failed | commit-failed`) AND at least one is `tracking`: go to `tracking`.
  - Otherwise: stay in current phase (the effect should NOT force `tracking` if no entries reached tracking ‚Äî `handleCommitAll`'s current `hasTracking` guard).

Note: the audit only flags the React warning; it doesn't require a specific transition algorithm. The simplest correct refactor keeps the same transition decisions but moves them out of the updater function.

`setPhase` at line 48 is a memoized wrapper around `_setPhase` that calls `onPhaseChange?.(p)`:

    const setPhase = useCallback((p: BatchPhase) => {
      _setPhase(p);
      onPhaseChangeRef.current?.(p);
    }, []);

So calling `setPhase(...)` from a `useEffect` is structurally identical to calling it from an event handler ‚Äî both run after the commit.
