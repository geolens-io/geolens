## Summary

This is a **frontend-only, three-fix consistency phase**. The root cause is verified exactly as the audit describes: the **builder layer stack** excludes `render_mode:"terrain"` DEM layers (`map-stack.ts:543` filters them out of `makeDataEntries`) and surfaces terrain once as a synthetic `relief:terrain` entry (`map-stack.ts:389-468`), while **both legend surfaces** filter only on visibility + `show_in_legend` and never check `render_mode`/`is_dem`:
- builder `LegendPlugin.tsx:80-83` — `ctx.layers.filter((l) => l.visible && l.show_in_legend !== false)`
- viewer `LayerLegend.tsx:166-172` — `createViewerLayerEntries(layers).filter(({layer}) => layer.show_in_legend !== false)`

The render-side truth is already centralized: `isDemTerrainVisualSuppressed(layer)` (`map-sync.ts:52-58`) returns `layer.is_dem === true && style_config.render_mode === 'terrain'`, and the builder paint loop already `continue`s on it (`map-sync.ts:922-924`). Both legend surfaces must consume this same predicate (D-02) plus a synthetic "3D terrain" entry driven by `terrain_config` (D-01). The viewer surface needs `terrain_config` plumbed into `LayerLegend` — it currently does not receive it.

For Fix 2 (dedup), terrain bind/unbind is **already non-duplicating in `use-builder-layers.ts`**: `handleDEMTerrainBind` (`:1056-1065`) and `handleDEMTerrainUnbind` (`:1067-1077`) only mutate `localTerrainConfig` (map-level), never create layers. `handleSwitchMode` in `DEMEditorScene.tsx:165-189` mutates the *existing* layer's `style_config` in place. The duplicate-accumulation seen on map `8dd6a129` came from a **"Duplicate rendering" action** (`handleDuplicateRendering`, `:1090-1133`) being applied to a terrain DEM layer (the audit §10 confirms the M-004 "rename" suggestion spawned a 3rd copy). The real Fix-2 work is: (a) guard `handleDuplicateRendering` / render-mode switch so a 2nd `render_mode:"terrain"` layer can never be created on a dataset that already has one; (b) clear `terrain_config` when the terrain source layer is deleted (`handleRemove` `:287-330` and `handleBulkDelete` `:611-696` currently leave `terrain_config` dangling).

For Fix 3 (D-06/D-07), the builder's `applyTerrainConfig` (`BuilderMap.tsx:389-393`) hard-requires `render_mode === 'terrain'` on the DEM layer; relaxing it to match the viewer's `source_dataset_id`-only lookup is safe **but the POLISH-02 guard (`map-sync.ts:621-649`) currently suppresses the hillshade raster-dem consumer whenever terrain is enabled on the same dataset**, so a visible hillshade overlay on the terrain DEM will not paint. The resolution (below) is to use a **distinct raster-dem source instance** for the hillshade so both consumers coexist — the constraint is per-source, not per-dataset.

**Primary recommendation:** Add one shared predicate-consuming helper that both legend surfaces call (consuming `isDemTerrainVisualSuppressed`), inject `terrain_config` into the viewer legend, prepend a single synthetic "3D terrain" legend entry when `terrain_config.enabled`, guard duplicate-terrain-layer creation, clear `terrain_config` on terrain-source delete with a toast, and for Fix 3 give the hillshade overlay its own raster-dem source id (keyed per-layer) so it does not collide with the terrain mesh's `terrain-dem` source.

## D-07 Resolution (TOP-PRIORITY OPEN QUESTION) — definitive

### What the POLISH-02 guard does (verified)

`isHillshadeTerrainBound(layer, terrainConfig)` (`map-sync.ts:621-630`):
```ts
return (
  layer.is_dem === true &&
  terrainConfig?.enabled === true &&
  terrainConfig.source_dataset_id === layer.dataset_id
);
```
It is applied inside `syncRasterLayer` (`map-sync.ts:644-649`):
```ts
// POLISH-02: skip the hillshade raster-dem consumer when this DEM is already
// powering a terrain source. Two raster-dem consumers on the same DEM with
// mismatched tile sizes cause backfillBorder "dem dimension mismatch" errors.
if (useHillshade && datasetId != null && isHillshadeTerrainBound({ dataset_id: datasetId, is_dem: adapterInput.is_dem }, terrainConfig)) {
  return;
}
```
`useHillshade` is `adapterInput.is_dem === true && renderMode === 'hillshade'` (`:642`). So: **when terrain is enabled on dataset X, any `render_mode:"hillshade"` layer on dataset X is skipped entirely — no GL layer, no source.** That is exactly why map `8dd6a129` "lost" its visible hillshade overlay once both DEM layers became terrain-mode (audit §10).

### What HYG-01 (v1033) actually fixed (verified by the inline comment + source-id seam)

The error is MapLibre `backfillBorder` **"dem dimension mismatch"** (`map-sync.ts:615`, `:645`). Root cause documented in the code: **two `raster-dem` consumers on the *same MapLibre source* with mismatched `tileSize`**. The terrain mesh source is created by `ensureRasterDemTerrainSource` (`map-sync.ts:92-137`) with a fixed id `terrain-dem` and `tileSize: options.tileSize ?? 256`, `encoding: 'mapbox'`. A hillshade layer's raster-dem source is created by the hillshade adapter on the per-layer source id `source-${layer.id}` (`getSourceIdForLayer` `:553-555`) with `tileSize: token.tile_size ?? 256` (`syncRasterLayer:660`). **The two sources are different ids**, but `setTerrain` and a `hillshade` layer reading from two raster-dem sources that have *different tileSize values for the same underlying tiles* triggers MapLibre's shared-border DEM neighbour lookup to compare mismatched grid dimensions → the backfillBorder crash. `[VERIFIED: map-sync.ts:613-619, 644-649 inline docs]` `[ASSUMED: exact MapLibre internal mechanism — the dimension-mismatch is documented in our own comment, not independently re-confirmed against MapLibre source this session]`

### MapLibre constraint: can one source back both `setTerrain` and a hillshade/raster layer?

`[ASSUMED]` Based on training knowledge of MapLibre GL JS v5: **a single `raster-dem` source CAN technically back both `map.setTerrain({source})` and a `hillshade`-type layer** — this is a documented MapLibre pattern (terrain + hillshade off one DEM). The crash our codebase hit was NOT inherent to sharing — it was the **tileSize mismatch between two separate source instances** of the same DEM. The safest, already-supported path in our code is to keep them as **distinct source instances with matched tileSize**, OR share the single `terrain-dem` source. Given our `ensureRasterDemTerrainSource` owns `terrain-dem` independently of the layer-sync loop, and the hillshade adapter creates `source-${layer.id}`, the cleanest fit with existing architecture is **distinct sources** (option b below). This claim needs the D-07 live-MCP gate to confirm no `dem dimension mismatch` reappears.

### Recommended approach — Option (b): distinct raster-dem source instance for the visible hillshade

**Do NOT simply relax the guard to always run the hillshade consumer** (option a) — that re-creates the v1033 HYG-01 crash if the two sources' tileSize differ. Instead:

1. **Relax the builder mesh lookup (D-06):** Change `BuilderMap.tsx:389-393` to drop the `render_mode === 'terrain'` clause, matching `use-viewer-terrain.ts:32-39` (resolve the DEM by `dataset_id === terrain_config.source_dataset_id && isTerrainCapableDemLayer(layer)` only). This lets a `render_mode:"hillshade"` layer on the same dataset still drive the mesh via `terrain_config`. `[CITED: use-viewer-terrain.ts:32-39 — the proven viewer pattern]`

2. **Relax the POLISH-02 guard for the intended visible-hillshade case, conditioned on matched tileSize:** In `syncRasterLayer` (`map-sync.ts:644-649`), only skip the hillshade consumer when keeping it would re-introduce the mismatch. Concretely, allow the hillshade consumer to run when it uses its own per-layer source (`source-${layer.id}`, already the case via `getSourceIdForLayer:553-555`) **and** the hillshade source's `tileSize` equals the terrain source's `tileSize`. Because `ensureRasterDemTerrainSource` and `syncRasterLayer` both default `tileSize` from the same raster token (`token.tile_size ?? 256`), they will match for the same dataset in the normal case — so the guard can be narrowed from "skip whenever terrain-bound" to "skip only when tileSize would mismatch."

   Minimal change: pass the active terrain source's tileSize (or the terrain token) into `syncRasterLayer` and compare; if equal, do NOT early-return. The lines to change are `map-sync.ts:644-649` (the guard) and the `syncRasterLayer` signature `:633-640` (add the comparison input), plus the two call sites `BuilderMap.tsx:573` / `:629` and `map-sync.ts:954` that already thread `terrainConfig`.

3. **Why not share the single `terrain-dem` source:** sharing would require the hillshade layer to read from `terrain-dem` (a fixed id outside the layer-sync desired-set), which `removeStaleSourcesAndLayers` (`map-sync.ts:864-896`) and the SF-04 dedupe contract don't track — high risk of the hillshade layer being orphaned or the terrain source torn down underneath it. Distinct per-layer source is the lower-risk fit with the existing teardown machinery.

**Fallback / minimal-safe variant (if live-MCP shows the mismatch still fires):** keep the POLISH-02 guard intact (no visible hillshade on the *exact* terrain dataset) and document that mesh + hillshade requires the hillshade to point at a **second DEM dataset**. This preserves all v1033/v1035 guarantees at the cost of the Fix-3 convenience. The planner should treat Option (b) as primary and this as the documented fallback gate outcome.

**D-07 verdict:** Fix 3 is **safe as a narrowed-guard variant (Option b)**, NOT as a blanket guard removal. The MapLibre constraint is per-source-tileSize, not per-dataset — distinct sources with matched tileSize coexist. The phase MUST include a live-MCP assertion that `map.getTerrain()` stays attached AND the hillshade layer paints AND the console shows zero `dem dimension mismatch` / `backfillBorder` errors after enabling both on one DEM.

## Shared-Derivation Seam (Claude's-discretion recommendation)

**Recommendation: factor ONE shared helper, consumed by both legend surfaces.** A natural seam exists. Create a small pure function (suggested home: `frontend/src/components/builder/map-sync.ts` next to `isDemTerrainVisualSuppressed`, OR a new `frontend/src/lib/legend-entries.ts`) of the shape:

```ts
// Pseudocode — returns { dataLayers, hasActiveTerrain } given layers + terrain_config.
// Consumes isDemTerrainVisualSuppressed (single source of truth, per D-02).
export function deriveLegendLayers<T extends { is_dem?; style_config? }>(
  layers: T[],
  terrainConfig: MapTerrainConfig | null | undefined,
): { entries: T[]; showSyntheticTerrain: boolean } {
  const entries = layers.filter((l) => !isDemTerrainVisualSuppressed(l));
  const showSyntheticTerrain = terrainConfig?.enabled === true && !!terrainConfig.source_dataset_id;
  return { entries, showSyntheticTerrain };
}
```

Rationale:
- The two surfaces have **different input shapes** (`MapLayerResponse` w/ `visible`+`show_in_legend` for builder vs `SharedLayerResponse` w/ `show_in_legend` for viewer) and **different render markup**, so a fully shared *component* is not feasible. But the **predicate layer** (which entries to drop + whether to show the synthetic terrain row) is identical and is the part most prone to drift — exactly what this phase exists to prevent.
- `isDemTerrainVisualSuppressed` is already generically typed (`{ is_dem?; style_config?: Pick<StyleConfig,'render_mode'> }` at `map-sync.ts:52-55`) so it accepts both `MapLayerResponse` and `SharedLayerResponse` with no casts. The shared helper inherits that genericity.
- Keep the per-surface JSX for the synthetic entry (builder uses the `builder` i18n namespace + design-token swatch; viewer uses `common`/viewer namespace) — only the *decision* is shared.

This satisfies the CONTEXT preference ("prefer a shared seam if one exists naturally, to keep legend ↔ stack ↔ map-sync in lockstep") while respecting the real shape divergence.

### Pattern 1: Synthetic terrain entry mirrors the stack row
**What:** Prepend one entry representing active 3D terrain, driven by `terrain_config`, not by any DEM layer.
**When:** `terrain_config.enabled === true && terrain_config.source_dataset_id` truthy.
**Reference shape (mirror this):** `map-stack.ts:433-467` — `id: 'relief:terrain'`, title from the source layer's display name or "Terrain source missing", `visible: enabled`, a `Terrain` badge. The legend version is simpler (label + swatch/icon) but should use the same enable/source logic. `[CITED: map-stack.ts:389-468]`

### Pattern 2: Viewer terrain plumbing already exists — reuse the prop
**What:** `LayerLegend` does not currently receive `terrain_config`. `ViewerMap.tsx` already holds it (it passes `terrainConfig` to `useViewerTerrain`). Thread the same `terrain_config` into `LayerLegend` props.
**Reference:** `use-viewer-terrain.ts:17-29` shows `terrainConfig?: MapTerrainConfig | null` is the established viewer-side type. `[VERIFIED]`

### Pattern 3: Terrain bind/unbind is config-only (no layer creation)
**What:** Binding terrain mutates `localTerrainConfig`, never creates a layer.
**Reference:** `handleDEMTerrainBind` `use-builder-layers.ts:1056-1065` sets `{enabled:true, source_dataset_id, exaggeration}`; `handleDEMTerrainUnbind:1067-1077` sets `{enabled:false, source_dataset_id:null,...}`. `handleSwitchMode` (`DEMEditorScene.tsx:165-189`) mutates the existing layer's `style_config` in place (`onStyleConfigChange`) — it does NOT add a layer. **So Fix 2's dedup work is about the Duplicate action + delete cleanup, not bind/unbind.** `[VERIFIED]`

### Anti-Patterns to Avoid
- **Re-deriving the suppression predicate per surface** — violates D-02. Always call `isDemTerrainVisualSuppressed`.
- **Removing the POLISH-02 guard wholesale** — re-introduces the v1033 HYG-01 `dem dimension mismatch` crash. Narrow it instead (D-07 Option b).
- **Sharing the `terrain-dem` source id for the hillshade layer** — it lives outside the layer-sync desired-set and will be torn down or orphaned.
- **A blocking confirm dialog on terrain-source delete** — D-05 mandates a non-blocking toast.

## Common Pitfalls

### Pitfall 1: Viewer legend has no `terrain_config` today
**What goes wrong:** Adding the synthetic entry to `LayerLegend` without plumbing the prop = no terrain row in the viewer.
**How to avoid:** Add `terrainConfig` to `LayerLegendProps` (`LayerLegend.tsx:20-26`) and pass it from the caller (`ViewerMap.tsx`, which already owns it for `useViewerTerrain`). `[VERIFIED: LayerLegend.tsx:20-26 has no terrain prop]`

### Pitfall 2: `LegendPlugin` uses `builder` namespace; `LayerLegend` uses `common`
**What goes wrong:** Adding the label to the wrong namespace → missing-key in one surface, breaking 4-locale parity gate (D-10).
**How to avoid:** `LegendPlugin.tsx:78` → `useTranslation('builder')`; `LayerLegend.tsx:163` → `useTranslation('common')`. Add the synthetic-terrain label key to **both** namespaces across en/es/fr/de (8 JSON edits). `[VERIFIED]`

### Pitfall 3: `createViewerLayerEntries` does NOT filter — it only assigns keys
**What goes wrong:** Assuming the shared seam lives inside `createViewerLayerEntries` (`layer-identity.ts:25-32`). It is a pure key-mapper (33-line file); the filter is applied by the *caller* at `LayerLegend.tsx:169`. The viewer-side suppression filter must be added at the call site (or the new shared helper applied there), not inside `createViewerLayerEntries`.
**How to avoid:** Apply `deriveLegendLayers` at `LayerLegend.tsx:166-172` and at `LegendPlugin.tsx:80-83`. `[VERIFIED]`

### Pitfall 4: Duplicate-rendering is the real dedup vector, not bind/unbind
**What goes wrong:** Hardening bind/unbind for dedup (they're already safe) while leaving `handleDuplicateRendering` (`:1090-1133`) able to clone a `render_mode:"terrain"` layer.
**How to avoid:** Guard `handleDuplicateRendering` (and any render-mode-switch-to-terrain path) so it refuses/avoids producing a 2nd `render_mode:"terrain"` layer on a dataset that already has one (D-04). `[VERIFIED: audit §10 + use-builder-layers.ts:1090]`

### Pitfall 5: `PATCH /maps/{id}/layers` nulls `style_config` on omit
**What goes wrong:** Any save path that patches a terrain DEM layer without re-sending `style_config` wipes `render_mode:"terrain"` (documented project footgun; hit live in the audit remediation §intro and MEMORY.md). If Fix 2's dedup/cleanup patches a layer, it must re-include `style_config`.
**How to avoid:** Always re-send `style_config` when patching a layer that has one. `[CITED: map-audit-8dd6a129-20260531.md intro + MEMORY.md "PATCH /maps/{id}/layers nulls style_config on omit"]`

### Pitfall 6: Don't regress v1035 BLDR-03 stack suppression
**What goes wrong:** Changing `map-stack.ts:541-543` or the `relief:terrain` entry while wiring the legend.
**How to avoid:** The stack is already correct — Fix 1 only adds legend-side suppression + a legend synthetic entry. **Do not touch `makeDataEntries` / `makeTerrainReliefEntry`.** Mirror their logic in the legend; don't alter them. `[VERIFIED: map-stack.ts:389-543]`
