# Phase 999.17 Plan 01: Legend Terrain Consistency (Fix 1) Summary

Made the builder and viewer legends agree with the layer stack about terrain-suppressed DEM
layers: phantom `render_mode:"terrain"` DEM entries no longer appear, and a single synthetic
"3D terrain" entry (driven by `terrain_config`) is shown on both surfaces — derived from one
shared helper that consumes the existing `isDemTerrainVisualSuppressed` predicate.

- `e065fcbf` — Task 1: shared `deriveTerrainLegendEntry` helper + unit test (7/7)
- `50af118f` — Task 2: builder `LegendPlugin` terrain consistency + builder i18n + `LegendPlugin.terrain.test.tsx` (6/6)
- `9c8f4d2a` — Task 3: viewer `LayerLegend` terrain consistency + viewer i18n + extended `LayerLegend.test.tsx` (7/7)

## What Was Built

**Task 1 — Shared helper (`terrain-legend.ts`)**
- Re-exports `isDemTerrainVisualSuppressed` from `./map-sync` (single source of truth, D-02 — never re-derived).
- `deriveTerrainLegendEntry(terrainConfig, { labelKey })` returns one synthetic descriptor
  (`id: 'relief:terrain'`, `role: 'surface-terrain'`, mirroring `map-stack.ts:434`) when terrain
  is active (`enabled === true && source_dataset_id` truthy — the SAME truth the stack uses), else `null`.
- Surface-agnostic: carries an i18n label KEY, no react-i18next dependency inside the helper.

**Task 2 — Builder `LegendPlugin.tsx`**
- Filter extended to exclude `isDemTerrainVisualSuppressed` layers (D-02).
- Synthetic "3D terrain" row prepended at the TOP of the legend (advisory A1), using a `Mountain`
  lucide icon as the swatch and `t('plugins.legend.terrain3d')`.
- Painting hillshade-mode relief layers remain normal entries (D-03 — they are not suppressed).
- `terrainConfig` threaded into `PluginContext` (smallest plumbing change) and supplied from
  `pluginCtx` in `MapBuilderPage.tsx` via the already-available `layers.localTerrainConfig`.
- `plugins.legend.terrain3d` added to all four `builder.json` locales.

**Task 3 — Viewer `LayerLegend.tsx`**
- Filter extended to exclude `isDemTerrainVisualSuppressed` layers (D-02).
- Synthetic entry injected at the top of the `<ul>` (A1) using the same Mountain icon and
  `t('viewer.legend.terrain3d')`. New `terrainConfig` prop, threaded from both
  `PublicViewerPage` and `PublicMapViewerPage` (`data.terrain_config ?? null`).
- Synthetic entry is LEGEND-ONLY: injected in LayerLegend's local derivation, NOT in the shared
  `createViewerLayerEntries` (which `ViewerMap.tsx` also consumes for real map-layer wiring) — so
  ViewerMap never treats the synthetic entry as a paintable layer. `layer-identity.ts` untouched.
- `viewer.legend.terrain3d` added to all four `common.json` locales.

## Deviations from Plan

**1. [Helper-naming] `terrain-legend.ts` re-exports the predicate directly**
- The action suggested "a thin re-export (or pass-through wrapper)". Chose the direct
  `export { isDemTerrainVisualSuppressed }` re-export (cleaner, identical type) rather than a
  redundant wrapper. Both legend surfaces import it from `terrain-legend.ts`.

**2. [Test file name] Builder test added as `LegendPlugin.terrain.test.tsx`, not `LegendPlugin.test.tsx`**
- The plan's `files_modified` named `LegendPlugin.test.tsx`, but an existing
  `LegendPlugin.test.ts` (8 pure-logic helper tests for `expressionColumn`/`displayColumn`)
  already lives in that `__tests__` dir. To avoid clobbering it and keep render-based React tests
  in a `.tsx` file, the new suite is `LegendPlugin.terrain.test.tsx`. The existing
  `LegendPlugin.test.ts` was left untouched and still passes (8/8).

**3. [layer-identity.ts NOT modified — intentional]**
- The plan frontmatter listed `layer-identity.ts` in `files_modified`, but the Task 3 action body
  and RESEARCH explicitly require NOT injecting the synthetic entry into the shared
  `createViewerLayerEntries` (also consumed by `ViewerMap.tsx`). The synthetic entry is therefore
  kept in LayerLegend's local derivation and `layer-identity.ts` is unchanged. This follows the
  authoritative task instruction over the frontmatter file list.

**4. [i18n namespace — common.json, not a viewer.json]**
- Viewer keys went into `common.json` (`viewer.legend.terrain3d`) because `LayerLegend` uses
  `useTranslation('common')`. This matches the plan's confirmed_seams; there is no separate
  `viewer.json` namespace.

**5. [Plumbing scope] PluginContext + two public viewer pages gained a small prop**
- Beyond the named files, `PluginContext` (`types.ts`) gained an optional `terrainConfig` field and
  `MapBuilderPage.tsx` supplies it in `pluginCtx`; both public viewer pages pass
  `terrainConfig={data.terrain_config ?? null}` to `LayerLegend`. These are the minimal plumbing
  changes required to drive the synthetic entry (Rule 3 — blocking: the surfaces had no terrain
  config otherwise). All listed in key-files above.
