# Phase 999.17 Plan 02: DEM Dedup + Delete Teardown Summary

Two systemic fixes to the builder terrain flow so the duplicate-DEM-layer bug (map `8dd6a129` drifted to 3 terrain layers) cannot recur, and dangling `terrain_config` is cleared when its backing DEM layer is deleted — both with non-blocking toasts and full en/es/fr/de i18n parity.

## What shipped

### Task 1 — Terrain DEM dedup (D-04) — commit `6ea5aa5e`
- Added a guard at the top of `handleDuplicateRendering` (`use-builder-layers.ts`): if the source layer is a `render_mode:"terrain"` DEM layer (detected via the already-imported `isDemTerrainVisualSuppressed`), the duplicate is refused (early return) with a non-blocking `toast.info(t('toasts.terrainDuplicateBlocked'))`. A DEM dataset can only back 3D terrain once; `buildDuplicateRenderingInput` copies `dataset_id` + `style_config` verbatim, so duplicating a terrain layer would always create a second terrain layer on the same dataset — the exact accumulation vector identified in RESEARCH Pitfall 4.
- Bind/switch (`DEMEditorScene.handleSwitchMode` → `onTerrainBind`/`handleDEMTerrainBind`) was **verified already non-duplicating**: `handleSwitchMode` mutates the existing layer's `style_config.render_mode` in place via `onStyleConfigChange`, and `handleDEMTerrainBind` only sets `localTerrainConfig` (map-level), never appends a layer. No change required there (matches RESEARCH Pattern 3).
- Reducer tests added to `use-builder-layers.test.ts` ("terrain DEM dedup (D-04)" describe): refuse-to-duplicate-terrain asserts exactly one terrain layer per dataset and that the create mutation never fires; hillshade-DEM and ordinary-vector duplication remain unaffected (regression pins).
- i18n key `toasts.terrainDuplicateBlocked` added to all four locales.

### Task 2 — Delete-teardown (D-05 + Advisory A2) — commit `645980b1`
- New helper `shouldClearTerrainOnDelete(remainingLayers, terrainConfig)` in `builder-layer-mutations.ts`: returns true only when terrain is enabled with a `source_dataset_id` AND no remaining layer is a DEM (`is_dem === true`) layer for that dataset. Keys on dataset identity, not "a DEM was deleted" (Advisory A2).
- Wired into both delete paths in `use-builder-layers.ts`:
  - `handleRemove` (single delete) — computes remaining layers, clears `localTerrainConfig` (`{enabled:false, source_dataset_id:null, exaggeration: normalized}`), marks dirty, fires `toast.success(t('toasts.terrainDisabledSourceRemoved'))`.
  - `handleBulkDelete` (batch delete) — same teardown applied against the post-batch remaining set.
  - `localTerrainConfig` added to both callbacks' dependency arrays.
- New test file `use-builder-layers.terrain-teardown.test.ts` (8 cases across `handleRemove` + `handleBulkDelete`): clear-on-terrain-source-delete + toast; **A2 negative** (delete a different DEM in a multi-DEM map → `terrain_config` preserved, no terrain toast); same-dataset sibling survives → preserved; vector delete → preserved. Mocks `sonner` (toast spies) and `@/api/maps` `bulkDeleteLayersApi`.
- i18n key `toasts.terrainDisabledSourceRemoved` added to all four locales.

## Deviations from Plan

### Auto-fixed / scope clarifications (no architectural changes)

1. **[Plan interpretation] Dedup enforced at the duplicate vector, not bind/unbind.** The plan's T1 wording targets bind/switch reuse. On inspection (and per RESEARCH Pattern 3 / Pitfall 4), bind/switch is already non-duplicating — the real and only duplicate-`terrain` vector is `handleDuplicateRendering`. The guard was placed there. Bind/switch left unchanged (intentionally) and documented.
2. **[Helper placement]** Teardown predicate factored into `builder-layer-mutations.ts` (`shouldClearTerrainOnDelete`) rather than inlined twice, so `handleRemove` and `handleBulkDelete` share one dataset-identity rule — keeps the two paths in lockstep and unit-testable.

No other deviations. POLISH-02 guard (`map-sync.ts`), `BuilderMap.tsx` terrain resolution, and Plan 1 legend code were **not touched**.
