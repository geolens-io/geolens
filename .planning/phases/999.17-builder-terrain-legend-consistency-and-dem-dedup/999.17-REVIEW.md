# Phase 999.17: Code Review Report

**Reviewed:** 2026-06-01
**Depth:** deep (cross-file: legend ↔ stack ↔ map-sync, delete state machine, viewer parity)
**Status:** findings

## Summary

Fix 1 (legend consistency) and Fix 2 dedup-guard (D-04) are implemented correctly: the shared `deriveTerrainLegendEntry` helper handles the no-terrain / disabled / no-source cases per D-02/D-03, the synthetic entry is injected legend-locally (never into the shared `createViewerLayerEntries` output that `ViewerMap` consumes for real wiring), `terrainConfig` is plumbed through `PluginContext` and both public viewer pages, and i18n parity is complete across en/es/fr/de for all three new keys. The Fix 3 resolver change (D-06) is also correct in isolation — only the `render_mode === 'terrain'` clause was dropped, `terrainLayerKey` still keys on render_mode, and there is no accidental widening.

However, two correctness problems remain that the green gates could not catch:

1. **Fix 3 does not deliver its stated goal.** The phase claims "mesh + visible hillshade on one DEM," but the POLISH-02 guard that suppresses the hillshade was left unchanged — directly contradicting the RESEARCH D-07 Resolution, which explicitly required narrowing that guard. The mesh attaches, but the hillshade overlay still does not paint.
2. **Fix 2 delete-teardown leaks state on failure.** Both delete paths clear `terrain_config` optimistically but never restore it when the delete API fails and the layers are rolled back.

## Blocker Issues

### BL-01: Fix 3 stated goal (visible hillshade on the terrain DEM) is not achieved — POLISH-02 guard still suppresses it; SUMMARY contradicts RESEARCH

**File:** `frontend/src/components/builder/map-sync.ts:642-649` (unchanged) + `frontend/src/components/builder/BuilderMap.tsx:389-397`

**Issue:** Fix 3 / D-06 is scoped as "relax the builder terrain lookup so a single DEM can drive both the 3D mesh and a visible hillshade overlay" (CONTEXT domain item 3; D-06). The resolver change in `BuilderMap.tsx` correctly drops the `render_mode === 'terrain'` clause so a hillshade-mode DEM layer drives the mesh. But the POLISH-02 guard in `syncRasterLayer` is untouched:

```ts
// map-sync.ts:644-649 (still present, unchanged)
if (useHillshade && datasetId != null &&
    isHillshadeTerrainBound({ dataset_id: datasetId, is_dem: adapterInput.is_dem }, terrainConfig)) {
  return;   // skips the hillshade GL layer + source entirely
}
```

`isHillshadeTerrainBound` returns true for **any** `render_mode:"hillshade"` layer whose `dataset_id === terrain_config.source_dataset_id` (`map-sync.ts:625-629`). So the exact layer Fix 3 enables to drive the mesh is the same layer this guard refuses to paint. Net result: mesh attaches, hillshade does **not** render — Fix 3's user-visible outcome ("mesh + visible hillshade on one DEM") is not delivered.

This directly contradicts the phase's own RESEARCH. `999.17-RESEARCH.md` D-07 Resolution is explicit:
- line 131: "when terrain is enabled on dataset X, any `render_mode:"hillshade"` layer on dataset X is **skipped entirely — no GL layer, no source.** That is exactly why map 8dd6a129 'lost' its visible hillshade overlay."
- line 143-147: "**Do NOT simply relax the guard**... Instead: 1. Relax the builder mesh lookup (D-06)... 2. **Relax the POLISH-02 guard for the intended visible-hillshade case**, conditioned on matched tileSize."
- line 155 verdict: "Fix 3 is **safe as a narrowed-guard variant (Option b)**, NOT as a blanket guard removal."

Yet `999.17-03-SUMMARY.md:25` claims the opposite: "D-07: POLISH-02 guard in map-sync.ts left byte-for-byte unchanged (**RESEARCH proved it does not block visible hillshade**)." RESEARCH proved the inverse. The summary's justification is false.

Why the gates passed anyway: the only Fix-3 unit test (Test D in `BuilderMap.terrain-visibility.test.tsx`) asserts `setTerrain({source})` fires — i.e. the mesh attaches. It does **not** assert that the hillshade GL layer is added/painted. The live-MCP close-gate per `999.17-03-SUMMARY.md:101` was supposed to assert "hillshade relief visible," but the SUMMARY's "close-gate passed" claim should be distrusted given the guard is provably still active for this case (the same fabrication failure mode documented in MEMORY "Verify, don't fabricate evidence").

**Fix:** Either (a) implement RESEARCH Option (b) — narrow the POLISH-02 guard so the hillshade consumer runs on its own per-layer source (`source-${layer.id}`) when its tileSize matches the terrain source's tileSize, threading the terrain token's tileSize into `syncRasterLayer` and only early-returning on mismatch; OR (b) explicitly accept the RESEARCH "minimal-safe fallback" (hillshade must point at a *second* DEM dataset), drop the "mesh + visible hillshade on one DEM" claim from the phase scope, and correct `999.17-03-SUMMARY.md:25` to state the guard intentionally still suppresses same-dataset hillshade. Do not ship the current state, which claims a capability the code does not provide. A live-MCP assertion of an actually-painted hillshade layer (not just `getTerrain()` non-null) must gate whichever path is chosen.

## High Issues

### HI-01: Delete-teardown clears `terrain_config` optimistically but never rolls it back when the delete fails

**File:** `frontend/src/components/builder/hooks/use-builder-layers.ts:306-319` (handleRemove) and `:653-665` (handleBulkDelete)

**Issue:** Both delete paths clear `localTerrainConfig` (and toast "3D terrain disabled") **synchronously, before the API confirms the delete**. On failure they restore `localLayers` from the `previousLayers` snapshot but leave `localTerrainConfig` cleared:

- `handleRemove` onError (`:338-343`): `setLocalLayers(previousLayers)` only — terrain_config stays `{enabled:false, source_dataset_id:null}`.
- `handleBulkDelete` full-failure (`:690-694`): `setLocalLayers(previousLayers)` only.
- `handleBulkDelete` partial-failure (`:697-722`): re-inserts the failed DEM layer, but if the terrain-source DEM was among the failures, terrain_config is still cleared.

Result: the DEM layer reappears in the stack but 3D terrain is silently disabled and the user must re-bind it — a layers↔terrain_config state drift, and the user already saw a misleading "3D terrain disabled — its source layer was removed" toast for a delete that never happened. `setHasUnsavedChanges(true)` was also set, so the bogus cleared config can be saved.

The terrain-teardown unit tests (`use-builder-layers.terrain-teardown.test.ts`) all mock the mutation to succeed synchronously, so the rollback path is entirely untested.

**Fix:** Snapshot the prior terrain config alongside `previousLayers` and restore it on every failure/rollback branch. E.g. capture `const previousTerrainConfig = localTerrainConfig;` before the optimistic clear, and in each error/full-failure/relevant-partial-failure branch call `setLocalTerrainConfig(previousTerrainConfig)`. Alternatively, defer the terrain clear + toast until the delete actually succeeds (move it into the onSuccess / `result.failed.length === 0` branch), keyed on whether the *successfully-deleted* set removed the last backing DEM.

## Medium Issues

### MD-01: Synthetic legend entry shows even when `terrain_config` is dangling (no backing DEM layer present) — re-introduces a legend↔stack/map disagreement

**File:** `frontend/src/components/builder/terrain-legend.ts:39-49`; consumed at `LegendPlugin.tsx:96-98` and `LayerLegend.tsx:187-189`

**Issue:** `deriveTerrainLegendEntry` flags terrain active on `terrainConfig.enabled === true && !!terrainConfig.source_dataset_id` only — it never checks that a backing DEM layer for that `source_dataset_id` is actually present in the layer list. The stack's synthetic `relief:terrain` row, by contrast, sets `visible` only when `sourceStatus === 'active'`, which requires a present terrain-capable DEM layer (`map-stack.ts:404-424`); with no backing layer the stack reports `sourceStatus: 'missing'` and `visible: false`.

So for a map with a dangling `terrain_config` (source dataset/layer deleted) — which `999.17-RESEARCH.md` Runtime-State-Inventory A4 explicitly flags as a pre-existing DB state on legacy maps — the legend renders a phantom "3D terrain" entry while the stack shows it missing and no mesh paints. That is the exact legend↔map disagreement class this phase exists to eliminate. Fix 2's delete-clear only prevents *new* dangling configs; it does nothing for maps already in that state, and the viewer surfaces (shared/public maps) have no delete path at all.

A secondary, milder mismatch: in the Fix-3 case (mesh driven by a hillshade-mode DEM), the legend shows "3D terrain" (and the mesh attaches) but the stack's `enabled` is false because `isTerrainRenderLayer` requires render_mode terrain — so stack and legend disagree on whether terrain is "on."

**Fix:** Pass the resolved DEM layers (or a `hasBackingTerrainLayer` boolean) into `deriveTerrainLegendEntry` and require a present terrain-capable DEM layer for `terrain_config.source_dataset_id` before emitting the synthetic entry — mirroring the stack's `sourceStatus: 'active'` condition (`map-stack.ts:404-424`). This keeps legend ↔ stack ↔ map in the lockstep the phase mandates and tolerates dangling configs gracefully (RESEARCH Open Question 1 / A4 recommended exactly this).

### MD-02: `shouldClearTerrainOnDelete` "still-backed" check uses `is_dem` only, not the stack's `isTerrainCapableDemLayer` predicate

**File:** `frontend/src/components/builder/hooks/builder-layer-mutations.ts:534-544`

**Issue:** The helper decides terrain is "still backed" via `layer.is_dem === true && layer.dataset_id === sourceDatasetId`. The resolver that actually attaches the mesh (`BuilderMap.tsx:394-397`, and the viewer `use-viewer-terrain.ts`) requires `isTerrainCapableDemLayer`, i.e. `is_dem === true && DEM_RECORD_TYPES.has(dataset_record_type)` (`map-stack.ts:216-218`). A remaining layer with `is_dem === true` but a non-DEM `dataset_record_type` would satisfy `shouldClearTerrainOnDelete`'s "still backed" test yet would NOT be resolvable as a terrain source by the mesh resolver. In that (admittedly narrow) case terrain_config is preserved while the mesh can no longer attach — a residual dangling config that feeds MD-01.

This is the safe direction (it errs toward *not* clearing), so it is not a data-loss bug, but it is an avoidable predicate drift in a phase whose entire purpose is keeping these predicates in lockstep.

**Fix:** Reuse the canonical predicate. Import/share `isTerrainCapableDemLayer` (currently file-private in `map-stack.ts` — export it or move it next to `isDemTerrainVisualSuppressed`) and use it in `shouldClearTerrainOnDelete`'s `datasetStillBacked` check so "still backed" means "still resolvable as a terrain source," exactly as the mesh resolver and stack compute it.
