### BL-01 (blocker) — narrowed the POLISH-02 guard so a hillshade overlay paints on the terrain DEM
**Commit:** `7ec15858`
**Files:** `frontend/src/components/builder/map-sync.ts`, `frontend/src/components/builder/__tests__/map-sync.raster.test.ts`

Per RESEARCH D-07 Option (b), step 2. The old guard early-returned (suppressing the
hillshade consumer) whenever the DEM was terrain-bound — so Fix 3's "mesh + visible
hillshade on one DEM" never delivered. Narrowed:

- Added pure `shouldSkipHillshadeForTerrain({ isTerrainBound, hillshadeTileSize, terrainSourceTileSize })`:
  skip ONLY when the hillshade source's tileSize MISMATCHES the active terrain source's
  tileSize. When they MATCH (the normal case — both default from `token.tile_size ?? 256`),
  the hillshade runs on its own per-layer source `source-${layer.id}` alongside the mesh.
- Threaded the active terrain source's tileSize into `syncRasterLayer` from `syncLayersToMap`,
  resolved from `tokenMap.get(terrain_config.source_dataset_id)` (same default the terrain mesh
  source uses). The two existing call sites (`BuilderMap.tsx:573/:629`) already pass `terrainConfig`
  via `syncOptions`; `tokenMap` already flows through — no call-site edits required.
- Kept `isHillshadeTerrainBound` and the POLISH-02 doc comment; only gated the early-return.
- The hillshade keeps its distinct per-layer source (NEVER the shared `terrain-dem` id), per
  RESEARCH lines 151/238.

**map-sync.ts change is ONLY the narrowed guard — no unrelated edits** (verified via `git diff`).

Tests: rewrote Test E to assert a MATCHING-tileSize terrain-bound hillshade IS created
(`source-dem-layer-1`, layer type `hillshade`); kept Test F (terrain disabled → paints);
added 5 pure-helper cases for `shouldSkipHillshadeForTerrain` (not-bound, match, undefined→256
default match, mismatch→skip, default-vs-512 mismatch→skip). Through `syncLayersToMap` a
terrain-bound hillshade and the terrain source share a dataset → same token → always match, so
the mismatch branch is pinned at the pure-helper level.
