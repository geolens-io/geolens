# Phase 999.17: Builder terrain/legend consistency & DEM dedup - Context

**Gathered:** 2026-05-31
**Status:** Ready for planning

<domain>
## Phase Boundary

Fix the cross-surface inconsistency where DEM layers with `style_config.render_mode === "terrain"` are excluded from the builder layer stack but still listed in the legend (phantom entries with no controllable row), and stop the terrain-bind flow from accumulating duplicate DEM layers on the same dataset. All three fixes from the phase brief are **in scope** (Fix 3 was elevated from optional during discussion):

1. **Legend consistency** — legend (builder `LegendPlugin` + viewer `LayerLegend`) must agree with the stack about terrain-suppressed DEM layers.
2. **DEM dedup** — terrain-bind / render-mode switch reuses one DEM layer; deleting the terrain source clears `terrain_config`.
3. **Mesh + visible hillshade on one DEM** — relax the builder terrain lookup so a single DEM can drive both the 3D mesh and a visible hillshade overlay.

Shared-surface UI behavior affecting every terrain map — needs regression tests + the builder live-MCP close-gate. Fits the *Map Builder Extensibility Refactor* active plan.

**Out of scope:** any new terrain/DEM capability beyond these three fixes; back-end terrain pipeline changes; non-meter SRID terrain exaggeration (separate known limitation).

</domain>

<decisions>
## Implementation Decisions

### Fix 1 — Legend representation
- **D-01:** Represent active 3D terrain as a **single synthetic "3D terrain" legend entry driven by `terrain_config`**, in BOTH the builder `LegendPlugin.tsx` and the viewer `LayerLegend.tsx` (via `layer-identity.ts`). Mirror the shape of the stack's existing synthetic `relief:terrain` row (`map-stack.ts:434`) so legend and stack visually agree, and viewers of published/shared maps can see terrain is active.
- **D-02:** Exclude per-layer `render_mode:"terrain"` DEM layers from the legend entirely (no per-DEM-layer entry). Consume the existing suppression truth `isDemTerrainVisualSuppressed` (`map-sync.ts:52`) as the single source of truth — do NOT re-derive the predicate independently in each legend surface.
- **D-03:** Hillshade-mode relief layers that actually paint MUST still appear as normal legend entries. Only terrain-suppressed (mesh) DEM layers are collapsed into the single synthetic entry.

### Fix 2 — DEM dedup + delete behavior
- **D-04:** Terrain-bind / render-mode switch must **reuse the existing DEM layer** for the dataset — never create a 2nd `render_mode:"terrain"` layer on the same DEM dataset. This kills the duplicate-accumulation bug (`DEMEditorScene.tsx:165-189` `handleSwitchMode`/`onTerrainBind`; `use-builder-layers.ts` terrain bind/unbind ~:1067-1077).
- **D-05:** When the terrain source layer (or last DEM layer) is deleted, auto-**clear `terrain_config`** and surface a **non-blocking toast** (e.g., "3D terrain disabled — source layer removed"). No blocking confirm dialog. Fix the delete paths that currently leave dangling config (`use-builder-layers.ts` ~:287-329, ~:611-696).

### Fix 3 — Mesh + visible hillshade on one DEM (IN SCOPE)
- **D-06:** Relax the builder `demLayer` lookup (`BuilderMap.tsx` ~:389-401) to resolve terrain by `terrain_config.source_dataset_id` — matching the viewer's `use-viewer-terrain.ts:52-74` — dropping the hard `render_mode === 'terrain'` requirement. This lets one DEM dataset drive the 3D mesh while a hillshade-mode layer on the same DEM adds visible relief.
- **D-07:** Fix 3 must coordinate with the **POLISH-02 anti-crash guard** (`map-sync.ts` ~:621-649) that suppresses a hillshade overlay on the same DEM as active terrain. Researcher/planner must determine whether the guard relaxes for the intended visible-hillshade case, or whether mesh + hillshade can coexist without re-introducing the dual-consumer error spam that HYG-01 (v1033) fixed. **This path needs extra live-MCP verification.**

### Execution & Gate
- **D-08:** Execute via `/gsd-autonomous` (discuss → plan → execute per phase) using the **Playwright MCP server as the live builder close-gate** — per user direction.
- **D-09:** **Playwright MCP is orchestrator-only.** gsd-executor and other GSD subagents lack `mcp__playwright__*` tool access (see project memory [[playwright-mcp-orchestrator-only]]). The orchestrator MUST drive the live close-gate directly — subagents asked to run MCP either write `.spec.ts` files (not the same) or fabricate PASS/FAIL. Do NOT delegate the MCP gate.
- **D-10:** Deterministic gates per project convention: typecheck 0, vitest, i18n 4-locale parity (en/es/fr/de), `e2e:smoke:builder`, backend tests only if backend touched (this is frontend-only). Required unit tests: legend entry derivation excludes terrain-suppressed layers (both `LegendPlugin` AND viewer `LayerLegend`); terrain-bind reducer reuses the layer (no duplicate).

### Claude's Discretion
- Exact label / icon / swatch for the synthetic "3D terrain" legend entry (subject to design tokens + i18n parity).
- Toast copy wording (D-05).
- Whether to factor a single shared derivation helper used by both `LegendPlugin` and viewer `LayerLegend`, or keep per-surface predicates — prefer a shared seam if one exists naturally, to keep legend ↔ stack ↔ map-sync in lockstep.

</decisions>

### Established Patterns
- Terrain is already modeled as a **single synthetic scene/surface entry** in the stack — the legend's "3D terrain" entry follows the same idea rather than inventing a new concept.
- Suppression logic is centralized in `map-sync.ts` — legend consistency is achieved by reusing that predicate, keeping legend ↔ stack ↔ map-sync in lockstep (the phase's core invariant).
- i18n: any new legend label needs en/es/fr/de parity (`LegendPlugin` is in the builder namespace; viewer `LayerLegend` in the viewer namespace). 4-locale parity is a deterministic gate.
- Recoverable non-blocking toasts/banners already exist in the builder (introduced v1009) — reuse for D-05.
