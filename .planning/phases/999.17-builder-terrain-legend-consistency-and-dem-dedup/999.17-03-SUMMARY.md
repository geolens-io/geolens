# Phase 999.17 Plan 03: Builder Terrain Resolver-Alignment (Fix 3) Summary

Aligned the builder's `applyTerrainConfig` DEM lookup with the proven viewer resolver ‚Äî terrain now resolves by `terrain_config.source_dataset_id` + `isTerrainCapableDemLayer` only, so a DEM layer in hillshade render-mode still drives the 3D mesh (mesh + visible hillshade on one DEM), without touching the POLISH-02 anti-crash guard.

## What Was Built

**Task 1 (TDD, FIX-3-RESOLVER):** Removed the third clause (`style_config.render_mode === 'terrain'`) from the `applyTerrainConfig` `.find()` predicate in `BuilderMap.tsx`. The lookup now matches `viewer/hooks/use-viewer-terrain.ts:32-39` ‚Äî `layer.dataset_id === currentTerrainConfig.source_dataset_id && isTerrainCapableDemLayer(layer)`. Every downstream line (raster `token` read, `effectiveTerrainEnabled` visibility-eye check, `ensureRasterDemTerrainSource`, `map.setTerrain(...)`) is unchanged. `terrainLayerKey` retains `render_mode` in its dependency string so `applyTerrainConfig` still re-runs on a hillshade‚Üîterrain flip.

**Advisory A3 (real assertion, not eyeball):** Extended `BuilderMap.terrain-visibility.test.tsx`. `makeDemLayer` now takes an optional `renderMode` ('terrain' | 'hillshade', default 'terrain'). Added **Test D (FIX-3)** asserting that a DEM layer in **hillshade** mode backing an enabled `terrain_config` + raster token still triggers `map.setTerrain({ source: TERRAIN_SOURCE_ID })`. RED was verified before the fix (Test D failed: `expected null to match object { source: 'terrain-dem' }`); GREEN after. Existing Test A (terrain-mode attach), Test B (hidden DEM ‚Üí detach), Test C (disabled config ‚Üí detach) stay green.
