<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| operator → local monorepo | Operator runs docker compose + seeder in the geolens repo |
| capture script → local GeoLens | Local-only HTTP to http://localhost:8080 and http://localhost:8001 |
| screenshot content → public git repo | PNGs are committed and will eventually ship to getgeolens.com — they are PUBLIC assets |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-216-02-01 | Information Disclosure | Screenshot content | mitigate | Before commit in Task 2, visually confirm no PII, no real user emails, no API keys visible on screen. The seeded demo data is synthetic (Natural Earth, GEBCO) so this is low-risk but not zero. Per D-14, the AI query must be benign ("show aquifers in California" — not "what is my API key?"). |
| T-216-02-02 | Information Disclosure | Admin session cookie in Playwright storage | accept | Playwright creates an ephemeral browser context — storage is discarded when the script exits. No persistent session files are written to disk. |
| T-216-02-03 | Spoofing | Wrong GeoLens URL → script captures attacker-controlled page | accept | GEOLENS_URL is operator-controlled and defaults to localhost. If an attacker can set env vars in the operator's shell, they can already do worse things. Out of this phase's threat model. |
| T-216-02-04 | Tampering | PNG file gets replaced with a malicious file after capture | accept | Committed to git; tamper evidence via git history. No runtime code execution from PNG assets — Astro treats them as image inputs only. |
| T-216-02-05 | Denial of Service | Capture script hangs on unresponsive GeoLens | mitigate | Plan 01 added try/catch + default 30s timeouts on all waitForLoadState calls. Worst case: script runs for ~3.5 minutes total (7 captures × 30s) before exiting. |
| T-216-02-06 | Information Disclosure | AI chat screenshot shows LLM response with stale/private data | mitigate | Per D-14: the LLM query is a benign geospatial question ("show aquifers in California"). Response content is the LLM's output on a clean session — no prior context. Reviewer (operator) is responsible for visually confirming the captured conversation before committing. |

All screenshots are treated as PUBLIC marketing assets. The threat model is primarily about what shows up ON SCREEN at capture time — mitigated by (a) using seeded synthetic data, (b) benign AI query, (c) operator visual review before commit.
</threat_model>
