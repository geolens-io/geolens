<!-- Active-link detection contract (research Q12) -->

In Nav.astro frontmatter, compute the current path once and derive active states:

```typescript
--- <!-- astro frontmatter fence -->
import { GEOLENS_GITHUB_URL } from '../../lib/links';

const pathname = Astro.url.pathname;

// Exact match for homepage; prefix match for subpages of /features and /quickstart
const isHome = pathname === '/';
const isFeatures = pathname === '/features' || pathname === '/features/' || pathname.startsWith('/features/');
const isQuickstart = pathname === '/quickstart' || pathname === '/quickstart/' || pathname.startsWith('/quickstart/');
--- <!-- astro frontmatter fence -->
```

Per research Q12 — both the bare `/features` and trailing-slash `/features/` variants should match (Astro's SSG can serve either depending on `build.format` config). The `startsWith('/features/')` pattern additionally handles hypothetical sub-routes like `/features/map-builder` if they're ever added.

The homepage match is EXACT only (`pathname === '/'`). Do NOT use startsWith for home — it would match every page.

<!-- Active state styling (D-07) -->

Each link has 3 states:
- Default: `color: var(--muted-foreground)`
- Hover: `color: var(--foreground)` (use Tailwind hover: modifier)
- Active: `color: var(--primary-700)` + underline OR bottom-border OR slightly heavier weight

Apply `aria-current="page"` when active. Use `aria-current={isFeatures ? 'page' : undefined}` — undefined omits the attribute entirely on inactive links, which is cleaner than `aria-current="false"`.

Per A11Y carry-forward: primary-700 is the WCAG AA contrast-compliant shade for text on var(--background) white. Do NOT use primary-500 (decorative only per STATE.md Phase 212-01 decision).

<!-- Mobile hiding (D-07) -->

Desktop (sm+, ≥640px): logo + Features + Quickstart + GitHub icon
Mobile (<640px): logo + GitHub icon ONLY — Features and Quickstart are hidden

Use Tailwind `hidden sm:flex` (or `hidden sm:block`) on the Features + Quickstart links. This is a pure CSS hide — no JS.

Per D-07 rationale: "No hamburger menu — mobile users navigate via homepage hero CTAs and footer links."

<!-- Security check (research) -->

Research security notes that Nav additions are INTERNAL links:
- Features and Quickstart: internal routes, NO `target="_blank"`, NO `rel="noopener noreferrer"`
- Only the GitHub link (existing) needs the noopener rel

<!-- Layout target -->

Current Nav structure (Phase 215):
```
<header>
  <nav class="flex items-center justify-between ...">
    <a href="/" ...>Logo + GeoLens wordmark</a>
    <a href={GEOLENS_GITHUB_URL} ...>GitHub icon</a>
  </nav>
</header>
```

Target Nav structure (Phase 216):
```
<header>
  <nav class="flex items-center ...">
    <a href="/" ...>Logo + GeoLens wordmark</a>

    <!-- Desktop subnav (hidden on mobile) -->
    <div class="hidden sm:flex items-center gap-6">
      <a href="/features" aria-current={isFeatures ? 'page' : undefined}>Features</a>
      <a href="/quickstart" aria-current={isQuickstart ? 'page' : undefined}>Quickstart</a>
    </div>

    <a href={GEOLENS_GITHUB_URL} ...>GitHub icon</a>
  </nav>
</header>
```

The `justify-between` still works: with 3 child items (logo, subnav div, GitHub button), flex distributes them with logo on the left, GitHub on the right, and subnav div taking natural space in between. At mobile, the subnav div is hidden, and justify-between returns to the 2-item logo-vs-GitHub layout from Phase 215.

Alternative layout (acceptable): group the subnav links INSIDE a flex wrapper with the GitHub button, so the "right side" of the nav becomes a single flex container. Either approach is fine — the first is cleaner for Tailwind readability.
