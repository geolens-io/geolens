The Astro `<Picture>` import pattern (verified in research Q4):

```astro
--- <!-- astro frontmatter fence -->
import BrowserFrame from './BrowserFrame.astro';
import { Picture } from 'astro:assets';
import searchScreenshot from '../../assets/screenshots/search.png';
--- <!-- astro frontmatter fence -->

<BrowserFrame url="app.geolens.io/search" class="w-full">
  <Picture
    src={searchScreenshot}
    formats={['avif', 'webp']}
    alt="GeoLens catalog search page showing dataset cards with filter tabs for vector, raster, and table types"
    widths={[448, 896]}
    sizes="(max-width: 640px) 100vw, 448px"
    loading="lazy"
    class="w-full block"
  />
</BrowserFrame>
```

CRITICAL details (from research + CONTEXT.md):
- `src={searchScreenshot}` — MUST be the imported variable, NEVER a string path (research Q4, Pitfall 1)
- `formats={['avif', 'webp']}` — AVIF first, WebP second, PNG is the default fallback (research Q4)
- `widths={[448, 896]}` — 1x and 2x of the 448px BrowserFrame desktop target
- `sizes="(max-width: 640px) 100vw, 448px"` — full viewport on mobile, 448 fixed at sm+
- `loading="lazy"` — ProductPreviewSection on the homepage places SearchPreview above the fold; planner call whether this specific component uses eager — research Q7 recommends lazy for all non-above-fold. Keep lazy unless visual regression shows a flash.
- `class="w-full block"` — applies to the inner `<img>`, preventing inline whitespace gap (research Pattern 1)
- `BrowserFrame class="w-full"` — carries forward Phase 215 Plan 04 responsive fix
- NO `min-height` anywhere — delete all min-height styles from the old SVG preview bodies (Pitfall 4)

Alt text requirements (A11Y carry-forward):
- Must be DESCRIPTIVE, not "screenshot of X". Describe what's visibly rendered.
- Reference the actual content: layers, dataset names, filter tabs, buttons
- Max ~160 chars so it doesn't become a paragraph

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| Astro build → image pipeline | Imported PNGs are processed at build time; output is hashed AVIF/WebP in dist/_astro/ |
| Screenshot asset → public web | All 3 screenshots will eventually ship as marketing content |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-216-03-01 | Tampering | String-path import regression | mitigate | Task verify greps for `import ... from '../../assets/screenshots/`. If the executor accidentally used `<Picture src="/screenshots/..." />` (string path), the grep fails and the build ships unoptimized images (Pitfall 1). TypeScript's `src: ImageMetadata` type also catches this at build time — string paths produce a type error. |
| T-216-03-02 | Information Disclosure | Weak alt text leaking system state | accept | Alt text is author-controlled and describes the screenshot content. No user input flows into alt text. A11Y risk (not security risk) is addressed by the "descriptive, not generic" guidance. |
| T-216-03-03 | Elevation of Privilege | Picture component XSS via filename | accept | Astro sanitizes image imports at build time. Even a maliciously-named PNG cannot inject script tags through the Picture component. |
| T-216-03-04 | Spoofing | BrowserFrame URL pill showing a misleading URL | mitigate | The URL pill is decorative (not a real link). Per research Pitfall 5, the MapBuilder URL is corrected from `/builder/` to `/maps/` to honest-match the real app. Task verify greps both strings — must contain `/maps/`, must NOT contain `/builder/`. |

All dispositions are appropriate. The primary failure mode is Pitfall 1 (string-path import bypassing optimization) which is mitigated by both the TypeScript type system AND the explicit grep in the verify command.
</threat_model>
