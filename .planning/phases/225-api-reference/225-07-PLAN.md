````markdown
# OpenAPI Snapshot

This directory holds the committed OpenAPI 3.x snapshot that the docs site
renders as the API reference. The snapshot is the **source of truth** for the
build — CI does not re-fetch it from a running backend. Refresh it manually
before each GeoLens release.

- **Snapshot file:** [`geolens.json`](./geolens.json)
- **Source endpoint:** `GET /api/openapi.json` on a running GeoLens FastAPI instance
- **Refresh script:** [`docs/scripts/fetch-openapi.mjs`](../../../scripts/fetch-openapi.mjs)
- **Refresh cadence:** before every GeoLens release; after any change to `backend/app/api/main.py` tags; after any router-signature change

## When to refresh

Re-run the fetch script in any of these situations:

- A new GeoLens release is being prepared.
- `backend/app/api/main.py`'s `_OPENAPI_TAGS` block was added to, removed from, or renamed.
- Any FastAPI router signature changed — added or removed parameters, changed response shape, new endpoint.
- Reader feedback indicates the rendered docs no longer match the live API.

## How to refresh

The script HTTP-fetches `/api/openapi.json` from a running GeoLens instance and
overwrites `geolens.json` with a deterministic pretty-printed copy. You need a
running backend on port 8000 (or anywhere reachable; override the URL with
`GEOLENS_API_URL`).

```sh
# Terminal 1 — start the GeoLens backend (in the geolens repo).
cd /path/to/geolens/backend
docker compose up api          # wait for healthy

# Terminal 2 — run the fetch (in the getgeolens.com repo).
cd /path/to/getgeolens.com/docs
npm run fetch-openapi          # writes src/content/openapi/geolens.json
```

Override the source URL when fetching from a non-local instance:

```sh
GEOLENS_API_URL=https://staging.geolens.example.com/api/openapi.json npm run fetch-openapi
```

The script exits non-zero with a `FAIL:` prefix if the response is unreachable
(exit 2) or fails structural validation (exit 3 — missing `openapi`,
`info.version`, or `paths`). In either case, `geolens.json` is left untouched.

## How to verify the snapshot is current

After the fetch, inspect the diff and the timestamps:

```sh
# Snapshot version that will ship in this build:
node -e 'const s = require("./src/content/openapi/geolens.json"); console.log("openapi=" + s.openapi, "version=" + s.info.version, "paths=" + Object.keys(s.paths).length);'

# Last commit that touched the snapshot:
git log -1 --format="%h %ad %s" --date=short docs/src/content/openapi/geolens.json

# Last commit that touched the FastAPI app:
git log -1 --format="%h %ad %s" --date=short --follow path/to/geolens/backend/app/api/
```

If the FastAPI app's last touch is newer than the snapshot's last touch, the
snapshot is stale — re-run `npm run fetch-openapi` and commit the diff.

## Diff review

Treat the snapshot diff like any other code review: scan for unexpected
endpoint additions, removals, or schema shape changes that don't match the
release notes. Commit the snapshot in its own commit (don't co-mingle with
unrelated changes) so the diff stays reviewable.

## Future: automated drift detection

Automated drift-detection between the live OpenAPI spec and the committed
snapshot is tracked as **OASDIFF-01** in
[`.planning/REQUIREMENTS.md`](https://github.com/geolens-io/geolens/blob/main/.planning/REQUIREMENTS.md)
under "Future Requirements (Deferred from v15.0)". When that ships, manual
cadence enforcement (this README) becomes a fallback rather than the only
gate.

Until then: refresh manually, review the diff, and commit.
````
