## Layout Contract

Fully inherited from Phase 1034. No new layout columns or breakpoints.

The three-column grid (`340px sidebar · 380px flyout · 1fr map`) is the shell. The empty state
lives entirely inside the 340px sidebar column; it does not affect the flyout or map columns.

**Empty-state layout within the sidebar:**

The sidebar structure is unchanged. When `isEmpty === true` (zero user layers), the layer stack
scroll area renders the empty-state content instead of `DndContext`. The `basemap-dock` footer
remains pinned at the bottom of the stack area at all times — both in the empty and populated state.

**Transition to populated state:**

change between states.

### 2. `SuggestCard` component

A single hand-curated dataset entry in the empty-state suggested list.

**Grid:** `32px 1fr 22px`, `gap: 10px`, `align-items: center`, `padding: 8px 10px`

| Column | Content |
|--------|---------|
| Type icon (32×32px) | Rounded square, same color scheme as Phase 1034 type icons but 32px; geometry-matched glyph |
| Name + meta | Name: 14px regular, truncated. Meta: 10px muted (e.g. "Vector · 850K · EPSG:4326") |
| Add button (22px) | Lucide `Plus` icon, `--primary` color, no background at rest; `--primary-50` bg on hover |

**States:**
| State | Visual |
|-------|--------|
| Rest | `--surface-1` bg, `1px solid --border` |
| Hover | `--surface-2` bg, `border-color: --primary/30`, `box-shadow: var(--elevation-sm)` |
| Adding (after click) | Add button shows Lucide `Loader2` spinning, card `pointer-events: none`, `opacity: 0.7` |
| Added | Add button shows Lucide `Check`, `--primary` color, card remains in list (user can add again for duplicate rendering) |

**Click behavior — the card:**
- Clicking the card body (not the `＋` button) opens the full Add Data modal with the dataset name pre-filled in the search query (same effect as typing the name in the inline search and pressing Enter).

**Click behavior — the `＋` button:**
- Immediately adds the dataset as a new layer (same as the "Add to map" action in the modal). Does NOT open the modal.
- Post-add: layer lands at z-position 0 (topmost user layer position); LayerEditorPanel flyout auto-opens on it (see Interaction Contract).

**Hand-curated suggestion list (client-side, v1):**
Four cards reflecting dataset breadth. The client-side constant `SUGGESTED_DATASETS` is a typed array of `SuggestedDataset`:
```ts
interface SuggestedDataset {
  id: string;          // dataset UUID — used to fetch/add via existing API
  name: string;        // display name
  record_type: 'vector_dataset' | 'raster_dataset' | 'vrt_dataset';
  geometry_type?: string;  // e.g. "MultiPolygon", "Point"
  feature_count?: number;
  crs?: string;        // e.g. "EPSG:4326"
}
```

The list is defined in `frontend/src/components/builder/suggested-datasets.ts` (new file).
It contains datasets known to exist in the org's catalog. If a dataset ID is not found by the API,
the card renders with a "Not available" muted overlay and the `＋` button is disabled.

**Availability check:** A single `GET /api/datasets/{id}` per suggested dataset is fired in parallel
on mount (via TanStack Query with `staleTime: 60_000`). This avoids referencing stale/deleted datasets.
If the request returns 404, the card is hidden (not shown as an error — the list simply collapses that slot).

### 3. Inline search input → modal pre-fill routing

The inline search `<input>` in the empty state shares no internal state with the modal.

**Behavior:**
1. User types a query into the inline input.
2. No autocomplete, no inline results — this is a single-field trigger, not a search interface.
3. On `Enter` keypress OR when the user clicks the Lucide `Search` icon: the Add Data modal opens with `initialQuery` set to the typed value.
4. On `blur` with non-empty text: the query is retained in local state so it is still present if the user re-focuses.
5. On modal close: the inline input is cleared (reset to empty string).

**`initialQuery` prop contract for `DatasetSearchPanel`:**
```ts
interface DatasetSearchPanelProps {
  // ... existing props ...
  initialQuery?: string;  // NEW — pre-fills the internal query state on mount
}
```

When `initialQuery` is provided:
- `useState('') → useState(initialQuery ?? '')` — internal `query` state is seeded with the value.
- The search fires immediately (no debounce delay on the initial value — set `debouncedQuery` synchronously for the first render).
- The search input inside the modal shows the pre-filled value, fully selected, so the user can immediately replace it.

When `initialQuery` is `undefined` or `''`: behavior is unchanged from the current implementation.

### 4. `DatasetSearchPanel` — audit finding fixes

The following changes align the modal to the unified stack model (BSR-18).

#### 4a. Raster discovery parity

**Current state:** The "Raster" tab exists (`activeTab === 'raster'`) but applies a client-side filter
over the same `/collections/datasets` query — it does NOT send `record_type=raster_dataset` to the
backend when the All tab query already includes all types. This means the results depend on the
page-20 limit, and raster datasets that do not appear in the first 20 results are invisible on the
Raster tab.

**Required fix:** The Raster tab must send `record_type=raster_dataset` to the backend as a dedicated
query (same pattern as the Vector tab sending `record_type=vector_dataset`). The current
`tabRecordType()` function already returns `'raster_dataset'` for the raster tab — the bug is that
the query `enabled` guard (`enabled: activeTab !== 'basemap'`) is correct, but the `queryKey` and
`queryFn` must use the `recordType` value directly without the post-fetch client-side filter.

**Spec:** Remove the post-fetch `Array.filter()` for raster in `useMemo(() => results)`:
```ts
// BEFORE (broken — client-side filter on a 20-item page)
if (activeTab === 'raster') {
  return features.filter((record) => isRasterRecord(record.properties.record_type));
}

// AFTER (correct — backend query with record_type param already in searchParams)
// No client-side filter needed — backend returns only the correct type.
return features;
```

The `record_type` param is already added to `searchParams` via `tabRecordType()`. The fix is to
remove the duplicate client-side filter, not to add a new query.

#### 4b. Post-add layer z-position

**Spec:** When a dataset is added via the modal (or via a suggest-card `＋` button), the new layer
must land at z-position 0 — the topmost position in the user layer list, directly below the basemap
group if present.

**Current state:** `handleAddDataset` in `MapBuilderPage.tsx` posts with `sort_order: -1` (line ~714).
The backend interprets `sort_order: -1` as "append to end" behavior. The correct behavior for
new layers is to prepend — new layers should appear on top so they are visible immediately.

**Required contract:**
- New layers added via the modal must use `sort_order: 0` (or the next value above the current topmost user layer if the stack uses non-zero-based ordering).
- If the unified stack model uses a different z-order encoding, the planner must derive the correct `sort_order` value from `buildMapStack()` at the time of add. The contract is: **new layer appears visually at the top of the user layers section**.

#### 4c. LayerEditorPanel flyout auto-opens on add

**Spec:** After a dataset is successfully added (`addLayer` mutation resolves), the newly created
layer must be auto-selected and the LayerEditorPanel flyout must open to it automatically.

**Trigger:** The `onAddDataset` success callback in `MapBuilderPage.tsx` (currently closes the modal)
must additionally call `handleSelectLayer(newLayer.id)` so the flyout opens.

**Sequence:**
1. User clicks "Add to map" in the modal (or `＋` on a suggest-card).
2. `addLayer` mutation fires.
3. On success: modal closes AND `handleSelectLayer(newLayerId)` is called.
4. The layer appears in the stack, is highlighted (selected state), and the flyout opens to its editor.

**Focus:** After auto-open, focus moves to the flyout's close `×` button (same rule as all flyout opens — inherited from Phase 1034 Accessibility Contract).

#### 4d. Modal empty-state UX (zero-result and unfiltered-empty)

The modal currently shows `"No datasets found"` (i18n key `search.noResults`) only when
`debouncedQuery.trim().length > 0 && results.length === 0`. Two additional states need distinct copy:

**State A — Unfiltered empty (no query, no results):**
Condition: `debouncedQuery.trim().length === 0 && !isLoading && results.length === 0 && activeTab !== 'basemap'`
Copy: "Your catalog is empty. Upload a dataset to get started."
CTA: A ghost button "Upload a file →" that opens the upload flow (existing upload route or the Import tab within the modal if available).

**State B — Zero-result (query entered, no matches):**
Condition: `debouncedQuery.trim().length > 0 && !isLoading && results.length === 0`
Copy (heading): "No datasets match '{query}'"
Copy (body): "Try a different search term, or browse all datasets."
CTA: A ghost button "Clear search" that resets `query` to `''`.

**State C — Error:**
Condition: `isError`
Copy: "Failed to load datasets. Check your connection and try again."
(No CTA — the retry is implicit on next keystroke; the existing error state can reuse this copy.)

**Visual for States A and B:**
- Lucide icon centered above text: `Inbox` for State A, `SearchX` for State B
- Icon size: 32×32px, `--muted-foreground`
- Heading: 14px semibold, centered
- Body: 12px regular muted, centered
- CTA: shadcn `Button variant="ghost" size="sm"`, centered below body
- Container: `padding: 24px 16px`, `display: flex; flex-direction: column; align-items: center; gap: 8px`

## Interaction Contract

### Empty-state: inline search → modal open

1. User types in the inline search input.
2. No autocomplete, no inline results are shown.
3. On `Enter` or clicking the Lucide `Search` icon: `showAddData` is set to `true` AND `addDataInitialQuery` is set to the input value. The modal opens with the query pre-filled and fires a search immediately.
4. On modal close: `addDataInitialQuery` resets to `''` and the inline search input is cleared.
5. Pressing `Escape` inside the inline input: clears the input value (does NOT open the modal).

### Empty-state: suggest-card body click → modal open

1. User clicks the card body (not the `＋` button).
2. Modal opens with `initialQuery` set to the dataset name.
3. The "All" tab is active; the search result for the dataset name appears at the top.

### Empty-state: suggest-card `＋` button click → direct add

1. User clicks the `＋` button on a suggest-card.
2. The add-button shows `Loader2` spinner. The card is `pointer-events: none`.
3. `addLayer` mutation fires with the dataset ID and `sort_order: 0` (topmost position).
4. On success:
   - Card shows `Check` icon.
   - Empty state transitions to populated state (`.empty-area` unmounts, new layer appears).
   - `handleSelectLayer(newLayerId)` fires: layer is selected, flyout opens.
   - Focus moves to flyout close `×` button.
5. On error:
   - Card reverts to the `＋` button (not spinner).
   - A toast notification appears: "Failed to add dataset. Try again." (uses existing toast pattern — no new component).

### Empty-state: "Browse all datasets →" click

1. User clicks the ghost link.
2. `showAddData` is set to `true`, `addDataInitialQuery` remains `''` (modal opens with no pre-fill — shows all datasets).

### Add Data modal: `initialQuery` prop behavior

- `DatasetSearchPanel` receives `initialQuery?: string`.
- On mount (or when `initialQuery` changes from a new open), the internal `query` state is set to `initialQuery ?? ''`.
- If `initialQuery` is non-empty, the search input text is fully selected (for immediate replacement).
- The search query is fired synchronously (bypasses the 300ms debounce on first mount).

### Add Data modal: post-add flyout auto-open

After `addLayer` mutation resolves in `BuilderDialogs` / `MapBuilderPage`:
1. `onShowAddDataChange(false)` closes the modal.
2. `handleSelectLayer(newLayerId)` is called with the ID of the newly created layer.
3. Flyout opens to the layer editor for the new layer.
4. Focus moves to flyout close `×` (Phase 1034 focus contract).

This applies whether the dataset is added from the modal or from a suggest-card `＋` button.

### Add Data modal: z-position on add

- New layers are added with `sort_order: 0` (prepend, topmost user layer position).
- If the stack already has layers, the new layer slides to the top of the user-layer section in the sidebar (above all existing user layers, below the basemap group if present at the bottom of the stack).

### Keyboard navigation (empty state)

| Element | Key | Behavior |
|---------|-----|---------|
