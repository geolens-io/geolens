## Publishing

> **Phase 228 (2026-05) shipped the cold-start credentials.** PyPI uses **Trusted Publishing** (OIDC, no long-lived token); npm uses a **granular access token** with Bypass 2FA + Read/Write scoped to the `@geolens` org. ### One-time setup (already done as of v13.4)

These steps were completed during Phase 228 as the cold-start setup. Re-execute only if a credential is rotated, the npm org is recreated, or PyPI Trusted Publishing is re-pointed at a different repo.

#### PyPI Trusted Publishing setup

For each PyPI project (`geolens-sdk` and `geolens`), configure a pending publisher at <https://pypi.org/manage/account/publishing/>:

| Field | `geolens-sdk` value | `geolens` value |
|-------|---------------------|------------------|
| PyPI project name | `geolens-sdk` | `geolens` |
| Publisher type | GitHub Actions | GitHub Actions |
| Owner | `geolens-io` | `geolens-io` |
| Repository name | `geolens` | `geolens` |
| Workflow filename | `publish-sdks.yml` | `publish-cli.yml` |
| Environment name | *(blank)* | *(blank)* |

PyPI converts the pending publisher to a live publisher on the first OIDC handshake AND creates the project simultaneously — no `0.0.1` placeholder publish needed. After first publish, the workflow's `permissions: id-token: write` is sufficient to publish forever; no token rotation. Provenance attestations are emitted automatically and visible on each release page (e.g. <https://pypi.org/project/geolens-sdk/1.0.0/>).

#### npm `@geolens` org + granular access token

1. **Claim the org** at <https://www.npmjs.com/org/create>: organization name `geolens` (free plan). This claims the `@geolens` scope.
2. **Generate a granular access token** at <https://www.npmjs.com/settings/~/tokens>:
   - Token name: `geolens-sdk-ci-publish`
   - Permissions: **Read and write**
   - Packages and scopes: select **`@geolens`** org scope (NOT a specific package — the package picker won't show `@geolens/sdk` until first publish)
   - **Bypass two-factor authentication: ENABLED** (required for non-interactive CI; npm removed the "Automation" token type on 2025-12-09 — granular tokens with Bypass 2FA are the modern replacement)
   - Expiration: maximum (90 days)
3. **Add to GitHub repo secrets** as `NPM_TOKEN` via `gh secret set NPM_TOKEN --repo geolens-io/geolens` or the GitHub web UI.

**Token rotation:** npm granular tokens have a hard 90-day max lifetime. Set a calendar reminder ~85 days from creation. On rotation, the token can be narrowed from `@geolens` org scope to `@geolens/sdk` package scope specifically (the picker will show the package once it's published).

### Publishing via the GitHub Actions workflow

The `Publish SDKs` workflow (`.github/workflows/publish-sdks.yml`) is manual-trigger only:

1. Navigate to **Actions → Publish SDKs → Run workflow** in the GitHub UI (or run `gh workflow run publish-sdks.yml --ref main -f target=both -f dry_run=true`).
2. Choose `target`: `python`, `typescript`, or `both`.
3. **First, run with `dry_run=true`** to validate build artifacts without publishing.
4. Once the dry-run is green, re-trigger with `dry_run=false` to publish for real.

The workflow:

- For Python: `uv build` produces a wheel + sdist in `sdks/python/dist/`; `uv publish --trusted-publishing automatic` uploads to PyPI via OIDC (no token).
- For TypeScript: `npm ci && npm run build` produces compiled JS + type declarations in `sdks/typescript/dist/`; `npm publish --access public` uploads to npm using `NODE_AUTH_TOKEN` (from `secrets.NPM_TOKEN`).

Both jobs include a pre-flight name-availability gate that emits a warning if the package is already at a higher version than expected.

### Verifying a published version

`.github/workflows/verify-published.yml` runs `pip install` + `npm install` smoke tests inside fresh `python:3.13-slim` and `node:22-slim` Docker images — no repo context. Trigger via:

```bash
gh workflow run verify-published.yml --repo geolens-io/geolens
```

Use this after a publish to confirm downstream installability, or before a public announcement to confirm the package is reachable on the registry CDN.

### Publishing locally (escape hatch)

PyPI Trusted Publishing only works inside GitHub Actions (it requires `ACTIONS_ID_TOKEN_REQUEST_*` env vars that GitHub injects). For local publish (e.g. emergency hotfix), fall back to a PyPI account-scoped token:

```bash
# Python — requires temporary UV_PUBLISH_TOKEN
cd sdks/python
uv build
UV_PUBLISH_TOKEN=$YOUR_PYPI_TOKEN uv publish

# TypeScript — requires npm login or NPM_TOKEN
cd sdks/typescript
npm install
npm run build
npm publish --access public  # --access public REQUIRED for first publish of @scope/name
```

The `--access public` flag is required for the first publish of `@geolens/sdk` because npm scoped packages default to private (paid org tier required for private). The `package.json` already includes `"publishConfig": { "access": "public" }` as a belt-and-suspenders measure.
```
