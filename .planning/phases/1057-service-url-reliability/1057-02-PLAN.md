<objective>
Fix PROBE-05 (P1) and CLASS-07 (P2) together because they share the probe-response shape and re-architect ogcapi.py + wfs.py adapter behavior in lock-step.

**PROBE-05 correction (per D-04):** REQUIREMENTS.md says "short-circuit `try_all_probes()`" but that is misdiagnosed — the orchestrator at `probe.py:95-161` already short-circuits per-probe (each `if result is not None: return` exits immediately). The real bottleneck is per-layer ogrinfo enrichment in `enrich_ogcapi_layers` (`adapters/ogcapi.py:162-239`) and its WFS sibling: a per-layer `ogrinfo -json -so OAPIF:{url} {layer_name}` subprocess gated by `asyncio.Semaphore(5)` with 30s wait_for. 17 pygeoapi collections × ~3-4s each = ~60s wall clock.

**CLASS-07 fix (per D-09):** Add a backend-classified `kind: Literal['vector', 'raster']` field to `LayerInfo` so frontend stops re-deriving the rule from `geometry_type` string contents. Frontend `ServiceUrlForm.tsx:197` reads `layer.kind` instead of `geometry_type && !contains('raster')`.

Purpose: Drop ogrinfo enrichment from the probe phase per D-05 (lazy-enrich on user select — the preview path at `preview.py` already runs ogrinfo when a layer is picked). Add `kind` as the durable classification contract. ≤5s probe target becomes trivially achievable because the per-layer subprocess cost goes to zero.

Output: Probe-response schema augmented with `kind`; OGC API and WFS adapters return layers with null `geometry_type` + `feature_count` + classified `kind`; orchestrator no longer calls enrichment; frontend mirrors the schema and consumes `layer.kind` at the TypeTag site.
</objective>

| T-1057B-03 | Information Disclosure | Frontend layer.kind exposed via probe response | accept | `kind` is a 2-valued enum (`'vector' \| 'raster'`) classified from already-exposed `geometry_type` + collection-link metadata. No PII; no new information surface. |
