## Responsive Layout Contract

Source: `frontend/src/pages/MapBuilderPage.tsx` (lines 93-94, 928-937, 1138).
Breakpoints are LOCKED from sketch 008 A and v1008 BSR. Phase 1051 RESP-01..03
fix collision/overlap symptoms — they do NOT change the breakpoint values themselves.

| Breakpoint | Width | Behavior |
|------------|-------|----------|
| Full (default) | ≥1100px | 3-col layout: sidebar 340px + flyout 380px + map remainder |
| Rail | <1100px | sidebar minifies to 64px icon rail; flyout 380px still visible |
| Editor-hidden | <800px | flyout hidden; Sheet overlay for layer editing; rail at 64px |
| Out-of-scope | <480px | mobile phone portrait — NOT in scope for this milestone |

**RESP-01 fix contract (sidebar/zoom-controls collision):**
- Symptom: at viewport 800–1099px (rail mode), the collapsed 64px sidebar overlaps MapLibre `NavigationControl` anchored `position="top-right"`.
- Fix strategy: shift `NavigationControl` to `position="top-left"` OR add `marginRight` padding at narrow widths. Confirm exact fix via Playwright MCP measurement.
- Visual result: `NavigationControl` is fully visible and clickable at all widths ≥800px. No overlap with the 64px rail column.

**RESP-02 fix contract (coord readout overlap):**
- Symptom: `MapCoordReadout` pill (positioned `absolute` bottom-right of map canvas) overlaps the map-widget container at narrow widths.
- Fix strategy: apply responsive bottom/right offset via `isRail` boolean OR use `bottom-8` instead of `bottom-2` when `isEditorHidden` is true. Confirm via MCP.
- Visual result: coord readout and map-widget container are non-overlapping. Both visible and legible.

**RESP-03 fix contract (duplicate close buttons):**
- Symptom: basemap selector flyout renders 2 stacked "X" close buttons at narrow viewports.
- Fix strategy: identify whether the duplicate arises from shared flyout chrome + inner component each adding a button; remove exactly one.
- Visual result: EXACTLY 1 visible, functional close button in every right-sidebar-opened flyout (basemap selector, LayerEditorPanel, Settings Sheet). Verified by DOM assertion in vitest.

### BUG-03: Rename input focus state (new behavioral contract)

When the rename inline-input mounts (triggered by "Rename group" kebab action):
- The `<input>` element receives focus via `autoFocus` prop OR `inputRef.current?.focus()` in `useEffect` on mount.
- Visual focus state: `border-b border-primary bg-transparent outline-none focus:ring-1 focus:ring-ring` — matches the existing `FolderGroupRow.tsx:236` pattern exactly.
- No cursor flash or additional visual affordance needed — the blinking text cursor in the focused input is sufficient.

### UX-01: Caret hit-target (new size contract)

Caret button on group rows (`BasemapGroupRow.tsx`, `FolderGroupRow.tsx`):

| Property | Current | Required | How |
|----------|---------|----------|-----|

### UX-03: Draggable basemap row (new interaction contract)

**Drag handle:** The basemap group row (`BasemapGroupRow.tsx`) currently has `Cell 2: Grip — hidden` (`aria-hidden span`, not draggable per the AUD-04 comment at line 106-107). For UX-03, enable drag participation:

- Wire `dragHandleProps` (already threaded via `BasemapGroupRowWrapper` in `UnifiedStackPanel.tsx`) to an actual `<button>` or `<span>` with `cursor-grab` / `active:cursor-grabbing` — same pattern as `FolderGroupRow.tsx:192`.
- `GripVertical` Lucide icon at `h-3.5 w-3.5` (14px), `text-muted-foreground` — matches `FolderGroupRow` grip pattern.
- Opacity: `opacity-35 group-hover/row:opacity-70` (same as folder group grip) so the handle is visible on hover without cluttering the default row state.

**Drag-as-group semantics (sketch 007 A, LOCKED):**
- Dragging the basemap group row moves ALL basemap sublayers as a single unit.
- During drag: `isDragging && 'opacity-40 scale-[0.98]'` on the dragged group (already in `BasemapGroupRow.tsx:76`).
- Insertion line (plain reorder): 2px `bg-primary` horizontal line with `bg-[var(--primary-50)] blur-[2px]` bloom at 25% opacity — matches v1009 insertion-line token.

**Saved-map position encoding:**
- Prefer deriving basemap position from layer-order array index (no new schema field) per REQUIREMENTS.md Out-of-Scope row 6.
- Normalizer must handle legacy maps (no explicit basemap position) by defaulting to bottom of stack.

**Visual cue for "basemap at top" vs "basemap at bottom":**
- No special badge or indicator needed in v1011. Position is communicated purely by where the row appears in the stack list.
- On 3D maps: when basemap is at top, the sublayers render above data layers (MapLibre layer order controlled by `map-sync.ts` loop). The sidebar row order IS the visual cue.

### UX-04: Widget enable/disable toggles (updated visual contract)

**Location:** `SettingsEditorScene.tsx` → WIDGETS collapsible section (lines 146-194).

The existing implementation already uses a `Switch` component (`onCheckedChange={() => onToggleWidget(widget.id)}`). This means the core toggle is ALREADY a shadcn Switch. The UX-04 requirement is about clarifying the label so the purpose ("enables/disables widget availability on the map") is unambiguous — NOT a full UI rework.

**Current label pattern:** `t('settings.toggleWidget', { action, name })` — action is "Enable" or "Disable" based on state. This pattern is already correct per line 189-192.

**UX-04 change:** Verify that:
1. The toggle label clearly reads "Enable [widget name]" when off and "Disable [widget name]" when on (or equivalent unambiguous phrasing).
2. No duplicate on-map widget controls exist alongside the Settings toggle for the same widget. If they do exist (the "duplicate" the requirement references), the Settings section removes the duplicate control and retains only the on/off toggle.
3. On-map interaction controls (e.g., clicking the coord readout pill, dragging the scale bar) remain functional for enabled widgets — these are NOT controlled from Settings.

**i18n keys needed (en/de/es/fr) if new strings are introduced:**
- `settings.enableWidget` → "Enable {{name}}"
- `settings.disableWidget` → "Disable {{name}}"
- `settings.widgetsAvailabilityNote` → "Controls whether each widget appears on the map."

**Toggle row visual (unchanged from existing shadcn Switch row):**
- Row: `flex items-center gap-3 py-1.5` (24px height minimum, 36px preferred with padding)
- Icon: `h-4 w-4 text-muted-foreground shrink-0` (16px Lucide, matches line 184)
- Label: `flex-1 text-xs text-foreground` (matches line 185)
- Switch: shadcn `<Switch />` — checked = primary blue fill, unchecked = muted background

## Cross-Plan Visual Conflict Check

Per the `context_for_agent` instruction: flag any cross-plan visual conflicts.

**UX-02 indicator slot vs existing row chrome:**
- The sublayer row inside `BasemapGroupRow.tsx` currently contains a per-sublayer opacity slider.
- Removing the slider and adding `SublayerConfigIndicators` is a SLOT SWAP — not a new column. The indicators occupy the same trailing position.
- Risk: If the row uses a fixed grid, adding the indicator `flex` container may push other cells. Executor must verify the sublayer row grid template and adjust `grid-cols` if needed.
- LOCK: Maximum 4 indicator badges at 16px each = 64px + gaps. If the row is narrower than this at rail mode (64px sidebar), indicators must be hidden in rail mode (the sublayer rows are only visible in the expanded basemap group, which requires a non-rail sidebar).

**UX-03 basemap drag vs multi-selection lock (Phase 1041 POL-11):**
- `BasemapGroupRow.tsx:77` has `isMultiSelectionActive && 'cursor-not-allowed'` which signals the basemap is not participatory in multi-select.
- Drag MUST be disabled during multi-selection mode (consistent with the existing cursor signal). Executor: gate `dragHandleProps` on `!isMultiSelectionActive`.
- No new visual conflict — the `cursor-not-allowed` class already communicates non-draggable during multi-select.

**RESP-01 NavigationControl repositioning vs UX-03 basemap drag:**
- No conflict. These are independent surfaces (map canvas controls vs sidebar DnD).

**UX-04 widget toggles vs existing Settings scene:**
- `SettingsEditorScene.tsx` already has the widget toggle pattern. If UX-04 only requires label clarification, risk is minimal. If it requires removing duplicate controls, executor must grep for any duplicate widget control rendering outside `SettingsEditorScene` and remove them.
