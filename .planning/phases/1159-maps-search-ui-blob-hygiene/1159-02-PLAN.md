BLOB-URL-CACHE CONTRACT (frontend/src/lib/blob-url-cache.ts) — the module under test:
```
const registered = new WeakSet<QueryClient>();
const BLOB_QUERY_KEYS = new Set(['map-thumbnail', 'quicklook']);
export function registerBlobUrlRevocation(queryClient: QueryClient): void { ... }
```
It subscribes to `queryClient.getQueryCache()`. Behavior:
  - event.type === 'removed' AND queryKey[0] in {'map-thumbnail','quicklook'} → revoke the entry's blob URL (current data, or lastSeen[hash]).
  - data is a `blob:` string AND a prior different blob URL was seen for that queryHash → revoke the PREVIOUS one (refetch-replacement), keep the new one.
  - idempotent: a second `registerBlobUrlRevocation(sameClient)` returns early (WeakSet) → does NOT double-subscribe.

The subscription fires synchronously on cache mutation. (Note: `setQueryData` emits an `'updated'` cache event whose data is the blob string — this is the refetch-replacement path.)
