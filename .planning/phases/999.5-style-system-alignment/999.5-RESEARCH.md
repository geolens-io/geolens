## Summary

Phase 999.5 is a targeted CSS token and asset alignment across three GeoLens surfaces. The geolens app (`frontend/src/index.css`) already uses IBM Plex Sans Variable + IBM Plex Mono and defines the canonical `--primary-*` palette. The marketing site (`getgeolens.com/src/styles/global.css`) currently imports Inter Variable; the docs site (`getgeolens.com/docs/src/styles/custom.css`) also imports Inter Variable. Both need font packages replaced with IBM Plex equivalents, and the docs need Starlight Aside accent overrides plus a sidebar tint.

The logo situation is more nuanced than the HANDOFF implied: there are **three divergent SVG implementations** — a React component in the app (different viewBox, more detail), an inline SVG in the marketing nav (simplified, different proportions), and the same simplified SVG as `public/favicon.svg` in both repos. None of these is a true shared asset. The recommended canonical location and approach is documented in the Logo Extraction section.

The primary risk is Starlight's CSS cascade layer (`@layer starlight.base`) wrapping its default props. Overrides written in `:root` (outside a `@layer`) have higher specificity than layered rules by default, so the `customCss` approach already in place will win without any extra specificity tricks.

**Primary recommendation:** Proceed file-by-file in a single wave. The changes are mechanical and the cascade mechanics are verified; the only audit step before execution is deciding on the canonical SVG file content for the Starlight logo.

## Starlight 0.38 Aside Slot Reference (D-04)

### How Starlight Asides Work

[VERIFIED: read installed `@astrojs/starlight/style/asides.css` and `props.css`]

Each aside variant sets **three CSS custom properties** on the element via a modifier class:

```
.starlight-aside--{variant} {
    --sl-color-asides-text-accent: var(--sl-color-{hue}-high);  /* title text + link color */
    --sl-color-asides-border:      var(--sl-color-{hue});        /* left border color */
    background-color:              var(--sl-color-{hue}-low);    /* fill background */
}
```

The three consumer slots:
- `--sl-color-asides-text-accent` → title text color + inline link color within the aside
- `--sl-color-asides-border` → the 4px left border
- `background-color` (directly, not a custom property) → fill; computed from `--sl-color-{hue}-low`

### Aside Variant → Starlight Color Group Mapping

| Variant | `--text-accent` source | `--border` source | `background-color` source | Default hue |
|---------|----------------------|------------------|--------------------------|-------------|
| `note` | `--sl-color-blue-high` | `--sl-color-blue` | `--sl-color-blue-low` | hsl(234, ...) |
| `tip` | `--sl-color-purple-high` | `--sl-color-purple` | `--sl-color-purple-low` | hsl(281, ...) |
| `caution` | `--sl-color-orange-high` | `--sl-color-orange` | `--sl-color-orange-low` | hsl(41, ...) |
| `danger` | `--sl-color-red-high` | `--sl-color-red` | `--sl-color-red-low` | hsl(339, ...) |

### Override Strategy for D-04

Per the CONTEXT.md specification: override `tip` and `note` accent to brand blue; keep `caution` and `danger` at Starlight's semantic orange/red (so they remain meaningfully distinct warnings). The exact mapping:

**`tip`** (was purple → brand blue):
- `--sl-color-purple-high` → `var(--primary-800)` (`oklch(0.38 0.13 250)`) — title/link text
- `--sl-color-purple` → `var(--primary-700)` (`oklch(0.46 0.16 250)`) — border
- `--sl-color-purple-low` → `var(--primary-50)` (`oklch(0.97 0.02 250)`) — background fill

**`note`** (was Starlight blue → brand blue; Starlight's blue is hsl(234) which is close to our hue-250 but noticeably different):
- `--sl-color-blue-high` → `var(--primary-800)` — title/link text
- `--sl-color-blue` → `var(--primary-700)` — border
- `--sl-color-blue-low` → `var(--primary-50)` — background fill

**`caution`** (KEEP Starlight orange — intentional semantic warning color):
- No override. Orange carries "caution" semantics that brand blue cannot.

**`danger`** (KEEP Starlight red — intentional semantic danger color):
- No override. Red carries "danger" semantics that brand blue cannot.

Dark mode: existing `custom.css :root[data-theme='dark']` block already sets `--sl-color-accent` slots. The same dark-mode block needs to override the purple and blue hue groups:
- `--sl-color-purple-high` → `var(--primary-100)` (light text on dark bg)
- `--sl-color-purple` → `var(--primary-400)`
- `--sl-color-purple-low` → `var(--primary-900)` (dark fill)
- `--sl-color-blue-high` → `var(--primary-100)`
- `--sl-color-blue` → `var(--primary-400)`
- `--sl-color-blue-low` → `var(--primary-900)`

### Cascade Layer Specificity — No Issue

[VERIFIED: read `layers.css` and `asides.css`]

Starlight styles live in `@layer starlight.base` and `@layer starlight.components`. CSS custom property overrides written in `:root` (unlayered, as in `custom.css`) automatically win over any layered `:root` declaration — unlayered styles have higher priority than any `@layer` regardless of selector specificity. No `:where()` wrappers in aside CSS. **No specificity tricks needed.**

## Sidebar Tint Slot (D-02)

### Confirmed: `--sl-color-bg-sidebar` Is the Correct Slot

[VERIFIED: read `@astrojs/starlight/style/props.css` lines 159 and 44]

In Starlight 0.38, the sidebar background is controlled by `--sl-color-bg-sidebar`:
- **Light mode** (`:root[data-theme='light']`): defaults to `var(--sl-color-bg)` (which resolves to `hsl(0, 0%, 100%)` — pure white)
- **Dark mode** (`:root` — dark is default): defaults to `var(--sl-color-gray-6)` (`hsl(224, 14%, 16%)`)

Override for D-02 goes in the `:root` block (light mode, unlayered) in `custom.css`:
```css
--sl-color-bg-sidebar: oklch(0.99 0.003 85);
```

This applies to light mode only. In dark mode the value is set in the dark `:root` block; leaving it at the Starlight default (`--sl-color-gray-6`) is correct for this phase.

## Starlight Logo Wiring (D-05, `astro.config.mjs`)

### Current Config

[VERIFIED: read `getgeolens.com/docs/astro.config.mjs`]

The Starlight `starlight({})` config block currently has NO `logo` key. The config lives at `/Users/ishiland/Code/getgeolens.com/docs/astro.config.mjs`.

The `customCss` key on line 16 is the established pattern. Logo wiring follows Starlight's `logo` config option.

### Starlight Logo Config Format

Starlight 0.38 accepts:
```js
logo: {
  src: './path/to/logo.svg',      // single logo (both light+dark)
  // OR:
  light: './path-light.svg',
  dark: './path-dark.svg',
  // optional:
  alt: 'GeoLens',
  replacesTitle: false,           // false = logo + "GeoLens Docs" title shown together
}
```

[ASSUMED — based on Starlight documentation knowledge; specifics of the `replacesTitle` option behavior in 0.38.4 should be confirmed in `node_modules/@astrojs/starlight/schemas/config.ts` if needed]

### Path Resolution

`astro.config.mjs` lives at `/Users/ishiland/Code/getgeolens.com/docs/astro.config.mjs`. Paths in the `logo.src` key are resolved relative to the **project root** which for the docs subpackage is `/Users/ishiland/Code/getgeolens.com/docs/`. The proposed `public/logo.svg` lives at `/Users/ishiland/Code/getgeolens.com/public/logo.svg`.

Starlight resolves `logo.src` paths as Vite asset imports, not file system paths — the path must resolve from the Astro project root. Since the docs project root is `docs/`, to reference `getgeolens.com/public/logo.svg`:
```js
logo: {
  src: '../../public/logo.svg',   // relative to docs/ project root, traverses up to repo root
  alt: 'GeoLens',
  replacesTitle: false,
}
```

**Landmine:** Astro/Vite may not import files outside the project root by default. A safer alternative is to copy `logo.svg` into `docs/public/logo.svg` (or `docs/src/assets/logo.svg`). Starlight specifically recommends placing the logo in `src/assets/` for proper optimization.

**Recommended safe path:**
1. Create canonical file at `getgeolens.com/public/logo.svg` (serves marketing site at `/logo.svg`)
2. Copy / symlink to `getgeolens.com/docs/src/assets/logo.svg` for Astro's asset pipeline
3. In `astro.config.mjs`: `logo: { src: './src/assets/logo.svg', alt: 'GeoLens', replacesTitle: false }`

This avoids cross-project-root Vite import issues entirely.

for label_css in "app:$APP_CSS" "marketing:$MARKETING_CSS" "docs:$DOCS_CSS"; do
  label="${label_css%%:*}"

    font_errors=$((font_errors+1))
    continue

  fi

if [ "$font_errors" -gt 0 ]; then

## Common Pitfalls

### Pitfall 1: Inter References Left in `html` Font Fallback

**What goes wrong:** `global.css` line 117 has a hardcoded fallback `'Inter Variable'` in the `html` font-family declaration. If only the `@import` and `@theme` token are updated, the fallback string still requests Inter — which is now absent from `node_modules`, causing a console warning in some browsers.

**How to avoid:** Update line 117 to `'IBM Plex Sans Variable'` as documented in the File Change Map.

**Where it lives:** `getgeolens.com/src/styles/global.css` line 117 — the `html {}` rule.

---

### Pitfall 2: `@fontsource/inter` devDependency in Marketing May Be Used by OG Script

**What goes wrong:** `getgeolens.com/package.json` has `@fontsource/inter` as a devDependency. The OG image generation script (`scripts/capture-screenshots.ts`) and satori-based OG generator may load Inter font data from this package. Removing it breaks OG image generation.

**How to avoid:** Before removing `@fontsource/inter`, grep `getgeolens.com/scripts/` for any `inter` or `fontsource` import. If found, either (a) switch the OG script to IBM Plex Mono/Sans data, or (b) add `@fontsource/ibm-plex-sans` (non-variable, for satori buffer use) as a devDependency.

**Warning signs:** `Cannot find module '@fontsource/inter'` error during `npm run capture` or `npm run a11y`.

---

### Pitfall 3: Starlight Logo Path Resolution Across Project Root

**What goes wrong:** Starlight resolves `logo.src` as a Vite asset import. If the path traverses outside the Astro project root (`docs/`), Vite may reject it with "cannot import outside project root".

**How to avoid:** Place the logo file in `docs/src/assets/logo.svg`. Do not use `../../public/logo.svg` as the `logo.src` path.

**Warning signs:** `error: Cannot import 'logo.svg' from outside of project root` at build time.

---

### Pitfall 4: Drift CI App Path Fails in CI (Single-Repo Checkout)

**What goes wrong:** The extended `check-token-sync.sh` includes an app-CSS path (`$SCRIPT_DIR/../../../geolens/frontend/src/index.css`). In CI, only `getgeolens.com` is checked out; the `geolens` repo is absent.

**How to avoid:** Guard the app-CSS leg with `test -f "$APP_CSS"` and skip gracefully with an INFO message (not an error). Only fail CI for marketing ↔ docs drift (both files are in-repo).

**Warning signs:** `FAIL [font]: app CSS file not found` in CI logs despite font being correctly set in marketing and docs.

---

### Pitfall 5: `--sl-color-bg-sidebar` Has Different Defaults in Light vs. Dark

**What goes wrong:** Overriding `--sl-color-bg-sidebar` in `:root` (unlayered) affects both light and dark modes if Starlight's dark-mode declaration is also unlayered. However, Starlight's dark-mode props are in `@layer starlight.base`, so the unlayered `:root` override wins in both themes.

**How to avoid:** Add the sidebar tint override **only** in the `:root` block (light mode), NOT in `:root[data-theme='dark']`. Verify dark mode sidebar still reads dark after change.

**Warning signs:** Dark mode sidebar appears washed-out warm white instead of dark gray.

---

### Pitfall 6: OG Image Font Loading (Satori) May Not Support Variable Fonts

**What goes wrong:** Satori (used for OG image generation in `getgeolens.com`) does not support variable fonts from `@fontsource-variable` packages. It requires a static weight font buffer.

**How to avoid:** OG scripts that load fonts must use `@fontsource/ibm-plex-sans` (non-variable, specific weight) not `@fontsource-variable/ibm-plex-sans`. Check `getgeolens.com/scripts/` for satori font loading.

[ASSUMED — based on known satori limitation; verify by checking getgeolens.com scripts directory for satori usage before removing @fontsource/inter]
