# Handoff — Style System Alignment Across geolens, getgeolens.com, and Docs

## Central Question

**Should `getgeolens.com` (marketing) and `docs.getgeolens.com` (Starlight docs) adopt the same design system as the geolens app, with `geolens/docs/DESIGN-GUIDE.md` as the single north star — or should they diverge intentionally?**

If the answer is "same system across the board", this is a multi-day refactor across two repos. If the answer is "intentional divergence", that decision needs to be documented in DESIGN-GUIDE.md so future contributors know the marketing/docs surfaces are deliberately different.

## Why This Came Up (Context)

This handoff was created during Phase 225 verification when the user asked whether the docs site styling aligned with the rest of GeoLens. Initial investigation found:

- The geolens app and `getgeolens.com/docs` (Starlight) are visually different (different fonts, different background warmth, different theme defaults).
- The marketing site `getgeolens.com` itself diverges from the app guide (Inter instead of IBM Plex, cool white instead of warm atlas paper, light-only instead of dual-mode).
- Phase 224 (BRAND-01..03) wired a partial token bridge from marketing → docs, but the bridge stopped at the primary blue palette. The full atlas-paper neutrals, surface hierarchy, dark-mode tokens, and elevation shadows from the app guide were never bridged.

The user identified `geolens/docs/DESIGN-GUIDE.md` as the **north star**, which makes both marketing and docs the deviant surfaces (not the docs alone).

## The Strategic Question (Three Honest Options)

### Option A — Full alignment to the app guide

Make `docs/DESIGN-GUIDE.md` the literal source of truth for all three surfaces. Marketing and docs both adopt:
- IBM Plex Sans + IBM Plex Mono
- Warm atlas paper neutrals (hue 85)
- Full 4-tier surface hierarchy
- Dual mode (light + dark) with the app's `.dark` token block mirrored
- Elevation shadow tokens with dark-mode opacity boost
- Logo continuity via shared `<Logo>` component
- Aside / callout colors mapped to `--primary` instead of Starlight defaults

**Pros**
- One mental model. New contributors learn one design system.
- Brand consistency across every surface a user touches.
- Future `status-colors.ts` patterns land on docs/marketing for free if they ever need status badges.
- The drift CI (`check-token-sync.sh`) becomes meaningful — it can grep all token blocks across all three surfaces.

**Cons**
- Large refactor. Marketing + docs both need to add IBM Plex deps, swap neutral tokens, add dark mode (marketing), and override most Starlight `--sl-color-*` slots (docs).
- Marketing was explicitly designed light-only for v14.0 launch (`global.css:6`) — adding dark mode is real product work, not a CSS swap.
- Risk of accessibility regressions during the swap unless re-audited (the warm hue 85 was tuned for the app's data-dense surfaces, which may not flatter marketing's hero typography or docs' long-form prose).
- Expressive Code's own syntax theme is well-tuned; replacing it with the app's `--code-*` tokens could degrade code-readability if not done carefully.

**Effort estimate:** ~2–3 phases (1 marketing, 1 docs, 1 cross-surface QA).

### Option B — Partial alignment, keep documented divergence

Mirror tokens that *visually* unify the brand (primary palette ✓ already done, fonts, logo, Aside accent), but leave Starlight defaults for surfaces where the divergence is cheap to live with:
- Adopt IBM Plex Sans + Mono (fixes the most visible mismatch — typography)
- Adopt logo continuity
- Override Aside `--sl-color-purple` → `--primary-700`
- Skip warm atlas-paper backgrounds (Starlight's neutral palette is fine for long-form docs; warm hue 85 was tuned for *map-adjacent* surfaces)
- Skip elevation token bridge (Starlight's flat-ish surfaces work fine for docs)
- Keep marketing light-only as a deliberate v14.0 decision, document the rationale

**Pros**
- Smaller refactor. Highest-ROI changes only (~1 day across both surfaces).
- Acknowledges that docs and marketing are different *visual contexts* than a data-dense app — what works for an attribute table may not flatter a hero headline or a long technical paragraph.
- Keeps Expressive Code and Starlight defaults that are already tuned for their use case.

**Cons**
- DESIGN-GUIDE.md needs an "Out of scope for this guide" section explicitly listing the divergences (typography ✓, primary palette ✓, logo ✓; backgrounds ✗, elevation ✗, dark mode ✗ for marketing).
- Drift CI gets messier — it has to grep an explicit allowlist of "what should match" rather than "everything matches".
- Future contributors may re-litigate the divergence boundary.

**Effort estimate:** ~1 phase (split: half marketing, half docs).

### Option C — No alignment, fork the design systems

Accept that marketing, docs, and app are three independent products with their own visual identities. Stop bridging tokens. Remove the partial bridge in `docs/src/styles/custom.css`. Document that DESIGN-GUIDE.md applies only to the app.

**Pros**
- No refactor. Each surface evolves independently.
- Marketing teams (if any) and docs teams (if any) can move at their own velocity.

**Cons**
- Brand fragmentation. A user signing up via marketing → reading docs → using the app sees three different products.
- Token-bridge work in Phase 224 BRAND-01..03 was wasted.
- Sets a "we don't really care about visual continuity" precedent that's hard to walk back.

**Effort estimate:** ~half a day (rip out the bridge, update docs).

**Recommendation: Option B** is the most defensible for a small team. Option A is the right answer if there are dedicated design resources; otherwise it's likely to ship half-done. Option C is rarely the right answer for a single-product company.

## What Each Option Looks Like as Concrete Work

### If Option B (recommended) is chosen, here's the work:

**Marketing site (`getgeolens.com/src/`):**
1. Install `@fontsource-variable/ibm-plex-sans` + `@fontsource/ibm-plex-mono`. Update `global.css:9-10` `@import` and `--font-sans`. (~10 min)
2. Decide: add dark mode now, or formally defer with a comment in `global.css:6`. (~varies)
3. Confirm `<Logo>` component is shared with the app's logomark — if not, extract a shared SVG. (~15 min)
4. No background / surface / elevation changes — keep current cool-white aesthetic.

**Docs site (`getgeolens.com/docs/src/`):**
1. Install IBM Plex packages. Update `custom.css:18` `@import` and `--sl-font` + add `--sl-font-mono`. (~10 min)
2. Override `--sl-color-purple` → `var(--primary-700)` in `custom.css :root`. Aside `tip` callouts then read brand blue. (~1 line + visual QA)
3. Update Starlight `logo` config in `astro.config.mjs` to reference the shared marketing logomark SVG. (~5 lines)
4. Optional polish: override `--sl-color-bg-sidebar` to a slightly warmer tint (`oklch(0.99 0.003 85)`) so docs sidebar feels less Starlight-default-cool. (~1 line)
5. Extend `docs/scripts/check-token-sync.sh` to grep IBM Plex font-family declaration in addition to the primary palette. (~3 lines)

**Cross-cutting:**
1. Update `docs/DESIGN-GUIDE.md` with a new section: `## Scope and Cross-Surface Application`. Document that marketing and docs adopt typography + primary palette + logo + Aside accents, but **not** atlas-paper neutrals, surface hierarchy, or elevation tokens (the rationale: those tokens were tuned for data-dense map-adjacent UI; marketing/docs are content-display contexts).
2. Optionally: rename `docs/DESIGN-GUIDE.md` → `docs/DESIGN-GUIDE-APP.md` and add a sibling `DESIGN-GUIDE-MARKETING.md` codifying what's *intentionally* different.

### If Option A (full alignment) is chosen, the work expands:

In addition to Option B's items:

**Marketing site:**
- Swap `global.css :root --background` from `oklch(1 0 0)` → `oklch(0.985 0.003 85)`. Re-audit hero/CTA contrast (warm tint may shift WCAG calculations on existing primary buttons).
- Add a full `:root.dark { ... }` block mirroring `geolens/frontend/src/index.css .dark`. Add a theme toggle to the marketing nav. Add FOUC prevention script to `index.html`.
- Add the 4-tier surface system (`--surface-0..3`).
- Add the 3-tier elevation system; update `<Card>` + button shadows to use it.
- A11Y audit pass after the swap.

**Docs site:**
- Map all relevant `--sl-color-bg*` slots to the warm atlas paper tokens (`--sl-color-bg`, `--sl-color-bg-nav`, `--sl-color-bg-sidebar`, `--sl-color-bg-inline-code`, `--sl-color-bg-accent`, etc.).
- Mirror the app's `.dark` block onto `:root[data-theme='dark']` token slots — full inversion, not just accent slots.
- Map elevation tokens to `--sl-shadow-md` / `--sl-shadow-lg` with the dark-mode opacity boost.
- Decide whether to bridge `--code-*` tokens onto Expressive Code (likely no — keep Expressive Code's own theme).

### If Option C (fork) is chosen:

- Delete the token bridge in `docs/src/styles/custom.css`.
- Remove `docs/scripts/check-token-sync.sh`.
- Remove the bridge documentation from Phase 224 SUMMARY.md.
- Update `docs/DESIGN-GUIDE.md` to add a scope section: "This guide applies to the app only. Marketing and docs are independent products."

## Decision Inputs the Next Owner Needs

Before picking an option, the decision-maker should answer:

1. **Is there a design owner**, even part-time, across all three surfaces? If yes → Option A is feasible. If no → Option B.
2. **Does marketing intentionally want dark mode**, or is it light-only as a brand-deliberate choice? `global.css:6` says "NOT supported for v14.0 launch" — has that intent expired with v15.0?
3. **What's the cost of brand fragmentation** in user terms? (E.g., a developer landing on marketing → docs → trying the app — does it feel like one product?) The answer probably comes from user research, not from this handoff.
4. **What's the cost of design-system maintenance** over time? Three converged surfaces means every token change ripples through three repos; three forked surfaces means the brand drifts unless someone curates.
5. **Does Expressive Code's syntax theme need to match the app's `--code-*` tokens?** Probably no — Expressive Code is well-tuned for code readability and its theme is purpose-built. But document the decision.
