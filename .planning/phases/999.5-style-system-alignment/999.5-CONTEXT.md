# Phase 999.5: Style System Alignment - Context

**Gathered:** 2026-04-26
**Status:** Ready for planning

<domain>
## Phase Boundary

Reconcile the visual design system across three GeoLens surfaces — the geolens app, `getgeolens.com` (marketing), and `docs.getgeolens.com` (Starlight docs) — so the brand reads as one product without forcing every token to match. `geolens/docs/DESIGN-GUIDE.md` is the canonical north star for the app. Marketing and docs adopt a partial set of upstream tokens (typography, primary palette, logo, callout accents) and intentionally diverge on neutrals, surface hierarchy, elevation, and dark-mode policy.

**In scope:**
- Marketing site (`getgeolens.com`): swap to IBM Plex Sans Variable + IBM Plex Mono; formalize the existing light-only intent with documented rationale.
- Docs site (`getgeolens.com/docs`): swap to IBM Plex Sans + Mono; override Starlight Aside accents (`tip` and `note`/`caution`/`danger` slots) to brand primary instead of Starlight defaults; warmer sidebar tint; Starlight `logo` config.
- Logo continuity: a single shared SVG referenced by app, marketing, and docs (extraction TBD by researcher).
- New `getgeolens.com/DESIGN-GUIDE.md` at the repo root that documents the marketing + docs design system and **references upstream** to `geolens/docs/DESIGN-GUIDE.md` for shared tokens (no token duplication).
- Drift CI extension: `getgeolens.com/docs/scripts/check-token-sync.sh` greps IBM Plex font-family declarations across all three surfaces in addition to the existing primary-palette grep.

**Out of scope (intentional Option B divergence — document, don't implement):**
- Atlas-paper warm neutrals on marketing+docs (kept Starlight/cool-white).
- 4-tier `--surface-0..3` hierarchy on marketing+docs.
- Elevation shadow tokens on marketing+docs.
- Dark mode for marketing site (deferred with documented rationale; tracked separately).
- `--code-*` token bridge to Expressive Code (Expressive Code's own theme stays).
- Any change to `geolens/docs/DESIGN-GUIDE.md` itself — it remains the unmodified north star.

</domain>

<specifics>
## Specific Ideas

- **Typography target:** IBM Plex Sans Variable + IBM Plex Mono — not Inter. The app already uses IBM Plex; the swap is marketing+docs adopting the app font, not the app changing.
- **Aside `tip` color target:** `var(--primary-700)` instead of Starlight's default `--sl-color-purple`.
- **Sidebar tint target:** `oklch(0.99 0.003 85)` — slightly warmer than pure Starlight cool-white but much lighter than the app's atlas-paper neutrals (`oklch(0.985 0.003 85)`). Picked to suggest brand warmth without forcing the full app neutral palette into long-form docs.
- **New DESIGN-GUIDE.md location:** `getgeolens.com/DESIGN-GUIDE.md` (repo root), not `getgeolens.com/docs/src/content/...` — it's a meta-doc about the design system, not user-facing docs content.
- **Dark-mode comment style:** rewrite `global.css:6` to follow the format "Intentional: [decision]. Rationale: [why]. Revisit if: [trigger]." rather than the v14.0-era "NOT supported" phrasing.

</specifics>

<deferred>
## Deferred Ideas

- **Marketing dark mode** — backlog item (D-06). Not in 999.5 scope. Suggested home: a future backlog phase (e.g. 999.6) once a trigger condition fires (design owner hire, brand refresh, or product decision to support dark mode in marketing surfaces).
- **Full Option A alignment** — atlas-paper neutrals on marketing+docs, surface hierarchy, elevation tokens, full Starlight `--sl-color-bg*` override. Reconsider only if a design owner is added or marketing dark mode work happens; otherwise stays divergent.
- **`--code-*` token bridge to Expressive Code** — explicitly out of scope. Expressive Code's syntax theme is purpose-built for code readability; bridging app `--code-*` tokens risks degrading code readability without clear gain.
- **Three-way design guide split** (`DESIGN-GUIDE-APP.md` + `DESIGN-GUIDE-MARKETING.md` + `DESIGN-GUIDE-DOCS.md`) — considered but rejected. The new `getgeolens.com/DESIGN-GUIDE.md` covers both marketing and docs in one doc since they share a repo and a build pipeline.
- **A11Y re-audit of bridged tokens** — not formally scoped. Option B's typography swap is unlikely to shift WCAG calculations, but if Option A is ever chosen later, an explicit a11y audit phase is required.

</deferred>
