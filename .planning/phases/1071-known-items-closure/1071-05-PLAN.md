# Postgres image used by docker-compose
The application's docker-compose.yml uses `postgis/postgis:16-3.4` (or
similar) for the `db` service. The test script should use the SAME tag
to ensure migrations apply cleanly against the production-tier image, not
just any vanilla postgres image.

Read `docker-compose.yml` for the `db.image:` line to get the exact tag,
or read `.env.example` POSTGRES_* defaults to mirror the local-dev config.

# Cleanup discipline
The script MUST clean up after itself (docker rm -f the ephemeral container)
even if the migration fails. Use `trap` to register cleanup before starting
the container, so a Ctrl-C or unexpected exit still drops the container.

  <action>
  Create `backend/scripts/test_alembic_upgrade_clean_db.sh` as an executable
  POSIX shell script (chmod +x). The script must:

  1. Be runnable from `backend/` as `./scripts/test_alembic_upgrade_clean_db.sh`
     and from anywhere via the full path.
  2. Pick a unique container name (e.g. `geolens-alembic-test-$$` using `$$`
     for the PID) so concurrent runs don't collide.
  3. Pick a free local port for the throwaway PostgreSQL (avoid the
     5432/`${DB_PORT}` that the dev stack uses). Use 54399 by default with
     an `ALEMBIC_TEST_DB_PORT` env override.
  4. Read the postgis image tag from `docker-compose.yml`. If the script
     can't parse the compose file (POSIX shell, no yq), hard-code the
     image to match the compose value at script-authoring time and
     surface a comment explaining the discipline ("if compose changes the
     image tag, update this script in lockstep").
  5. Use a `trap` to register cleanup: `docker rm -f <name>` on ANY exit
     (success, failure, signal).
  6. Steps in order:
     a. Echo what it's doing.
     b. Pull the postgis image if not already present (optional — docker
        run does this implicitly).
     c. `docker run -d --name <name> -p <port>:5432 -e POSTGRES_USER=geolens
        -e POSTGRES_PASSWORD=test -e POSTGRES_DB=geolens postgis/postgis:<tag>`
     d. Poll `docker exec <name> pg_isready -U geolens -d geolens` until
        the DB is ready (max 60s, exit 1 with diagnostic on timeout).
     e. Export `DATABASE_URL=postgresql+asyncpg://geolens:test@localhost:<port>/geolens`
        (the format the app expects — confirm by reading `backend/app/core/config.py`
        for the URL shape).
     f. Run `uv run alembic upgrade head` from inside `backend/`. Capture
        exit code.
     g. On success: echo "✓ Alembic upgrade head applied cleanly against
        a fresh DB" and exit 0.
     h. On failure: dump container logs (`docker logs <name>`), the alembic
        stderr, and exit non-zero with diagnostic.

  7. Be idempotent — running it twice in a row should both pass (the trap
     ensures the previous run's container is gone before the next starts;
     if PID collision somehow happens, `docker run -d --name X` will fail
     fast and the script exits non-zero).

  Shebang: `#!/usr/bin/env bash`. Use `set -euo pipefail` at the top.

  Sample skeleton (executor must flesh out the polling loop, image-tag
  resolution, and error diagnostics):

  ```bash
  #!/usr/bin/env bash
  # test_alembic_upgrade_clean_db.sh — exercise the full migration chain
  # against a freshly-initialized PostGIS container.
  #
  # KNOWN-02 (v1016 Phase 1071): v1015's close-gate only verified migration
  # ordering via down_revision linkage. This script provides the semantic
  # complement — does the chain APPLY without error against a clean DB?
  #
  # Usage:
  #   cd backend && ./scripts/test_alembic_upgrade_clean_db.sh
  #
  # Env overrides:
  #   ALEMBIC_TEST_DB_PORT — local port for the throwaway DB (default 54399)
  #
  # Requires: docker daemon running, uv installed.

  set -euo pipefail

  CONTAINER_NAME="geolens-alembic-test-$$"
  PG_PORT="${ALEMBIC_TEST_DB_PORT:-54399}"
  POSTGIS_IMAGE="postgis/postgis:16-3.4"  # MUST match docker-compose.yml db.image
  WAIT_SECONDS=60

  cleanup() {
    docker rm -f "${CONTAINER_NAME}" >/dev/null 2>&1 || true
  }
  trap cleanup EXIT INT TERM

  # ... (start container, poll readiness, run alembic, dispatch on exit code)
  ```

  Verify the working-directory assumption: the script must be invokable
  from `backend/` (alembic.ini is at `backend/alembic.ini` and alembic's
  `script_location = alembic` is a relative path).

  One-shot maintenance scripts for the GeoLens backend. Each script is
  meant to be run from `backend/` as the working directory.

  ## Alembic clean-DB upgrade test

  **Script:** `test_alembic_upgrade_clean_db.sh`

  Runs `alembic upgrade head` against a freshly-initialized, throwaway
  PostGIS container. This is the semantic complement to the syntactic
  `down_revision` linkage check that v1015's close-gate ran — confirms
  the full migration chain (0001 → latest) actually applies without error
  on a clean DB, not just that the revision graph is well-formed.

  ### Prerequisites

  - Docker daemon running locally (the script spins up a container).
  - `uv` installed (alembic runs through `uv run`).
  - No other process listening on port 54399 (override with
    `ALEMBIC_TEST_DB_PORT` if the default collides).

  ### Usage

  ```bash
  cd backend && ./scripts/test_alembic_upgrade_clean_db.sh
  ```

  Exits 0 on success, non-zero with diagnostics on failure. The throwaway
  container is `docker rm -f`'d on every exit path (including Ctrl-C).

  ### When to run

  - At every milestone close-gate (Phase XXXX-CLOSE) after the last
    migration lands and before the version tag is cut.
  - When adding a new migration to v1016-and-later phases — local
    smoke before pushing the PR.
  - Suspected migration-chain rot (e.g. an out-of-order `down_revision`
    that the linkage check missed because both sides referenced the same
    revision).

  ### Limitations

  - Tests the schema-only path. Data migrations that depend on existing
    rows (rare in this repo; the convention is "expand/contract DDL with
    no row-mutation") would need a fixture step before `upgrade head`.
  - Does NOT run the application after migration — only confirms alembic
    exits 0. Application boot is exercised by the existing pytest
    `test_db_session` fixture and by `docker compose up`.
  - The `POSTGIS_IMAGE` tag is hard-coded in the script and MUST be kept
    in lockstep with `docker-compose.yml`'s `db.image:` value. If you
    bump the PostGIS image in compose, also bump it here.
  ```
