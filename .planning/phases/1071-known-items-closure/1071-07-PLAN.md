The inline 7-prefix tuple lives in `backend/app/processing/ingest/validation.py:113-121`:

```python
# Scan every <SourceFilename> for path-traversal markers.
vsi_prefixes = (
    "/vsis3/",
    "/vsicurl/",
    "/vsizip/",
    "/vsigs/",
    "/vsiaz/",
    "/vsitar/",
    "/vsimem/",
)
```

And the docstring above the function references the list at lines 83-85:

```
    EXCEPT recognized GDAL VSI prefixes (/vsis3/, /vsicurl/, /vsizip/)
    which the COG ingest path legitimately uses for managed-storage VRTs.
```

This is the ONLY consumer of the tuple today. The "dual-edit" risk the
v1015 followup mentions is about a FUTURE state where another module
needs to reference the same set — for example:
- An env-overlay constant like `GDAL_VSI_ALLOWED_HANDLERS` if a future GDAL
  version exposes a scheme allow-list at the env level.
- A VRT-source URL classifier for the import pipeline.
- A documentation generator that lists the supported VSI schemes.

The fix is to extract the tuple to a module-level constant in
`raster/vrt.py` (the module that already owns the VRT subprocess hardening),
and have `validate_vrt_body` import it.

## Why raster/vrt.py and not validation.py
`validation.py` is in the `ingest/` subpackage and is broader (it validates
all uploaded file content types — shapefiles, GPKGs, raster, etc.). The
VSI prefix list is specifically about VRT semantics, so it conceptually
belongs in `raster/vrt.py` alongside `_VRT_SAFE_ENV` and `_build_vrt`.

This also gives the import a one-way dependency direction:
`ingest.validation` already imports from elsewhere in the app; it can
import from `raster.vrt` cleanly (no circular risk; the raster module
has no need to import from ingest.validation).

## Naming
Use `VRT_VSI_ALLOWED_PREFIXES` (full, descriptive name) as the public
constant. Keep the prefixes as a `tuple[str, ...]` — `str.startswith()`
accepts a tuple natively (the consumer at validation.py:137 already
does `raw_path.startswith(vsi_prefixes)` — works against any tuple).

## Sort discipline
List the 7 prefixes alphabetically in the new constant. This is a
no-behavior-change cosmetic refactor for grepability — `str.startswith`
doesn't care about order, but a sorted list reads as a deliberate
"complete" allow-list rather than a "the order I happened to type" list.

      "/vsimem/",

  class TestVrtVsiAllowedPrefixes:
      def test_constant_shape_and_value(self):
          """The exported constant has exactly the 7 expected prefixes."""
          from app.processing.raster.vrt import VRT_VSI_ALLOWED_PREFIXES

          from app.processing.raster.vrt import VRT_VSI_ALLOWED_PREFIXES

          assert isinstance(VRT_VSI_ALLOWED_PREFIXES, tuple)

      def test_validate_vrt_body_consumes_shared_constant(self, tmp_path, monkeypatch):
          """Monkey-patch the constant and confirm validate_vrt_body picks it up

          monkeypatch.setattr(vrt_module, "VRT_VSI_ALLOWED_PREFIXES", patched)

          )

          vrt_file = tmp_path / "fake.vrt"

              b"  <SRS>EPSG:4326</SRS>\n"
              b'  <VRTRasterBand dataType="Byte" band="1">\n'
              b'    <SimpleSource>\n'
              b'      <SourceFilename relativeToVRT="0">' + test_prefix.encode() + b'fake.tif</SourceFilename>\n'
              b'      <SourceBand>1</SourceBand>\n'
              b'    </SimpleSource>\n'
              b'  </VRTRasterBand>\n'
              b"</VRTDataset>\n"
          )

          # If validate_vrt_body consumes the shared constant, this passes.
          # If it has a private copy, ValueError fires (absolute path rejected).
          validation_module.validate_vrt_body(str(vrt_file))  # MUST NOT raise

              b"</VRTDataset>\n"
