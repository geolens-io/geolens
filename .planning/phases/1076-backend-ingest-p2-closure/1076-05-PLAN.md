<objective>
Add an optional `strict_cog: bool` field to `RasterCommitRequest` (default `False`). When `True`, raster commit rejects non-COG TIFFs at the magic-byte rule by failing the job during `tasks_raster.ingest_raster` BEFORE `check_and_prepare_cog` would convert the file. Closes ING-07 / P2-09 from `INGEST-AUDIT-2026-05-21.md`.

Purpose: Today, `is_cog_compliant` is surfaced in preview but commit does NOT gate on it — `check_and_prepare_cog` (`cog.py:329`) converts non-COG TIFFs by design. Strict-mode users (e.g. SDI operators ingesting an externally-produced COG catalog) want commit to FAIL if the source isn't already COG, not silently rewrite it. Default `False` preserves all existing call sites and tests.

Out of scope:
- Default flip from `False` → `True` (deferred to a future coordinated CLI/manifest schema bump per REQUIREMENTS.md "Future Requirements")
- Frontend UI surface for the toggle (this plan adds the wire contract only; UI lives in v1077 or later)

Output:
- `RasterCommitRequest.strict_cog: bool = False` field with Pydantic Field description
- `tasks_raster.ingest_raster` checks `um.get("strict_cog")` and, when True, calls `check_cog_compliance(file_path)` BEFORE the COG conversion branch; on non-compliance, raises `ValueError` with a clear message
- `test_commit_request_schemas.py` extended with the new field cases (model_fields assertion + 3 unit tests)
- New `backend/tests/test_strict_cog_enforcement.py` covers the behavior gate
- Backwards compatible: every existing raster commit test continues to pass without supplying `strict_cog`
</objective>
