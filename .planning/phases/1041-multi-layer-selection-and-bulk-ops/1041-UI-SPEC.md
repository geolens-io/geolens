## Interaction Contract

This section is the primary deliverable for Phase 1041. It defines every visual state and interaction rule the executor must implement.

### Decision A — Bulk action bar position: stack footer

**Decision (Claude's discretion):** The bulk action bar is anchored to the **bottom of the `UnifiedStackPanel`** (inside the sidebar, above the panel bottom edge), not the header.

Rationale:
- The header already has two CTAs (`⚙ Settings` + `＋ Add data`). Adding a third group of controls (visibility / opacity / group / ungroup / delete) would overload the 340px header chrome and force icon-only compression.
- Footer placement follows the "action bar rises when selection is active" pattern used in file managers and data tables — users recognize this idiom.
- The stack footer already has `padding-bottom: 80px` (from `sidebar-structure.md` `.sb-stack` rule) leaving room for a floating footer without reflow.
- The bar animates in when `selectedCount >= 2` and animates out on clear — it does not permanently occupy space.

**Bulk action bar anatomy:**

```
[ {N} selected  |  👁 Visibility  |  ▕◀▶▏ Opacity  |  ⊞ Group  |  ⊡ Ungroup  |  🗑 Delete ]
```

- Left: `"{N} selected"` count label — `text-[13px] font-medium text-muted-foreground`; occupies ~80px.
- Visibility: `Button` variant `ghost` size `sm`, `Eye` / `EyeOff` icon from lucide-react, label "Visibility". Toggles ALL selected layers to the OPPOSITE of the majority state (if > half are visible → hide all; else → show all).
- Opacity: compact horizontal `Slider` from shadcn, range 0–100, `w-20`. Shows the AVERAGE opacity of selected layers at open. Dragging sets all selected layers to the new value. Live (no debounce during drag on optimistic state; debounce API calls 300ms).
- Group: `Button` variant `ghost` size `sm`, `FolderPlus` lucide icon, label "Group". Only enabled when ALL selected rows are loose layers (not already in a group — enabling on grouped-layer selection would require ungroup-first).
- Ungroup: `Button` variant `ghost` size `sm`, `FolderMinus` lucide icon, label "Ungroup". Only enabled when ALL selected rows are group rows (folder-group type). Disabled when selection is mixed or contains only loose layers.
- Delete: `Button` variant `ghost` size `sm`, `Trash2` lucide icon, label "Delete". Always enabled when `selectedCount >= 2` AND no basemap rows in selection. Uses `text-destructive` class (no separate button variant needed — ghost + destructive text color). See § Bulk delete confirmation.

**Bar container:**

```css
.bulk-action-bar {
  position: sticky; bottom: 0;
  height: 48px;
  display: flex; align-items: center; gap: 4px;
  padding: 0 12px;
  background: var(--surface-2);
  border-top: 1px solid var(--border);
  border-bottom-left-radius: var(--radius-md);
  border-bottom-right-radius: var(--radius-md);
}
```

Enter animation: `translate-y-0 opacity-100 duration-150` from `translate-y-2 opacity-0`. Exit: reverse. Use Tailwind `transition-all duration-150`.

### 1. Selection model (POL-06)

**Storage:** `selectedIds: Set<string>` in React component state inside `UnifiedStackPanel`. Not zustand — selection is ephemeral builder session state, not persisted. The set holds layer IDs only (no group IDs, no sublayer IDs from within the basemap group).

**Mouse interactions:**

| Gesture | Effect |
|---------|--------|
| Plain click on a row | Clears selection; sets `selectedLayerId` (single-selection for flyout) — existing behavior |
| `Cmd/Ctrl` + click on a row | Toggles that row in `selectedIds`; does NOT change `selectedLayerId` |
| `Shift` + click on a row | Range-selects from the last toggle anchor to the clicked row (inclusive); does NOT change `selectedLayerId` |
| Click outside the stack panel | Clears `selectedIds` (outside-click handler on `document`) |

**Keyboard interactions (on a focused row):**

| Key | Effect |
|-----|--------|
| `Space` | Toggles the focused row in `selectedIds` |
| `Shift + ArrowUp` | Extends selection upward to the previous row |
| `Shift + ArrowDown` | Extends selection downward to the next row |
| `Escape` | Clears `selectedIds`; focus stays on the last focused row |
| `Enter` | Opens `LayerEditorPanel` for the focused row (single-selection); does NOT change multi-selection |

**Anchor tracking:** A `lastToggleAnchor: string | null` ref tracks the ID of the last `Cmd-click` or `Space`-toggled row. `Shift-click` / `Shift+Arrow` extends from this anchor.

**Boundary enforcement (POL-11):** The selection model refuses basemap group rows and basemap sublayer rows. Implementation: before adding any ID to `selectedIds`, check `isBasemapLayer(id)` (helper that checks `basemapGroup.id` and `basemapGroup.sublayers[*].id`). If true, ignore the gesture without error or toast — the cursor hint (see below) is the signal.

**Cross-boundary visual signal:** When the user attempts to `Cmd-click` or hover-shift-click a basemap row, the cursor shows `not-allowed` only on the basemap row itself. No toast. The bulk delete cannot target the basemap because the basemap IDs are never in `selectedIds`.

### 2. Selection state visual (POL-07)

**Rest state (no selection):** Row renders as today — eye + grip + type-icon + name + opacity + kebab in the 7-column grid.

**Single-selection focus (existing behavior, unchanged):** The row that opened `LayerEditorPanel` renders `background: var(--primary-50)` + `box-shadow: inset 2px 0 0 var(--primary)`. No checkbox. `aria-selected="true"` is already set via `StackRow.tsx:146`.

**Multi-selected row (new for Phase 1041):**

- Background: `var(--primary-50)` (same as single-selection — they share the tint)
- Left rail: `box-shadow: inset 2px 0 0 var(--primary)` (same as single-selection)
- Checkbox: a `h-3.5 w-3.5` checkbox in the **caret column** (16px wide):
  - On rows that are NOT group rows (no caret today): the caret column is `visibility: hidden` — replace with a `Checkbox` component (shadcn `components/ui/checkbox`, already installed).
  - On rows that ARE group rows (folder-groups with a caret): the checkbox replaces the caret ONLY during multi-selection mode (when `selectedIds.size >= 1`). When selection is cleared the caret returns.
  - Checkbox is `checked` when the row is in `selectedIds`.
  - Clicking the checkbox is equivalent to `Cmd-click` on the row — toggles the row without affecting `selectedLayerId`.
- `aria-selected="true"` on the row element.
- `aria-checked="true"` on the checkbox element.

**Decision (Claude's discretion) — checkbox visibility:** Checkboxes are ONLY visible when at least one row is in `selectedIds` OR when the user holds `Cmd/Ctrl` (tracked via `keydown/keyup` document listeners). At rest (no multi-selection in progress) the caret column returns to its original state. This keeps the rest-state clean — no persistent checkbox column — and matches the "reveal on activation" idiom used by macOS Finder and linear.app.

**Visual disambiguation table:**

| State | Background | Left rail | Checkbox |
|-------|------------|-----------|----------|
| Rest | none | none | hidden |
| Hover | `var(--surface-2)` | none | hidden |
| Single-selected (flyout open) | `var(--primary-50)` | 2px primary | hidden |
| Multi-selected | `var(--primary-50)` | 2px primary | visible + checked |
| Multi-hover (not yet selected) | `var(--surface-2)` | none | visible + unchecked (during active multi-selection mode) |

### 3. Bulk action bar conditions (POL-08)

The bulk action bar is rendered when `selectedIds.size >= 2`. With 0 or 1 selected rows, the bar is unmounted (not merely hidden) to avoid focus-trap issues.

**Per-action enable/disable rules:**

| Action | Enabled when |
|--------|-------------|
| Visibility toggle | `selectedIds.size >= 2` AND no basemap rows (enforced by model) |
| Opacity slider | `selectedIds.size >= 2` AND no basemap rows |
| Group | ALL selected rows are loose layers (not inside a folder-group, not a folder-group row itself) |
| Ungroup | ALL selected rows are folder-group rows (layer_type `'group:folder'`) |
| Delete | `selectedIds.size >= 2` AND no basemap rows |

Disabled buttons: `opacity-40 cursor-not-allowed pointer-events-none`. Add a `Tooltip` (shadcn, already installed) with a one-line explanation: e.g., "Cannot group layers that are already in a group."

### 4. Atomic bulk operations (POL-09)

All bulk operations use **existing per-layer PATCH callbacks** (no new backend endpoints). The pattern is:

1. Snapshot `previousLayers` from current state.
2. Apply optimistic update to ALL selected layers in local state (one React setState call — not N calls).
3. Fire N parallel PATCH requests (`Promise.allSettled`).
4. On ANY rejection: call `setLayers(previousLayers)` (rollback); call `toast.error(...)` (one toast, not N).
5. On all-success: no toast (state is already correct from optimistic update).

**Rollback scope:** Rolls back ALL layers in the selection, not just the failed one. Partial success is not surfaced as partial state — the atomic contract requires all-or-nothing from the user's perspective.

**Implementation location:** A `handleBulkOp(op, selectedIds)` function in `use-builder-layers.ts` (or a co-located `use-bulk-ops.ts` hook). It receives the current snapshot, the selection set, and the op type.

**Selection clear after bulk op:** Selection clears on successful bulk op completion (all API calls resolved). Selection does NOT clear if rollback occurs — the user sees the error toast and can retry.

### 5. Bulk delete confirmation

**Decision (Claude's discretion):** No modal. Use an inline confirmation within the bulk action bar itself.

Rationale:
- A modal for bulk delete would block the entire stack panel, preventing the user from reviewing which layers are selected before confirming.
- Inline confirmation within the bar (two-step click: Delete → confirm red "Delete {N} layers?" + Cancel) is sufficient for a builder tool where the undo path is clear (map autosave is optimistic). This pattern is used by Airtable, Notion, and Linear for bulk-destructive actions in dense list UIs.
- A modal IS appropriate for single-layer delete (current `StackRow.tsx:380-386` AlertDialog pattern) because single delete has no selection context — the user must confirm intent without visual batch context. Multi-select already provides that context.

**Inline confirmation sequence:**

1. User clicks "Delete" in the bulk action bar.
2. The bar transitions to a confirmation state:
   - Left label changes to `"Delete {N} layers? This cannot be undone."` in `text-destructive text-sm`.
   - Visibility / Opacity / Group / Ungroup buttons are hidden (bar collapses to just the confirmation row).
   - Two buttons appear: `Cancel` (ghost, sm) on the left and `Delete {N} layers` (ghost + `text-destructive`, sm) on the right.
   - `autoFocus` is set on the `Cancel` button (safe default — prevents accidental double-click deletion). Source: AUD-09 finding pattern.
3. User clicks `Delete {N} layers` → execute the bulk delete op (§4).
4. User clicks `Cancel` OR presses `Escape` → bar returns to normal bulk-action state; selection is preserved.

**Copy (confirmation state):**

| Element | Copy |
|---------|------|
| Confirmation label | `"Delete {N} layers? This cannot be undone."` |
| Cancel button | `"Cancel"` |
| Confirm button | `"Delete {N} layers"` |

### 6. Selection clearing rules (POL-10)

| Event | Effect |
|-------|--------|
| `Escape` key (anywhere in the stack panel) | Clear `selectedIds`; keep `selectedLayerId` unchanged |
| Click outside the stack panel | Clear `selectedIds` (document `mousedown` listener, excluded when target is inside the stack panel or the bulk action bar) |
| Route change (React Router `useEffect` on `location.pathname`) | Clear `selectedIds` on unmount |
| Drag start (`onDragStart` from DndContext) | Clear `selectedIds` — drag and multi-select are mutually exclusive interactions |
| Successful bulk op | Clear `selectedIds` |

**Outside-click disambiguation:** Plain click on a ROW is handled by the row's own `onClick` (which sets `selectedLayerId` and clears multi-selection). The document-level `mousedown` listener fires BEFORE the row's `onClick`, so it would incorrectly clear selection on row clicks. Guard: the document listener checks `if (stackPanelRef.current?.contains(e.target)) return` before clearing.

### 7. Focus management and ARIA (POL-23 foundation)

The stack panel's `role="listbox"` (currently at `UnifiedStackPanel.tsx:689`) must gain:

```
aria-multiselectable="true"
aria-label="Map layers"
```

Each row gains:
- `aria-selected="true"` when the row is in `selectedIds` (in addition to the existing single-selection `aria-selected` already wired at `StackRow.tsx:146`)
- `aria-checked` on the `Checkbox` element

The bulk action bar element:

```
role="toolbar"
aria-label="Bulk actions for {N} selected layers"
aria-live="polite"   ← announces the {N} count change
```

When `selectedIds` count changes, the `aria-live` region announces: `"{N} layers selected."` (polite — does not interrupt ongoing speech).

### 8. Keyboard-only path for multi-select

The executor MUST wire keyboard sensors for multi-selection without requiring a mouse. The contract:

| Interaction | Path |
|-------------|------|
| Start multi-selection | Hold `Cmd/Ctrl` or use `Space` on a focused row |
| Extend selection | `Shift + ArrowUp` / `Shift + ArrowDown` from any focused row |
| Cancel selection | `Escape` |
| Activate bulk op | Tab into the bulk action bar (`tabIndex={0}` on each button), then `Enter` or `Space` |

The stack panel rows use `tabIndex={selected ? 0 : -1}` for roving tabindex within the listbox. When multi-selection is active, all selected rows have `tabIndex={0}`.

### 9. Interaction flow diagram (prose)

**Happy path — select and group:**
1. User `Cmd-clicks` Row A → Row A gets multi-selected tint + checkbox. Count badge "1 selected" not shown (need 2).
2. User `Cmd-clicks` Row B → Row B gets multi-selected. Bulk action bar rises from bottom: "2 selected | Visibility | Opacity | Group | Ungroup | Delete".
3. User clicks "Group" → Group op fires. Both rows gain `parent_group_id: newGroupId`. Optimistic: a new folder-group row appears in the stack with both rows as children. API PATCH fires for each. Bar dismisses. Selection clears.

**Happy path — bulk delete:**
1. User `Shift-clicks` from Row A to Row D → 4 rows multi-selected.
2. User clicks "Delete" → bar enters confirmation state: "Delete 4 layers? This cannot be undone." [Cancel] [Delete 4 layers].
3. User clicks "Delete 4 layers" → 4 rows disappear optimistically. 4 DELETE requests fire in parallel. All succeed. Bar dismisses.

**Error path — one layer fails:**
1. User multi-selects 3 rows and clicks Visibility toggle.
2. All 3 rows hide optimistically.
3. One PATCH returns 500.
4. `toast.error("Failed to update visibility — changes rolled back.")` fires.
5. All 3 rows revert to their previous visibility.
6. Selection remains (user can retry).

## What to Avoid

Source: sketch decisions, REQUIREMENTS.md Future Requirements, phase context, BUILDER-UX-AUDIT findings.

- **Modal for bulk delete.** Inline confirmation in the bar is the contract. Modals block the stack and prevent reviewing the selection before confirming.
- **Partial-success state for bulk ops.** The contract is atomic — all succeed or all roll back. Do NOT show "{N} of {M} succeeded" states; that adds complexity the user cannot act on (they cannot selectively rollback N of M).
- **Checkbox column always visible.** Checkboxes ONLY appear when multi-selection is active (`selectedIds.size >= 1`) or when the user holds `Cmd/Ctrl`. Rest state must remain clean.
- **New design tokens.** v1009 hard constraint: all work uses existing `sketch-findings-geolens` token set.
- **New backend endpoints.** Hard constraint: use existing per-layer PATCH callbacks from `use-builder-layers.ts`.
- **Co-selecting basemap group or sublayers.** The model enforces this silently — no toast, no modal.
- **Bulk-selecting DEM raster rows for a "Group" op.** DEM layers cannot be put in a folder-group (they are standalone raster layers). The "Group" button must be disabled whenever a DEM/raster layer is in the selection (add to the disable condition: `!selectedLayers.every(l => l.dataset_record_type === 'vector_dataset')`).
- **Drag during multi-selection.** Drag-start clears selection — the two modes do not compose. Attempting to drag a multi-selected row initiates a single-layer drag (of the dragged row) and clears selection.
- **Persistent "X selected" counter in header.** The header is already at density limit (AUD-01). The count lives in the bulk action bar only.
- **zustand for selection state.** Selection is ephemeral per-session UI state, not persisted. Component state (`useState`) is the right scope. Using zustand would couple selection state to the global map store, making route-change cleanup fragile.
