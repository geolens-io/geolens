<objective>
GPKG-01 (P0 silent-data-swap fix). End the silent fallthrough to layers[0] on the File-path reupload by plumbing a user-chosen `layer_name` from the ReuploadDialog through both `/preview` and `/commit` endpoints into the ingest worker.

Purpose: A user reuploading a multi-layer GPKG today silently picks `layers[0]` regardless of which layer the dataset was originally ingested from — same dataset ID + slug, different logical data. This plan inserts a layer-select step into ReuploadDialog's state machine that mirrors the Service URL path at `ReuploadDialog.tsx:581`, defaults the selection to the previous `source_layer` recorded on the most-recent IngestJob, and warns the user when the previously-ingested layer is missing from the new file.

Output:
- New `all_layers` and `previous_source_layer` fields on `ReuploadPreviewResponse` (backend schema + frontend type).
- New optional `layer_name` field on `ReuploadCommitRequest`.
- `selecting-layer` step rendered between `uploading` and `preview` when `all_layers` length > 1.
- Backend `reupload_preview` endpoint accepts `layer_name` query param, passes through to `run_ogrinfo_preview`; backend `reupload_commit` persists `layer_name` to `job.source_layer` so the worker uses it.
- `_do_reupload_file` worker (`tasks_reupload.py`) reads `job.source_layer` and passes it as `layer_name=...` to `run_ogr2ogr`, `run_ogrinfo`, and `run_ogrinfo_preview`.
- Backend + frontend regression tests covering the five truths above.
</objective>

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| frontend → /datasets/{id}/reupload/{job_id}/preview | User-controlled `layer_name` crosses HTTP boundary into backend |
| frontend → /datasets/{id}/reupload/{job_id}/commit | User-controlled `layer_name` persisted to DB (`IngestJob.source_layer`) |
| backend → `ogrinfo` / `ogr2ogr` subprocess | `layer_name` flows into argv list of GDAL subprocess invocations |
| worker → PostgreSQL `IngestJob.source_layer` column | DB write of user-controlled string (already validated by Pydantic max_length=500) |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-1058A-01 | Tampering | `layer_name` field in `ReuploadPreviewRequest` / `ReuploadCommitRequest` | mitigate | Pydantic `Field(max_length=500)` enforces upper bound; matches existing `IngestJob.source_layer` DB column constraint (`String(500)` in jobs/models.py:52). Pydantic rejects non-string types at request validation. |
| T-1058A-02 | Tampering | `layer_name` argv passed to `ogrinfo` / `ogr2ogr` subprocess | mitigate | `asyncio.create_subprocess_exec(*cmd, ...)` invocation is argv-list (no shell interpretation); GDAL tokens are positional. Even if user supplies a malicious string like `"; rm -rf /"`, GDAL just sees a layer name and fails with "layer not found" — no shell expansion possible. Defense-in-depth from D-15 threat model. |
| T-1058A-03 | Tampering | `layer_name` not present in actual file's layer list | mitigate | In Task 1, after `_extract_common_layer_metadata` returns, the preview endpoint validates that `request.layer_name` (when non-null) appears in the returned `all_layers` names. If absent, return HTTP 422 with detail "Layer '{name}' not found in this file". This guards against the pathological case of a user crafting a layer_name pointing to an unrelated file accessible to the GDAL process. (Practical impact is low because GDAL would also fail with "layer not found"; this just produces a cleaner error.) |
| T-1058A-04 | Information Disclosure | `previous_source_layer` leak | accept | Returning the prior `source_layer` is intentional product behavior (D-02 default selection). The value is metadata about the same dataset the requesting user already has `edit_metadata` permission on — no cross-tenant leak. |
| T-1058A-05 | Repudiation | Reupload swaps logical layer silently | mitigate | This is the very bug GPKG-01 closes. After this plan: `IngestJob.source_layer` is persisted on every reupload; existing audit infra (`audit_log` for `reupload.commit`, per `_assert_compatible_record_type` docstring) records the action. No new audit field needed. |
| T-1058A-06 | Denial of Service | Malformed GPKG with hundreds of layers blows up `all_layers` payload | accept | `_extract_common_layer_metadata` already materializes the layer list inline; the response is bounded by file content. No new attack surface — same exposure as current ogrinfo preview. Cap-and-truncate would be a separate hardening pass; out of scope per CONTEXT D-12. |
| T-1058A-SC | Tampering | npm/pip installs | n/a | No new npm or pip packages added by this plan. Existing dependencies only. |
