# Phase 999.4: Shared Vector Staging Pipeline - Context

**Gathered:** 2026-04-12
**Status:** Ready for planning

<domain>
## Phase Boundary

Extract the duplicated validate → ogrinfo → ogr2ogr → post-process → metadata pipeline from `ingest_file` (~lines 542-770) and `reupload_file` (~lines 1137-1345) in `backend/app/ingest/tasks.py` into a single shared helper. Both callers retain their divergent downstream steps: `_finalize_ingest` (dataset creation) for ingest and `_apply_reupload_swap` (atomic table swap + versioning) for reupload.

This is a pure backend refactor — no API contract changes, no frontend changes, no new capabilities.

</domain>

<decisions>
## Implementation Decisions

### Extraction Boundary
- **D-01:** The shared helper (`_ingest_vector_into_staging`) includes everything from file validation through metadata extraction, 3D detection, and `promote_z_to_elev`. It returns a result tuple (metadata dict, sample_values dict, three_d info dict) so callers can feed results into their divergent downstream steps.
- **D-02:** Table name is passed in by the caller. `ingest_file` generates a new table name via `generate_table_name()`; `reupload_file` uses `{table_name}_staging`. The helper does not own table naming.
- **D-03:** The helper accepts the ogr2ogr target table, session, job, file_path, and ogrinfo results as inputs. CRS detection (`run_ogrinfo`) runs before the helper is called — both callers already do this identically but ingest needs the `layer_name` parameter from user metadata while reupload does not.

### Feature Parity
- **D-04:** User geometry override (`_detect_and_override_geometry`) stays ingest-only. It is driven by the commit-step UI metadata (`x_column`, `y_column`, `geom_column`) which does not apply to reupload. The shared helper accepts an optional `geometry_override_result` parameter (the override geometry type if applicable, or None) so ingest can run the override before calling the helper and pass the result in. Reupload passes None.
- **D-05:** The `assumes_4326` CRS fallback logic and missing-CRS error handling stay in the callers (ingest checks it pre-helper; reupload has different CRS handling with `srid_override` defaulting to 4326). The helper receives `effective_srid` and `has_geometry` as resolved inputs.

### Test Strategy
- **D-06:** Add unit tests for the new shared helper function itself, verifying it calls the expected pipeline steps in order with correct arguments (mock-based, testing the orchestration logic).
- **D-07:** Add 1-2 integration tests exercising the shared helper through both ingest and reupload paths using the `_point_ogr2ogr_at_test_db` conftest fixture (real ogr2ogr, tiny shapefile). These verify behavioral parity with the pre-refactor paths.
- **D-08:** Existing test suites (`test_ingest.py`, `test_reupload.py`) must continue passing unchanged — they serve as regression coverage for the callers.

### Error/Cleanup Contract
- **D-09:** The shared helper raises exceptions on failure — it does not catch or swallow errors. Cleanup responsibility stays with each caller: `ingest_file` keeps uploaded files for retry (N2 contract); `reupload_file` drops the staging table on failure.
- **D-10:** The helper does not commit the session. Callers own transaction boundaries (ingest commits via `_finalize_ingest`; reupload commits after `_apply_reupload_swap`). The helper may flush but not commit.

### Claude's Discretion
- Helper function signature design (dataclass vs keyword args vs named tuple for return)
- Whether to introduce a `StagingResult` dataclass for the return value or use a plain tuple/dict
- Internal decomposition of the helper (single function vs sub-helpers)
- Import organization within the helper

</decisions>

## Existing Code Insights

### Reusable Assets
- `IngestContext` dataclass — Prior art for parameter bundling. The new helper may follow a similar pattern for its inputs.
- `_finalize_ingest` — Already extracts the post-ogr2ogr-to-dataset-creation pipeline for ingest + service ingestion. The new helper extracts the pre-`_finalize_ingest` duplication.
- `_append_job_warning` helper — Used by both paths for warning accumulation.
- `_resolve_effective_srid` helper — Already extracted, used by ingest. Reupload has inline equivalent.

### Established Patterns
- Lazy imports inside task functions (Procrastinate worker isolation requirement)
- `structlog` for task-level logging with `_bind_task_log_context`
- `_qtable()` for safe SQL table quoting (from metadata.py)
- Exception propagation with rollback + status update in caller's except block

### Integration Points
- `ingest_file` calls shared helper → then `_finalize_ingest` → then `_archive_original_file`
- `reupload_file` calls shared helper → then `_apply_reupload_swap` → then `_archive_original_file`
- Both paths end with embedding generation (`defer_embedding`)
