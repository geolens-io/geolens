**Reviewed:** 2026-04-13T00:00:00Z

## Summary

This phase introduces `_ingest_vector_into_staging`, a shared helper that deduplicates the ogr2ogr → rename → DBF → geometry → metadata pipeline between `ingest_file` and `reupload_file`. The refactor is structurally sound and the unit test suite (test_staging_pipeline.py) is thorough. One critical security issue was found: raw SQL string interpolation of an unquoted table name inside `_detect_and_override_geometry`. Three warnings cover a staging-table name truncation bug that silently produces wrong names, a missing `@pytest.mark.asyncio` decorator on integration tests, and a `user_wants_geom=True` + `user_metadata=None` edge case that raises `AttributeError` instead of a clean error. Two info items flag a duplicate import and a magic number.

## Critical Issues

### CR-01: SQL injection via unquoted table name interpolation in `_detect_and_override_geometry`

**File:** `backend/app/ingest/tasks.py:293-297`
**Issue:** The raw SQL query inside `_detect_and_override_geometry` uses an f-string to interpolate `table_name` into the query without quoting it, even though `_validate_table_name` is called three lines earlier. `_validate_table_name` presumably rejects names with SQL metacharacters, but the defence relies entirely on that upstream guard. If any future caller bypasses the guard or the validation regex has a gap, the query becomes injectable. All other SQL builders in `metadata.py` use `_qtable()` (which quotes the name with `data."<name>"`). This one does not.

The pattern also silently diverges from the surrounding codebase convention (`_qtable` usage), making it a maintenance hazard even if not currently exploitable.

**Fix:**
```python
from app.ingest.metadata import _qtable

result = await session.execute(
    _text(
        f"SELECT GeometryType(geom) FROM {_qtable(table_name)} "
        f"WHERE geom IS NOT NULL LIMIT 1"
    )
)
```

## Warnings

### WR-01: Staging table name truncation produces wrong name when `dataset.table_name` is exactly 55 characters

**File:** `backend/app/ingest/tasks.py:1330`
**Issue:** `staging_tn = f"{dataset.table_name[:54]}_staging"` truncates the base name to 54 characters before appending `_staging` (8 chars), yielding a 62-character identifier — comfortably under PostgreSQL's 63-char limit. However, when `dataset.table_name` is between 55 and 63 characters, the slice silently changes the base name, so `staging_tn` no longer derivable from `table_name` by any deterministic rule. A name like `"my_long_dataset_name_exactly_55chars_here_padding"` (55 chars) would be sliced to 54, making the staging table name impossible to reconstruct without reading the code. The same formula appears at line 1535 in `reupload_service`.

More concretely, if the cleanup path (`DROP TABLE IF EXISTS {_qtable(staging_tn)}`) runs between a restart and a retry, it drops the wrong table if the in-memory derivation used a different slice boundary than whatever was persisted. The correct maximum base length for a `_staging` suffix is 63 - len("_staging") = 55 chars, so the slice should be `[:55]`.

**Fix:**
```python
_STAGING_SUFFIX = "_staging"
_MAX_BASE_LEN = 63 - len(_STAGING_SUFFIX)  # 55
staging_tn = f"{dataset.table_name[:_MAX_BASE_LEN]}{_STAGING_SUFFIX}"
```

Apply identically at line 1535 in `reupload_service`.

### WR-02: Integration tests missing `@pytest.mark.asyncio` — tests silently pass without running

**File:** `backend/tests/test_staging_pipeline_integration.py:122`
**Issue:** `TestStagingPipelineIntegration` methods are `async def` but the class has no `@pytest.mark.asyncio` decorator (or `asyncio_mode = "auto"` equivalent). Without it, pytest-asyncio does not drive the coroutines — they return coroutine objects and are immediately collected as passing. The three integration tests (`test_ingest_path_spatial_geojson`, `test_reupload_path_staging_table`, `test_nonspatial_csv_path`) will show green in CI without actually executing.

The unit test class `TestIngestVectorIntoStaging` in `test_staging_pipeline.py` correctly applies `@pytest.mark.asyncio` per-method; the integration test class needs either per-method decorators or a class-level marker.

**Fix:**
```python
@pytest.mark.asyncio
class TestStagingPipelineIntegration:
    ...
```
Or add `@pytest.mark.asyncio` to each individual method.

### WR-03: `user_wants_geom=True` with `user_metadata=None` raises `AttributeError` inside `_detect_and_override_geometry`

**File:** `backend/app/ingest/tasks.py:413-421`
**Issue:** `_ingest_vector_into_staging` accepts `user_metadata: dict | None = None`. When `user_wants_geom=True` is passed without `user_metadata`, the call on line 417 passes `user_metadata or {}` to `_detect_and_override_geometry`, which is safe. However, the docstring says "Required when user_wants_geom=True" but there is no assertion or raised error to enforce this. The silent fallback to `{}` means the geometry override silently does nothing (returns `None`), leaving the user thinking geometry was constructed when it was not.

This is a correctness hazard: a caller that sets `user_wants_geom=True` but forgets `user_metadata` will get no error and no geometry override — the ingest silently succeeds as non-spatial when the user expected spatial output.

**Fix:** Add a guard at the top of `_ingest_vector_into_staging`:
```python
if user_wants_geom and not user_metadata:
    raise ValueError(
        "_ingest_vector_into_staging: user_metadata is required when user_wants_geom=True"
    )
```

### IN-02: Magic number `54` used for staging table truncation without explanation

**File:** `backend/app/ingest/tasks.py:1330` and `1535`
**Issue:** The literal `54` appears twice for the staging-table base name slice without a comment explaining the arithmetic (63 PostgreSQL identifier limit minus 8 chars for `_staging` minus 1 = 54, or the intended bound of 55 with an off-by-one as noted in WR-01). Named constants make the intent clear and prevent future drift.

**Fix:** Define once at module level:
```python
_STAGING_SUFFIX = "_staging"
_STAGING_BASE_MAX = 63 - len(_STAGING_SUFFIX)  # 55
```
Then use `dataset.table_name[:_STAGING_BASE_MAX]` at both call sites.
