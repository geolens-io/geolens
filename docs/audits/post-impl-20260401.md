## Executive Summary

GeoLens is a well-maintained codebase — zero lint violations, zero TODO comments, clean test suite (830 frontend + 650 backend passing), and strong docstring coverage. The biggest structural debt is in two areas: (1) **KISS violations** in the ingestion pipeline (`ingest_file` is a 300-line monolith) and map builder hooks (6 functions >40 lines in `use-builder-layers.ts`), and (2) **type safety gaps** in Pydantic schemas (missing `max_length` on user-input fields, bare `dict`/`list` types) and frontend-backend type drift. The single crash-risk finding is WebGL context loss in map components (no handler on any of 3 map components). The top recommendation is to add input validation constraints to Pydantic schemas and WebGL context loss handling before the next release.

## 1. KISS & Simplification

### Top Findings

1. **`ingest_file()` is 300 lines** (`backend/app/ingest/tasks.py:33`) — a monolithic pipeline with 9 numbered steps, 5 levels of nesting, and inline imports. `ingest_service()` (line 356) duplicates many of the same steps. Extract a shared post-ingestion helper to eliminate ~150 lines of duplication.

2. **`syncLayersToMap()` reimplemented in ViewerMap** (`frontend/src/components/viewer/ViewerMap.tsx:281-384`) — 103-line useEffect that duplicates `map-sync.ts:50-196` with only an ID prefix difference. Parameterize and reuse.

3. **`update_user_metadata()` has 19 kwargs** (`backend/app/datasets/service.py:358`) — the router already passes a `DatasetMeta` Pydantic model. Accept it directly instead of destructuring every field.

4. **Route handlers contain business logic** (`backend/app/datasets/router.py:188-281`) — `get_single_dataset()` is 93 lines with inline RasterAsset queries, VRT source counts, and STAC assembly. This belongs in service.py.

5. **Mobile/desktop sidebar JSX duplicated** (`frontend/src/pages/MapBuilderPage.tsx:248-513`) — `LayerPanel` + `BasemapPicker` blocks are rendered identically in both Sheet (mobile) and sidebar (desktop). Extract a `<SidebarContent>` component.

### Recommendations

1. Extract shared post-ingestion pipeline from `ingest_file` and `ingest_service`
2. Parameterize `syncLayersToMap()` and call from ViewerMap instead of reimplementing
3. Accept Pydantic model directly in `update_user_metadata()` and `create_dataset()`

## 2. Performance

### Top Findings

1. **N+1 in `auto_detect_relationships`** (`backend/app/datasets/service.py:1021`) — fires per-column SELECT + per-match existence check. Should batch into single IN queries.

2. **`get_facet_counts` fires 5 sequential queries** (`backend/app/search/service.py:179-560`) — record_type, keyword, org, SRID, and collection facets could run concurrently or be combined into fewer queries.

3. **`SearchResultCard` not memoized** (`frontend/src/components/search/SearchResultCard.tsx:124`) — rendered in a list via `.map()`, each card calls `useQuicklook`. Parent re-renders cause all cards to re-render.

4. **Stable data hooks use short staleTime** — `useAllSettings`, `useContacts`, `useKeywords`, `useRelatedDatasets` rely on the 30s global default. Settings/reference data should use 5+ minutes.

5. **`get_collection_datasets` missing `joinedload(Dataset.record)`** (`backend/app/collections/service.py:187`) — accessing `dataset.record` triggers N+1 lazy loads.

### Recommendations

1. Batch relationship detection queries into single IN operations
2. Add `React.memo` to `SearchResultCard`, hoist constant objects outside component
3. Set explicit long `staleTime` on stable data hooks (settings, contacts, keywords)

## 4. Type Safety

### Top Findings

1. **Missing `max_length` on user-input Pydantic fields** — `DatasetMeta` (19 fields), `ChatHistoryMessage.content`, `RegisterRequest.table_name` accept unbounded strings. `table_name` is especially risky as it becomes a PostgreSQL identifier (63-char limit).

2. **Frontend-backend type drift** — `DatasetResponse` frontend type missing `stac_assets`, `stac_extensions`, `language`. `OGCRecordProperties` missing 11 fields. `ChatAction.geojson` typed as `Record<string, unknown>` instead of `GeoJSON.FeatureCollection`.

3. **`SearchParams.limit` has no bounds** (`backend/app/search/schemas.py:24`) — description says "1-100" but no `ge`/`le` constraint. Caller can request `limit=999999`.

4. **6 `as unknown` casts in feature editing** — all stem from `extractSingleGeometry` accepting `Record<string, unknown>` instead of `GeoJSON.Geometry`. Fixing the root function eliminates all downstream casts.

5. **8 routes missing `response_model`** — settings, ingest, and dataset data endpoints return typed responses but don't declare `response_model`, reducing OpenAPI accuracy and disabling output validation.

### Recommendations

1. Add `max_length` to all user-input string fields, especially `table_name` (63), `title` (500), `summary` (5000)
2. Add `ge=0` to `SearchParams.offset`, `ge=1, le=100` to `SearchParams.limit`
3. Retype `extractSingleGeometry` to accept/return `GeoJSON.Geometry`

## 5. Error Handling & Resilience

### Top Findings

1. **WebGL context loss unhandled (CRASH)** — None of 3 map components (`BuilderMap`, `DatasetMap`, `ViewerMap`) listen for `webglcontextlost`. GPU memory exhaustion or OS reclamation causes a blank map with no recovery.

2. **Missing route-level `errorElement`** — `SearchPage`, `CollectionsPage`, `MapsPage`, `SettingsPage`, `/login`, `/register` have no `errorElement`. Errors bubble to app-level boundary.

3. **No retry on external service calls** — Titiler proxy (`tiles/router.py:244`), LLM calls (`ai/llm_loop.py:204,302`), and embedding generation (`embeddings/service.py:69`) make single attempts with no retry on transient failures.

4. **`pool_pre_ping` defaults to False** — stale connections after DB restarts cause first-request failures.

5. **No streaming size enforcement on uploads** — `upload_file` endpoint accepts full file before checking size. Malicious client can stream arbitrarily large file to disk.

### Recommendations

1. Add `webglcontextlost`/`webglcontextrestored` handlers to all map components
2. Add `errorElement` to all leaf routes
3. Change `pool_pre_ping` default to `True`
